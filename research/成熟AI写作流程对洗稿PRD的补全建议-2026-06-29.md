# 成熟 AI 写作流程对洗稿 PRD 的补全建议

日期：2026-06-29

> 当前状态：本文是 PRD v3 的前置研究记录，核心结论已吸收到 `v2-restart/PRD_v3.md`。后续执行以 `PRD_v3.md` 为准。

## 研究目的

这次研究不是为了把外部项目搬进来，也不是为了把当前项目做成原创小说工具、聊天前端或工程流水线。

当前项目的锚点仍然是：

> 基于一个已被验证或值得参考的短剧源本，把它洗成新壳下仍然好看、能追、能付费的新剧本。

外部项目只用来反查一件事：

> 我们从“源本分析”到“写出新剧本”之间，是不是缺少必要的产品工作稿和审稿面。

## 研究结论

当前 PRD 的大方向是对的：先读源本，再判断变量，再重建事件载体，再写首批，再自检和 reviewer。

但当前 PRD 的“写首批新本”太薄。它只有四句话：

1. 拿到每集叙事功能；
2. 判断新壳里什么场面能承载；
3. 写成新场景和对白；
4. 检查每集是否有变化。

这个方向没错，但中间缺少能被主创、作者自检、独立 reviewer 共同使用的工作稿。缺这层，就容易出现现在样本里暴露的问题：新本看起来不是洗飞，而是“太像源本，只是改了名字和背景”。

成熟 AI 写作项目共同显示：写正文前，通常会有一层“写前任务/义务”；写正文后，通常会有一层“是否兑现任务”的审稿；长内容推进时，还会有“当前事实和未兑现事项”的追踪。

对我们来说，这些不能照搬成工程系统，但可以收成产品流程里的四份工作稿。

## 建议补的四份工作稿

### 1. 源本节点拆解稿

作用：把“源本有效性摘要”从泛泛总结变成可迁移的写作依据。

它不需要写成长篇报告，但每个关键节点至少要回答：

- 这一集或这一段发生了什么；
- 它让观众继续看的功能是什么；
- 它制造的是留存、爽点、压迫、反转、信息差，还是关系张力；
- 它的情绪作用是什么；
- 它用了什么具体事件、道具、证据、场面或人物行动承载；
- 哪些外形太有源本辨识度，后续必须避开。

可借依据：

- `oh-story` 的情节节点：事件、情绪强度、涉及人物、手法、原文锚点。
- `Long-Novel-GPT` 的剧情提炼：先把正文提炼成一行行剧情，再扩写。

对我们的改造：

不是“剧情摘要”，而是“源本为什么有效”的节点账。

### 2. 迁移义务与同构红线稿

作用：把变量迁移表从“哪些保留/替换/删除”升级成“新本必须兑现什么体验，同时必须避开什么源本外形”。

每个关键节点要说清：

- 新本必须让观众感受到什么；
- 源本具体事件里哪些东西不能照搬；
- 新壳里用什么新的压力、权力关系、证据、行动或场面承载；
- 如果继续保留原桥段顺序、原道具、原揭露方式，会不会让人觉得只是改名；
- 这个节点如果没写好，应该回到源本分析、新壳承载，还是正文重写。

可借依据：

- `AI-Novel-Writing-Assistant` 的“章节义务”思路：写前明确本章必须完成什么，写后检查缺了什么。
- 当前 PRD 已经有“事件载体重建与同构扣除”，但还没有把它明确成写正文前的硬门槛。

对我们的改造：

这一步不是增加规则，而是防止“保留功能”滑回“保留桥段外形”。

### 3. 新本分集/场景草案

作用：正文前先确认新本每集每场要完成什么戏剧任务。

它不应该是原创大纲，也不是让模型自由发挥。它是源本节点迁移后的新壳草案。

每集至少回答：

- 本集承接源本哪个追看功能；
- 本集的新壳事件是什么；
- 本集推进了关系、局势、信息、情绪里的哪一项；
- 本集有没有阶段性兑现或新追问；
- 本集结尾为什么能让人看下一集。

每场至少回答：

- 这场戏的目的是什么；
- 谁想要什么，遇到什么阻碍；
- 观众能看到什么动作、选择、冲突或后果；
- 场景结束时局面和开始前有什么不同；
- 是否太像源本原场面。

可借依据：

- `short-drama` 标准库的分集目录和单集写作顺序。
- `NovelForge` 的阶段大纲、章节工作稿和章节审核。
- `how-to-make-script` 的 scene / outline 输出契约。

对我们的改造：

不做复杂场景卡，不做工程化字段，只保留“写正文前必须想清楚的戏剧任务”。

### 4. 首批审核稿

作用：让作者自检和独立 reviewer 不再泛泛评价“好不好”，而是按前面的工作稿反查。

审核结论不需要打总分，建议用：

- `pass`：可以进入用户/导演确认；
- `revise`：方向对，但有必须修改项；
- `block`：不能继续写后续，必须回到前面某一步。

重点检查：

- 源本有效性是否迁移；
- 同构风险是否过高；
- 新壳是否承载得住；
- 每集是否有关系、局势、信息或情绪变化；
- 是否硬堆钩子和爽点；
- 是否继承了水分烂点；
- 是否有不可拍的抽象描述；
- 台词是否解释腔、AI 味、翻译腔。

可借依据：

- `NovelForge` 的章节审核稿：pass / revise / block。
- `how-to-make-script` 的硬失败、弱项、复查计划分离。
- 当前 PRD 的独立 reviewer 原则。

对我们的改造：

自检不是证明作者正确，而是找出“前面承诺的体验有没有兑现、有没有太像原本”。

## 对当前三张表的 lock 判断

### 分析表

方向可以锁，字段不能锁。

五层分析是对的：

- 留存骨架；
- 情绪爽点；
- 主线推进；
- 人物压力；
- 干净度。

但它现在还缺“节点化输出”。如果只输出源本有效性摘要，写作时还是容易丢细节或变成标签化理解。

建议：保留五层，但新增“源本节点拆解稿”作为分析输出的一部分。

### 变量迁移表

分级可以保留，不能 final lock。

强继承、等效迁移、自由改写、删除修复这四类是对的。但变量表必须补一个产品判断：

> 继承的是观众体验功能，不是源本事件外形。

建议：变量表后面接“迁移义务与同构红线稿”。否则模型容易把强继承理解成照着原桥段走。

### 自检表

方向对，但不能 lock。

当前自检表已经覆盖洗飞、动机、空转、不可拍、AI 味等问题。但它还不是“按写作前承诺反查”的审核稿。

建议：自检表改成 `pass / revise / block` 的首批审核稿，明确必须改项和回退位置。尤其要把“同构太高、只是改名”提升成硬失败或至少重警告。

## 对 PRD 的最小改动建议

不要扩成大工程文档，也不要继续堆规则。

只建议在 `PRD_v2.md` 的核心工作流里，把“写首批新本”拆成更清楚的产品流程：

```text
源本有效性分析
  -> 源本节点拆解稿
  -> 变量迁移策略
  -> 迁移义务与同构红线稿
  -> 事件载体重建
  -> 洗稿方案
  -> 新本分集/场景草案
  -> 首批新本
  -> 作者自检
  -> 独立 reviewer
```

这里的“稿”都是内部工作稿，不是工程产物，也不一定全部给用户看。

用户需要看的仍然应该很简单：

- 洗稿方案；
- 首批新本；
- 简短改动说明；
- 自检和 reviewer 结论；
- 需要用户判断的问题。

## 不应照搬的东西

以下东西会让项目漂移：

- 自动原创整本小说；
- 雪花法、世界观自动扩张、角色自动扩张；
- 一键生成几十章或完整长剧；
- 把所有状态做成复杂平台；
- 纯电影/长剧排版工具的制片级拆解；
- 把短剧标准库比例、数字、集数卡点当硬规则；
- 把节点账本做成机械模板，导致模型只填表不思考。

## 最终判断

这条补全方向不是从低星项目拍脑袋来的，也不是某个高星项目可以原封不动背书的行业标准。

它是基于三类证据做出的产品判断：

1. 当前样本已经暴露“改名感/同构过高”问题；
2. 本地短剧和拆文资产都指向“节点拆解、追踪、反查”；
3. 高星 AI 写作项目也普遍存在“写前任务、写后验收、正文与规划对齐、完成后状态回收”。

所以当前最稳的做法不是继续扩规则，而是补一条写作桥梁：

> 源本拆解清楚，迁移义务说清楚，新本写前有草案，写后按同一套承诺审。

这仍然是产品设计，不是实现方案。
