## 下一步开发建议

建议从小而稳的开发任务开始：

1. 冷启动演练：新用户只读正式版 README/QUICKSTART，主控是否能正确跑到蓝图确认。
2. `/dialogue-polish` clean run 验证：证明台词精修环节接入后仍然稳定。
3. `batch-state` 续批验证：跑 11-20 的最小链路，确认长线状态不会漂。
4. issue 模板和 release 规则：让 GitHub 上的用户反馈能进入 `codex/issue-*` 分支。

每个任务都尽量单独开分支或至少单独提交。不要把多个方向混成一锅。

## 合回 main 的标准

只有满足以下条件，才考虑合回 `main`：

- 对新用户有直接价值；
- 不依赖旧实验语境；
- 不暴露开发素材、历史复盘、旧路线名或内部审计；
- 已通过必要检查；
- README / QUICKSTART / AGENTS 同步更新。

`main` 是展示柜，`codex/next` 是工作台。这个边界不要再打穿。
