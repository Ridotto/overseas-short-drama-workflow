# V3 / V5 完整对读与合并方案

> 日期：2026-07-01  
> 状态：合并依据，不是执行 workflow。  
> 目的：完整读完 V3 / V5 后，判断哪些能力应该进入新版 workflow，哪些不能照搬。

---

## 0. 先说结论

之前把 V3 / V5 结合坏了，原因不是“没想到某条规则”，而是抽得太粗。

不能说：

```text
V3 保方向，V5 管链路，当前版管纪律。
```

这句话太粗。V3 和 V5 每个版本内部都有多种能力，不能每版只抽一个标签。

正确合并方式是：

```text
以 workflow_spec_v2 的完整产物链为主骨架；
拿 V3 的产品边界、迁移义务、同构红线和商战稳态做护栏；
拿 V3 / V5 里多条有效链路做 writer 输入和 reviewer 判据；
拿当前执行协议做 manifest、reviewer 和版本对比纪律；
明确最终正文必须是 screenplay，不是带内部标签的说明书。
```

这里不能再把 V5 简化成一条“创作链”。

V3 / V5 至少有这些链路必须同时保住：

| 链路 | 作用 | 在 V3 / V5 里的正向证据 |
|---|---|---|
| 留存 / 钩子 / 卡点链 | 保住观众下一集还想看 | V3 分集功能表；V5 Opening Hook / End Button / Do Not Consume |
| 情绪 / 爽点 / 压抑释放链 | 保住短剧的痛、爽、气、急和兑现周期 | V3 情绪爽点；V5 爽点与压抑释放、Max Spike |
| 主线期待 / 信息差链 | 保住观众追真相、追身份、追误会解除的动力 | V3 主线期待；V5 信息差与真相路径 |
| 人物关系 / 拉扯链 | 保住男女主、主反、男配之间的靠近、退开、误会、试探和压迫 | V3 人物压力；V5 关系 / 权力变化、Story memory |
| 权力变化 / 打脸链 | 保住谁被踩、谁翻身、谁当众失去东西 | V3 新壳承载；V5 Action chain / Power shift / 可见后果 |
| 同构扣除 / 新壳承载链 | 防止改名复刻，也防止避同构避到洗飞 | V3 迁移义务与同构红线；V5 禁止复用外形 / 新壳场面迁移卡 |
| 场面动作链 | 把信息、证据和情绪写成现场动作和后果 | V5 Scene packet / Evidence triggers action |
| 正文表达链 | 确保最终文本像剧本，不像说明书、审计表或流程单 | V5 正文虽有强动作，但内部标签泄露；新版必须修正 |
| 连续性 / 伏笔链 | 保证下一批不乱写、不提前消费、不让角色状态回退 | V5 Story memory checkpoint |
| 审稿 / 版本对比链 | 保证不是作者自己说过了就过了 | V3 reviewer；V5 reviewer 对比；当前 manifest / reviewer 协议 |

新版 workflow 不是找一条最强链路，而是让这些链路在同一条执行流程里分别有入口、出口和回查点。

## 2. V3 到底好在哪

V3 不是最终答案，但它有几个后来版本反而丢掉的优点。

### 2.1 V3 的写作前稿保住了产品常识

V3 很清楚地把任务放在“洗稿”里：

- 保源本追看功能；
- 换源本高辨识外形；
- 不把豪门商战壳写成黑帮词替换；
- 不把原剧本有效性洗飞。

它的写作前稿有这些关键层：

- 源本有效性摘要；
- 留存骨架；
- 情绪爽点；
- 主线期待；
- 人物压力；
- 源本节点拆解；
- 变量迁移；
- 新壳承载判断；
- 迁移义务与同构红线；
- 阶段骨架；
- 分集功能表；
- 关键场景草案。

这些东西不是多余的。它们保证 writer 不会忘记：这个项目不是原创，而是把源本里让观众继续付费看的东西迁过来。

### 2.2 V3 的商战边界更稳

V3 第 10 集用的是直播刑责陈述、公开认罪、交权和法律后果，比较贴豪门商战 / 法务壳。

这点比 V5 第 10 集枪响黑屏稳。V5 的枪很短剧，但类型边界容易滑回黑帮 / 犯罪。

### 2.3 V3 也有明显问题

V3 reviewer 结论是 `revise`，不是 pass。

主要问题：

- 第一集“孕白月光进门 + 装弱 + 男主误判”同构风险仍明显；
- 第 7-8 集偏股东大会、证人、律师、文件宣读；
- 女配反咬承担了一部分男主罪责回扣；
- 商业机制有时像权限开挂；
- 高压节点还偏“证明了什么”，不够“现场发生了什么”。

所以 V3 不能直接当默认 workflow 或质量上限。

## 3. V5 到底好在哪

V5 是当前最重要的正向证据。它不是因为“更刺激”才有价值，而是因为它证明了 `workflow_spec_v2` 那套产物顺序可以同时承载多条内容链，并把源本有效性推到正文。

### 3.1 V5 的核心不是关键场面卡，也不是单条写作链

V5 的写作前工作稿不是一张卡，也不只是一条线性链。

它表面上有一条产物顺序：

```text
源本拆文包
-> 洗稿边界包
-> 新本 beat sheet / outline
-> 新壳场面迁移卡
-> Episode function map
-> Scene packet
-> 正文前策略审核
```

但它真正有效，是因为这条产物顺序里同时跑着多条内容链：

- 留存链：Opening Hook / End Button / Do Not Consume；
- 爽点链：压抑、看衰、打脸、兑现；
- 关系链：谁靠近、谁退开、谁误会、谁被迫承认；
- 信息差链：谁知道、谁不知道、观众等哪一层真相；
- 权力链：谁失去钱、身份、席位、控制权、体面；
- 场面链：证据出现后触发什么行动和后果；
- 连续性链：写完后保存关系、信息差、伏笔和禁提前消费。

后面失败版本只恢复了“关键场面迁移卡”的壳，没有恢复这些链路之间的相互牵引。

这就是为什么后面写出来越来越像流程证明：writer 拿到的是“要过哪些点”，不是“这集怎么写成戏”。

### 3.2 V5 的源本拆文包更完整

V5 明确拆了：

- 故事核；
- 核心期待；
- 首批源本分集功能；
- 高价值节点；
- 钩子和卡点；
- 爽点与压抑释放；
- 关系 / 权力变化；
- 信息差和真相路径；
- 写作手法；
- 水分、烂点和禁止复用外形。

这部分必须保留。后续不能只保“场面强不强”，否则会把留存、爽点周期、关系拉扯、信息差和情绪节奏全部挤掉。

### 3.3 V5 的 Scene packet 给到了 writer 能写正文的材料

V5 每集 scene packet 不是简单说“本集要揭露什么”，而是给：

- Scene purpose；
- Characters；
- Location；
- Starting power state；
- Action chain；
- Evidence / object；
- Evidence triggers action；
- Relationship shift；
- Power shift；
- End button。

这就是 V5 比 V3 第 7-8 集更好的原因。

V3 是“公开揭露材料”；V5 变成了：

- 继承墙刻名灯熄灭；
- 基金状态 Frozen；
- 认证台强制启动；
- 孩子名字被红色封条覆盖；
- Chloe 胸针被摘；
- 男主听见自己的切电授权；
- 第 9 集亲手砸开物证室。

信息没有消失，但信息都触发了现场动作和后果。

### 3.4 V5 的 Story memory 是后续连续性的必要产物

V5 写完后保存了：

- 关系状态；
- 信息差状态；
- 伏笔 / 证据状态；
- 角色状态；
- 未兑现承诺；
- 禁止下批提前消费；
- 下一批安全入口。

这不是形式主义。短剧洗稿如果要继续 11 集以后，必须知道谁已经知道什么、谁还误会什么、谁已经不能再被写回旧状态。

### 3.5 V5 也不能原样照搬

V5 的问题也很明确：

- 最终正文里出现了 `Action / Performance / Purpose / Keywords / Hook Type` 这类内部标签，读起来不像干净剧本；
- 第 10 集枪响黑屏短剧冲击强，但豪门商战类型边界不稳；
- 第 7 集医疗授权 / 信托冻结逻辑有理解成本；
- 有些命名和角色对应关系混乱；
- pass 的含义是“允许导演 review”，不是“workflow 已经稳定解决所有问题”。

所以 V5 的产物链要拿，V5 的最终正文格式不能拿。

## 4. 后来版本为什么越合越差

根因不是“少了一条视听语言规则”，也不是“少了一条场面链”。

更准确是：

```text
V3 / V5 的多条有效链路被压成了单条流程；
V3 的洗稿边界和商战稳态被抽成一句方向；
当前执行协议又把注意力推向证明每步已完成；
最后 writer 拿到的不是戏，而是合规任务。
```

典型断点：

1. 源本分析里有钩子、爽点、拉扯、情绪密度，但到 writer 输入时只剩“要揭露 X / 要证明 Y”。
2. Scene packet 本来应该给动作链，后来变成审计包、证明包或字段包。
3. Screenplay draft 本来应该是可看可听的剧本正文，后来混进内部标签或写成说明书。
4. Reviewer 能指出问题，但返修经常只加身体反应、加视听描述、加 gate，没有回到上游重建 scene packet。
5. 版本对比经常只比某个症状是否改善，没有整体比较读感、留存、同构、短剧场面、台词、continuity。
6. 留存链、情绪链、关系链、信息差链、权力链和场面链之间没有互相校验：某一链看似过了，另一链已经断了。

## 5. 合并原则

### 5.1 主骨架用 V5 / workflow_spec_v2，但按多链路回收

保留：

```text
源本拆文包
-> 洗稿边界包
-> beat sheet / outline
-> 新壳场面迁移卡
-> Episode function map
-> Scene packet
-> Screenplay draft
-> Rewrite report
-> Dialogue polish
-> 作者 quality gate
-> Story memory checkpoint
-> 干净 reviewer
-> 版本对比
```

### 5.2 V3 进入护栏层

保留 V3 的：

- 源本有效性摘要；
- 迁移义务；
- 同构红线；
- 新壳承载判断；
- 阶段骨架；
- 分集功能意识；
- 第 10 集商战 / 法务 / 直播式卡点方向。

不保留 V3 的：

- 第一集高同构装弱桥段；
- 第 7-8 集偏证据听证会的推进；
- 过多商业权限开挂；
- “证明材料公开”代替“现场动作后果”。

### 5.3 V5 进入 writer 输入层

保留 V5 的：

- 源本拆文包结构；
- 钩子 / 爽点 / 卡点 / 关系 / 信息差 / 权力变化；
- Episode function map；
- Scene packet；
- Evidence triggers action；
- 可见后果；
- Story memory。

不保留 V5 的：

- 正文里的内部标签；
- 第 10 集枪响作为默认解；
- 规则逻辑过复杂的表达方式；
- 把 pass 理解成可交付终稿。

### 5.4 当前执行纪律只管验收，不管创作

保留：

- manifest-first；
- 作者自检；
- 干净 reviewer；
- 主控版本对比；
- 不能把 manifest complete 当质量 pass。

但不能让执行纪律反过来驱动创作。

manifest 是证据，不是剧本发动机。

## 6. 新候选 workflow 应该长什么样

新版不应该再重写一套大理论。

它应该是：

```text
workflow_spec_v2 的完整链条
+ V3 的洗稿边界护栏
+ V5 的 writer-facing scene packet
+ 当前执行协议
+ final screenplay clean contract
```

最关键的三条连接：

### 6.1 源本拆文包不能停在分析层，也不能只传一条链

源本拆出来的：

- 钩子；
- 卡点；
- 爽点；
- 情绪密度；
- 关系拉扯；
- 误会 / 信息差；
- 角色状态；
- 权力变化；
- 付费追问。

必须进入：

```text
Episode function map
-> Scene packet
-> Screenplay draft
-> Reviewer 判断
```

而且不能只传“场面动作链”。

Episode map 至少要接住：

- 本集留存链；
- 本集情绪 / 爽点链；
- 本集关系 / 拉扯链；
- 本集信息差 / 真相链；
- 本集权力变化链。

Scene packet 至少要把这些链压成：

- 进入状态；
- 动作推进；
- 压力变化；
- 关系变化；
- 退出状态；
- 集尾最后一拍。

### 6.2 Scene packet 不能变成审计表

Scene packet 给 writer 的不是：

```text
本集必须证明 X；
本场必须出现 Y；
本节点必须回指 Z。
```

而是：

```text
这一场从哪里进；
谁压谁；
谁想逃；
谁被迫行动；
什么东西被当众夺走 / 砸碎 / 暴露 / 截断；
关系和权力怎么变；
最后停在哪个画面 / 声音 / 动作上。
```

### 6.3 最终正文不能带内部标签

最终剧本必须像剧本：

- 场景标题；
- 动作行；
- 人物名；
- 台词；
- 必要声音；
- 必要屏幕文字。

不能像：

```text
Purpose:
Action:
Performance:
Hook Type:
End Button:
```

内部字段要在正文里被消化成可见可听的东西。
