# V15 完整流程失效审计

> 日期：2026-07-01  
> 目的：不是再审 PRD 方向，也不是继续修某一版剧本，而是反查：为什么完整 workflow 跑完、manifest complete、reviewer pass 后，仍会漏掉用户能一眼发现的问题。

## 1. 总结论

V15 的问题不是“没跑完整 workflow”。

V15 确实跑了完整流程，也有两轮 reviewer：

- 第一轮只读正文：`creative_pass`
- 第二轮源本迁移审计：`source_audit_pass`
- manifest：`complete`

但这三个结论不能推出“交付形态正确”或“产物已经最优”。

真正失效点是：

```text
execution complete
creative pass
source audit pass
delivery candidate
```

这四个状态被混在一起了。

V15 证明了流程闭合、正文能读、源本没洗飞；但没有证明：

- 输出语言 / 审稿语言 / 正文用途已确认；
- 正文满足 AI 出海短剧的 page-to-screen 合同；
- V15 文本整体优于 V14；
- 完整 workflow 真的比局部文本修稿更会产出好剧本。

## 2. 不是“文档完全没有”，而是“没有进入阻断型 gate”

好莱坞格式、动作行、可见可听、对白不能裸奔、情绪落到行为上，这些在 v2 里早就有。

失效不在“从来没写”，而在：

| 要求 | 文档里有没有 | V15 是否卡住 |
|---|---|---|
| 输出语言 / 审稿语言 | 后来补进 PRD/v2/v4，V15 当时需求摘要没有 | 没卡住 |
| 正文用途：人审 / AI 生成 / 字幕版 / 双语版 | 后来补进 PRD/v2/v4，V15 当时没有 | 没卡住 |
| Hollywood screenplay 基本容器 | v2 有，v4 有弱表达 | V15 只用了外形，没有 gate |
| page-to-screen：剧本页基本等于最终画面/声音 | v2 后来已补硬，v4 后来已补硬 | V15 当时没有 smoke test |
| AI 出海短剧可视化执行 | PRD/v2/v4 后来补清楚 | V15 reviewer 没审这个 |

结论：这是**合同没有进入验收层**，不是用户突然提出新要求。

## 3. 哪些 gate 给了 pass，但没覆盖问题

### Step 0 最小需求确认是假 pass

V15 manifest Step 0 判了 pass，但当时只确认了：

```text
豪门商战
集数不动
完整跑
完整对比
```

没有确认：

```text
对白语言
动作/场景描述语言
审稿语言
正文用途：人审稿 / AI 视频生成 / 字幕台词版 / 双语版
```

这就是为什么“默认英文对白”和“给谁用”后来还能被用户抓出来。

### Step 8 Screenplay Draft pass 只验了产出，不验交付合同

V15 Step 8 的 pass 证据是：

```text
正文完整产出
高压场锚点：E2 glass door、E7 Daddy、E9 second band、E10 live button
```

它没有验：

```text
是否满足输出语言
是否满足正文用途
是否满足 Hollywood screenplay 容器
是否满足 page-to-screen
动作行是否足够给 AI 生成画面
```

所以 Step 8 是“有稿且有高压锚点”的 pass，不是“交付合同 pass”。

### Reviewer 第一轮 pass 只能证明正文能读

第一轮 reviewer 只读需求摘要和正文，它判断的是：

- 想不想继续看；
- 高压场有没有戏；
- 台词是否水；
- 情绪是否被看见；
- 卡点是否停住。

它没有职责检查“需求摘要是否缺字段”，也没有职责检查“这是不是 AI 生成链路交付格式”。

所以 `creative_pass` 不能推出 `delivery_pass`。

### Reviewer 第二轮 pass 只能证明没洗飞

第二轮 reviewer 审的是：

```text
Source Bible
-> Source Retention Anchor
-> Story Operating State
-> Episode Plan
-> Pressure-chain Writer Brief
-> 正文
```

它判断的是源本留存、同构、短剧降级、提前消费。

它不会天然比较 V15 是否比 V14 文本更强，也不会审输出语言或 AI 生成用途。

所以 `source_audit_pass` 不能推出 `best_version` 或 `delivery_pass`。

## 4. V15 产物层真实情况

V15 不是没进步。

真实进步：

- E6 不再让护士直接递完整证据，Adrian 有追查阻力；
- E7 不再靠轻伤输血硬爆雷，改成孩子本能喊 Bennett + guardian form；
- E9 主动追旧物更强；
- V15 没有退回 V8/V12 那种机制审判机器。

但 V15 也不是文本层面碾压 V14。

V14 仍有优势：

- E8 公开围观感更强；
- E10 空 press room、两部 phone、signet ring 更简洁；
- 有些地方更像短剧强场，而不是完整流程下的规整稿。

所以产物层结论是：

```text
V15 是有效候选稿；
不是证明 v4 workflow 已稳定胜出的铁证；
更不是导演终稿。
```

## 5. 根因分层

### 根因 1：pass 语义混用

`complete / creative_pass / source_audit_pass / candidate checkpoint` 被放在一起后，主控容易误读成“这版已经整体过关”。

实际上：

- manifest complete = 流程跑完；
- creative_pass = 正文作为短剧能读；
- source_audit_pass = 没洗飞；
- delivery candidate = 交付形态和产物优选也成立。

前面三个不能自动推出第四个。

### 根因 2：合同从 writer 移走后，没有稳稳移到 reviewer / 主控

之前重构的方向是对的：不要把 source ref、证明链、大表塞给 writer。

但拿掉 writer 证明链后，需要把关键合同放到 reviewer / 主控侧验收。

V15 漏掉的是这个迁移：

```text
不能污染 writer
≠
没人负责验合同
```

### 根因 3：reviewer 更擅长防错，不擅长选优

两轮 reviewer 能抓：

- 洗飞；
- 改名复刻；
- 短剧降级；
- 结构不成立；
- 台词水。

但它们没有被要求做：

```text
V13 / V14 / V15 盲读排序
哪个版本更想追
哪个版本更有短剧冲击
哪个版本更适合 AI 生成链路
```

所以它们可以正确给 V15 pass，但仍不能证明 V15 最优。

## 6. 最小修法，不回到规则怪谈

不要扩 PRD，不要把证据链塞给 writer，不要让正文写证明。

只补三个轻量动作。

### 6.1 Run Contract：每轮开跑前锁交付头

manifest / 需求摘要顶部必须有：

```text
对白语言：
动作/场景描述语言：
审稿语言：
正文用途：人审稿 / AI 视频生成 / 字幕台词版 / 双语版
正文容器：Hollywood screenplay basic / 其他
```

缺任何一项，Step 0 不能 pass，只能写：

```text
needs_user_confirmation
```

或：

```text
unconfirmed_default
```

### 6.2 Delivery Contract Smoke Test：只抽 3 场，不做大表

Screenplay Draft 后，主控或 reviewer 抽：

```text
E1 开场
一个中段高压场
E10 终钩
```

每场只问三句：

1. 画面/声音是否能直接被 AI 成片生成？
2. 是否有心理判断、剧情功能、作者意图替代动作？
3. 语言和格式是否符合 Run Contract？

失败则：

```text
delivery_contract_status = revise
```

但不重跑全流程。

### 6.3 Product-Level Blind Rank：用来防程序正义

如果有多个候选版本，reviewer 不能只判 pass/fail。

必须盲读排序：

```text
哪个更想看下一集？
哪个更像短剧？
哪个场面压力更强？
哪个人物互逼更清楚？
哪个更适合 AI 成片生成？
```

这一步不替代 workflow，但能防止“流程更完整，文本却不一定更好”。

## 7. 绝不能做的事

不要把这些塞给 writer：

- manifest；
- pass/fail gate；
- 版本对比；
- reviewer 报告；
- 完整 Source Bible；
- source ref 表；
- carry-through evidence；
- 历史失败清单；
- 大段 Hollywood / page-to-screen 理论；
- “证明你迁移了某源本节点”的审计语言。

writer 只需要：

```text
交付头
故事状态
压力链简报
一句写法底线：写给 AI 出海短剧成片，正文只写画面和声音里能出现的东西
```

抽检、证据、失败归因留在 reviewer / 主控层。

## 8. 当前判断

这次不是再发现一个“新规则”。

这是发现：

```text
workflow 已经能防很多错，
但还没有把交付合同和产物优选放到足够高的位置。
```

下一步如果修机制，只修：

- pass 语义拆开；
- Step 0 交付头；
- Screenplay 后三场 smoke test；
- 候选版本盲读排序。

不要继续扩设计提纲，不要继续为了 V15 无限修正文。
