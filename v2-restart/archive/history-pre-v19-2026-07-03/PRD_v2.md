- 第一阶段做什么、不做什么；

`设计提纲_v8.md` 保留为设计推演和规则池，不再作为当前主文档继续加内容。

---

## 2. 产品一句话

把一个已被市场验证过的短剧源本，洗成一个新壳下仍然好看、能追、能付费的新剧本。

产品交付的是剧本文字，不是分析报告。

分析、变量判断、自检、多 agent，都只是为了服务一件事：

**新本没有洗飞源本有效性，同时作为一个新剧本自己站得住。**

---

## 3. 产品不是

- 不是评分器。
- 不是从零原创工具。
- 不是短剧标准库套模板工具。
- 不是查词替换式换皮工具。
- 不是一键生成完整长剧的黑箱工具。
- 不是视频、分镜、投流片段或成片工具。
- 不是把多 agent 过程展示给用户看的 agent 展示产品。

---

## 4. 目标用户与场景

目标用户是写短剧本的人：用户本人，或团队里的编剧。

典型场景：

用户手上有一部爆款或可参考的短剧源本，想把它改成另一个题材、身份、背景、世界观或集数版本。用户不想从零原创，也不想让大模型把源本好看的东西洗丢。

用户可能很清楚方向，例如“把黑手党复仇洗成豪门商战”；也可能只有模糊想法，需要先对齐。

---

## 5. 产品形态路线

### 第一阶段：skill / workflow 原型

第一阶段不先做网页 App。

先把创作流程做成可重复执行的 skill / workflow，用真实样本验证它能不能稳定产出可用剧本。

原因：

- 当前最大风险是剧本质量，不是界面。
- 过早做网页 App 容易反过来把创作流程写死。
- 现在还需要通过样本验证继续调整分析、写作、自检之间的关系。

### 中期：轻量工具化

当流程稳定后，可以做轻量本地工具或半自动流程，固定：

1. 上传或选择源本；
2. 需求对齐；
3. 输出源本分析；
4. 输出变量迁移策略；
5. 输出首批新本；
6. 独立 reviewer 审稿；
7. 用户确认后续写。

### 长期：网页 App

网页 App 是最终用户入口，但不是创作能力本身。

网页 App 负责：

- 上传源本；
- 填或对话确认新壳、集数、改写力度；
- 展示源本分析和迁移策略；
- 展示首批新本；
- 收集用户反馈；
- 管理版本、批次、审稿记录。

网页 App 不应该让用户看到内部 agent 分工，也不应该把创作流程压成机械表单。

---

## 6. 内部 agent 分工

多 agent 是后台执行方式，不是用户可见产品形态。

### 主控 agent

负责：

- 和用户对齐需求；
- 选择当前流程；
- 决定什么时候需要用户拍板；
- 汇总主创和 reviewer 的结果；
- 最终对用户交付。

主控 agent 拍板，不让多个 agent 各自扩大规则。

### 主创 agent

负责：

- 读源本；
- 分析源本有效性；
- 做变量迁移策略；
- 写新本。

原则：**源本分析和写剧本必须由同一个主创 agent 承担。**

原因是写作需要继承分析里的创作判断。如果分析和写作彻底拆开，写作 agent 容易只拿到标签，丢掉源本为什么好看的细节。

主创写完首批后，必须先做作者自检。作者自检不是为了证明自己正确，而是先确认：有没有按迁移方案写完整、有没有明显洗飞、有没有漏掉自己设定的迁移基准。

### 独立 reviewer agent

负责：

- 不参与写作；
- 在干净上下文里审首批新本；
- 检查有没有洗飞；
- 检查新本自己好不好看、站不站得住；
- 指出机械复刻、空转、钩子堆叠、动机不成立、不可拍等问题。

原则：**作者自检可以有，但不能替代独立 reviewer。**

独立 reviewer 第一轮不要读取主创自检，避免被作者解释带偏。Reviewer 先独立审完，再由主控 agent 把作者自检和 reviewer 意见对照，判断哪些是必须改、哪些等用户或导演反馈。

### 用户

用户负责：

- 定新壳；
- 定集数或改写力度；
- 确认关键取舍；
- 判断首批方向是否继续。

---

## 7. 第一阶段范围

第一阶段只跑通：

```text
源本 + 用户需求
  -> 需求对齐
  -> 源本有效性分析
  -> 变量迁移策略
  -> 事件载体重建
  -> 洗稿方案
  -> 首批新本
  -> 独立自检
  -> 用户确认
```

第一阶段不一次性洗完整本。

默认先交付首批。首批通过后，再继续分批写。

---

## 8. 输入

### 必填输入

1. 源剧本。
2. 改写目标。

### 需求对齐必须问清

| 问题 | 作用 |
|---|---|
| 想换成什么新壳 | 决定题材、身份、世界观、场景怎么重建 |
| 改写力度多大 | 决定哪些高度贴源本，哪些等效迁移 |
| 集数是否改变 | 决定留存节点和节奏是否重排 |
| 目标市场 | 决定哪些设定需要本地化或替换 |
| 用户特别想保留什么 | 锁定用户主观偏好 |
| 用户特别不想要什么 | 避免继承源本烂点或用户雷点 |

---

## 9. 核心工作流

### 1. 需求对齐

不是让用户填长表，而是用自然语言问清关键决策。

输出：需求确认摘要。

### 2. 分析源本有效性

要回答：源本为什么让观众继续看？

分析按五层看：

| 层 | 看什么 |
|---|---|
| 留存骨架 | 开头、每集钩子、关键卡点、付费或留存节点 |
| 情绪爽点 | 压抑、释放、爽点兑现、高潮递进 |
| 主线推进 | 核心期待、信息差、秘密揭露、每集/每场新进展 |
| 人物压力 | 主角驱动力、关系功能、反派层级、压迫升级 |
| 干净度 | 水分、烂点、AI 味、非视听语言、不可拍描述 |

输出：只写影响改写的源本有效性摘要，不输出长篇分析报告。

### 3. 形成变量迁移策略

把源本内容标成四类：

| 策略 | 含义 |
|---|---|
| 强继承 | 源本追看功能、情绪走势、关键期待不能无故丢 |
| 等效迁移 | 原形式不适合新壳，但功能必须保留 |
| 自由改写 | 外壳、身份、职业、地点、对白、桥段包装可换 |
| 必须删除或修复 | 水分、烂点、假动机、虚假悬念、不适配设定 |

输出：变量迁移策略表。

### 4. 判断新壳承载

要回答：用户的新题材和新背景，能不能承载源本有效性？

如果承载不了，不是硬套，而是做等效迁移。

输出：新壳承载判断。

### 5. 事件载体重建与同构扣除

进入写正文前，必须先完成事件载体重建，避免新本只是源本换名。

事件载体重建要回答：

- 源本这一节点真正要保留的功能是什么；
- 源本用了什么具体事件、场景、道具、证据、人物行动来承载；
- 这些具体形式里哪些不能复刻；
- 新壳下用什么新的权力关系、压力来源、行动方式和信息揭露来承载同一功能；
- 同构风险高不高。

输出：事件载体迁移表。

如果事件表看起来仍然只是把黑手党、医院、亲子鉴定等源本表层元素换成董事会、合同、审计报告，不能进入正文写作。

### 6. 生成洗稿方案

方案要说清：

- 新壳怎么落；
- 哪些强继承；
- 哪些等效迁移；
- 哪些自由改写；
- 哪些删掉或修掉；
- 首批覆盖哪个留存闭环。

方案需要用户确认后，再进入写作。

### 7. 写首批新本

写作不是把源本文字改名替换。

写作步骤是：

1. 先拿到每集的叙事功能；
2. 再判断新壳里什么场面能承载这个功能；
3. 再写成新的可拍场景和对白；
4. 最后检查每集有没有关系、局势、信息或情绪变化。

首批不是固定 3 集、5 集或 10 集，而是覆盖第一个完整留存小闭环。

### 8. 作者自检

主创 agent 自己先检查：

- 有没有洗飞；
- 有没有按自己的迁移方案写完整；
- 有没有按事件载体迁移表重建，而不是换词；
- 有没有漏掉迁移基准；
- 有没有机械复刻；
- 有没有只虐不爽；
- 有没有太快和解；
- 有没有信息空转；
- 有没有不可拍抽象描述。

作者自检是第一道底线检查，不是最终质量判断。

### 9. 独立 reviewer 自检

独立 reviewer 在干净上下文里审。

第一轮 reviewer 输入不包含作者自检，只包含：用户需求、源本有效性摘要、变量迁移策略、事件载体迁移表、必要源本片段、新本首批、PRD 验收标准。

它不是重写者，主要输出：

- 硬失败；
- 重警告；
- 可选优化；
- 是否建议进入用户确认。

Reviewer 完成后，主控 agent 再对比“作者自检 vs reviewer 意见”，判断主创漏了什么、reviewer 是否误判，以及是否需要先改稿再给用户或导演。

### 10. 主控交付给用户

用户看到的是：

1. 首批新本；
2. 简短改动说明；
3. 自检结果；
4. 需要用户判断的问题。

---

## 10. 首批交付原则

首批要覆盖第一个完整留存小闭环。

一个完整小闭环至少包括：

1. 开头抓人；
2. 核心压力或期待建立；
3. 至少一次阶段性兑现，或一次足够强的悬置；
4. 新一轮追剧问题成立。

判断标准：

观众看完首批，既觉得已经看到一点东西，又想知道接下来会怎样。

---

## 11. 自检与验收

自检不是评分器，不合成总分。

### 硬底线

源本有效性是否迁移。

洗飞就失败。

### 质量标准

新本自身是否：

- 自洽；
- 好看；
- 不假；
- 不 AI；
- 能拍出来；
- 没有为了刺激硬堆钩子。

### 硬失败

| 硬失败 | 处理 |
|---|---|
| 源本有效性没迁过去 | 回到源本分析和变量策略 |
| 强继承项无故丢失 | 回到迁移基准 |
| 新壳承载不住 | 回到承载判断 |
| 新本被外部读者判断为“只是改名字” | 回到变量迁移策略，重建事件载体 |
| 主线期待断掉或空转 | 回到主线推进 |
| 人物动机站不住 | 回到人物和场景重写 |
| 水分烂点被继承 | 删除或修复 |
| 为刺激硬堆钩子爽点 | 回到剧本重写 |
| 抽象不可拍 | 回到场景重写 |

---

## 12. 第一阶段交付物

| 交付物 | 用户是否可见 | 作用 |
|---|---|---|
| 需求确认摘要 | 是 | 确认方向 |
| 源本有效性摘要 | 是 | 说明源本靠什么有效 |
| 变量迁移策略表 | 是 | 说明保留、替换、删除什么 |
| 事件载体迁移表 | 可见 | 证明新本保留功能但重建具体事件 |
| 洗稿方案 | 是 | 写首批前给用户拍板 |
| 首批新本 | 是 | 第一批可读剧本文字 |
| 改动说明 | 是 | 解释新壳怎么落 |
| 作者自检 | 可见 | 说明主创自查结果 |
| 独立 reviewer 意见 | 可见 | 给用户更可信的质量判断 |

---

## 13. 成功标准

第一阶段成功，不等于已经能自动产出完整爆款。

第一阶段成功是：

1. 用户给源本和方向后，产品能完成需求对齐。
2. 产品能读出源本有效性骨架。
3. 产品能给出合理变量迁移策略。
4. 产品能在正文前给出事件载体迁移表，证明不是换名复刻。
5. 用户能看懂并确认洗稿方案。
6. 产品能写出首批新本。
7. 首批新本没有洗飞源本有效性。
8. 独立 reviewer 没有指出硬失败。
9. 用户能明确判断：继续、调整后继续，或停止。

## 14. 当前样本验证结论

已用飞书三星样本《黑手党少主痛哭求原谅，前妻她杀疯了》做了一次端到端小跑。

用户确认的新壳：

- 豪门商战 / 家族企业；
- 集数不变。

已产出：

- `样本验证/飞书三星样本/三星样本小跑_黑手党少主.md`
- `样本验证/飞书三星样本/豪门商战版_首批1-10集.md`

当前结论：

1-10 集已完成主创稿和独立 reviewer 审稿，但导演反馈认为“现在就是改个名字，其他东西都是一模一样的，太明显了”。

这不是继续润色能解决的问题，而是事件载体同构过高。下一步不继续写 11-37 集，先回到事件载体迁移表，重建 1-10 集的具体事件链，再写新稿。

## 15. 风险

| 风险 | 表现 | 应对 |
|---|---|---|
| PRD 变摆设 | 设计文档越来越重，PRD 不再指导执行 | PRD_v2 作为主文档，设计提纲降级为参考 |
| 机械套模板 | 变量表被当成硬规则 | 强调功能迁移，不照抄形式 |
| 改名感太强 | 源本事件链整串保留，新本看起来只是换人名和场景名 | 保留功能链，重建事件载体；外部读者说“只是改名”视为不通过 |
| 多 agent 混乱 | 每个 agent 各自放大规则 | 主控拍板，主创写，reviewer 审 |
| 作者自检自证 | 写作者给自己找理由 | 增加独立 reviewer |
| 写作和分析断裂 | 分析 agent 给标签，写作 agent 写飞 | 主创 agent 同时负责分析和写作 |
| 后段继承烂点 | 源本后半段弱点被照搬 | 水分烂点单独分层，后段允许更大改写 |
| 网页 App 过早 | 界面把创作流程写死 | 先 skill/workflow 验证，再工具化 |

---

## 16. 下一步

1. 为当前豪门商战版 1-10 集补做事件载体迁移表。
2. 按迁移表重写 1-10 集，重点解决“只是改名”的同构问题。
3. 主创先做作者自检。
4. 单开干净 reviewer 审新稿，重点审同构风险。
5. 根据导演和 reviewer 意见，判断是否继续 11-37 集大纲。
6. 后续把稳定流程沉淀为 skill / workflow。
