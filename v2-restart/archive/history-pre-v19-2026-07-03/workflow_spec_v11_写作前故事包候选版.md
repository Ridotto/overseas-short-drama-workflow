## 0. 这版解决什么

v11 不重新定义产品，也不复活旧方案。

产品仍然是洗稿：

```text
基于已验证或值得参考的短剧源本，
迁移源本让人继续看、继续付费的有效性，
同时换成新壳下能成立的新剧本。
```

v10 的主要问题不是“分析不够”，而是：

```text
分析层、创作层、审计层混在一起。
writer 拿到的不是可写故事包，而是一堆证明链、禁区和检查项。
```

V20 的失败集中说明：

- 源本有效性迁移了；
- 多条链路也写进了工作稿；
- 但 Ep7-Ep8 仍变成“认证台 / 屏幕结果 / 反派否认 / 女配反咬”；
- Ep8 还提前消费 Ep9 的火葬场信息。

所以 v11 只修一个大接口：

```text
Source analysis / Adaptation Boundary
  -> 新本创作开发包 / 阶段骨架 / 分集追剧提纲
  -> 写作前故事包 / 可写剧本简报
  -> Screenplay
```

不是加平台式大纲，不是再加一堆 gate。

---

## 0.1 外部平台处理原则

外部商业平台的导出产物只作为成品形态观察，不作为本 workflow 设计依据。

本项目不做：

- 不照抄平台字段；
- 不倒推平台黑箱中间链路；
- 不把平台的“信息流 / 人物设定 / 粗纲 / 集纲 / 正文”当本项目主流程；
- 不因为平台有某类产物，就强行加到本项目里。

本项目中间产物可以比平台更强，也可以更薄。

唯一判断标准：

```text
这个产物是否帮助把源本追看机制和新壳场面变成可写正文。
```

## 1. 总 workflow

```text
0. 最小需求确认
1. 读源本
2. 源本有效性包 / Source Bible
3. 二次需求确认
4. 洗稿边界包 / Adaptation Boundary
5. 新本创作开发包 / Creative Development Pack
6. 首批 / 全剧阶段骨架 / Stage Skeleton
7. 分集追剧提纲 / Episode Pursuit Map
8. 高压场面候选与优选 / Scene Option Pitch + Choice Gate
9. 写作前故事包 / Playable Writer Brief
10. Screenplay draft
11. Rewrite report
12. Pressure-chain patch brief
13. Dialogue polish
14. 作者 ready check
15. Story memory checkpoint
16. 独立 reviewer
17. 主控版本对比与交付判断
```

v11 的关键不是步骤更多，而是每一步谁消费、失败回哪里更清楚。

### 1.1 本版小修原则：先接线，不建大引擎

本版不把 0xsline 的钩子库、爽点矩阵、反派体系和评分表整套搬进来。

只在现有链路里补 4 个二元存活问和 1 个 reviewer lens：

```text
追问还在吗？
压抑释放还在吗？
现场戏还在吗？
对抗燃料还在吗？
reviewer 能否按层指出回滚点并给出 suggestion？
```

这 4 个问题不是新表，也不是评分项。它们只负责检查上游分析里的短剧能力传到下游时有没有死掉。

如果某项回答弱，不默认加规则、加分数或加场面；先回到对应上游产物重接。

---

## 2. 四条线和材料可见性

| 线 | 产物 | 谁看 | 作用 |
|---|---|---|---|
| 分析线 | 源本有效性包、洗稿边界包、源本锚点 | 主创 / 主控 / reviewer | 保住源本为什么有效 |
| 创作线 | 新本创作开发包、阶段骨架、分集追剧提纲、场面候选、写作前故事包、正文 | 主创；writer 只读写作前故事包 | 把有效性变成新本戏 |
| 审计线 | rewrite report、ready check、reviewer report | 主控 / reviewer | 判断失败层 |
| 治理线 | manifest、版本对比、决策记录 | 主控 | 证明流程，不证明质量 |

规则：

```text
writer 不读完整审计底稿。
writer 只读写作前故事包。
```

同一个主创可以完成分析和写作，但在“正文起草”这个动作里，输入必须是薄的、可写的故事包。

## 3. Step 0. 最小需求确认

目标：拿到足够开工的信息，不把用户问爆。

必确认：

```text
任务类型：洗稿 / 改编 / 重写 / 诊断 / 审稿
源本：
新壳：
目标市场：
集数是否变化：
改写力度：
本轮输出范围：
输出语言 / 审稿语言：
正文用途：AI 视频生成用剧本 / 人审稿 / 字幕台词版 / 双语版
已知禁区：
```

默认：

```text
台词英文；
动作和场景描述中文；
内部分析中文；
目标是 AI 出海短剧剧本文字。
```

但必须显式问：

```text
台词默认英文，是否需要中文 / 英文 / 中英双语？
动作行和场景描述用中文、英文，还是双语？
本轮交付是给导演审稿，还是直接进入 AI 视频生成链路？
```

---

## 4. Step 1-2. 源本有效性包 / Source Bible

目标：读出源本为什么好看、为什么能追、为什么能付费。

必须输出：

```text
1. 一句话故事核
2. 核心期待
3. 首批分集功能
4. 高价值节点
5. 钩子与卡点
6. 爽点 / 压抑-释放
7. 关系 / 权力变化
8. 信息差与真相路径
9. 写作手法 / 表达手法
10. 水分 / 烂点 / 禁止复用外形
```

每个高价值节点必须写：

```text
客观事件：
观众体验功能：
强度来源：
关系 / 权力变化：
信息差状态：
卡点停法：
源本高辨识外形：
下游写作抓手：
```

通过标准：

```text
下游能知道：保什么体验、避什么外形、在哪里停、谁压谁、谁退谁进、观众为什么想看下一集。
```

如果源本自身在关键事实、身份、血缘、时间线、人物动机上有冲突或含混，Source Bible 必须显式标注：

```text
源本冲突点：
可能解释：
不允许原样下传的风险：
需要新本创作开发包拍板的问题：
```

Source Bible 可以保留源本原貌，但不能把源本矛盾抹平成一串并列信息后交给下游。

注意：这一步可以厚，可以保留 source ref，但它不是 writer brief。

---

## 5. Step 3. 二次需求确认

目标：读完源本后，再判断用户初始需求是否能承载源本有效性。

主创先判断，再让用户拍板。

必须确认：

```text
新壳是否能承载源本最强追看机制；
本轮改写力度；
哪些体验必须强继承；
哪些事件外形必须换；
哪些变量可自由改；
哪些烂点必须删；
集数变化时留存阶段如何等效重排；
用户特别想保留或避开的东西。
```

如果用户需求会明显削弱源本有效性，必须直接说。

---

## 6. Step 4. 洗稿边界包 / Adaptation Boundary

目标：把源本分析压成薄边界，防止 writer 既洗飞又复刻。

输出：

```text
必须保留的体验功能：
必须避开的源本外形：
可以自由换的外壳变量：
必须删除 / 修复的烂点：
新壳承载风险：
本轮用户拍板项：
```

禁止：

- 恢复成大变量矩阵；
- 把强继承理解成照抄事件；
- 为了避同构，把源本好看的钩子、节奏、关系和情绪洗掉；
- 只写禁区，不写替代承载方向。

---

## 7. Step 5. 新本创作开发包 / Creative Development Pack

目标：在进入分集和正文前，让新本先成为一个成立的故事，而不是从源本功能直接跳到单集任务。

这一步不是照抄平台式栏目，也不是写设定百科。它只收纳会影响后续写作的创作决定。

输出：

```text
Run Contract：
- 新壳：
- 目标市场：
- 目标受众 / 口味：只写会影响创作的部分
- 题材类型：
- 风格元素：只写会影响场面、台词、尺度的部分
- 输出语言：
- 正文用途：
- 改写力度：

New Story Premise：
- 新本一句话故事：
- 核心设定：
- 核心期待：
- 核心亮点：
- 核心梗概：
- 世界观：仅当题材需要时写，现代现实题材不厚写

Story Engine：
- 主角想要什么：
- 阻碍是什么：
- 赌注是什么：
- 观众追什么：
- 核心误会 / 信息差：
- 核心关系张力：

Core Truth + Pursuit / Payoff Model：
- 全剧主追问：
- 当前阶段追问：
- 单集追问的生成方式：
- 真实事实是什么：
- 角色当前误以为什么：
- 初始信息差：开局时观众被允许知道什么、误信什么、暂时不知道什么：
- 误导方伪造 / 遮蔽 / 扭曲了哪一层：
- 哪一层事实在哪个阶段兑现：
- 哪一层事实、关系变化、情绪击穿不能提前兑现：
- 兑现后关系 / 权力 / 情绪 / 局势至少哪一项变化：
- 已播事实之间是否自洽：

Character Operating State：
- 主要角色当前欲望：
- 主要角色误判：
- 主要角色底线：
- 主要角色会怎样行动，而不是只贴什么标签：

Relationship / Power State：
- 谁高位，谁低位：
- 谁靠近，谁后退：
- 谁握有信息，谁被蒙在鼓里：
- 本批结束前必须发生的关系 / 权力变化：

Character Operating Notes：
- 主要角色小传：只写欲望、误判、底线、行动方式，不写履历百科
- 主要反派为什么敢继续作死：
- 主角关系为什么能持续拉扯：
```

### 7.1 核心真相自洽门

Creative Development Pack 不能只“写了核心真相”，必须先过这一道门，才能进入 Stage Skeleton。

只检查三件事：

```text
事实账本：真实事实、对外谎言、伪造层分别是什么；
角色错信：主要角色各自误以为什么，且这些误信能驱动行动；
初始信息差：开局时观众知道 / 误信 / 不知道什么。
```

不检查揭示排程。

“第几集揭”“哪一集不能提前消费”属于 Stage Skeleton / Episode Pursuit Map 的信息边界，不塞进核心真相门。

不通过时：

```text
不得进入 Stage Skeleton；
不得把源本矛盾原样下传；
必须先在 Creative Development Pack 拍板。
```

通过标准：

- 主创能知道这版新本到底是什么故事；
- 后续阶段骨架和分集提纲能从这里长出来；
- writer brief 能从这里抽取当前集需要的内容；
- 每个设定、受众判断、人物小传都必须说明会逼出什么人物行动、冲突或后续场面；
- 核心真相和追问兑现模型必须自洽；如果源本有冲突，必须在这里拍板，不得原样下传；
- 不是人物百科；
- 不是世界观设定集；
- 不是平台式卖点包装；
- 每个设定都要能推动后续场面；如果不能推动，删掉。

## 8. Step 6. 首批 / 全剧阶段骨架 / Stage Skeleton

目标：在单集之前先确定跨集推进，防止每集各写各的，也防止提前消费后续火葬场、反转、关系击穿。

默认首批先做首批阶段骨架；如果用户要求全剧规划，再扩到全剧阶段骨架。

输出：

```text
阶段范围：
- 例如 1-3 / 4-6 / 7-8 / 9-10，或按本项目实际集数重排

阶段起点：
- 本阶段开始时，关系、权力、信息差、情绪状态是什么

阶段任务：
- 本阶段观众主要追什么
- 本阶段要压低什么
- 本阶段要兑现什么
- 本阶段要留下什么追问

阶段变化：
- 关系怎么变
- 权力怎么变
- 信息差怎么变
- 情绪怎么升级

阶段边界：
- 哪些真相不能提前说
- 哪些关系变化不能提前完成
- 哪些爽点 / 打脸 / 火葬场必须留到对应位置
```

通过标准：

- 分集提纲能从阶段骨架拆出来；
- 每个阶段有明确追问、兑现和留白；
- 不把阶段骨架写成剧情流水账；
- 每个阶段都必须写出压力来源、阶段末交给下一阶段的追问，以及本阶段绝不能提前消费的内容；
- 不强行规定秒点，但要保住源本留存惯性和情绪递进。
- 阶段计划必须承接 Creative Development Pack 的事实账本、角色错信和初始信息差，不能在阶段里新造一套核心事实。

## 9. Step 7. 分集追剧提纲 / Episode Pursuit Map

目标：每集写“如何留住观众”，不是只写“发生了什么”。

字段：

| 字段 | 要回答 |
|---|---|
| Episode | 第几集 |
| Title | 集标题 |
| Summary | 本集核心推进，1-2 句 |
| Opening Hook | 开头凭什么不被划走 |
| Core Friction | 本集主要摩擦 / 冲突 |
| Max Spike | 本集最大刺激点 |
| Change | 关系 / 局势 / 信息 / 情绪至少哪一项改变 |
| End Button | 结尾钩子，停在答案前一拍 |
| Next-Episode Payoff | 下一集必须兑现什么，不能只是继续吊着 |
| Source Function Ref | 对应源本追看功能，不是源本事件外形 |
| Forbidden Shape | 本集不能复用的源本外形 |
| Do Not Consume | 本集绝不能提前消费的后续内容 |
| Pressure Progression | 起压 / 升压 / 爆点 / 收住 |
| Opposition Fuel | 本集结束后，谁还有资源、筹码或误判能继续反扑 |
| Chain Carry | 本集真正承担的 2-4 条主链 |

`Do Not Consume` 不能只写道具或证据。

必须包括：

```text
不可提前说出的事实；
不可提前完成的关系变化；
不可提前释放的情绪击穿；
不可提前给答案的追问。
```

通过标准：

- 每集有 Max Spike；
- 每集有 End Button；
- End Button 必须对应一个下一集可兑现的问题，不只是最后一帧有画面；
- 每集有 Do Not Consume；
- 本集至少有一处压抑释放：释放后有人可见地失去位置、脸面、权力、关系或安全感；
- 本集结束时仍有下一轮对抗燃料，除非这是明确收束集；
- 连续两集不能只是同一种场面形态重复；
- 高压集能继续进入 Step 8 做候选。

## 10. Step 8. 高压场面候选与优选

目标：同一个源本强点，新壳下先想 2-3 种可写法，再选最有记忆点的可见后果载体，不默认选最安全、最容易解释的机制。

只做高风险节点：

- 前 1-3 集；
- 每集结尾卡点；
- 强羞辱、强危机、强误会、强打脸；
- 身份揭露；
- 关系质变；
- 同构风险高的源本桥段；
- 上一版导演 / reviewer 指出的失败点。

每个候选必须写：

```text
候选名：
场面一句话：
谁压谁：
谁旁观：
谁被迫选择：
身体 / 关系 / 社会性代价：
爽点前怎么压低：
爆点是什么：
可见后果 / 失去 / 反扑 / 卡点是什么：
卡点停在哪个画面 / 动作 / 声音：
信息边界：
同构风险：
短剧降级风险：
AI 成片可视化风险：
```

### 10.1 场面存活问

高压候选必须通过一个二元问题：

```text
删掉屏幕 / 文件 / 录音 / 报告之后，这场高压戏还剩不剩现场人物行动、空间压力、身体反应、公开后果或不可逆选择？
```

如果答案是否定，说明候选只是证据链或机制链，不是短剧场面。

这时不是给证据链贴身体反应，而是回 Step 8 重选场面载体。

不合格候选：

- 只是换证据；
- 只是换地点；
- 只是把源本桥段改名；
- 只靠文件、会议、屏幕或报告完成高潮；
- 只能证明剧情通顺，看不出哪一下让人想追；
- 事件顺序仍然像源本，只是道具换了。

这里的“可见后果”是节点级二元判断：高压节点整体有没有一个观众能记住的失去、反扑、关系断裂、公开选择或卡点。有即可进入风险权衡；不要下沉成逐句台词口味审，也不要和 V5 等更强版本做比较级打分。

优选输出：

```text
选中候选：
为什么选它：
淘汰候选：
淘汰原因：
仍有风险：
如果正文写弱，回滚到哪里：
```

这里允许创作判断。

不要把创作判断伪装成客观打分。

---

## 11. Step 9. 写作前故事包 / Playable Writer Brief

目标：writer 真正拿到的是可写戏的故事包，不是审计底稿。

### 11.1 内部开发包与 writer brief 分开

内部开发包可以厚，包括源本拆文、洗稿边界、新本创作开发包、阶段骨架、分集追剧提纲、场面候选、风险记录。

writer brief 必须薄，只保留正文写作需要的信息。

writer brief 可见：

```text
Run Contract：
- 新壳 / 目标市场 / 输出语言 / 正文用途 / 改写力度

Current Story State：
- 上一集结束后，关系、权力、信息差、情绪状态是什么

Episode Pursuit：
- 本集观众等什么
- 本集兑现什么
- 本集最后卡什么
- 下一集必须兑现什么
- 本集绝不能提前消费什么

Selected Scene Carrier：
- 选中的高压场面一句话
- 谁压谁
- 谁被迫选择
- 谁失去什么
- 谁获得什么
- 删掉文件 / 屏幕 / 报告后，现场戏还剩什么

Character Pressure：
- 每个关键角色此刻想要什么
- 误判什么
- 怕失去什么
- 会用什么行动保住自己
- 本集结束后谁还有反扑资源 / 筹码 / 误判

Action Chain：
- 起手动作
- 压迫升级
- 反扑 / 反转
- 不可逆后果

Scene Sequence：
- 本集先写哪场，再写哪场
- 普通过渡场如何把人物推到高压场
- 哪些场可以合并或删除

Information Boundary：
- 本集可以公开什么
- 本集只能暗示什么
- 本集绝不能说穿什么
- 哪个追问必须留到后面

Last Frame / Sound / Action：
- 本集最后停住的可拍瞬间
- 这个瞬间对应的未答问题

Screenplay Contract：
- 正文只写观众能看见、听见、AI 能生成的东西
- 台词短、准、有功能
- 高压场必须把情绪落到动作、身体、空间、声音或当众后果上
- 不能靠多人站着低信息对话撑场
```

writer brief 不可见：

```text
完整 Source Bible；
完整 source ref 大表；
manifest；
reviewer 报告；
版本对比；
外部研究材料；
历史失败清单；
证明链；
所有候选的审计过程。
```

### 11.2 通过标准

```text
writer 能只看这个 brief 写本集正文；
reviewer 能根据这个 brief 回查正文有没有写飞；
brief 里没有 manifest / reviewer / 历史争论噪音；
brief 给的是场面发动机，不是“必须做到 X、Y、Z”的任务清单。
brief 必须让 writer 看见四个存活问题的答案：追问、压抑释放、现场戏、对抗燃料。
```

如果 writer brief 只剩：

- 禁止项；
- 检查项；
- 源本 ref；
- 证明链；
- 抽象情绪词；

则不通过，回 Step 8 或 Step 9 重做。

## 12. Step 10. Screenplay draft

目标：交付真正剧本正文，不是说明书。

核心合同：

```text
正文页上写出来的东西，应该基本等于最终 AI 成片里能看见和听见的东西。
```

正文容器：

```text
# Episode N: Title

INT. / EXT. LOCATION - DAY / NIGHT

可被观众看见或听见的动作、环境、物件、身体反应、空间变化。
每段动作行尽量短，只写正在发生的事情，不写作者解释。

CHARACTER
(必要时才写简短括号提示)
英文台词，除非需求确认指定中文或双语。

SFX: 必要声音。
ON SCREEN: 必要屏幕文字。
```

写作质量底线：

- 高压场不能只靠文件、屏幕、录音、报告或对白完成高潮；
- 角色不能站桩聊天，场面里必须有人逼近、后退、打断、夺走、暴露、失控、选择或承受后果；
- 情绪不能只写“崩溃 / 黑化 / 心痛 / 愤怒”，必须落到观众能看见或听见的动作、身体反应、空间变化、声音、物件或当众后果；
- 台词必须推动关系、权力、信息、选择或卡点；多轮低信息对话必须删或合并；
- 高压节点人物不能像冷静开会，除非“冷静”本身是角色主动压迫对方的手段，并且现场后果足够强；
- 每场结束时，关系、权力、信息、情绪或局势至少有一项发生可见变化。

正文不能出现内部字段：

- `Source Function Ref`
- `Forbidden Shape`
- `End Button`
- `Do Not Consume`
- `Chain Carry`
- `Evidence triggers action`
- `Purpose`
- `Action`
- `Performance`
- `Dialogue Functions`

但字段背后的约束必须被消化进正文。

## 13. Step 11-13. 返修

返修先判失败层，不直接润色。

输出：

```text
Failure Layer：
- source_analysis / boundary / creative_pack / stage_skeleton / episode_map / scene_option / writer_brief / screenplay / dialogue / continuity / reviewer

Root Symptoms：
- 至少 2 条

Prioritized Actions：
- 先改什么
- 后改什么

Rollback Point：
- 回哪一步
```

返修原则：

```text
新本故事不成立 -> 回 Step 5
阶段推进错 / 提前消费 -> 回 Step 6
场面发动机错 -> 回 Step 8
writer brief 不能写戏 -> 回 Step 9
正文表达弱 -> 回 Step 10
台词水 -> Step 13 Dialogue polish
状态漂移 -> 回 Step 5 / Step 15
源本洗飞或复刻 -> 回 Step 2 / Step 4
```

不得默认“加料”。

如果一个场面靠机制、屏幕、文件撑起来，修法优先是重选场面载体，而不是往机制上贴更多身体反应。

## 14. Step 14-15. 作者 ready check 与 Story memory

作者 ready check 只给三种结论：

```text
ready_for_reviewer
needs_rewrite
blocked
```

ready check 必须说明：

- 哪些高风险节点已从源本有效性迁到正文；
- 哪些节点仍有风险；
- 如果失败，应该回哪一步。

Story memory 记录：

```text
角色状态；
关系距离；
权力位置；
信息差；
已兑现追问；
未兑现追问；
后续不可提前消费内容；
伏笔和物件状态。
```

Story memory 不是剧情复述，是给后续批次防状态漂移。

## 15. Step 16. 独立 reviewer

每个首批版本单开干净 reviewer。

Reviewer 两轮：

1. 第一轮只读最小需求 + 正文，判断剧本效果；
2. 第二轮再读源本有效性包、洗稿边界、新本创作开发包、阶段骨架、writer brief 和必要源本片段，查洗飞、同构、降级、信息边界。

Reviewer 不读作者自检作为第一轮依据。

Reviewer 输出：

```text
pass / revise / block

失败层：
- source_analysis / boundary / creative_pack / stage_skeleton / episode_map / scene_option / writer_brief / screenplay / dialogue / continuity

红线判断：
- 没洗飞源本有效性：
- 不是改名复刻：
- 新本自己能追：

finding 分级：
- 致命阻断：
- 应改风险：
- 可选优化：

每条 finding 必须带：
- 回滚层级：
- 修改建议：
- 是否需要重审全稿：

不该修改项：
证据：
```

Reviewer 不能只说“剧本不好看”。必须指出失败发生在哪一层。

Reviewer 判定规则：

```text
致命阻断 = 任一：
- 三条产品红线之一失败；
- 核心真相 / 核心反转 / 主要人物动机自相矛盾；
- 和正文已播事实冲突，导致追剧逻辑断裂；
- 硬边界被破坏到影响“不是改名复刻”或“新本自己能追”。

不属于致命阻断：
- “观众可能看不懂”这类软判断；
- 动机可以再解释一点；
- 局部相似偏高但没有形成改名复刻第一印象；
- 台词口味、金句密度、伏笔可更早；
- 某个高压节点已有可见后果但还能更强。

Reviewer 不得只写“更短剧 / 更刺激 / 更好看”。

必须说明失败落在哪一层，并给出可执行 suggestion。

应改风险和可选优化不能在下一轮自动升级成新的重审入口，除非它引入或暴露了致命阻断。
```

Verdict 规则：

```text
无致命阻断 -> pass，可带风险登记；
一个明确致命阻断，且修复范围清晰 -> pass-pending-lock / single-blocker revise；
多个致命阻断，或需要回上游重排阶段/故事发动机 -> revise；
源本有效性洗飞、新壳不成立、正文不可读不可拍 -> block。
```

`pass-pending-lock` 的处理：

```text
只修被点名的 lock；
修完后只做窄验证：确认 lock 已修对，且没有引入新的主线断裂；
不得重新打开全稿 review；
风险登记项不能在 lock 后升级成新一轮重审理由。
```

## 16. Step 17. 主控版本对比与交付判断

主控不能只看最终剧本。

必须看：

- manifest；
- 新本创作开发包；
- 阶段骨架；
- 写作前故事包；
- 正文；
- 作者 ready check；
- reviewer 两轮；
- 与关键旧版本的整体对比。

交付判断分开写：

```text
workflow execution：
creative outcome：
source carry-through：
delivery outcome：
```

manifest complete 只说明流程执行完成，不说明剧本质量 pass。

## 17. v11 的最小验证方式

下一次如果跑样本，不要只检查“v11 有没有照步骤跑完”。

必须看：

1. 新本创作开发包是否让新本先成为一个成立的故事，而不是一串功能任务；
2. 阶段骨架是否保住跨集追问、兑现、留白和信息边界；
3. writer brief 是否真的不含审计噪音；
4. 正文是否像剧本，不像流程说明；
5. Ep7-Ep8 这类高风险节点是否不再默认滑成认证 / 证据 / 屏幕 / 反派否认；
6. 信息边界是否保住后续火葬场、关系击穿和追问；
7. reviewer 能否把失败明确打回某个中间产物。

如果这些做不到，v11 也不能正式化。
