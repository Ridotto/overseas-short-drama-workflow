# PRD v3 vs 设计提纲 v8 漂移 review

> 审查对象：`PRD_v3.md` 对 `设计提纲_v8.md` 的吸收是否发生产品方向漂移。
> 审查结论只看产品设计与执行规则，不看工程实现。

## 1. 总结论：小修

PRD v3 没有从 v8 的核心方向漂移。

它仍然表达的是“把爆款源本的有效性迁移到新壳里”，不是从零原创、不是分析报告、不是模板化短剧生成器，也不是查词替换式换皮。

需要小修的点主要是：PRD v3 为了落地当前样本问题，新增了几类写作前工作稿，这是对的；但少数 v8 里的执行硬点在 PRD 里被压缩了，后续主创执行时可能不够醒目。

## 2. 是否有产品定义漂移

没有明显漂移。

v8 的核心定义是：拿一个验证过的爆款源本，读懂它为什么抓人，再把这套有效性迁移到新壳里；保留的是“源本有效性骨架”，不是人物名字、具体场景、原句对白或机械卡点。

PRD v3 对应吸收得比较准：

- 产品一句话仍然是“已被市场验证过的短剧源本 -> 新壳下仍然好看、能追、能付费的新剧本”。
- 明确说产品交付剧本文字，不是分析报告。
- 明确排除了评分器、从零原创工具、标准库套模板工具、查词替换式换皮工具。
- 第一阶段定位为 skill / workflow 原型，而不是先做网页 App 或工程 spec。

所以 PRD v3 的产品定义没有变成“原创重构工具”。它的核心仍是商业有效性迁移。

## 3. 是否有变量控制漂移

基本没有漂移，但有轻微压缩。

PRD v3 保留了 v8 的四类策略：

- 强继承；
- 等效迁移；
- 自由改写；
- 必须删除或修复。

它也正确写了：改写力度不是整部剧一刀切成“中度”或“深度”，而是按变量分别判断；同构红线不是默认深度重构，而是防止源本高辨识事件外形整串照搬。

风险在于，PRD v3 的变量表比 v8 更压缩。v8 把“付费/关键留存节点、爽点兑现周期、信息差结构、钩子类型、关键道具/秘密/证据、人物关系功能”等拆得更细；PRD v3 多数都吸收了，但有些被合并到大项里。作为主文档执行时，主创可能漏掉“付费点/强留存点”和“信息密度/兑现周期”这类具体检查。

这不是方向错误，是执行颗粒度稍弱。

## 4. PRD v3 已正确吸收 v8 的地方

1. 核心目标吸收正确。
   PRD v3 仍以“源本有效性没有洗飞，同时新本自己站得住”为目标，没有把产品改成原创工具或报告工具。

2. 五层分析吸收正确。
   留存骨架、情绪爽点、主线推进、人物压力、干净度都进入了 PRD v3。

3. 变量迁移吸收正确。
   强继承、等效迁移、自由改写、删除/修复四类策略完整保留，并且强调强继承的是观众体验功能，不是源本事件外形。

4. 同构红线表达正确。
   PRD v3 多处写明：同构红线不是默认深度重构；要改的是承载方式，不是放弃源本好看的原因。

5. 首批验证逻辑吸收正确。
   首批不固定集数，而是覆盖第一个完整留存小闭环；首批不过，不继续往后写。

6. 自检硬底线吸收正确。
   “源本有效性是否迁移”被放在硬底线位置；洗飞就是失败。

7. 多 agent 边界吸收正确。
   主控拍板、主创分析并写作、独立 reviewer 干净上下文审稿，这和 v8“可以并行分析，不能并行拍板”的精神一致。

8. 新增工作稿方向正确。
   源本节点拆解稿、迁移义务与同构红线稿、新本阶段骨架、分集功能表、关键场景草案，不是偏离 v8，而是在补 v8 到正文写作之间的断层。

## 5. PRD v3 仍缺或表达不够准的地方

1. 变量表颗粒度略粗。
   v8 单独列出的“付费/关键留存节点”“爽点兑现周期”“信息差结构”“钩子类型”“关键道具/秘密/证据”等，在 PRD v3 里多被合并进大项。执行时可能不如 v8 那么防漏。

2. “参考资产使用边界”没有单独落到 PRD。
   v8 明确说 short-drama 标准库、oh-story、how-to-make-script、shortdrama-pipeline、Dramatron 都只能借结构或做参照，不能压过源本。PRD v3 只在风险里写了“机械套模板”，但没有把参考资产的使用边界写清。

3. 首批闭环的表达弱了一点。
   v8 写的是观众看完首批要“看到一点兑现，又想知道接下来会怎样”。PRD v3 写成“看到一点东西”。“东西”太松，容易让只有信息铺垫、没有阶段性兑现的首批误过。

4. 源本节点拆解稿还可以更明确纳入“信息增量/兑现周期”。
   PRD v3 已要求节点说明功能、承载事件和高辨识外形，但没有明确要求标记这个节点带来的信息增量、情绪兑现或下一轮追看问题。v8 对信息密度和推进效率更敏感。

5. 迁移义务与同构红线稿可以加一个更硬的输出判断。
   PRD v3 已写“同构风险高不高”，也写了如果只是表层替换不能进入正文。但执行时最好要求每个高风险节点给出“可进入正文 / 必须重建”的明确结论，避免只写分析不触发返工。

## 6. 不建议改的地方

1. 不建议删掉新增的写作前工作稿。
   当前样本失败点正是“分析到正文之间断层太大”。新增源本节点拆解稿、迁移义务与同构红线稿、阶段骨架、分集功能表、关键场景草案，是必要补桥，不是流程膨胀。

2. 不建议把 PRD v3 改回 v8 那种完整设计推演。
   v8 已经降级为规则池，PRD v3 需要保持可执行，不应重新变成大段设计提纲。

3. 不建议把“同构红线”改成“必须大改剧情顺序”。
   这会反向洗飞源本留存骨架。正确做法仍然是保留功能链，重建事件载体。

4. 不建议把 reviewer 变成共同创作者。
   Reviewer 的价值是干净审稿，不是参与主创过程。第一轮不读作者自检这个规则应该保留。

5. 不建议第一阶段提前收敛到网页 App。
   当前风险是剧本质量和流程稳定性，不是界面入口。

## 7. 最小修改建议

1. 在 PRD v3 的变量迁移表里补 3-5 个防漏项：付费/关键留存节点、爽点兑现周期、信息差结构、关键道具/秘密/证据、钩子类型。

2. 把首批交付标准里的“看到一点东西”改成“看到一次阶段性兑现或足够强的悬置”。

3. 在“源本节点拆解稿”要求里补一句：每个关键节点必须标记信息增量、情绪兑现状态、下一轮追看问题。

4. 在“迁移义务与同构红线稿”里补一个明确结论字段：可进入正文 / 必须重建事件载体。

5. 增加一个很短的“参考资产使用边界”小节：外部标准库只做参照和异常提醒，不能压过源本有效性；拆文结构可借，不能把产品变成报告。
