## 第一部分 · 产品是什么

### 1. 一句话定义
**二次精洗器**：输入一个"已经照着爆款洗过一遍、但写得还不够强"的短剧本子，诊断它、重构它，输出一个**更能让人追下去、更像真人写、更有镜头感**的本子。

### 2. 关键事实（决定整个设计）
- **输入不是原创稿，是"洗稿稿"**：用户/编剧已经照着真实爆款洗过一道，结构上已接近爆款，**主要问题在执行层**（信息密度/镜头感/cliche/对白/AI味）。
- **所以主战场是"精洗执行层"，不是"重建结构"**。但要留一个保险：万一某稿其实是结构本身烂（伪装成执行问题），要能识别并退出提示。
- **最终输出 = 纯文本剧本**（给人读 / 给用户下游环节对接）。不输出 JSON、不碰视频。
- **产品是通用的**：方法论 + 标准库支撑**任意短剧赛道**，不聚焦单一题材。"想写什么赛道就写什么赛道"。

### 3. 用户
写本子的人（用户 / 编剧）。不服务导演、监制、甲方。

### 4. 它不是什么（防跑偏）
- ❌ 不是评分器（评分只是内部环节，不是产品）。
- ❌ 不是从零生成器（是"改已有本"，不是"凭空写新本"）。
- ❌ 不是一键成片 / 全链路视频工具。
- ❌ 不是"专门拟合导演那18个本"的分析器（那批数据将来或可做验证参考，不是产品本体）。

## 第二部分 · 产品怎么工作（核心 · workflow 主干）

> 一个本子进来，6 个环节走出去。每个环节都有【输入 → 做什么 → 产出】，环节间用产出物串联，全程对照"诊断库"和"需求单"，不跑偏。
> 架构特征（借三套 workflow）：**文件驱动**（产出落盘成 source of truth）+ **写评分离**（重写和评审不在同一手里，防自我批准）+ **全程防漂移**（追踪文件）。

```
┌──────────────────────────────────────────────────────────────┐
│  输入：一个洗稿短剧本子（.txt）                                    │
└──────────────────────────────────────────────────────────────┘
            ↓
┏━ 环节 0 · 需求对齐 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ 输入：用户 + 本子                                              ┃
┃ 做什么：问清楚要什么（不对齐不动手）                            ┃
┃   · 想达到什么效果？（更抓人 / 某段不满意 / 整体提质）           ┃
┃   · 题材/风格/情绪基调 要变还是保持？                           ┃
┃   · 改多狠？（L1只润色执行 / L2改桥段反转 / L3大改故事）         ┃
┃   · 绝对不能动的？（某人设/某条线/某卖点）                      ┃
┃   · 特别想要的？（更虐/节奏更快/某个爽点）                      ┃
┃ 产出：【需求单】← 后面所有环节对照它                            ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
            ↓
┏━ 环节 1 · 拆解落盘 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ 输入：本子 + 需求单                                            ┃
┃ 做什么：把本子彻底读透（先不评好坏），并逆向拆成可改写的结构      ┃
┃   1a 事实层：集/场/角色/人物关系网/卡点位置/篇幅                 ┃
┃   1b 六维质量抽取：                                            ┃
┃      ①情绪节拍（画情绪曲线±9，标空窗/塌点）                     ┃
┃      ②钩子/卡点（开篇钩+集尾扣，按钩子5类标）                   ┃
┃      ③反转/爽点（密度+意外度，按爽点矩阵标，查爽点闭环）         ┃
┃      ④结构/期待线（核心期待立没立、转折布置）                   ┃
┃      ⑤人物欲望/驱动力（主角想要啥、强不强清不清）               ┃
┃      ⑥信息密度（每场新信息、有无废戏注水）                     ┃
┃   1c 专项扫描：反派四层 / cliche点 / AI味·非视听语言 / 可拍性    ┃
┃   1d 逆向拆结构（借Dramatron）：logline/人物表/beat表/对白      ┃
┃ 产出：【诊断库】= source of truth，后续全程对照                 ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
            ↓
┏━ 环节 2 · 诊断分层 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ 输入：诊断库 + 需求单 + 方法论标准库                            ┃
┃ 做什么：判断这本子"现在到底什么状态"（不预设它弱）              ┃
┃   先定状态（三选一）：                                          ┃
┃     · 已经够好 → 微调即可（可能就是好本子）                     ┃
┃     · 局部有短板 → 哪几维弱，精洗能提                           ┃
┃     · 结构性问题 → 执行层救不了，触发"结构退出口"提示大改        ┃
┃   有短板则分层定位（借how-to-make-script，不许只说症状）：       ┃
┃     病在 概念层 / 结构层 / 场景层 / 对白层 哪一层？             ┃
┃     每个问题分级：必改 / 建议 / 可不改                          ┃
┃   对照的是【方法论标准库】，不是凭空，也不拟合谁的分            ┃
┃ 产出：【诊断结论】= 什么状态、若改改哪层、改到什么程度           ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
            ↓
┏━ 环节 3 · 改稿单 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ 输入：诊断结论 + 需求单                                        ┃
┃ 做什么：动笔前写死"这一稿怎么改"（防瞎改/改飘）                 ┃
┃   产物有固定骨架（借how-to-make-script契约思想），逐项填：        ┃
┃     · 保留什么（优点/用户要的，不准动）                         ┃
┃     · 改什么（哪几场/哪条线怎么改、为什么）                     ┃
┃     · 新增什么（缺的钩子/反转/信息补在哪）                      ┃
┃     · 禁止什么（不许用的套路、不许碰的设定）                    ┃
┃ 产出：【改稿单】= 重写施工图                                    ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
            ↓
┏━ 环节 4 · 逐层重生（重写）━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ 输入：改稿单 + 诊断库的结构 + 创作工艺库                        ┃
┃ 做什么：按改稿单重写，且保证"改一处、全篇一致"                  ┃
┃   引擎（借Dramatron）：改上层（logline/人物/beat）自动级联下层   ┃
┃   工艺（创作工艺库）：信息差锁/物件钩子/打脸三段式/              ┃
┃                      身体反应替代心理描写/围观反应链…            ┃
┃   硬标准：镜头感（情绪标签句不许出现）+ 去AI味（禁用词/去独白）   ┃
┃   边写边更新【追踪文件】（借oh-story）：伏笔/角色状态/时间线      ┃
┃ 产出：【重写稿】                                               ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
            ↓
┏━ 环节 5 · 独立评审 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ 输入：重写稿 + 改稿单 + 方法论标准库                            ┃
┃ 关键：评审独立于重写（不在同一手里，防自我批准——借写评分离）     ┃
┃ 做什么：                                                       ┃
┃   · 命门检查：开篇钩子/情绪起伏/主线 有无硬伤                    ┃
┃   · 六维回量：对照方法论标准，比原版强了吗                      ┃
┃   · 改稿单核对：该改的改了？禁止的犯了？                        ┃
┃   · 镜头感/去AI味 过硬门槛了吗                                  ┃
┃   · 复查计划：改完后哪些地方要重查（防改A破坏B）                ┃
┃ 不过 → 带问题回环节4（有自改次数上限，防无限循环）              ┃
┃ 过  → 输出                                                     ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
            ↓
┌──────────────────────────────────────────────────────────────┐
│  输出：精洗后的剧本（纯文本）                                     │
└──────────────────────────────────────────────────────────────┘

【全程大脑】方法论 + 标准库（被各环节调用，不是流程本身）
【全程记忆】诊断库（source of truth）+ 追踪文件（防漂移）
```

### workflow 的三个底层特征（为什么这样设计）
1. **文件驱动**：拆解产物落盘成"诊断库"，后续环节都读它、对照它 → 不会改着改着忘了原诊断（借 oh-story）。
2. **写评分离**：环节4重写、环节5评审，独立两手，不自我批准 → 防"自己改自己判说挺好"（借 how-to-make-script，也符合用户全局规则"writer pass 与 reviewer pass 分开"）。
3. **全程防漂移**：追踪文件锁伏笔/角色状态/时间线 → 多集精洗不前后矛盾（借 oh-story）。

## 第三部分 · 大脑：方法论 + 标准库

> 这不是流程，是被 workflow 各环节**调用**的知识库。原"三机制"框架降级到这里——它本来就是零件分类，不是工作流。
> 内容已从调研 + 拆解开源项目得到，可直接查用（详见独立文件 `短剧标准库.md` 及参考资产）。

| 库 | 内容 | 被哪个环节用 |
|---|---|---|
| 分析维度体系 | 6维质量+权重 / 分层诊断(概念-结构-场景-对白) / 情绪曲线±9 | 环节1、2 |
| 短剧标准库 | 钩子5类×5子模式 / 爽点矩阵5类 / 节奏四段配比 / 反派四层 / 开场6模板 / 留存卡点 | 环节1、2、4 |
| 创作工艺库 | 11手法（信息差锁/物件钩子/打脸三段式/身体反应替代心理…） | 环节4 |
| 镜头感标准 | 情绪标签句 hard-fail / 可拍画面写法 | 环节4、5 |
| 去AI味标准 | 禁用词表 / 7Gate / 去非视听语言（内心独白/上帝视角） | 环节4、5 |
| cliche清单 | 套路点识别（反转维度的负面镜像） | 环节1、4 |
| 海外赛道参考 | S/A级赛道 + 中式叙事陷阱替代表 | 环节0、2、4 |
