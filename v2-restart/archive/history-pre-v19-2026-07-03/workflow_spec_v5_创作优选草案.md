# 海外短剧洗稿器 · workflow spec v5 创作优选草案

> 日期：2026-07-01  
> 状态：草案，不是当前正式执行版  
> 上游：`PRD_v4.md`  
> 参照：`workflow_spec_v2.md`、`workflow_spec_v4_机制重构草案.md`、`失效审计_2026-07-01_V15完整流程为何仍漏问题.md`  
> 目的：把 workflow 的主轴从“证明没错”改成“在不洗飞、不复刻的前提下，主动选出更有戏的新本写法”。  

---

## 0. 动手前根因校验

任何后续修订、跑样本、返修剧本前，先问一句：

```text
我现在要改的是症状，还是机制？
```

如果只是为了补：

- 视听语言；
- 情绪外化；
- 拉扯感；
- 钩子；
- 台词；
- page-to-screen；
- reviewer gate；

先停。

这些都可能是真问题，但都不是最上层根因。

当前确认的机制根因是：

```text
源本的好看机制在层层转译时被压扁：
源本里的活戏 -> 中间产物里的功能词 -> 正文里的任务完成 -> reviewer 的 pass。

workflow 很会证明“没洗飞、没同构、流程跑完”，
但不够会保证“这一版更有戏、更短剧、更想追”。
```

因此 v5 不继续补单项规则。

v5 只改主轴：

```text
源本强戏机制
  -> 新壳场面候选
  -> 场面优选
  -> 可写剧本简报
  -> page-to-screen 正文
  -> 版本优选
```

防错仍然保留，但只做护栏，不做主引擎。

---

## 1. 产品锚点

本项目仍然是洗稿器，不是从零原创短剧生成器。

目标仍然是：

```text
把一个已被市场验证或值得参考的短剧源本，
洗成一个新壳下仍然好看、能追、能付费的新剧本。
```

v5 不能漂移成“写一个好剧本”。

它必须同时满足：

1. 源本让人付费追下去的东西没有丢；
2. 新本不是改名复刻；
3. 新本自己作为短剧有戏；
4. 同一关键节点下，优先选择更想追、更短剧、更适合 AI 成片的写法。

---

## 2. v4 已经解决了什么

v4 不是错的。

v4 解决了几件重要问题：

- 不再把 writer 塞进 source ref / manifest / 证明链；
- 不再把文件、系统、key、董事会流程当高潮本体；
- 让故事状态和压力链进入正文前；
- 让 V15 避免退回 V12 的机制审判机器。

但 v4 没完全解决：

```text
同一个源本强点，新壳下到底哪种场面最有戏？
```

v4 更像：

```text
压力链写作 + 防错审稿
```

v5 要补成：

```text
场面优选 + 压力链写作 + 防错审稿 + 版本优选
```

---

## 3. 四种语言必须保真

这个 workflow 里有四种语言。

过去的问题不是没有产物，而是四种语言互相转译时没人负责保真。

| 层 | 正常应该表达什么 | 常见坏法 |
|---|---|---|
| 分析语言 | 源本为什么让人痛、爽、急、气、想追 | 压成“本集有反咬/揭露/跪求/钩子” |
| 规划语言 | 新壳下哪种场面能产生同等强度 | 压成“这一集要完成某剧情任务” |
| 剧本语言 | 观众能看见、听见、感到压力的动作和对白 | 写成任务说明、证据链、冷对白 |
| 验收语言 | 哪版更有戏，哪层坏了，回哪里改 | 只写 pass / revise / complete |

v5 的核心检查不是“表填没填”，而是：

```text
源本那个最让人想看的东西，穿过四层后还活不活？
```

---

## 4. 总 workflow

```text
0. Run Contract / 最小需求确认
1. Source Strong-Scene Read / 源本强戏阅读
2. 二次需求确认
3. Adaptation Boundary / 洗稿边界
4. Source Drama Engine Cards / 源本强戏机制卡
5. Story Operating State / 故事运行状态
6. Episode Pursuit Map / 分集追剧曲线
7. Scene Option Pitch / 高压场面候选
8. Scene Choice Gate / 场面优选门
9. Playable Writer Brief / 可写剧本简报
10. Screenplay Draft / 剧本正文
11. Writer Self-Trim / 作者减法自检
12. Rewrite Routing / 返修路由
13. Reviewer A / 干净正文效果审
14. Reviewer B / 源本迁移审计
15. Version Preference Review / 版本优选
16. Story Memory + Manifest / 状态与执行记录
```

v5 的新增主轴是第 7、8、15 步。

它们不是为了多写表。

它们只解决一个问题：

```text
不要让“功能完成”冒充“场面有戏”。
```

---

## 5. 0. Run Contract / 最小需求确认

### 目标

先确认本轮交付形态，避免写完后才发现语言、用途或审稿形态错。

### 必须确认

```text
任务类型：洗稿 / 改编 / 重写 / 诊断 / 审稿
源本：
新壳：
目标市场：
集数是否变化：
本轮输出范围：
改写力度 / 变量边界：
已知禁区：
输出语言：
审稿语言：
动作/场景描述语言：
正文用途：用户/导演审稿 / AI 视频生成用剧本 / 字幕台词版 / 双语版
```

### 默认值

如果用户没有特别指定：

```text
英文对白 + 中文场景/动作描述 + 中文内部分析和审稿材料。
```

但默认值必须写进本轮 Run Contract，不能藏在模型假设里。

---

## 6. 1. Source Strong-Scene Read / 源本强戏阅读

### 目标

不是复述剧情，也不是只写分集功能。

这一环必须回答：

```text
源本为什么让人想继续看？
哪些节点是靠“活戏”成立的？
```

### 输出

`Source Strong-Scene Read`

每个高价值节点必须写：

```text
节点发生了什么：
观众在等什么：
这一节点最强的体验：痛 / 爽 / 急 / 气 / 期待 / 拉扯
谁压谁：
谁误判谁：
谁被迫选择：
谁失去什么：
谁在场看见：
爽点前如何压低：
卡点停在哪一秒：
源本具体外形哪些不能复用：
如果洗淡，最容易淡在哪里：
```

### 不合格

以下不算合格：

```text
本集有反转；
本集有钩子；
本集是火葬场；
本集白月光翻车；
本集男主跪求。
```

这些是功能词，不是强戏机制。

---

## 7. 2. 二次需求确认

### 目标

读完源本后，主控先判断用户需求是否会损害源本强戏，再让用户确认。

### 必须给用户拍板的情况

- 新壳无法承载源本最强节点；
- 集数变化会改变关键卡点阶段；
- 用户要求保留的东西和“不能复刻”冲突；
- 用户要求删掉的东西正好是源本付费冲动来源；
- 输出用途需要改变正文写法。

### 原则

不要把专业判断甩给用户。

主控先给建议，再让用户确认。

---

## 8. 3. Adaptation Boundary / 洗稿边界

### 目标

只做最薄洗稿适配，不复活万能变量表。

### 输出

```text
必须保留的体验：
必须替换的源本外形：
可以自由改的外壳变量：
必须删除 / 修复的烂点：
```

### 通过标准

- 不把“强继承”理解成照抄；
- 不把“去同构”理解成丢强度；
- 不把“深度改写”理解成从零原创。

---

## 9. 4. Source Drama Engine Cards / 源本强戏机制卡

### 目标

把源本高价值节点压成可以传给规划层的“强戏机制”，而不是功能摘要。

这一步是 v5 的关键。

### 只做高压节点

不每集都厚写。

适用：

- 前 1-3 集；
- 每集强卡点；
- 强羞辱、强危机、强打脸、强误会、身份揭露、关系质变；
- 上一版失败节点；
- 用户特别要求保留的节点。

### 输出结构

```text
Source Drama Engine Card

源本节点：
源本功能：
强戏机制：
  - 谁被压：
  - 谁施压：
  - 谁旁观：
  - 谁误判：
  - 谁被迫选择：
  - 选择代价：
  - 身体/关系/社会性损失：
  - 爽点前压低方式：
  - 爆点：
  - 卡点最后一秒：
不可复用外形：
新壳迁移时最容易丢掉什么：
```

### 例子口径

差：

```text
E8 功能：白月光反咬男主同罪。
```

好：

```text
E8 强戏机制：白月光在公开场合失去保护伞，为保命把男主也拖成凶手；
男主不是单纯知道真相，而是在众人面前从审判者变成共犯；
观众爽点是“他终于不能把自己包装成被骗者”。
```

---

## 10. 5. Story Operating State / 故事运行状态

### 目标

告诉后续写作：这个故事现在活到哪儿了。

不是 source ref，不是 reviewer 表。

### 必须回答

```text
主角此刻想要什么：
主角此刻怕什么：
主角身体 / 情绪 / 社会位置：
对手此刻想要什么：
对手怕什么被拆穿：
核心关系卡在哪里：
谁靠近 / 谁退开：
谁掌握主动权：
观众正在等什么：
本批不能提前兑现什么：
```

### 通过标准

后续每个高压场面候选，都必须能看出它来自这个状态，而不是孤立桥段。

---

## 11. 6. Episode Pursuit Map / 分集追剧曲线

### 目标

不是分集梗概。

而是回答：

```text
这一集怎么让人继续看？
```

### 字段

```text
Episode：
Opening Hook：
Core Friction：
Max Spike：
Change：
End Button：
Do Not Consume：
Source Drama Engine Ref：
本集追问：
本集最不能洗淡的强度：
```

### 通过标准

- 每集至少有一个关系、局势、信息或情绪变化；
- 关键集必须能回到 Source Drama Engine Card；
- `Max Spike` 不能只写“揭露某信息”，必须写观众爽/痛/急/气在哪；
- `End Button` 必须是最后画面、动作或声音，不是概念问题。

---

## 12. 7. Scene Option Pitch / 高压场面候选

### 目标

高压节点不能直接进 writer。

必须先设计 2-3 个新壳场面候选，再选最有戏的。

这是 v5 和 v4 的最大区别。

v4 问：

```text
这个新场面能不能承载？
```

v5 追问：

```text
同一个源本强点，新壳下哪种写法最有戏？
```

### 每个候选必须写

```text
候选名：
场面一句话：
谁压谁：
谁旁观：
谁被迫选择：
身体 / 关系 / 社会性代价：
爽点前怎么压低：
爆点是什么：
卡点停在哪个画面 / 动作 / 声音：
同构风险：
短剧降级风险：
AI 成片可视化风险：
```

### 不合格候选

- 只是换证据；
- 只是换地点；
- 只是把源本桥段改名；
- 只靠文件、会议、屏幕或报告完成高潮；
- 只能证明剧情通顺，看不出哪一下让人想追。

---

## 13. 8. Scene Choice Gate / 场面优选门

### 目标

从候选里选一个最有戏的，不是选一个最安全、最方便、最容易解释的。

### 选择标准

按顺序判断：

1. 是否保住源本强戏机制；
2. 是否避免源本外形复刻；
3. 是否比其他候选更短剧、更想追；
4. 是否有清楚最后画面 / 动作 / 声音；
5. 是否适合 AI 成片；
6. 是否不会把剧情推进变成审计材料。

### 输出

```text
选中候选：
为什么选它：
淘汰候选：
淘汰原因：
仍有风险：
如果正文写弱，回滚到哪里：
```

### 原则

这里可以有创作判断。

不要把创作判断伪装成“客观评分”。

---

## 14. 9. Playable Writer Brief / 可写剧本简报

### 目标

writer 真正拿到的不是审计底稿，而是可写戏的简报。

### writer 可见

```text
Run Contract 交付头；
当前故事运行状态；
本集追剧曲线；
选中的高压场面；
人物此刻的欲望、误判、阻碍、选择、代价；
必须停住的最后画面 / 动作 / 声音；
一句底线：正文只写观众能看见和听见、AI 能生成的东西。
```

### writer 不可见

```text
完整 Source Bible；
完整 source ref；
manifest；
reviewer 报告；
版本对比；
外部研究材料；
历史失败清单；
证明链；
所有候选的审计打分。
```

### 重要澄清

“主创负责分析和写作”不等于“正文起草上下文必须塞满分析底稿”。

更准确是：

```text
主创负责创作判断；
正文起草阶段只吃可写剧本简报；
写完后再由主创 / 主控拿分析底稿回查。
```

---

## 15. 10. Screenplay Draft / 剧本正文

### 目标

交付真正剧本正文，不是说明书。

核心合同：

```text
正文页上写出来的东西，应该基本等于最终 AI 成片里能看见和听见的东西。
```

### 正文容器

使用 Hollywood screenplay 的基本容器：

```text
INT. / EXT. LOCATION - DAY / NIGHT

动作行：只写画面和声音里正在发生的事。

CHARACTER
英文对白，除非 Run Contract 指定其他语言。

SFX:
ON SCREEN:
```

### 不允许

- 内部字段进正文；
- 解释剧情功能；
- 用“她崩溃了 / 他意识到 / 气氛凝重”替代动作；
- 连续低信息对白；
- 文件、屏幕、报告替人物完成高潮；
- 结尾把答案说完。

---

## 16. 11. Writer Self-Trim / 作者减法自检

### 目标

作者不能给自己质量 pass。

作者只做减法自检：

```text
ready for reviewer / needs rewrite / blocked
```

### 只查五件事

1. 有没有写成说明书；
2. 有没有高压场靠文件/屏幕/报告完成；
3. 有没有裸对白；
4. 有没有提前说完卡点；
5. 有没有明显违背 Playable Writer Brief。

作者不能用自检证明自己“写得好”。

---

## 17. 12. Rewrite Routing / 返修路由

### 目标

先判坏在哪层，再改。

### 路由

| 问题 | 回哪里 |
|---|---|
| 源本强戏机制没读出 | Source Strong-Scene Read |
| 强戏机制被压成功能词 | Source Drama Engine Cards |
| 新壳候选都没戏 | Scene Option Pitch |
| 选错候选 | Scene Choice Gate |
| writer brief 变任务书 | Playable Writer Brief |
| 正文不像剧本 | Screenplay Draft |
| 只是台词水 | Dialogue Polish / 局部修 |
| 版本更完整但更不好看 | Version Preference Review |

### 禁止

- 场面发动机错，只加身体反应；
- 台词结构错，只删几个字；
- 候选选错，还在正文里补补丁；
- 发现版本不如旧版，却只写“可 polish”。

---

## 18. 13. Reviewer A / 干净正文效果审

### 输入

只读：

```text
Run Contract
正文
```

不读：

```text
Source Bible
Writer Brief
作者自检
manifest
旧版本对比
```

### 目标

判断正文作为短剧是否有戏。

### 输出

```text
creative_pass / creative_revise / creative_block
failure_layer:
callback_target:
reason:
```

### 必须判断

- 哪几集最想追；
- 哪几场最有短剧冲击；
- 哪些地方像任务完成；
- 哪些台词水；
- 哪些卡点说完了；
- 哪些地方虽然逻辑通，但不好看。

---

## 19. 14. Reviewer B / 源本迁移审计

### 输入

读取：

```text
Source Strong-Scene Read
Source Drama Engine Cards
Adaptation Boundary
Story Operating State
Episode Pursuit Map
Scene Choice Gate
Playable Writer Brief
正文
必要源本片段
```

### 目标

判断：

- 有没有洗飞；
- 有没有改名复刻；
- 有没有短剧降级；
- 有没有源本强戏机制被压扁；
- 有没有场面候选选错；
- 有没有提前消费；
- 有没有继承烂点。

### 输出

```text
source_audit_pass / source_audit_revise / source_audit_block
failure_layer:
callback_target:
reason:
```

### 关键要求

Reviewer B 不能只说“功能都在”。

必须抽查至少 3 个高压节点：

```text
源本强戏机制
-> 场面候选
-> 选中理由
-> writer brief
-> 正文最后画面 / 动作 / 声音
```

任何一层只剩功能词，都不能直接 pass。

---

## 20. 15. Version Preference Review / 版本优选

### 目标

防止“流程更完整，但文本更弱”。

如果本轮有旧版可比，必须做版本优选。

### 输入

隐藏版本号，至少对比：

```text
当前版
上一版
基准版
```

不需要全本逐字盲读，可以只抽关键高压节点。

### 判断问题

```text
哪版更想看下一集？
哪版更短剧？
哪版人物互逼更强？
哪版场面更干净、更狠？
哪版更适合 AI 成片？
哪版虽然逻辑完整但没戏？
```

### 结论

```text
current_preferred / previous_preferred_locally / baseline_better / inconclusive
```

如果旧版局部更强，不能只写“polish 参考”。

必须写：

```text
吸收旧版哪一点；
不吸收哪一点；
为什么。
```

## 21. 16. Story Memory + Manifest

### Story Memory

只记录本批写完后的故事状态：

```text
关系状态：
人物状态：
信息差：
伏笔：
未兑现承诺：
禁止提前消费：
下一批入口：
```

### Manifest

manifest 只证明执行。

它不能证明：

- 剧本好；
- 能付费；
- 版本最优；
- 可以送导演。

manifest 状态必须拆开：

```text
execution_status:
```

## 22. 外部成熟产物合同归属

这不是给 writer 的表。

这是主控用来防止“只借名字没借合同”的登记。

| 成熟产物 | 谁消费 | 谁验收 | 失败回哪里 |
|---|---|---|---|
| `scene_draft` | Scene Option / Writer Brief | Reviewer A / B | Scene Option Pitch / Playable Writer Brief |
| `screenplay_draft` | Screenplay Draft | Reviewer A / Delivery smoke test | Screenplay Draft |
| `rewrite_report` | Rewrite Routing | 主控 | 对应 failure layer |
| `dialogue_polish` | 局部台词修订 | Reviewer A | Dialogue Polish / Screenplay Draft |
| `quality_gate_report` | Reviewer / 主控 | 主控 | 对应 failure layer |
| `story_memory_checkpoint` | 下一批写作前 | 主控 | Story Operating State |

如果某轮没有按合同验收，就不要声称使用了该成熟产物。

## 23. v5 不做什么

v5 不做：

- 不扩设计提纲；
- 不把短剧标准库变硬规则；
- 不给每场戏加固定表格；
- 不让 writer 看完整审计底稿；
- 不把 reviewer pass 当最终质量证明；
- 不把版本优选伪装成客观打分；
- 不为了一个样本问题改成永久规则。

---

## 24. 何时能说 v5 有效

不能因为文档看起来完整就说有效。

至少要用真实样本验证：

1. 同一源本，同一要求，按 v5 跑一版；
2. 对比 V15、V14 和基准版；
3. 重点看 E5 / E8 / E10 这类过去暴露问题的节点；
4. 如果 v5 只让流程更厚，但文本没更有戏，则 v5 不成立；
5. 如果 v5 能让高压节点更稳定地保住源本强戏机制，同时不回到证明链写作，才可考虑正式化。

## 25. 一句话总结

v5 的目标不是继续防错。

v5 的目标是：

```text
在洗稿边界内，主动选出最有戏的新壳场面，
再把它写成能直接进入 AI 出海短剧成片链路的剧本正文。
```
