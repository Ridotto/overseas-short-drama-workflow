## 1. 这份补充解决什么

这份补充不重新定义产品。

产品仍然是：

```text
把一个已被市场验证或值得参考的短剧源本，
洗成一个新壳下仍然好看、能追、能付费的新剧本。
```

V20 证明：v10 已经能把源本有效性、多条内容链路、同构扣除和 reviewer 机制带回来，但在“分析结果怎么交给正文写作”这一层仍然断。

具体表现是：

```text
分析层知道：哪里不能复刻、哪里不能提前消费、哪里要保源本强度。
正文层却仍会写成：认证台、屏幕结果、证据上台、反派否认、女配反咬。
```

所以问题不是“还缺更多分析字段”，而是：

```text
源本分析和审计材料没有先变成新本创作开发包、阶段骨架，
再压成 writer 能直接写戏的故事包。
```

---

## 2. 不从商业平台倒推流程

外部商业平台的导出结果只能作为“成品形态观察”，不能作为本项目 workflow 的来源。

不能做：

- 不照抄它的字段；
- 不从它的最终产物倒推黑箱中间链路；
- 不把它的粗纲、集纲、人物设定结构当成我们必须补齐的模板；
- 不因为它有某些栏目，就把我们的 PRD 写厚。

可以参考的只有一个常识：

```text
前置产物最终必须服务正文，不能和正文脱节。
```

我们的中间产物可以比平台更多、更少，或者更强。判断标准不是“像不像平台”，而是：

```text
它是否帮助把源本的追看机制、情绪强度、钩子卡点、关系拉扯和新壳承载，
变成一集一集能写出来的短剧正文。
```

---

## 3. 新增产品口径：新本创作开发包 + 写作前故事包

PRD v4 已有“单集写作包”。本补充把它前面的层级补清楚：

```text
源本有效性包
-> 洗稿边界包
-> 新本创作开发包
-> 首批 / 全剧阶段骨架
-> 分集追剧提纲
-> 写作前故事包
-> 剧本正文
```

新本创作开发包回答：

- 这版新本到底是什么故事；
- 给谁看；
- 属于什么题材；
- 哪些设定会真正推动剧情；
- 主角追什么；
- 核心真相是什么，角色误判什么，观众在等什么，哪一层何时兑现；
- 关系怎么拉扯；
- 权力结构和信息差怎么运转。

阶段骨架回答：

- 首批或全剧分几个阶段推进；
- 每个阶段压什么、兑现什么、留下什么；
- 每个阶段承接哪个主追问 / 阶段追问，哪些答案只暗示不揭开；
- 哪些真相、关系变化、火葬场和爽点不能提前消费。

写作前故事包才是 writer 真正输入：

```text
单集写作包不是审计表，也不是证明链。
它是写作前故事包：把源本有效性和新壳方案翻译成 writer 能直接写戏的输入。
```

它要回答：

- 这一集观众等什么；
- 这一集兑现什么；
- 这一集最后卡什么；
- 谁在逼谁；
- 谁想保住什么；
- 谁必须做选择；
- 哪个信息只能暗示，不能说穿；
- 哪个关系、权力或情绪状态必须发生变化；
- 最后停在哪个能看见、能听见、能生成的画面 / 动作 / 声音上。

它不应该塞给 writer：

- 完整 source ref 大表；
- manifest；
- reviewer 报告；
- 版本对比；
- 历史失败清单；
- 证明自己遵守 workflow 的证据；
- 外部研究材料。

## 4. 关键分层

本项目需要四条线，但四条线不能混在同一个 writer prompt 里。

| 线 | 服务对象 | 可以厚吗 | 作用 |
|---|---|---:|---|
| 分析线 | 主创 / 主控 / reviewer | 可以 | 读出源本为什么能追、能付费 |
| 创作线 | 主创；writer 只读写作前故事包 | 开发包可适度厚，writer brief 必须薄 | 让新本先成立，再让正文能直接写成戏 |
| 审计线 | reviewer / 主控 | 可以 | 查洗飞、同构、降级、状态漂移 |
| 治理线 | 主控 | 可以 | 记录 manifest、版本、失败回滚点 |

关键原则：

```text
主创可以同时负责分析和写作，但正文起草阶段不能吃全量审计底稿。
```

原因不是要把分析和写作拆成两个互不相关的 agent，而是避免正文写作者被证明链、禁区表和版本历史污染，最后写成“为了过检查而写”的文本。

## 5. 当前关键交付物调整

PRD v4 的交付物不推翻，只改清楚其中几个产物的关系。

| 交付物 | 作用 | 关键要求 |
|---|---|---|
| 源本有效性包 | 读出源本为什么好看 | 可厚，可保留源本节点和分析证据 |
| 洗稿边界包 | 防洗飞、防复刻 | 薄，压成必须保、必须换、可自由、必须删 |
| 新本创作开发包 | 让新本先成为成立的故事 | 只写会影响创作的受众、题材、核心设定、核心亮点、核心真相、追问兑现、角色行动方式、关系/权力/信息差 |
| 首批 / 全剧阶段骨架 | 控制跨集推进 | 防单集孤岛化，防提前消费后续真相、关系击穿、火葬场和爽点 |
| 分集追剧提纲 | 控制每集为什么继续看 | 不是剧情流水账，必须有追问和变化 |
| 高压场面候选 | 对强节点做二创选择 | 不是只找一个最安全载体，而是选最有记忆点的可见后果载体 |
| 写作前故事包 | writer 真正输入 | 只给能写戏的信息，去掉审计噪音 |
| 剧本正文 | 最终交付核心 | page-to-screen：写出来基本就是成片可见可听 |
| 作者自检 | 主创自查 | 先判断坏在哪层，不直接润色 |
| 独立 reviewer | 干净审稿 | 第一轮只读正文，第二轮再查源本迁移；结论必须分致命阻断、应改风险、可选优化 |

## 6. 为什么不是再加规则

这次不是新增一堆“禁止项”。

过去的问题是：

```text
发现一个症状 -> 补一个检查项 -> writer 为了过检写得更像程序。
```

这次改的是中间产物职责：

```text
分析材料负责解释源本。
新本创作开发包和阶段骨架负责让新本先成为能运转的故事。
写作前故事包负责把单集压成可写场面。
reviewer 负责回查是否写飞。
manifest 只负责证明流程执行。
```

判断是否有效，不看新增了多少字段，只看三件事：

1. 新本创作开发包和阶段骨架能不能让新本先成为一个能运转的故事；
2. writer 能不能只看写作前故事包写出像剧本的正文；
3. reviewer 能不能明确指出失败应该回到开发包、阶段骨架、故事包、场面候选、正文、台词，还是源本分析，并且只把伤到产品红线的 finding 判为阻断。

## 7. 成功标准补充

在 PRD v4 原成功标准基础上，增加一条：

```text
分析结果必须先变成新本创作开发包和阶段骨架，再压成可写的故事包；
正文必须像剧本，不像审计报告、流程说明或程序执行记录。
```

如果某版 workflow manifest complete，但正文仍然：

- 主要靠认证 / 文件 / 屏幕播报推进；
- 人物只是对系统结果做反应；
- 核心真相、角色误判或已播事实互相冲突；
- 关键追问被台词一次说完；
- 多条链路在分析里存在，正文里消失；
- 新本没有清楚骨架，单集各写各的；
- reviewer 找到的问题无法回到明确中间产物，或把风险 / 优化项当成阻断；

那它不能算产品能力通过。
