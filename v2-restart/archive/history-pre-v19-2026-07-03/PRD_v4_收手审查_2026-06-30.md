# PRD v4 草案收手审查

日期：2026-06-30  
审查对象：`PRD_v4_草案_吸收外部workflow版.md`

---

## 1. 结论

**内容可以收手。**

现在不该继续找外部资料，不该继续补导演反馈规则，也不该继续扩 PRD。

v4 草案已经把当前最关键的断点补上了：

```text
源本分析
  -> 源本事件载体拆解
  -> 新事件载体迁移
  -> 正文前策略审核
  -> 单集写作包
  -> 正文初稿
  -> 短剧化返修
  -> 分层 reviewer
```

这条链已经能解释并处理当前三类核心失败：

1. **改名复刻**：源本事件链整串保留，只换名词。
2. **短剧降级**：为了避同构，把强场面洗成文件、会议、冷对白。
3. **正文长剧化**：台词水、人物冷、站桩聊天、卡点说完。

所以产品方向不需要再大改。

---

## 2. 有没有变成规则怪谈

**有风险，但还没失控。**

v4 草案确实有很多表：源本事件载体拆解表、新事件载体迁移表、关键场面草案、单集写作包、状态追踪、reviewer 分层审稿。

但它们不是同一层的“每集必填规则”，而是分布在不同工序里：

| 层 | 作用 | 是否全量执行 |
|---|---|---|
| 源本事件载体拆解 | 拆关键源本节点 | 只拆关键节点 |
| 新事件载体迁移 | 防改名复刻 | 只做高风险/关键节点 |
| 正文前策略审核 | 写前拦方向错误 | 只审高风险方案 |
| 分集功能表 | 控制每集留存和推进 | 每集薄写 |
| 关键场面草案 | 防短剧场面降级 | 只写高压节点 |
| 单集写作包 | 控制当前集输入 | 每集薄写 |
| 短剧化返修 | 删水词、修场面、校卡点 | 正文后执行 |

因此它不是“所有规则每集检查一遍”，而是把不同问题放到不同环节解决。

但正式版要注意：**不要把 v4 继续加厚。**

---

## 3. workflow 有没有明显漏环

目前没有明显漏掉的主环节。

从产品流程看，v4 已覆盖：

- 需求对齐；
- 源本有效性分析；
- 源本事件载体拆解；
- 变量迁移；
- 新事件载体迁移；
- 正文前审核；
- 用户确认洗稿方案；
- 阶段骨架；
- 分集功能表；
- 关键场面草案；
- 状态追踪；
- 单集写作包；
- 正文写作；
- 正文后短剧化返修；
- 作者自检；
- 独立 reviewer；
- 用户 / 导演反馈回流。

这条链已经比 v3 更完整。

唯一需要保留的轻微疑问是：

**“不同题材/男频女频/赛道套路”现在没有作为独立流程进入 v4。**

我判断这不是阻塞问题。原因是：

- 本项目是洗稿，不是从零原创；
- 题材/赛道信息应该在“需求对齐”和“新壳承载判断”里使用；
- 不应该把家庭、商战、黑帮、穿越、种田等套路库塞进主 PRD；
- 后续如果需要，可以作为参考资产或题材库，不进入主 workflow。

所以不因为这个继续扩 PRD。

---

## 4. v4 最大优点

v4 第一次把“写剧本到底怎么写”补清楚了。

之前的问题是：

```text
分析源本
-> 得到一堆判断
-> 直接写正文
```

现在改成：

```text
分析源本
-> 拆源本事件载体
-> 设计新事件载体
-> 写前审核
-> 每集生成写作包
-> 写正文
-> 台词/场面/卡点返修
```

这个变化是实质性的，不是换概念。

---

## 5. v4 最大风险

最大风险不是方向错，而是**文档形态太像 PRD + workflow spec + 执行手册混在一起**。

这会带来两个问题：

1. 后续 agent 可能把所有字段都当硬规则，执行变重。
2. 用户或 reviewer 看 PRD 时，会被内部执行细节淹没。

所以正式落地时，建议不要再重写方向，而是做一次文档分层：

```text
PRD_v4.md
  只保留产品定义、主流程、关键交付物、硬失败、成功标准。

workflow_spec_v1.md
  放详细字段：源本事件载体拆解表、新事件载体迁移表、单集写作包、短剧化返修、reviewer 分层审稿。
```

这不是继续扩文档，而是防止主 PRD 太厚。

---

## 6. 不建议再做什么

不建议继续做：

- 继续搜更多开源项目；
- 继续补“短剧味”抽象规则；
- 继续把导演反馈逐条变成新字段；
- 继续讨论男频、女频、家庭、商战、黑帮、穿越等类型套路是否进入主流程；
- 继续改设计提纲。

这些都会把项目重新拖回规则怪谈。

---

## 7. 建议下一步

下一步只做一个收尾动作：

**把 v4 草案整理成正式版本。**

推荐方式：

1. `PRD_v4.md`：正式主文档，控制在更短篇幅。
2. `workflow_spec_v1.md`：执行工序细节，给 agent 跑流程用。
3. 更新 `当前工作入口.md`：把主入口从 `PRD_v3.md` 改为 `PRD_v4.md`。
4. 更新项目 `AGENTS.md`：说明当前以 `PRD_v4.md + workflow_spec_v1.md` 为准。

这一步之后，就可以停止文档扩张，进入用 v4 workflow 重跑样本。

---

## 8. 最终判断

**v4 草案内容层面可以定。**

不要再找更多资料，也不要继续补规则。

剩下的不是产品方向问题，而是文档落位问题：把“主 PRD”和“执行工序”分开，让后续 agent 不乱读、不乱跑。
