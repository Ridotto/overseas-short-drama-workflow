# 海外短剧洗稿器 · workflow spec v1

> 日期：2026-06-30
> 对应主文档：`PRD_v4.md`
> 用途：给主控 / 主创 / reviewer 执行流程用
> 注意：这是执行说明，不是产品定义；产品定义以 `PRD_v4.md` 为准

---

## 0. 执行总原则

不要从源本分析直接进入正文。

必须先把源本有效性转成：

1. 源本事件载体拆解；
2. 新事件载体迁移；
3. 正文前策略审核；
4. 新本阶段骨架、分集功能表、关键场面草案；
5. 状态追踪；
6. 单集写作包；
7. 正文初稿；
8. 短剧化返修；
9. 作者自检和独立 reviewer。

所有表格都是为了服务写剧本，不是为了展示分析能力。

不要每集每场都填厚表。  
高风险节点重点写，普通过渡集薄写。

---

## 1. 需求确认摘要

### 目标

确认这次到底怎么洗，不让模型默认深度重构，也不让它默认改名复刻。

### 必须问清

| 问题 | 作用 |
|---|---|
| 新壳是什么 | 决定身份、职业、场景、证据形态怎么重建 |
| 集数是否变化 | 决定留存节点是否重排 |
| 改写力度 | 决定哪些贴近源本、哪些必须大换 |
| 目标市场 | 决定常识、本地化、敏感设定 |
| 用户特别想保留什么 | 防止误删用户在意的爽点 |
| 用户特别不想要什么 | 防止继承雷点 |

### 输出格式

```text
需求确认摘要

源本：
新壳：
集数：
改写力度：
目标市场：
必须保留：
必须避免：
本轮首批范围：
仍需用户拍板的问题：
```

### 通过标准

- 用户能确认方向；
- 主控能判断这次更接近贴近改写、中度改写，还是深度重构；
- 不把“改写力度”当成单一档位，后续按变量分别判断。

---

## 2. 源本有效性摘要

### 目标

回答源本为什么让观众继续看。

### 分析五层

| 层 | 看什么 |
|---|---|
| 留存骨架 | 开头、每集钩子、关键卡点、付费或留存节点 |
| 情绪爽点 | 压抑、释放、爽点兑现、高潮递进 |
| 主线推进 | 核心期待、信息差、秘密揭露、每集/每场新进展 |
| 人物压力 | 主角驱动力、关系功能、反派层级、压迫升级 |
| 干净度 | 水分、烂点、AI 味、非视听语言、不可拍描述 |

### 输出格式

```text
源本有效性摘要

一句话有效性：
核心期待：
主要留存链：
主要情绪链：
主要关系/权力链：
主要信息差：
前段为什么抓人：
首批必须迁移的体验：
源本水分/烂点：
```

### 通过标准

- 不写长篇剧情复述；
- 只保留后续改写必须用的判断；
- 能说清“这本为什么能让人继续看”。

## 3. 源本事件载体拆解

### 来源

吸收 Toonflow 的事件提取，以及 oh-story 的情节节点拆解。

### 目标

把“源本为什么有效”拆成可迁移、可禁止复刻、可改写的节点。

### 适用范围

默认拆：

- 前 1-3 集；
- 每集结尾钩子；
- 付费点或强留存点；
- 大误会、大羞辱、大危机、大反转、大打脸；
- 身份揭露、火葬场、关系质变；
- 所有同构风险高的桥段。

不要求全剧每场都拆，避免流程拖死。

### 字段

| 字段 | 要回答什么 |
|---|---|
| 源本节点编号 | 第几集 / 哪一段 / 大致位置 |
| 源本事件载体 | 具体发生了什么事件，不是功能词 |
| 源本场面承载 | 观众直接看到了什么动作、画面、身体反应、空间压力、道具或证据 |
| 观众体验功能 | 让观众爽、急、痛、气、期待、误会、想追哪一种 |
| 信息差状态 | 谁知道、谁不知道、观众知道多少 |
| 关系 / 权力变化 | 这一段结束后谁压过谁，谁靠近谁，谁误会谁，谁失去主动权 |
| 情绪走势 | 从哪里压低，到哪里爆发，或在哪里悬住 |
| 结尾钩子 / 卡点 | 如果是集尾，停在什么问题或动作上 |
| 源本高辨识外形 | 哪些事件、道具、证据、场景、台词太像源本，后续不能复用 |

### 示例

```text
源本节点：第二集开头车祸
源本事件载体：女主流产/危机与男主冷酷选择绑定在同一高压事件里。
源本场面承载：车祸、血、身体伤害、救不救的即时选择。
观众体验功能：危机、心疼、男主误判/无情、追悔火葬场伏笔。
信息差状态：观众逐渐知道女主被伤害，男主仍误判。
关系/权力变化：女主被推到更弱位置，男主债务加深。
情绪走势：从被抛弃压到身体危机。
源本高辨识外形：车祸、医院抢救、同样的流产证据链。
```

### 通过标准

- 不能只写“危机”“反转”“爽点”这种抽象词；
- 必须写清源本靠什么具体事件和场面成立；
- 必须标出不能照搬的外形。

## 4. 变量迁移策略

### 目标

决定每类内容如何处理。

### 四类策略

| 策略 | 含义 |
|---|---|
| 强继承 | 追看功能、情绪走势、关键期待不能无故丢 |
| 等效迁移 | 原形式不适合新壳，但功能必须保留 |
| 自由改写 | 外壳、身份、职业、地点、对白、桥段包装可换 |
| 必须删除或修复 | 水分、烂点、假动机、虚假悬念、不适配设定 |

### 执行口径

强继承不是继承事件外形，而是继承观众体验和结构功能。

### 输出格式

```text
变量迁移策略

强继承：
等效迁移：
自由改写：
必须删除或修复：
需要用户拍板：
```

## 5. 新事件载体迁移

### 来源

吸收 Toonflow 的 adaptation strategy。

### 目标

把“不能照搬源本外形”真正落到新事件上，防止新本只是换名。

这是 v4 最关键的新中间产物。

### 字段

| 字段 | 要回答什么 |
|---|---|
| 对应源本节点 | 回指源本事件载体拆解 |
| 必须保留的体验 | 观众必须感受到什么 |
| 源本不能复用的外形 | 哪些事件、道具、证据、场景、台词不能再用 |
| 新事件载体 | 新壳下具体发生什么事 |
| 新压力来源 | 谁在压迫，压力从哪里来 |
| 新证据形态 | 用什么承载信息：人证、物证、视频、直播、账目、监控、身体痕迹等 |
| 新行动链 | 人物不是说了什么，而是做了什么 |
| 同构风险 | 高 / 中 / 低，为什么 |
| 场面强度风险 | 新事件会不会变成文件、会议、冷对白 |
| 结论 | 可进入骨架 / 需重建 / 放弃该方案 |

### 示例

```text
源本节点：车祸流产危机
必须保留体验：女主身体受伤 + 男主冷酷误判 + 观众心疼/愤怒 + 后续追悔伏笔。
不能复用：车祸、医院抢救、同样的二选一事故。
新事件载体：必须找一个豪门商战壳里同样即时、可见、有身体代价的危机场面。
场面强度风险：如果只改成“安全档案室交钥匙”，就是短剧场面降级。
结论：不能直接进入正文，需重建事件载体。
```

### 通过标准

- 如果只是把源本名词替换成新壳名词，不通过；
- 如果新事件只能靠文件和对白解释，不通过；
- 如果新事件保不住源本体验，也不通过；
- 必须能说明“为什么这不是改名复刻”。

## 6. 正文前策略审核

### 来源

吸收 Toonflow 的 supervision。

### 目标

正文写出来之前，先拦掉方向性错误。

### 审核对象

- 源本事件载体拆解；
- 新事件载体迁移；
- 变量迁移策略。

### 审核问题

| 问题 | 判定 |
|---|---|
| 源本有效体验是否明确 | 不明确则回到源本分析 |
| 源本高辨识外形是否扣掉 | 没扣掉则回到事件载体拆解 |
| 新事件载体是否只是换名 | 是则打回 |
| 新事件是否能承载同等短剧冲击 | 不能则打回 |
| 新壳是否自洽 | 不自洽则重建 |
| 是否为了去同构把源本钩子/情绪/关系功能洗没 | 是则打回 |

### 执行方式

- 普通节点由主控审核；
- 高风险节点可单开临时 reviewer；
- 审核不写正文，只判断能不能进入写作。

### 输出结论

| 结论 | 含义 |
|---|---|
| 通过 | 可以进入洗稿方案和骨架 |
| 局部返修 | 少数节点需要重建 |
| 阻断 | 当前改写方案不能进入正文 |

## 7. 洗稿方案

### 目标

让用户确认方向，而不是让用户看内部大表。

### 用户可见内容

- 新壳怎么落；
- 哪些体验强继承；
- 哪些源本外形不会照搬；
- 哪些强节点会用新事件承载；
- 首批覆盖哪个留存闭环；
- 是否存在需要用户拍板的取舍。

### 通过标准

用户能看懂：

```text
这版为什么不会洗飞？
这版为什么不是换名复刻？
这版首批会覆盖什么留存闭环？
```

---

## 8. 新本阶段骨架

### 来源

吸收 0xsline 的 plan / outline 思路，以及 Toonflow 的 story skeleton。

### 目标

先确定全剧大方向和阶段比例。

### 必答

- 前段怎么抓人；
- 第一个留存小闭环在哪里；
- 第一次大爽点、反转或关系质变在哪里；
- 中段靠什么持续拉扯；
- 后段怎么揭真相、兑现、收尾；
- 如果集数改变，哪些节点按阶段比例重排；
- 哪些源本有效节点必须留在相似阶段，哪些可以重排。

### 注意

阶段骨架不是固定三幕模板。它服从源本和用户需求。

---

## 9. 分集功能表

### 来源

吸收 0xsline 的 episode-directory，以及 Toonflow 的逐集决策。

### 目标

保证每集都有功能，不空转，不让模型一口气写飞。

### 字段

| 字段 | 要回答什么 |
|---|---|
| 集数 | 第几集 |
| 源本功能引用 | 对应源本哪个追看功能，不是照搬事件 |
| 本集新事件 | 新壳下这一集发生的核心事件 |
| 本集变化 | 关系、局势、信息、情绪至少哪一项发生变化 |
| 最大刺激点 | 这一集最强的一下是什么 |
| 结尾钩子 | 观众为什么必须看下一集 |
| 禁止复用 | 本集不能照搬的源本外形 |

### 通过标准

- 不能只写剧情梗概；
- 每集必须有变化；
- 每集结尾必须有追看理由；
- 不要求每集都爆炸，但不能空转。

---

## 10. 关键场面草案

### 来源

吸收 Toonflow 改编策略，以及 ReelForge 的 visual executability / power shift。

### 目标

高压节点写正文前，先确认这一场如何成为短剧场面。

### 适用范围

- 前 1-3 集；
- 首批每集结尾；
- 强危机、强羞辱、强误会、强打脸；
- 身份揭露；
- 大爽点兑现；
- 关系质变；
- 同构风险高的桥段。

### 字段

| 字段 | 要回答什么 |
|---|---|
| 场面目的 | 这一场要制造什么体验 |
| 新事件载体 | 这场具体发生什么 |
| 观众直接看到什么 | 不靠解释，画面上有什么 |
| 人物动作 | 主角、反派、关键配角分别做什么 |
| 身体反应 | 哭、跪、摔、退、僵住、颤抖、失控、被拖走等，按剧情需要使用 |
| 空间压力 | 当众、封闭空间、门口、台上台下、众人围观、直播、监控等 |
| 证据/道具 | 证据怎么被看见，不只是被读出来 |
| 权力变化 | 这场结束谁压过谁 |
| 关系变化 | 谁靠近、谁退开、谁误会更深、谁第一次动摇 |
| 台词原则 | 哪些必须说，哪些必须用动作替代 |
| 卡点 | 停在哪个动作、信息、选择或后果前 |

### 通过标准

- 高压节点不能只有文件、会议、冷对白；
- 如果把台词静音，观众仍应大致看懂这场发生了什么情绪/权力变化；
- 不是每场都夸张，但高压节点不能端着。

## 11. 状态追踪

### 来源

吸收 oh-story 的关系状态、时间线、伏笔、角色状态。

### 目标

分批写作时防止人物、误会、信息差、伏笔漂移。

### 四个轻量追踪表

| 追踪表 | 记录什么 |
|---|---|
| 关系状态 | 主角之间、主角与反派之间的信任/误会/依赖/敌意变化 |
| 信息差状态 | 观众知道什么、主角知道什么、反派知道什么 |
| 伏笔与证据 | 哪些证据已出现、谁看见了、还没兑现什么 |
| 角色状态 | 身体伤害、情绪债、身份暴露程度、当前目标 |

### 通过标准

- 首批写作前至少初始化主角关系、核心误会、核心秘密、关键证据；
- 每批写完更新一次；
- reviewer 可以用它判断人物反应是否漂移。

---

## 12. 单集写作包

### 来源

吸收 Toonflow 逐集输入，以及 0xsline episode-directory。

### 目标

主创写某一集时，不直接吞全量源本和全量分析，避免被源本事件链拖走。

### 字段

| 字段 | 内容 |
|---|---|
| 上一集结束状态 | 上一集卡在哪里，人物/信息/关系状态是什么 |
| 本集功能 | 本集继承哪个源本追看功能 |
| 本集新事件载体 | 新壳下具体发生什么 |
| 必须保留体验 | 爽、痛、急、误会、反转、拉扯等 |
| 必须发生的变化 | 关系/局势/信息/情绪哪一项变化 |
| 禁止复用外形 | 不准照搬的源本桥段、道具、证据、台词 |
| 关键场面落点 | 本集高压场面怎么拍出来 |
| 台词提醒 | 哪些信息不要靠对白解释 |
| 结尾钩子 | 卡在哪个动作/信息/选择前 |

### 通过标准

- 主创能只看这个包写当前集；
- 不需要反复翻大段源本；
- 写出来后能回查是否执行了本集功能。

---

## 13. 首批正文初稿

### 目标

写出可读剧本文字。

### 写作原则

- 主创 agent 必须同时负责分析和写作，不能把源本理解交给另一个 agent 后自己只按标签写。
- 正文必须执行单集写作包。
- 高压节点必须执行关键场面草案。
- 普通过渡集可以克制，但必须快、清楚、少废话。
- 文件、会议、录音、报告、对白可以使用，但只能辅助场面，不能替代关键情绪和冲击。

### 输出

`首批正文初稿`

## 14. 正文后短剧化返修

### 来源

吸收 how-to-make-script 的 dialogue polish / rewrite doctor，以及 ReelForge 的 badcase repair。

### 目标

正文初稿写完后，不直接进入作者自检，先做一次短剧化返修。

这一步不是重新设计全剧，只改正文层坏味道。

### 14.1 台词压缩

处理：

- 删重复信息；
- 删寒暄；
- 删解释腔；
- 删“我现在很痛苦/我黑化了/我很愤怒”这种抽象直说；
- 把能用动作、物件、画面、身体反应表达的台词改掉。

判断：

```text
这句台词有没有功能？
它是在逼迫、攻击、揭露、反击、选择、拒绝、威胁、试探，还是只是聊天？
连续两三轮对白有没有新信息、新压力或新关系变化？
```

### 14.2 场面强化

处理：

- 高压节点补动作、身体反应、空间压迫、当众后果；
- 文件/证据尽量转成可见证据，不只让角色读；
- 让权力变化被看见；
- 让关系拉扯被动作化。

例子：

```text
弱：男主看完文件，沉默。
强：男主听见录音，同时监控画面里出现女主倒地的瞬间；他手指停在暂停键上，会议室里没人敢出声。
```

### 14.3 卡点修正

处理：

- 不把结果说完；
- 卡在选择落下前、身份揭开前、惩罚执行前、直播按钮按下前、证据播放前；
- 如果一集已经把期待兑现完，补新一轮追问。

### 14.4 重复功能场删除

处理：

- 如果一场戏只是在重复上一场的信息或情绪，删掉或合并；
- 能用一句话 cover 的场，不单开一场；
- 例如导演指出的 2-1，如果男主一句“把档案室钥匙交还”能完成冷酷功能，就不需要拖一场。

### 输出

`首批正文修订稿`

---

## 15. 作者自检

### 目标

主创先对自己的稿子负责。

### 自检问题

| 问题 | 如果失败 |
|---|---|
| 是否洗飞源本有效性 | 回到源本有效性摘要 |
| 是否改名复刻 | 回到新事件载体迁移 |
| 是否短剧场面降级 | 回到关键场面草案 |
| 是否台词水、解释腔、电视剧聊天 | 回到短剧化返修 |
| 是否状态漂移 | 回到状态追踪 |
| 是否每集有变化和钩子 | 回到分集功能表 |

### 必须说明

- 自己最担心的问题是什么；
- 哪些地方可能需要 reviewer 重点看；
- 哪些导演反馈风险已经处理过。

---

## 16. 独立 reviewer 分层审稿

### 来源

吸收 how-to-make-script 的 quality gate / rewrite report 思路。

### 输入

Reviewer 第一轮不要读作者自检。

输入包括：

- 用户需求；
- 源本有效性摘要；
- 源本事件载体拆解；
- 新事件载体迁移；
- 正文前策略审核意见；
- 新本阶段骨架；
- 分集功能表；
- 关键场面草案；
- 状态追踪；
- 首批正文修订稿；
- PRD 验收标准。

### 结论

| 结论 | 含义 |
|---|---|
| `pass` | 可给用户 / 导演确认 |
| `revise` | 方向对，但有必须修改项 |
| `block` | 不能继续写后续，必须回到前面某一步 |

### 必须分层判错

| 失败层 | 说明 |
|---|---|
| 源本分析失败 | 没读出源本为什么有效 |
| 事件载体迁移失败 | 新本仍是源本事件换名，或新事件承载不了 |
| 场面承载失败 | 高压节点变文件、会议、冷对白、站桩 |
| 分集功能失败 | 单集空转，没有关系/局势/信息/情绪变化 |
| 正文对白失败 | 台词水、解释腔、长剧感 |
| 卡点失败 | 集尾说完，没停在下一集欲望点 |
| 状态追踪失败 | 人物、误会、证据、伏笔前后不接 |

Reviewer 不能只说“加强视听语言”。必须指出具体失败层和返修入口。

## 17. 主控汇总交付

### 主控对比

- 作者自检；
- 独立 reviewer 意见；
- 必要时的导演反馈。

### 主控决定

- 直接交付；
- 局部返修；
- 回到某个上游工序；
- 停止继续写后续。

### 用户看到

1. 首批新本；
2. 简短改动说明；
3. 作者自检摘要；
4. reviewer 结论；
5. 需要用户判断的问题。

---

## 18. 用户 / 导演反馈回流

### 目标

反馈不是无限测试，而是回流到具体工序。

### 分类

| 反馈类型 | 回流位置 |
|---|---|
| 只是改名 | 新事件载体迁移 |
| 视听冲击弱 | 关键场面草案 |
| 台词太多 | 短剧化返修 |
| 拉扯不够 | 关键场面草案 + 关系状态追踪 |
| 节奏拖 | 分集功能表 + 重复功能场删除 |
| 卡点不准 | 分集功能表 + 卡点修正 |
| 人物反应怪 | 状态追踪 + 角色动机 |

反馈不能直接变成新规则。必须先定位失败层。

## 19. 当前样本问题对应

### 第 2 集：车祸 -> 档案室降级

应该在两个地方拦住：

- `新事件载体迁移`：判断档案室是否能承载车祸级别的身体危机和情绪冲击。
- `关键场面草案`：如果只能靠交钥匙、文件和对白推进，判定短剧场面降级。

### 第 7 集：台词多、像电视剧聊天

应该在短剧化返修处理：

- 删重复对白；
- 标每句台词功能；
- 把解释性信息转成动作、物件、表情、空间压迫；
- 如果关系没有变化，回到关键场面草案。

### 第 8 集：不够 dramatic、站桩

应该在关键场面草案处理：

- 明确人物身体反应；
- 明确空间压迫和当众后果；
- 明确权力变化；
- 明确人物是否跪、退、失控、被踢出局，按剧情需要选择，不机械撒狗血。

### 第 3 集：女主回归不该说太多

应该在卡点修正处理：

- 如果视觉姿态和众人反应已经能形成期待，就少说；
- 集尾卡在“她出现，但还没兑现反击”的瞬间。

### 第 10 集：导演觉得好

可以作为正例：

- 压力到位；
- 拉扯升级；
- 动作卡点明确；
- 可以卡在直播按钮按下前。
