## 0. 本版解决什么

本版承接 `PRD_v4.md` 的产品形态：用户只面对主控 agent；内部用主创链、skill 工位和干净 reviewer 执行。

本版解决的是执行方式问题：

```text
不要再让一个 agent 一口气扮演所有角色。
```

每一环必须按对应 skill 工位运行。可以是同一个主创 agent 连续执行，但每一步都要显式切换角色、输入、输出、交接和禁区。

## 1. 产品锚点

产品仍然是：

```text
把已被市场验证或值得参考的短剧源本，
洗成新壳下仍然好看、能追、能付费的新剧本。
```

不是：

- 从零原创；
- 评分器；
- 查词替换；
- 套模板；
- 视频分镜工程；
- reviewer 驱动的返修机器。

## 1.5 产品运行形态

产品形态以 `PRD_v4.md` 第 7 节为准。本 spec 只规定执行含义：

- 主控 agent 维护 run state / manifest，并负责反馈路由；
- 主创链连续执行 S1-S9，但每一步必须切换 skill 工位；
- Reviewer 只执行 S10，干净隔离，不生产短剧感；
- S11 由主控汇总，不让 writer 或 reviewer 直接对用户拍板。

## 2. 执行链路

```text
0. 建 manifest
1. S0 需求确认
2. S1 源本拆文
3. S2 洗稿边界 / 变量迁移
4. S3 新本故事架构
5. S4 场面承载原料
6. S5 writer 单集施工稿
7. S6 screenplay 正文写作
8. S7 台词 / 视听 polish
9. S8 连续性 / 真相 / 信息边界
10. S9 作者自检与返修路由
11. S10 干净 reviewer
12. S11 主控版本对比与拍板
```

本版比 v2 少了一些表面编号，但没有删除能力。v2 的能力被重新归入对应 skill 工位。

## 3. Manifest 要求

使用本候选版跑样本时，manifest 必须以本版 0-12 为准。

manifest 必须记录：

- 本轮使用 `workflow_spec_v14_skill工位链候选版.md`；
- 每个 skill 的输入文件；
- 每个 skill 的输出文件；
- 每个 skill 的 pass / fail；
- 如果同一个 agent 执行多个 skill，必须写明何时切换角色；
- 如果 reviewer 是独立 agent，必须写明第一轮读取了什么、没有读取什么；
- 如果某个 skill 跳过，必须说明为什么不影响本轮。

没有 manifest，不允许声称完整跑完。

## 4.5 0xsline 分阶段加载

0xsline 的材料只作为短剧手艺库接入，不当作本项目主流程。

本项目不是从零原创，而是洗稿。所以每次使用 0xsline 参考时，都必须先服从源本有效性迁移、洗稿边界和 Do Not Consume。

| Step | Skill | 参考加载 | 边界 |
|---|---|---|---|
| 1 | S0 需求确认 | `genre-guide.md` | 只辅助题材、市场、受众和语感确认 |
| 2 | S1 源本拆文 | `rhythm-curve.md` / `hook-design.md` / `satisfaction-matrix.md` | 只帮助拆“为什么能追”，不能压过源本 |
| 3 | S2 洗稿边界 | 不固定加载 | 以源本迁移、防洗飞、防复刻为主 |
| 4 | S3 新本故事架构 | `genre-guide.md` / `rhythm-curve.md` / `paywall-design.md` / `villain-design.md` | 只辅助故事底盘、追问兑现和反派燃料 |
| 5 | S4 场面承载原料 | `satisfaction-matrix.md` / `hook-design.md` | 只给可写戏的原料，不替 writer 排戏 |
| 6 | S5 writer 施工稿 | `opening-rules.md` / `rhythm-curve.md` / `hook-design.md` / `satisfaction-matrix.md` | 在写施工稿时加载短剧手艺 |
| 7 | S6 正文写作 | S5 的施工稿与短剧手艺 | 从施工稿写成可看可听的剧本 |
| 8 | S7 台词/视听 polish | `genre-guide.md` + 本项目台词/视听要求 | 删水词、保声线、强动作和声音 |
| 9 | S8 连续性/真相/信息边界 | 不固定加载 | 只查事实、伏笔、状态、信息边界 |
| 11 | S10 干净 reviewer | `rhythm-curve.md` / `hook-design.md` / `satisfaction-matrix.md` | 只作审稿 lens，不使用 0xsline 的 50 分制硬打分 |

不允许把 0xsline 全库一次性塞给 writer。每一阶段只加载当阶段需要的手艺。

## 5. 核心交接规则

### 5.1 S1 到 S5 必须保活

S1 拆出的以下能力，必须在 S5 施工稿里看到对应痕迹：

- 观众正在追什么；
- 爽点前如何压低；
- 本集钩子停在哪；
- 谁压谁、谁退、谁被迫行动；
- 哪个信息不能提前消费；
- 角色状态和关系拉扯如何变化。

如果 S1 有、S5 没有，S5 不能 pass。

### 5.2 S5 到 S6 必须保强度

S5 施工稿里的核心强度，必须进入 S6 正文：

- 现场压迫；
- 人物动作；
- 身体 / 空间 / 声音 / 物件；
- 当众后果；
- 关系 / 权力 / 信息变化；
- 最后一拍卡点。

如果施工稿像戏，但正文写成说明书，S6 不能 pass。

### 5.3 S8 不是 reviewer

S8 只查连续性、真相、信息边界。

它不评价“够不够爽”，也不替 writer 改戏。

### 5.4 S10 reviewer 降权

S10 只查红线和明显风险。

Reviewer 可以指出“这场强度漏损”，但必须说明回哪一层修，不能直接变成逐句挑刺。

### 5.5 用户 / 导演反馈回流

用户和导演反馈只进主控 agent。

主控先判断反馈属于哪一层：

- 需求不清：回 S0；
- 洗飞或复刻：回 S1 / S2；
- 真相、追问、阶段骨架不稳：回 S3；
- 场面载体承载不住：回 S4；
- 有骨架但不像短剧戏：回 S5；
- 施工稿强，正文弱：回 S6；
- 台词水、视听弱、人物站桩：回 S7；
- 伏笔、信息边界、角色状态错：回 S8；
- 自检或 reviewer 误判：回 S9 / S10 口径。

不能把每条反馈直接写成新规则。先归层，再返修。

### 5.6 Run state

主控必须维护本轮 run state，至少包括：

- 当前需求确认；
- 当前 workflow spec；
- 本轮输出范围；
- 每个 skill 的产物位置；
- 当前 pass / fail；
- 当前阻断点；
- 用户 / 导演反馈归层；
- 返修是否只回对应层。

run state 是给主控编排用的，不是给 writer 的完整输入。

## 6. 不允许的失败形态

以下情况不能判完整通过：

1. S1 分析很厚，但 S5 施工稿只剩剧情任务。
2. S4 场面承载替 writer 排完整正文，导致 S5 变成抄写。
3. S5 施工稿写成字段表。
4. S6 正文变成报告、会议、屏幕、冷对白。
5. S7 把台词修成金句堆砌。
6. S8 把“还能更好”当连续性错误。
7. S10 reviewer 把风险项当致命阻断。
8. S11 只看最终剧本，不看 manifest 和中间产物。

## 7. 和 sub-agent 的关系

本版不强制每个 skill 一个 sub-agent。

建议：

- 主控 agent：S0、manifest、S11；
- 主创 agent：S1-S9 连续执行，但每步必须切 skill；
- Reviewer：S10 单开干净 sub-agent；
- Continuity：长剧本或真相复杂时，S8 可以单开干净 sub-agent。

原因：

```text
创作连续性需要保留；
审稿独立性需要隔离。
```

所以不是“agent 越多越好”，而是该连续的连续，该隔离的隔离。

## 8. 本版下一次样本验证

如果继续，应对同一飞书三星样本完整跑首批 1-10 集。

对比对象：

- V5：短剧感和场面强度基准；
- V22：相对稳定但偏平的近期稿；
- V24：完整链路但 writer 渲染感明显；
- 本版新稿。

对比维度：

- 主线是否清楚；
- 是否洗飞；
- 是否改名复刻；
- 是否保住源本追看功能；
- 钩子 / 卡点是否有效；
- 爽点压放是否成立；
- 人物关系 / 拉扯是否推进；
- 高压场是否像戏；
- 台词是否有功能；
- 视听表达是否足够；
- 信息边界是否稳定；
- 是否优于 V24；
- 是否至少不低于 V5 的短剧感优势。
