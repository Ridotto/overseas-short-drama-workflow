## 0. 文档定位

这份 spec 不是 workflow，不是工程 spec，不是 reviewer 规则，不是打回机制。

这份 spec 只做一件事：

```text
把 PRD_v4 完整编译成稳定的产品契约。
```

稳定产品契约包含两层：

1. 用户可见产物契约：用户每一步看到什么，凭什么确认，最终拿到什么。
2. 内部生产产物契约：系统内部每一步必须自然产出什么，如何支撑下一步稳定写出好剧本。

这份 spec 的完成标准不是“更像流程”，而是：

```text
后续任何 agent 只读 PRD_v4、项目基础说明和本 spec，
就能继续编 workflow，
但不会把产品重新写成字段碎片、gate 表、reviewer 返工机或机械模板。
```

### 1.2 地基约束

`项目基础说明.md` 不是执行细则，但它约束所有上层文档。

本 spec 必须遵守以下地基原则：

- 这是一个帮用户把爆款短剧洗成新本子的工具。
- 最终产物是能用的剧本文字，不是分析报告。
- 本质是写好剧本，只是带一个市场验证源本作为底子。
- 源本永远是底子，不能脱缰乱编。
- 换壳必须自洽，不能查词替换，不能生搬硬套。
- 一切为了观众想追、想付费，但不能靠硬堆钩子、爽点、卡点。
- 正确结果是扎实好剧本加关键钩子、爽点、卡点、情绪节奏写得好。
- 交互方式是先对齐、再动手、边做边确认，不是一键黑箱生成。
- 用户负责方向和拍板，系统负责读透源本、保住内核、落地成新本。
- 系统要让用户信得过：能说明改了什么、为什么改、源本有效内核是否保住。

## 2. Product Contract

### 2.1 产品一句话

产品必须稳定完成：

```text
把一个已被市场验证或值得参考的短剧源本，
洗成一个新壳下仍然好看、能追、能付费的新剧本。
```

“好看、能追、能付费”不能只停在口号。它在产物中必须落成：

- 源本有效性没有被洗飞；
- 新壳里人物动机和事件载体成立；
- 观众能理解、能代入、想继续看；
- 高压节点靠行动、关系、权力、体面、身体、空间或物件后果承载；
- 不是靠文件、会议、冷对白、解释性信息播报撑强度；
- 新剧不是源本改名复刻；
- 新剧作为独立短剧站得住。

### 2.2 产品交付物

产品交付的是剧本文字。

分析、摘要、蓝图、控制包、施工稿、reviewer 结论都只能服务剧本文字，不能反客为主。

任何 workflow 编译都必须守住：

```text
用户购买的是能用的新剧本，
不是一堆看起来很专业的内部分析。
```

### 2.3 产品不是

产品不得变成：

- 评分器；
- 从零原创工具；
- 短剧标准库套模板工具；
- 查词替换式换皮工具；
- 导演或用户反馈拆规则工具；
- 视频、分镜、投流片段或成片工具；
- 一键生成完整长剧的黑箱工具；
- 多 agent 过程展示产品；
- reviewer 驱动返修机器；
- 只会产出分析报告的研究助手。

### 2.4 目标用户

目标用户是写短剧本的人：

- 用户本人；
- 团队里的编剧；
- 需要把已验证源本改成新题材、新身份、新背景、新世界观或新集数版本的人。

系统必须默认用户有创作判断和拍板权。系统不是替用户做所有决定，而是把用户的方向高质量落地。

### 2.5 典型场景

系统必须支持以下真实输入状态：

- 用户有完整爆款或可参考源本，想正式洗稿。
- 用户有新壳方向，例如黑手党复仇洗成豪门商战。
- 用户方向模糊，需要先对齐。
- 用户可能带自备角色设定、梗概、粗纲、分集大纲、导演反馈、市场偏好、对标参考。
- 用户可能只想先判断源本值不值得洗。
- 用户可能要长篇多批次交付。
- 用户可能只想局部修几集或某个卡点。

## 3. Product Shape Contract

### 3.1 第一阶段产品形态

第一阶段是 skill / workflow 原型阶段。

第一阶段不做：

- 网页 App；
- 工程 spec；
- 多 agent 产品 surface；
- 一键黑箱生成；
- 完整自动化平台。

第一阶段要做：

- 稳定创作流程；
- 用真实样本验证；
- 验证源本分析、创作蓝图、场景施工、正文是否能稳定产出可用剧本；
- 先证明剧本质量，而不是先固化界面。

### 3.2 中期形态

流程稳定后，用户入口可以工具化为：

1. 上传或选择源本；
2. 需求对齐；
3. 输出源本有效性摘要；
4. 输出洗稿方案摘要；
5. 输出创作蓝图包；
6. 输出首批正文修订稿；
7. 收集用户反馈；
8. 分批继续写。

中期工具化不得改变第一阶段验证出的创作能力边界。

### 3.3 长期形态

网页 App 是最终用户入口，不是创作能力本身。

网页 App 只能负责：

- 上传源本；
- 确认新壳、集数、改写力度、语言；
- 展示用户可见交付；
- 管理版本、批次和反馈；
- 让用户在正确节点确认；
- 隐藏内部工位细节。

## 4. Supported Modes

系统固定支持四种模式。主控只能在这四种里选择，不能临场发明第五种默认模式。

### 4.1 诊断 / 蓝图模式

只跑到创作蓝图包，不进正文。

适用条件：

- 用户先想判断源本值不值得洗；
- 用户只想确认方向、角色、粗纲和当前批戏稿；
- 输入不是完整文本源本，只是摘要、片段、零散材料；
- 系统和用户对源本为什么好看还没对齐。

产物上限：

- 需求确认摘要；
- 源本有效性摘要；
- 洗稿方案摘要；
- 创作蓝图包。

### 4.2 标准首批模式

默认模式。

产物目标：

```text
跑完整链路到首批 1-10 集正文。
```

标准首批模式的验证重点不是“流程跑完”，而是：

- 源本有效性保住；
- 新壳自洽；
- 创作蓝图包可确认；
- 场景施工稿能支撑正文；
- 正文能追、能爽、有继续看的冲动；
- 首批整体体验至少保住既有强样本的关键体验信号。

### 4.3 长篇多批次模式

适用条件：

- 用户要 100 分钟、100 集或多批次交付；
- 不能一次性写完整本；
- 每一批都需要重新确认当前批细层蓝图。

固定原则：

```text
全剧粗，当前批细，批次滚动推进。
```

任何批次不得重置人物关系、误会、伏笔、证据、承诺和 DNC。

### 4.4 局部改稿 / 定点返修模式

只接受已有项目上下文。

适用条件：

- 用户只想修某几集；
- 用户只想改某个卡点；
- 用户只想重写某段场面；
- 已有蓝图和正文基础，不需要重跑整条正式链。

这条模式不是默认 workflow 的正向验证证据。

## 5. User Interaction Contract

### 5.1 用户只和主控 agent 对话

用户不直接面对：

- 内部工位；
- reviewer 全报告；
- run log；
- 故事控制包；
- 场景施工稿；
- 多 agent 协作过程。

主控负责：

- 需求对齐；
- 解释用户可见产物；
- 收集用户反馈；
- 翻译必要 reviewer 结论；
- 对用户拍板；
- 保护用户不被内部噪音淹没。

### 5.2 默认用户可见主链

用户视角的主链是：

```text
需求确认
-> 源本有效性摘要 + 洗稿方案摘要
-> 创作蓝图包
-> 首批正文修订稿
-> 完整版交付 / 用户反馈回流
```

PRD 中出现的 `场景施工稿` 和 `reviewer 轻版硬伤检查` 是执行链节点，不是默认用户展示节点。

用户确认的是：

- 方向；
- 蓝图；
- 正文结果；
- 是否继续、调整后继续，或停止。

用户不确认：

- 内部工位细节；
- agent 分工；
- reviewer 全量意见；
- run log；
- proof-flow；
- gate 表；
- 质量检查表。

### 5.3 必填输入

需求对齐必须问清以下内容。

| 输入项 | 产品含义 | 后续用途 |
|---|---|---|
| 源本是什么 | 决定是否能进入正式写稿链 | 源本有效性分析 |
| 想换成什么新壳 | 决定题材、身份、世界观、场景重建 | 洗稿方案 |
| 改写力度与变量边界 | 决定变量贴近、迁移、自由改写、删除修复 | 洗稿边界 |
| 集数是否改变 | 决定留存节点和节奏是否重排 | 粗纲和批次规划 |
| 目标市场 | 决定常识、本地化、敏感设定 | 新壳自洽 |
| 输出语言 / 审稿语言 | 用户可见材料默认中文，正文语言必须确认 | 正文写作 |
| 正文用途 | 用户/导演审稿，或进入 AI 视频生成链路 | 文体和 page-to-screen 密度 |
| 用户特别想保留什么 | 锁定主观偏好 | 冻结点 |
| 用户特别不想要什么 | 避免继承源本烂点或用户雷点 | DNC / 禁区 |

缺少这些信息时，系统可以先进入诊断 / 蓝图模式，但不能承诺正式正文质量。

### 5.4 额外输入材料

用户可额外提供：

- 角色设定；
- 梗概；
- 粗纲 / 分集大纲；
- 导演反馈；
- 市场偏好；
- 对标参考；
- 其他参考源本。

这些材料不得直接塞给 writer。

主控必须先判断它们属于：

- 需求层；
- 边界层；
- 蓝图层；
- 风险提示；
- 补充偏好。

正式写稿链默认只接受一个主源本。其他参考源本不能进入主链迁移，只能作为补充偏好或风险提示。

### 5.5 用户权限

用户可以：

- 补充事实；
- 纠正系统理解；
- 对中间材料提修改意见；
- 覆盖系统建议，但必须显式确认风险。

用户不可以：

- 直接改系统生成的蓝图文本；
- 跳过蓝图确认直接进正式正文链；
- 让 reviewer 直接对话拍板。

## 6. User-visible Deliverable Contract

用户可见对象固定分开。

| 交付物 | 是否可见 | 产品作用 | 成品质感 |
|---|---|---|---|
| 需求确认摘要 | 是 | 确认方向 | 简洁、可拍板 |
| 源本有效性摘要 | 是 | 说明系统理解了源本为什么好看 | 不是剧情复述，而是有效机制说明 |
| 洗稿方案摘要 | 是 | 进蓝图前确认保什么、换什么、风险什么 | 是改编策略，不是抽象原则 |
| 创作蓝图包 | 是 | 正式写正文前的主确认文档 | 像商业项目大纲文档 |
| 首批正文修订稿 | 是 | 用户真正判断成品质量 | 可读、可拍、可继续反馈 |

默认不展示：

- 故事控制包；
- 场景施工稿；
- 完整 reviewer 报告；
- 后台 run log；
- 作者自检；
- proof-flow；
- gate 表；
- 候选演法表；
- 多 agent 分工记录。

如果用户主动要求看 reviewer 结论，主控只给精简摘要，不展示完整内部报告。

## 7. Main Confirmation Document: 创作蓝图包

### 7.1 定位

创作蓝图包是正文前的正式用户检查点。

它同时承担两件事：

1. 用户能读懂、能判断、能确认。
2. 下游能基于它继续扩写，不需要回头翻内部厚材料。

它不是：

- 四问表；
- 字段碎片；
- 内部分析包；
- reviewer 检查表；
- proof-flow；
- S 编号执行记录；
- 候选方案堆叠；
- 场景施工稿；
- 完整正文。

### 7.2 商业项目形态

创作蓝图包必须像商业项目里的中间确认文档。

默认结构：

```text
创作蓝图包

一、故事梗概
二、人物设定
三、全剧粗纲
四、当前批集纲
五、当前批状态与禁区说明
```

PRD 中的两层结构必须被编译到上述商业文档形态里：

| PRD 模块 | 创作蓝图包中的位置 |
|---|---|
| 一句话故事核 | 故事梗概开头 |
| 核心角色设定 | 人物设定 |
| 全剧梗概 | 故事梗概 |
| 阶段粗纲 | 全剧粗纲 |
| 主追问 | 故事梗概 / 全剧粗纲 |
| 当前批每集戏稿 | 当前批集纲 |
| 每集追看债务 | 当前批集纲 |
| 关键戏发动方式 | 当前批集纲 |
| 角色本批状态线 | 当前批状态与禁区说明 |
| 关系变化摘要 | 当前批状态与禁区说明 |
| DNC 摘要 | 当前批状态与禁区说明 |
| 禁用外形摘要 | 当前批状态与禁区说明 |

### 7.3 故事梗概

故事梗概必须让用户看懂：

- 新剧讲什么；
- 新壳是什么；
- 主角为什么必须行动；
- 对立压力来自哪里；
- 源本有效内核如何迁移；
- 本剧为什么有追问；
- 本剧为什么不是源本改名复刻。

故事梗概不得只是源本剧情压缩，也不得只是设定说明。

### 7.4 人物设定

人物设定必须包含：

- 核心角色；
- 角色欲望；
- 角色压力；
- 角色秘密或错信；
- 关键关系；
- 关系变化潜力；
- 与源本角色功能的迁移关系；
- 新壳下的人物合理性。

人物设定不得只有姓名、身份、标签。

人物设定必须服务后续行动和冲突，不能只服务世界观介绍。

### 7.5 全剧粗纲

全剧粗纲必须回答：

- 全剧大阶段如何推进；
- 每阶段核心压力是什么；
- 每阶段主追问如何变化；
- 关键爽点和释放大致落在哪里；
- 源本强节点如何被重排或迁移；
- 长篇多批次时，当前批处在全剧什么位置。

全剧粗纲必须比单集集纲更宏观、更完整。

全剧粗纲不能薄到只剩几句阶段标签。

### 7.6 当前批集纲

当前批集纲必须是“已经像戏”的分集内容。

每集至少自然包含：

- 本集发生什么具体人事；
- 谁逼谁行动；
- 本集如何承接上一集；
- 本集兑现什么；
- 本集留下什么追问；
- 本集爽点或压力在哪里；
- 本集结尾为什么让人继续看；
- 本集如何服务整批关系和状态变化。

当前批集纲不得只写：

- hook / spike / button；
- 先兑什么 / 再欠什么；
- 高冲动方案；
- 追看债务标签；
- 证据推进；
- 制度流程；
- 情绪关键词。

这些概念可以在内部帮助设计，但不能替代集纲本身。

### 7.7 当前批状态与禁区说明

当前批状态与禁区说明必须记录：

- 当前批开始时人物状态；
- 当前批结束时人物状态；
- 关系变化；
- 误会变化；
- 真相 / 证据 / 伏笔状态；
- 已经兑现的债；
- 仍然欠着的债；
- DNC；
- 禁用外形；
- 用户特别要求保留或避免的内容。

它的目的不是给用户展示内部复杂度，而是确保用户确认后，后续正文和下一批不会漂移。

---

## 8. Internal Production Contract

内部生产产物不是用户主文档，但它们决定用户看到的东西是不是垃圾。

内部产物的质量必须内建到产物定义里，不能靠末端 reviewer 或无限打回补救。

### 8.1 源本有效性摘要

源本有效性摘要必须说明：

- 源本为什么值得洗；
- 源本为什么能让观众追；
- 源本为什么能产生爽点；
- 源本哪些结构或节点支撑付费冲动；
- 源本哪些体验不能洗丢；
- 源本有哪些烂点不能继承；
- 源本节奏中哪些节点需要保留、迁移或重排。

它不是剧情复述。

下游使用方式：

- 洗稿方案用它决定保什么、换什么、风险什么；
- 创作蓝图用它判断源本有效内核有没有迁移；
- 正文不能脱离它乱编。

它不得把“源本很抓人”写成空泛判断，必须拆到可迁移的机制。

### 8.2 洗稿方案摘要

洗稿方案摘要必须说明：

- 新壳方向；
- 哪些源本变量贴近保留；
- 哪些源本变量等效迁移；
- 哪些源本变量自由改写；
- 哪些源本变量删除或修复；
- 新壳如何承载源本有效机制；
- 如何避免改名复刻；
- 如何避免洗飞；
- 用户偏好和禁区如何进入改编。

它不是新故事灵感列表。

下游使用方式：

- 故事控制包用它建立新剧内部逻辑；
- 创作蓝图包用它向用户展示“我们准备怎么改”；
- 场景施工稿和正文不得越过它重新发明新壳。

### 8.3 故事控制包

故事控制包只对内。

它承接：

- 真相账本；
- 角色错信；
- 源本强节点重排；
- 全剧阶段骨架；
- 批次地图；
- 每批追看债务；
- 反派 / 对立压力源；
- DNC；
- 新壳承载边界。

故事控制包的产品责任是：

```text
把源本有效性和洗稿方案压成新剧可持续运转的内部事实系统。
```

它不能直接给 writer 当正文输入。

它必须被编译成：

- 用户能确认的创作蓝图包；
- writer 能施工的场景施工稿。

如果故事控制包很完整，但创作蓝图包仍是碎片，视为产品契约没有落地。

### 8.4 场景施工稿

场景施工稿是正文前最后一层，只服务当前批 / 当前集 / 当前场景组。

它必须让 writer 不回头翻厚材料也能写正文。

场景施工稿必须自然包含：

- 每集 opener；
- 每集 scene chain；
- 每场 function；
- 每场 conflict；
- 每场 change point；
- 压力如何进入现场；
- 释放如何发生；
- 可见后果是什么；
- 结尾 hook；
- 下一债务；
- 不能写什么。

场景施工稿不得只是剧情 bullet。

它必须把蓝图中的追看、爽点、兑现、下一债务转成可以拍的场面。

### 8.5 正文

正文必须是可读、可拍、可继续反馈的短剧文本。

正文必须满足：

- page-to-screen；
- 人物行动推动；
- 对白不裸飘；
- 信息揭示服务冲突；
- 高压节点可见；
- 爽点有兑现；
- 结尾有追问；
- 状态连续；
- 不复刻源本；
- 不洗飞源本。

正文不是对场景施工稿的扩字，也不是对白版剧情提纲。

### 8.6 连续性状态

连续性状态必须支撑长篇多批次。

它必须记录：

- 关系状态；
- 误会状态；
- 证据状态；
- 伏笔状态；
- callback / payoff；
- 人物承诺；
- 已冻结设定；
- DNC；
- 禁用外形；
- 下一批必须承接的债。

它的作用是防止下一批重置、漂移或重复消费。

### 8.7 reviewer 摘要

reviewer 只做轻版硬伤兜底。

reviewer 可以查：

- 洗飞；
- 复刻；
- 提前消费；
- 真相矛盾；
- 明显强度漏损；
- 状态漂移。

reviewer 不负责：

- 生产短剧感；
- 逐句重写；
- 对用户拍板；
- 替上游补设计；
- 成为新增生产工位。

reviewer 结果默认只给主控使用。用户主动要求时，主控只给精简摘要。

---

## 9. Natural Handoff Contract

交接不是“验收打回表”。

交接的本质是：

```text
上一产物必须已经自然包含下一产物继续生产所需的材料，
下一产物不得靠临场补锅替上游完成本该完成的创作责任。
```

| 上一产物 | 下一产物如何直接使用 | 下一产物不得补什么 |
|---|---|---|
| 需求确认摘要 | 确定源本、新壳、语言、集数、市场、边界 | 不得替用户臆造核心方向 |
| 源本有效性摘要 | 判断源本有效机制、强节点、不能洗飞的体验 | 不得只靠短剧标准库重新发明追感 |
| 洗稿方案摘要 | 建立新壳迁移策略和防复刻边界 | 不得只换名词、身份、地点 |
| 故事控制包 | 建立新剧内部事实、阶段、债务、压力源 | 不得让蓝图自己补真相账本和新壳逻辑 |
| 创作蓝图包 | 给用户确认，并给场景施工提供故事、人物、粗纲、集纲 | 不得把字段碎片交给下游扩成戏 |
| 场景施工稿 | 给 writer 提供当前批 / 当前集 / 当前场景组的可拍施工材料 | 不得让 writer 回翻厚材料补场面发动机 |
| 正文 | 给 reviewer 和用户判断成品质量 | 不得把“还没做出来的短剧感”交给 reviewer 补 |
| 连续性状态 | 给下一批继续写，保持关系、误会、伏笔、DNC | 不得让下一批重新推理前文状态 |

---

## 10. Built-in Quality Contract

质量不是末端 reviewer 的责任。

每一层产物必须把自己的质量责任内建进去。

### 10.1 洗飞保护

源本让人追下去的有效性不能丢。

防洗飞必须发生在：

- 源本有效性摘要；
- 洗稿方案摘要；
- 故事控制包；
- 创作蓝图包；
- 场景施工稿；
- 正文。

正文末端只补钩子不能解决洗飞。

### 10.2 防改名复刻

新剧不能只是源本事件链和强桥段整串保留，只换身份和名词。

防复刻必须发生在：

- 洗稿方案；
- 新壳承载边界；
- 创作蓝图包；
- 场景施工稿。

判断标准不是“看起来换了设定”，而是：

```text
新事件载体、新人物动机、新场景逻辑能独立成立。
```

### 10.3 防短剧降级

短剧降级指：

```text
事件变新了，但场面变弱、变冷、变文件对白。
```

防短剧降级必须发生在：

- 创作蓝图包的当前批集纲；
- 场景施工稿；
- 正文。

高压节点必须尽量靠：

- 行动；
- 空间；
- 身体；
- 物件；
- 当众后果；
- 关系撕裂；
- 权力变化；
- 体面损失。

不能主要靠：

- 文件递交；
- 会议解释；
- 系统确认；
- 证据播报；
- 权限流程；
- 冷对白。

### 10.4 防正文长剧化

正文长剧化指：

```text
剧情逻辑对，但台词多、人物冷、站着聊天。
```

正文必须满足：

- 每场有行动目的；
- 对白有策略；
- 信息揭示服务冲突；
- 情绪通过行为和局面变化表现；
- 场面前后状态不同。

### 10.5 防状态漂移

状态漂移指：

```text
分批写作后，关系、误会、伏笔、人物状态接不上。
```

连续性状态必须在每批结束后更新。下一批必须继承，不得重新初始化。

### 10.6 失败不能互相抵消

以下情况都不能互相抵消：

- 没有复刻但洗飞；
- 没有洗飞但短剧降级；
- 剧情成立但正文长剧化；
- 单集好看但状态漂移；
- 中间产物完整但正文不抓人；
- run log complete 但剧本质量不达标。

## 11. Version Freeze and Change Management

用户确认后形成冻结点的对象：

- 源本有效性摘要；
- 洗稿方案摘要；
- 创作蓝图包。

冻结的含义：

- 已确认模块默认保持稳定；
- 下游不得静默覆盖；
- 只有上游事实或用户决定变化，才允许解冻；
- 用户只要求改一个模块时，不应扩大成整包重做；
- 用户大改核心方向时，必须重新确认受影响的上游。

用户在蓝图确认后大改以下内容，会改变产品基础：

- 新壳；
- 集数；
- 改写力度；
- 核心关系；
- 核心设定。

这种变化不是普通局部修改，而是上游变更。系统必须重新对齐受影响的用户可见产物。

注意：

```text
冻结与解冻是版本管理，不是 reviewer 打回机制。
```

## 12. Admission and Exception Contract

### 12.1 正式写稿链准入

正式写稿链默认只接受完整文本源本。

摘要、片段、口述、零散材料默认只能进入：

- 诊断 / 蓝图模式；
- 需求对齐；
- 方向判断；
- 源本有效性初判。

它们不能直接进入正式正文链。

### 12.2 坏源本

如果源本质量一般，或不适合直接洗，系统不能承诺正式正文质量。

坏源本红旗包括：

- 主追问弱；
- 强节点稀；
- 重复水戏多；
- 后半塌；
- 拉扯弱；
- 外部反馈差。

坏源本默认转诊断 / 蓝图模式。

### 12.3 高风险改编例外模式

如果用户坚持超出洗稿器边界、明显更像原创改编的方向，可以继续，但必须让用户知道：

- 这不是标准洗稿链；
- 质量保障下降；
- 不计入默认 workflow 正向验证证据。

### 12.4 用户覆盖系统建议

如果主控判断用户坚持的改法会伤短剧感或洗飞源本，可以继续，但必须显式说明会损失什么。

这个例外必须记录。

### 12.5 无源稿脚本医生模式

如果用户只给现成新稿，不给源本，却要求“改得更短剧”：

- 不进洗稿链；
- 只当脚本医生模式；
- 不计入默认 workflow 正向验证证据。

## 13. First-stage Success Contract

第一阶段成功不等于自动产出完整爆款。

第一阶段成功必须同时满足：

1. 用户需求被确认。
2. 源本有效性被对齐。
3. 洗稿方案摘要被确认。
4. 创作蓝图包被用户确认。
5. writer 不再拿任务补戏，而是拿场景施工稿写正文。
6. 高压节点由动作、空间、身体、物件、当众后果承载，而不是文件、会议、冷对白。
7. 每集有明确追看理由和结尾追问。
8. 正文经过台词压缩和视听 polish。
9. 状态追踪能支撑关系、误会、证据、伏笔连续。
10. 主创链先把明显失败处理掉，reviewer 没指出致命阻断。
11. 剧本正文满足 page-to-screen 合同。
12. 用户能判断：继续、调整后继续，或停止。

这些标准必须进入后续 workflow 的产物定义，而不是只留在 PRD 或 reviewer 表里。

---

## 14. Workflow Promotion Contract

新的候选 workflow 不能只跑一个样本就升默认。

升默认门槛：

- 先跑飞书三星样本首批 1-10 集；
- 再跑一个明显不同题材、不同气质的样本首批 1-10 集；
- 第二个样本由主控按规则选；
- 两个样本都过，才允许升默认 workflow。

“过”的双门槛：

1. 整体显著强于 V22 / V24 / V25 / V32。
2. 追看强度至少保住 V5。

这里的“保住 V5”不是逐集对齐，也不是照抄 V5。

它看首批整体体验信号：

- 开场能抓人；
- 压力持续升级；
- 至少一次阶段释放或足够强悬置；
- 结尾让人想继续看；
- 高压节点主要靠可见承载；
- 人物拉扯 / 对立压力源不是平的。

这个体验门由主控结合样本对照判断，不由 reviewer 单独拍板。

no-ship gate 在后续 workflow 中只能保留为极薄的主控发布判断：

- 只保留 4-6 条一票否决项；
- 来源只取历史复发根因和首批整体体验门；
- 不从零碎反馈里长；
- 一次性判定；
- 不在同一版里无限 review。

## 16. Thin Workflow Compilation Appendix

这一节不是 workflow。

它只防止后续把本 spec 再次编歪。

### 16.1 后续 workflow 必须保持的方向

后续 workflow 必须从以下产物链编译：

```text
需求确认摘要
-> 源本有效性摘要
-> 洗稿方案摘要
-> 故事控制包
-> 创作蓝图包
-> 场景施工稿
-> 正文
-> 连续性状态 / reviewer 轻版硬伤摘要
```

其中用户可见主链是：

```text
需求确认摘要
-> 源本有效性摘要 + 洗稿方案摘要
-> 创作蓝图包
-> 首批正文修订稿
-> 用户反馈回流
```

### 16.2 后续 workflow 禁止事项

后续 workflow 不得：

- 把创作蓝图包写成内部字段表；
- 把故事控制包展示给用户；
- 把场景施工稿展示给用户；
- 让 writer 回翻故事控制包补戏；
- 让 reviewer 生产短剧感；
- 把 gate / proof-flow / reviewer checklist 写进用户主文档；
- 把外部项目固定比例压过源本和用户边界；
- 把商业项目文档形态当成纯格式套壳；
- 把运行完成当质量通过；
- 把失败处理写成无限返工循环；
- 把当前 v19 的 S 编号和旧 gate 语言直接继承为产品事实。

### 16.3 后续 workflow 应该吸收的外部参照

商业项目用于校准：

- 故事梗概；
- 人物设定；
- 全剧粗纲；
- 当前批集纲；
- 正文。

外部项目用于校准：

- 分集目录；
- 节奏曲线；
- opening；
- hook；
- satisfaction；
- paywall / 追付费机制；
- villain pressure；
- scene-writing；
- dialogue-subtext。

吸收方式必须是：

```text
进入某个产物的结构和写法，
而不是作为一段“参考知识”挂在文档末尾。
```

### 16.4 后续 workflow 的最低静态检查

后续 workflow 编完后，必须静态检查：

- 只读用户可见产物，是否像商业项目文档；
- 只读内部产物，是否能解释每一步如何稳定产出好东西；
- 只读场景施工稿，writer 是否不需要回翻厚材料；
- 只读正文，是否能看到短剧感、追看理由、可见高压承载；
- 只读连续性状态，下一批是否不会重置；
- `reviewer` 是否仍只是兜底；
- `run log` 是否只证明执行，不证明质量；
- 是否仍有 `gate / proof-flow / S 编号` 污染用户主文档。

## 19. Terms

### 产品契约

把 PRD 编译成稳定产物、边界和质量责任的文档。

### 用户可见产物

用户默认会看到、确认或反馈的材料。

### 内部生产产物

系统内部用来稳定产出用户可见产物和正文的材料。

### 创作蓝图包

正文前的用户主确认文档，同时是下游主上游。它必须像商业项目的大纲文档。

### 故事控制包

内部事实和结构控制材料。它服务创作蓝图包和场景施工稿，不直接服务用户或 writer。

### 场景施工稿

正文前最后一层内部施工材料。writer 只吃这一层写正文。

### reviewer

轻版硬伤兜底角色。它不生产短剧感，不替上游补设计，不对用户拍板。

### 自然承接

上一产物本身已经包含下一产物继续生产所需材料，下一产物不需要临场补上游的创作责任。

### 内建质量责任

质量标准写进每个产物的成品形态，而不是依赖末端打回或 reviewer 兜底。
