## 1. 产品总合同

产品目标固定为：

> 把一个已被市场验证或值得参考的短剧源本，洗成一个新壳下仍然好看、能追、能付费的新剧本。

交付物是剧本文字，不是分析报告。

任何 workflow 都必须同时满足三条核心判断：

1. 新本没有洗飞源本有效性。
2. 新本不是源本改名复刻。
3. 新本作为一个新短剧自己站得住。

这三条不能互相抵消。
例如：新本很不像源本，但不好看，仍失败。新本很抓人，但整串复刻源本，仍失败。

---

## 2. 反目标

workflow 不能把产品实现成以下形态：

- 评分器：只判断好坏，不产出可用剧本。
- 从零原创工具：脱离主源本有效性重写。
- 短剧标准库套模板工具：用通用模板覆盖源本节奏。
- 查词替换式换皮工具：只换名字、身份、地点。
- 反馈规则堆叠器：把导演或用户反馈拆成一百条硬规则。
- 视频 / 分镜 / 投流 / 成片工具：本阶段不生产这些产物。
- 一键完整长剧黑箱：必须有确认点和分批推进。
- 多 agent 展示产品：用户不需要看到内部多 agent 过程。

后续任何 workflow 条款只要把系统推向这些反目标，必须删除或内收。

---

## 3. 目标用户与典型输入场景

目标用户是写短剧本的人：用户本人，或团队里的编剧。

系统必须支持以下场景：

- 用户有爆款或可参考短剧源本，想迁移到另一个题材、身份、背景、世界观或集数版本。
- 用户不想从零原创，也不想让模型洗丢源本好看的东西。
- 用户可能方向清晰，也可能只有模糊想法。
- 用户可能带自备材料：角色设定、梗概、粗纲、分集大纲、导演反馈、市场偏好、对标参考。

执行含义：

- 模糊想法必须先进入需求确认，不得直接开写。
- 自备材料必须先归类，再吸收；不能直接塞给 writer。
- 对标参考和其他参考源本只能做偏好或风险提示，不能进入主链迁移。

## 4. 产品形态路线

当前阶段是 `skill / workflow 原型`，不是网页 App，也不是工程实现阶段。

当前必须优先稳定主创链：

```text
源本分析 -> 创作蓝图 -> 场景施工 -> 正文
```

中期工具化只能在流程稳定后发生，固定用户入口：

```text
上传或选择源本
-> 需求对齐
-> 源本有效性摘要
-> 洗稿方案摘要
-> 创作蓝图包
-> 首批正文修订稿
-> 用户反馈
-> 分批继续写
```

长期网页 App 只负责入口、展示、版本、批次和反馈管理，不承担创作能力本身。

执行禁止：

- 不得因为未来会做网页 App，就提前把创作流程写成表单填空。
- 不得把网页 App 需要的字段反向塞回创作 workflow。
- 不得在第一阶段引入产品层多 agent 展示。

## 5. 四种运行模式

主控必须在启动时选择且记录模式。不得临场发明第五种默认模式。

### 5.1 诊断 / 蓝图模式

只跑到创作蓝图包，不进正文。

适用条件：

- 用户先想判断源本值不值得洗。
- 用户只想确认方向、角色、粗纲和当前批戏稿。
- 输入不是完整文本源本，只是摘要、片段、零散材料。
- 系统和用户尚未对齐“源本为什么好看”。

产物上限：需求确认摘要、源本有效性摘要、洗稿方案摘要、创作蓝图包。
不得声称已验证正文质量。

### 5.2 标准首批模式

默认模式。

跑完整链路到首批 `1-10` 集正文。

首批 `1-10` 只是验证和交付批次，不等于整部剧只有前十集需要抓人。整部剧每一批都必须有追看理由和结尾追问。

### 5.3 长篇多批次模式

适用于 100 分钟、100 集或多批次交付。

固定原则：

- 全剧粗，当前批细。
- 每一批都必须重新确认当前批细层蓝图。
- 前批冻结内容不能静默覆盖。
- 后批必须读取连续性 / 信息边界，不得重置人物状态。

### 5.4 局部改稿 / 定点返修模式

只接受已有项目上下文。

适用：

- 修某几集。
- 改某个卡点。
- 重写某段场面。
- 已有蓝图和正文基础，不需要重跑正式链。

这条模式不是默认 workflow 的正向验证证据。

## 6. 用户可见主链

用户只和主控 agent 对话。

默认主链固定为：

```text
需求确认
-> 源本有效性摘要 + 洗稿方案摘要（一次前置确认）
-> 创作蓝图包（用户确认）
-> 场景施工稿
-> 正文
-> reviewer 轻版硬伤检查
-> 完整版交付
-> 用户反馈回流
```

解释规则：

- `场景施工稿` 是执行链节点，不是默认用户展示物。
- 用户确认的是蓝图和结果，不是内部工位。
- reviewer 默认不直接展示给用户；主控只摘译必要结论。
- 用户反馈回流必须进入主控归因，不得直接覆盖 writer。

## 7. 输入准入与用户权限

### 7.1 必填输入

需求确认必须覆盖以下项：

| 输入项 | 执行用途 | 未确认时处理 |
|---|---|---|
| 源本是什么 | 判断是否能进入正式写稿链 | 不能进入正式正文链 |
| 想换成什么新壳 | 决定题材、身份、世界观、场景重建方向 | 继续需求确认 |
| 改写力度与变量边界 | 决定贴近源本、等效迁移、自由改写、删除修复 | 继续需求确认 |
| 集数是否改变 | 决定留存节点和节奏是否重排 | 默认不能重排留存节点和节奏关键点 |
| 目标市场 | 决定常识、本地化和敏感设定 | 继续需求确认 |
| 输出语言 / 审稿语言 | 用户可见材料默认中文，正文语言必须确认 | 正文不能开写 |
| 正文用途 | 判断是用户/导演审稿，还是进入 AI 视频生成链路 | 正文合同不能确定 |
| 用户特别想保留什么 | 锁定用户主观偏好 | 记录为空，不得脑补 |
| 用户特别不想要什么 | 避免继承烂点或雷点 | 记录为空，不得脑补 |

### 7.2 额外材料接纳

可接纳材料：

- 角色设定；
- 梗概；
- 粗纲 / 分集大纲；
- 导演反馈；
- 市场偏好；
- 对标参考；
- 其他参考源本。

接纳规则：

- 主控先归类到需求层、边界层或蓝图层。
- 正式写稿链默认只接受一个主源本。
- 其他参考源本不进主链迁移，只能做补充偏好或风险提示。
- 任何额外材料不得直接塞给 writer。

### 7.3 用户权限

用户可以：

- 补充事实；
- 纠正系统理解；
- 对中间材料提修改意见；
- 覆盖系统建议，但必须显式确认风险。

用户不可以：

- 直接改系统生成的蓝图文本；
- 跳过蓝图确认直接进正式正文链；
- 让 reviewer 直接对话拍板。

执行含义：

- 用户提出修改时，主控必须判断修改属于事实纠正、偏好覆盖、上游变更还是局部返修。
- 用户覆盖系统建议时，必须记录损失和例外类型。

## 8. 用户可见交付合同

用户可见交付物固定分开：

| 交付物 | 可见性 | 作用 | 冻结属性 |
|---|---|---|---|
| 需求确认摘要 | 是 | 确认方向 | 需求层基准 |
| 源本有效性摘要 | 是 | 说明系统理解了源本为什么好看 | 用户确认后冻结 |
| 洗稿方案摘要 | 是 | 进蓝图前确认保什么、换什么、风险什么 | 用户确认后冻结 |
| 创作蓝图包 | 是 | 正式写正文前的用户检查点 | 用户确认后冻结 |
| 首批正文修订稿 | 是 | 用户真正判断成品质量 | 可按反馈返修 |

默认不展示：

- 故事控制包；
- 场景施工稿；
- 完整 reviewer 报告；
- 后台 run log。

如果用户主动要求看 reviewer 结论，只给精简摘要，不展示完整内部报告。

---

## 9. 创作蓝图包合同

创作蓝图包是正文前的正式用户检查点。它同时是用户可见交付物和 writer 主上游。

### 9.1 全剧粗层

必须包含：

- 一句话故事核；
- 核心角色设定；
- 全剧梗概；
- 阶段粗纲；
- 主追问。

全剧粗层必须足够支撑长篇多批次推进。
如果全剧粗纲比单集集纲还薄，视为不合格。

### 9.2 当前批细层

必须包含：

- 当前批每集戏稿；
- 每集追看债务；
- 关键戏发动方式；
- 角色本批状态线；
- 关系变化摘要；
- DNC 摘要；
- 禁用外形摘要。

每集戏稿必须已经像戏，不能只是标签或功能表。

合格的每集戏稿至少应让下游看懂：

- 本集发生什么具体人事；
- 谁逼谁行动；
- 本集兑现什么；
- 本集留下什么追问；
- 这集如何服务整批关系和状态变化。

不合格形态：

- 只写“hook / spike / button”；
- 只回答“为什么追、先兑什么、再欠什么、最强承载是什么”；
- 只给功能标签；
- 只给候选方案表；
- 把 proof-flow、gate、reviewer 语言塞进主文档；
- 需要 writer 回翻故事控制包才能知道剧情怎么写。

### 9.3 用户可局部打回

用户可以分别拍板或打回：

- 角色设定；
- 全剧粗纲；
- 当前批戏稿。

不要求整包全退。被打回模块及其直接依赖解冻，其他已确认模块默认保持冻结。

### 9.4 内部附录边界

DNC、禁用外形、proof-flow 风险、候选演法、质量 gate 可以存在，但只能作为内部执行附录或主控检查项。
除非用户主动要求，否则不得进入蓝图主文档的阅读主体。

---

## 10. 内部关键产物合同

内部主产物固定为三层。

### 10.1 故事控制包

只对内，承接：

- 真相账本；
- 角色错信；
- 源本强节点重排；
- 全剧阶段骨架；
- 批次地图；
- 每批追看债务；
- 反派 / 对立压力源；
- DNC；
- 新壳承载边界。

故事控制包不得直接给 writer 当正文输入。它要先被压成创作蓝图包和场景施工稿。

### 10.2 创作蓝图包

用户可见，也作为 writer 的主上游。
它是“用户能确认、下游能扩写”的创作文档，不是内部分析包。

### 10.3 场景施工稿

正文前最后一层。
只服务当前批 / 当前集 / 当前场景组。

writer 正文只能吃这一层，不回头翻厚材料。

如果 writer 需要回头翻故事控制包才能理解：

- 人物状态；
- 场面目标；
- 冲突走向；
- 本集追问；
- 禁用事项；

则失败归因到创作蓝图包或场景施工稿，不归因到 writer。

---

## 11. 版本冻结与回滚

形成冻结点的用户确认物：

- 源本有效性摘要；
- 洗稿方案摘要；
- 创作蓝图包。

默认规则：

- 已确认模块默认冻结。
- 只有上游变更才解冻。
- 不能静默覆盖。
- 如果只打回一个模块，默认最小解冻，只解冻该模块及其直接依赖。

蓝图确认后，如果用户大改以下项，默认回上游重开，并整份蓝图重新确认：

- 新壳；
- 集数；
- 改写力度；
- 核心关系；
- 核心设定。

执行系统必须在 run log 或产物头部记录：

- 哪些模块已确认；
- 哪些模块被冻结；
- 本轮是否解冻；
- 解冻原因；
- 解冻影响范围。

## 12. 失败类型与处理入口

| 失败类型 | 判定口径 | 处理入口 | 不允许的处理 |
|---|---|---|---|
| 洗飞 | 源本让人追下去的有效性丢了 | 回到源本有效性分析 | 只在正文补钩子 |
| 改名复刻 | 源本事件链和强桥段整串保留，只换身份和名词 | 回到洗稿边界和蓝图层 | 用“致敬源本”包装 |
| 短剧降级 | 事件变新了，但场面变弱、变冷、变文件对白 | 回到蓝图或场景施工层 | 只让 reviewer 扣分 |
| 正文长剧化 | 剧情逻辑对，但台词多、人物冷、站着聊天 | 回到场景施工层或正文层 | 只压缩字数 |
| 状态漂移 | 分批写作后，关系、误会、伏笔、人物状态接不上 | 回到连续性层 | 静默改前文 |

失败不能互相抵消。
例如：没有复刻但短剧降级，仍失败。

---

## 13. reviewer 合同

reviewer 默认每次首批都跑，但只兜底，不负责生产短剧感。

reviewer 只查：

- 洗飞；
- 复刻；
- 提前消费；
- 真相矛盾；
- 明显强度漏损；
- 状态漂移。

reviewer 不负责：

- 把短剧感做出来；
- 逐句重写；
- 对用户拍板。

如果用户反馈和 reviewer 冲突：

- 只要不碰红线或自相矛盾，用户目标优先；
- 主控负责归因和决定是否采纳 reviewer。

执行含义：

- reviewer 不能替上游补设计。
- reviewer 不能成为新增生产工位。
- reviewer 报告默认内部使用。
- 主控必须把 reviewer 结论翻译成“是否阻断、回哪层、是否继续”的决策。

## 14. 准入规则与例外模式

### 14.1 正式写稿链准入

正式写稿链默认只接受完整文本源本。

摘要、片段、口述、零散材料：

- 默认只能进诊断 / 蓝图模式；
- 不能直接进正式正文链。

### 14.2 坏源本

如果主控判断源本质量一般，或不适合直接洗：

- 默认转诊断 / 蓝图模式；
- 不直接承诺正式正文质量。

坏源本判定口径：

- 红旗清单；
- 主控综合判断。

红旗示例：

- 主追问弱；
- 强节点稀；
- 重复水戏多；
- 后半塌；
- 拉扯弱；
- 外部反馈差。

### 14.3 高风险改编例外模式

如果用户坚持超出洗稿器边界、明显更像原创改编的方向：

- 允许继续；
- 主控必须明确告诉用户这不是标准洗稿链；
- 主控必须明确质量保障下降；
- 本轮不计入默认 workflow 正向验证证据。

### 14.4 用户覆盖系统建议

如果主控明确判断用户坚持的改法会伤短剧感或洗飞源本：

- 允许继续；
- 必须显式写出会损失什么；
- 记录为“用户覆盖系统建议”的例外。

### 14.5 无源稿脚本医生模式

如果用户只给现成新稿，不给源本，却要求“改得更短剧”：

- 不进洗稿链；
- 只当脚本医生模式处理；
- 不计入默认 workflow 正向验证证据。

## 15. 关键质量门

这不是新增 reviewer，而是每个关键工位自己的内置责任。

关键层固定为：

- S4 创作蓝图包；
- S5 场景施工稿；
- S6 正文；
- S8 连续性 / 信息边界。

默认规则：

- 每个关键工位先自检自己过没过线。
- 主控只做 gate，检查是否真的过线、证据够不够、是否带病下传。
- 过线证据直接写在产物头部，不只写在 run log。
- 如果某层没过线，不允许继续往下。

最低硬门：

| 层级 | 必须证明 | 不过线时 |
|---|---|---|
| S4 创作蓝图包 | 用户能确认，writer 能作为主上游；每集戏稿像戏 | 回 S4，不进 S5 |
| S5 场景施工稿 | 当前批 / 当前集 / 当前场景组可直接扩成正文 | 回 S5 或 S4 |
| S6 正文 | 不长剧化，满足 page-to-screen 合同 | 回 S6 或 S5 |
| S8 连续性 / 信息边界 | 关系、误会、证据、伏笔、状态能接住后批 | 回 S8 或上游冻结点 |

## 16. 第一阶段成功标准

第一阶段成功不等于自动产出完整爆款。

成功标准固定为：

1. 用户需求被确认。
2. 源本有效性被对齐。
3. 洗稿方案摘要被确认。
4. 创作蓝图包被用户确认。
5. writer 不再拿任务补戏，而是拿场景施工稿写正文。
6. 高压节点由动作、空间、身体、物件、当众后果承载，而不是文件、会议、冷对白。
7. 每集有明确追看理由和结尾追问。
8. 正文经过台词压缩和视听 polish。
9. 状态追踪能支撑关系、误会、证据、伏笔连续。
10. 主创链先把明显失败处理掉，reviewer 没指出致命阻断。
11. 剧本正文满足 page-to-screen 合同。
12. 用户能判断：继续、调整后继续，或停止。

这些标准必须被拆进 workflow 的验收，不得只留在 PRD。

## 17. workflow 升默认门槛

新的候选 workflow 不能只跑一个样本就升默认。

默认门槛：

- 先跑飞书三星样本首批 `1-10` 集；
- 再跑一个明显不同题材、不同气质的样本首批 `1-10` 集；
- 第二个样本由主控按规则选；
- 两个样本都过，才允许升默认 workflow。

“过”的双门槛：

1. 整体显著强于 `V22 / V24 / V25 / V32`。
2. 追看强度至少保住 `V5`。

“保住 V5”不是逐集对齐，也不是照抄 V5，而是首批整体体验信号成立：

- 开场能抓人；
- 压力持续升级；
- 至少一次阶段释放或足够强悬置；
- 结尾让人想继续看；
- 高压节点主要靠可见承载；
- 人物拉扯 / 对立压力源不是平的。

这个体验门由主控结合样本对照来判，不由 reviewer 单独拍板。

no-ship gate：

- 主控执行；
- 只保留 4-6 条一票否决项；
- 来源只取历史复发根因 + 首批整体体验门；
- 不从零碎反馈里长；
- 一次性判定，不过就退回链路层，不在同一版里无限 review。

## 19. 产物流与读取边界

### 19.1 正向运行流

```text
需求确认摘要
-> 源本有效性摘要
-> 洗稿方案摘要
-> 故事控制包（内部）
-> 创作蓝图包（用户确认 + writer 主上游）
-> 场景施工稿（内部，正文前最后一层）
-> 正文
-> reviewer 轻版硬伤检查
-> 完整版交付
-> 用户反馈回流
```

### 19.2 读取边界

| 角色 / 环节 | 允许读取 | 禁止依赖 |
|---|---|---|
| 主控 | 全部产物、用户反馈、run log | 不能把内部工位直接展示成产品 |
| 故事控制包生成 | 需求、源本、洗稿方案 | 不能绕过源本有效性 |
| 创作蓝图包生成 | 故事控制包、用户确认边界 | 不能输出内部分析包替代蓝图 |
| 场景施工稿生成 | 已确认创作蓝图包 | 不能回头改蓝图冻结项 |
| writer | 场景施工稿 | 不能回头翻厚材料补戏 |
| reviewer | 正文、必要摘要、冻结点 | 不能生产短剧感或替用户拍板 |

### 19.3 执行 owner

| 责任区 | owner | 对外可见 | 完成条件 |
|---|---|---|---|
| 需求确认 | 主控 | 是 | 必填输入已确认，模式已选择 |
| 源本有效性摘要 | 主控 / 源本分析 | 是 | 用户能看懂系统认为源本为什么好看 |
| 洗稿方案摘要 | 主控 / 洗稿边界 | 是 | 用户能确认保什么、换什么、风险什么 |
| 故事控制包 | 内部主创链 | 否 | 真相、错信、强节点、批次、DNC、新壳边界被内部承接 |
| 创作蓝图包 | 蓝图层 | 是 | 用户能确认，writer 能作为主上游 |
| 场景施工稿 | 场景施工层 | 默认否 | writer 不回翻厚材料也能写正文 |
| 正文 | writer | 是 | 剧本文字满足当前批交付目标 |
| 连续性 / 信息边界 | 连续性层 | 否 | 状态、误会、证据、伏笔能接住后续 |
| reviewer 轻版硬伤检查 | reviewer | 默认否 | 只给主控兜底，不替上游补设计 |

## 20. workflow / skill_chain 迁移规则

后续清理 v19 / skill_chain 时，必须按以下顺序：

1. 先删或内收会污染用户可见产物的内部执行项。
2. 再修每个产物的输入、输出、读取边界。
3. 再修关键质量门。
4. 最后才跑样本。

必须删除或内收的污染项：

- 把创作蓝图包写成四问表；
- 把三档候选放进用户可见主文档；
- 把 proof-flow、gate、run log、reviewer checklist 放进蓝图主体；
- 让 writer 回翻故事控制包；
- 让 reviewer 负责补短剧感；
- 用固定外部比例覆盖用户确认边界和源本节奏；
- 让额外参考源本进入主链迁移。

必须保留的 PRD 约束：

- 用户可见交付分开；
- 创作蓝图包是用户确认点；
- writer 只吃场景施工稿；
- 已确认模块冻结；
- 例外模式不计入默认 workflow 正向验证；
- 两样本过线才升默认。

## 21. 静态验收检查

本 spec 通过后，workflow 清理前必须执行以下静态检查。

### 21.1 PRD 覆盖检查

- `PRD_v4` 1-19 节必须全部映射到本 spec。
- 每节必须能回答：落到哪个执行规则、哪个产物、哪个 gate。
- 第 19 节必须按当前项目状态解释，不能机械复刻“新建 v19”。

### 21.2 产物形态检查

- 创作蓝图包必须像用户和下游都能吃的创作文档。
- 当前批每集戏稿必须像戏，不是标签或功能表。
- 全剧粗层必须足以支撑长篇多批次，不得比单集更薄。
- 场景施工稿必须能让 writer 不回翻厚材料。

### 21.3 污染检查

用户可见主文档不得出现：

- run log；
- reviewer checklist；
- proof-flow 风险表；
- 三档候选表；
- gate 术语；
- 旧版本对比；
- 厚分析术语。

这些可以存在于内部附录或主控检查文件，但不能替代用户可见创作文档。

### 21.4 样本前置检查

只有当 v19 / skill_chain 已按本 spec 清理，并通过静态检查后，才允许进入样本。

样本顺序仍按 PRD：

1. 飞书三星样本首批 `1-10` 集；
2. 一个明显不同题材、不同气质的样本首批 `1-10` 集。
