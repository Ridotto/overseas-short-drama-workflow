# 海外短剧洗稿器 · 全量要求层级交接对账表

> 日期：2026-07-01  
> 状态：候选 workflow 的强制对账表，不是新规则库，不是 writer 输入。  
> 用途：防止“文档里有、实际没传下去”，也防止每次修一个点丢一批能力。

---

## 0. 这张表解决什么

过去反复失败，不是因为完全没有这些要求，而是：

```text
要求散在 PRD、设计提纲、旧 workflow、外部项目、样本反馈里；
有些在分析层，有些在写作层，有些只在 reviewer 层；
一跑起来就变成“某些能力生效，另一些能力被挤掉”。
```

所以这张表只做一件事：

```text
把所有核心要求逐项放回正确层，并确认它如何交给下游、如何在正文里落地、失败回哪层。
```

它不允许用“大部分已覆盖”这种说法。每一项只能是：

- 已接线：有归属层、有产物、有下游交接、有失败回滚层；
- 未接线：文档里提过，但没有进入当前产物链；
- 接错层：放在 reviewer / 自检 / run log 里，没进入写作前或正文；
- 待验证：文档已接线，但还没被样本证明有效。

## 1. 使用原则

### 1.1 这不是给 writer 的大表

writer 不直接读取这张表。

这里必须区分两件事：

- S5 施工稿工位：只读取 S4 场面承载原料包；S4 必须已经消化 S1-S3 的必要信息；
- S6 正文 writer：只读取 S5 施工稿，不再打开 S1-S4 原材料。

如果 S6 正文 writer 还要直接读源本拆文、洗稿边界、故事开发包、场面原料包、旧版对比或 reviewer report，就会重新退回“综合材料写证明题”的旧问题。

这张表给：

- 主控：确认能力没有丢；
- S1-S5：确认分析能力压成了写作材料；
- S8/S9/S10：确认失败能回到正确层；
- S11：确认本轮不能因为“run log 完整”就误判质量通过。

### 1.2 当前产品目标

所有要求都服务同一个目标：

```text
把已被市场验证或值得参考的短剧源本，
洗成新壳下仍然好看、能追、能付费的新剧本。
```

所以不能只问：

```text
流程跑完了吗？
```

必须问：

```text
流程跑完后，源本有效性、短剧手艺、新壳承载和正文质量是否都活到了最终剧本里？
```

合法的最终状态只有两种：

1. 剧本达到当前阶段可交付标准；
2. 在某一层被明确打回。

不允许出现：

```text
流程完整跑完，但剧本明显不够好，还被说成完成。
```

## 4. 环节边界硬线

### 4.1 分析层不能干写作层的活

S1/S2/S3/S4 可以给：

- 体验功能；
- 故事状态；
- 压力来源；
- 可见后果方向；
- 禁止外形；
- 不可提前消费信息。

不能给：

- 完整高压戏；
- 具体台词；
- 成片最后一帧；
- 源本强桥段整串替换版。

否则 writer 会退化成抄写和渲染。

### 4.2 正文 writer 只能吃施工稿

S5 是施工稿工位，负责把上游材料消化成可写的一集戏。

S6 是正文 writer，只负责把 S5 施工稿写成可看可听的正文。

S6 不直接读取：

- 源本拆文包；
- 洗稿边界包；
- 新本故事开发包；
- 场面承载原料包；
- story memory 全文；
- 外部标准库；
- reviewer report；
- 作者自检；
- run log；
- 旧版对比。

S6 只读取：

```text
S5 单集短剧施工稿
```

如果正文 writer 还需要自己回头综合 S1-S4，说明 S5 没把材料消化好，必须回 S5，不是让 S6 加读材料。

### 4.3 写作层不能干审计层的活

S5/S6/S7 要写戏，不要证明流程正确。

正文里禁止出现：

- 工位名；
- 规则名；
- 该段承担什么功能；
- 与源本的对应说明；
- “为了表现她崩溃”这类作者解释。

### 4.4 审计层不能干创作层的活

S8/S9/S10/S11 可以判断失败层和回滚层。

不能承担：

- 生产短剧感；
- 逐句改写；
- 为每个“还能更好”开返修；
- 把 reviewer 偏好变成新规则。

### 4.5 短剧手艺不能压过洗稿主线

开场、节奏、爽点、钩子、台词、视听都必须生效。

但它们必须服务：

```text
源本有效性迁移 + 新壳等效承载。
```

如果为了短剧感，源本追问、卡点、关系、DNC、真相路径被洗掉，就是失败。

## 5. 运行前强制检查

在用 v15 或任何后续候选 workflow 跑样本前，主控必须先做一次对账：

1. 本轮是否已确认 S0 的源本、新壳、市场、语言、集数、力度、禁区、正文用途；
2. S1 是否产出源本有效性，而不只是剧情摘要；
3. S2 是否明确 forbidden shape 和强继承体验；
4. S3 是否有真相账本、错信、阶段骨架、分集功能，且没有具体高压外形；
5. S4 是否只给原料，没有替 writer 排完整戏；
6. S5 是否把短剧手艺写成一拍一拍的施工稿；
7. S6/S7 是否把施工稿写成可看可听正文，而不是说明书；
8. S8 是否能拦住真相、错信、DNC、重复信息问题；
9. S9 是否能把硬风险打回对应层；
10. S10 是否保持干净 reviewer，不生产短剧感；
11. S11 是否做横向对比，而不是只看 run log。

如果任一项缺失，不能说“workflow 已经准备好完整跑”。

## 6. 当前状态判断

截至本表建立时：

```text
文档层级已经有基础，但此前缺一张全量对账表。
```

所以不能再说“所有要求都达成”。

更准确的状态是：

```text
核心要求已被逐项归位；
是否真实生效，必须通过完整样本运行和横向对比验证。
```

这张表之后，若样本仍然失败，不能笼统说“短剧感不够”。

必须指出：

- 哪一项要求没传下去；
- 是哪一层产物不干净；
- 是 writer 没吃到；
- 是正文没落地；
- 还是检查层软放行。
