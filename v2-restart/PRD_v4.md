## 1. 文档定位

这份 PRD 只回答产品定义，不承担执行模板职责。

它负责定义：

- 产品是什么；
- 用户怎么和产品交互；
- 用户什么时候确认、确认什么；
- 内部生产链的大方向；
- 什么情况算失败；
- 什么情况才算通过。

它不负责：

- 具体字段模板；
- 每个工位的长表；
- 样本运行记录；
- agent 提示词；
- 工程实现细节。

详细执行合同放在候选 workflow spec 和 skill chain spec 里。

---

## 2. 产品一句话

把一个已被市场验证或值得参考的短剧源本，洗成一个新壳下仍然好看、能追、能付费的新剧本。

产品交付的是剧本文字，不是分析报告。

核心判断：

```text
新本没有洗飞源本有效性；
新本不是源本改名复刻；
新本作为一个新短剧自己站得住。
```

---

## 3. 产品不是

- 不是评分器。
- 不是从零原创工具。
- 不是短剧标准库套模板工具。
- 不是查词替换式换皮工具。
- 不是把导演或用户反馈拆成一百条规则的工具。
- 不是视频、分镜、投流片段或成片工具。
- 不是一键生成完整长剧的黑箱工具。
- 不是把多 agent 过程展示给用户看的产品。

---

## 4. 目标用户与场景

目标用户是写短剧本的人：用户本人，或团队里的编剧。

典型场景：

- 用户手上有一部爆款或可参考的短剧源本，想把它改成另一个题材、身份、背景、世界观或集数版本。
- 用户不想从零原创，也不想让大模型把源本好看的东西洗丢。
- 用户可能很清楚方向，例如“把黑手党复仇洗成豪门商战”；也可能只有模糊想法，需要先对齐。
- 用户有时会带自备材料进来，例如角色设定、梗概、粗纲、分集大纲、导演反馈、市场偏好、对标参考。

---

## 5. 产品形态路线

### 第一阶段：skill / workflow 原型

第一阶段不先做网页 App，不先写工程 spec。

先把创作流程做成可重复执行的 skill / workflow，用真实样本验证它能不能稳定产出可用剧本。

原因：

- 当前最大风险是剧本质量，不是界面。
- 过早做网页 App 容易反过来把创作流程写死。
- 现在需要先稳定“源本分析 -> 创作蓝图 -> 场景施工 -> 正文”的主创链。

### 中期：轻量工具化

流程稳定后，再把用户入口工具化，固定：

1. 上传或选择源本；
2. 需求对齐；
3. 输出源本有效性摘要；
4. 输出洗稿方案摘要；
5. 输出创作蓝图包；
6. 输出首批正文修订稿；
7. 收集用户反馈；
8. 分批继续写。

### 长期：网页 App

网页 App 是最终用户入口，但不是创作能力本身。

网页 App 负责：

- 上传源本；
- 确认新壳、集数、改写力度、语言；
- 展示用户可见交付；
- 管理版本、批次和反馈；
- 让用户在正确节点确认，不让用户看到内部工位细节。

## 6. 四种模式

产品固定支持四种模式：

### 6.1 诊断 / 蓝图模式

只跑到创作蓝图包，不进正文。

适用：

- 用户先想判断源本值不值得洗；
- 用户只想先确认方向、角色、粗纲和当前批戏稿；
- 输入不是完整文本源本，只是摘要、片段、零散材料；
- 系统和用户对“源本为什么好看”还没对齐。

### 6.2 标准首批模式

默认模式。

跑完整链路到首批 `1-10` 集正文。

### 6.3 长篇多批次模式

全剧粗、当前批细，批次滚动推进。

适用：

- 用户要 100 分钟、100 集或多批次交付；
- 不能一次性写完整本；
- 每一批都需要重新确认当前批细层蓝图。

### 6.4 局部改稿 / 定点返修模式

只接受已有项目上下文。

适用：

- 用户只想修某几集；
- 只想改某个卡点；
- 只想重写某段场面；
- 已有蓝图和正文基础，不需要重跑整条正式链。

这条模式不是默认 workflow 的正向验证证据。

默认仍以标准首批模式为主。  
主控按用户需求选模式，不临场发明新模式。

## 7. 用户可见主链

用户只和主控 agent 对话。

默认主链：

```text
需求确认
-> 源本有效性摘要 + 洗稿方案摘要（一次前置确认）
-> 创作蓝图包（用户确认）
-> 场景施工稿
-> 正文
-> reviewer 轻版硬伤检查
-> 完整版交付
-> 用户反馈回流
```

关键原则：

- 用户先确认方向，再看正文。
- 用户确认的是蓝图和结果，不是内部工位。
- reviewer 默认不直接对用户展示，只由主控摘译必要结论。

---

## 8. 用户输入与权限

### 8.1 必填输入

需求对齐必须问清：

| 问题 | 作用 |
|---|---|
| 源本是什么 | 决定是否能进入正式写稿链 |
| 想换成什么新壳 | 决定题材、身份、世界观、场景怎么重建 |
| 改写力度与变量边界 | 决定每类变量是贴近源本、等效迁移、自由改写还是删除修复 |
| 集数是否改变 | 决定留存节点和节奏是否重排 |
| 目标市场 | 决定常识、本地化和敏感设定 |
| 输出语言 / 审稿语言 | 用户可见材料默认中文；正文语言必须确认 |
| 正文用途 | 用户/导演审稿，还是进入 AI 视频生成链路 |
| 用户特别想保留什么 | 锁定用户主观偏好 |
| 用户特别不想要什么 | 避免继承源本烂点或用户雷点 |

### 8.2 正式接纳的额外输入材料

用户可以额外提供：

- 角色设定；
- 梗概；
- 粗纲 / 分集大纲；
- 导演反馈；
- 市场偏好；
- 对标参考；
- 其他参考源本。

这些材料不能直接塞给 writer。  
主控必须先把它们归到需求层、边界层或蓝图层，再决定是否吸收。

正式写稿链默认只接受一个主源本。  
其他参考源本只能作为补充偏好或风险提示，不进主链迁移。

### 8.3 用户权限

用户可以：

- 补充事实；
- 纠正系统理解；
- 对中间材料提修改意见；
- 覆盖系统建议，但必须显式确认风险。

用户不可以：

- 直接改系统生成的蓝图文本；
- 跳过蓝图确认直接进正式正文链；
- 让 reviewer 直接对话拍板。

---

## 9. 用户可见交付

用户可见对象固定分开：

| 交付物 | 是否可见 | 作用 |
|---|---|---|
| 需求确认摘要 | 是 | 确认方向 |
| 源本有效性摘要 | 是 | 说明系统理解了源本为什么好看 |
| 洗稿方案摘要 | 是 | 在进蓝图前确认保什么、换什么、风险什么 |
| 创作蓝图包 | 是 | 正式写正文前的用户检查点 |
| 首批正文修订稿 | 是 | 用户真正判断成品质量 |

默认不展示：

- 故事控制包；
- 场景施工稿；
- 完整 reviewer 报告；
- 后台 run log。

如果用户主动要求看 reviewer 结论，主控只给精简摘要，不展示完整内部报告。

---

## 10. 创作蓝图包

创作蓝图包是正文前的正式用户检查点。

它默认分两层：

### 10.1 全剧粗层

- 一句话故事核；
- 核心角色设定；
- 全剧梗概；
- 阶段粗纲；
- 主追问。

### 10.2 当前批细层

- 当前批每集戏稿；
- 每集追看债务；
- 关键戏发动方式；
- 角色本批状态线；
- 关系变化摘要；
- DNC 摘要；
- 禁用外形摘要。

蓝图里的“每集戏稿”必须已经像戏，不能只是标签或功能表。

用户可对以下模块分别拍板或打回：

- 角色设定；
- 全剧粗纲；
- 当前批戏稿。

不要求整包全退。

---

## 11. 内部关键产物

内部主产物固定为三层：

### 11.1 故事控制包

只对内，承接：

- 真相账本；
- 角色错信；
- 源本强节点重排；
- 全剧阶段骨架；
- 批次地图；
- 每批追看债务；
- 反派 / 对立压力源；
- DNC；
- 新壳承载边界。

### 11.2 创作蓝图包

用户可见，也作为 writer 的主上游。

### 11.3 场景施工稿

正文前最后一层。  
只服务当前批 / 当前集 / 当前场景组。

writer 正文只能吃这一层，不回头翻厚材料。

---

## 12. 版本冻结与回滚

一旦用户确认：

- `源本有效性摘要`
- `洗稿方案摘要`
- `创作蓝图包`

就形成明确版本冻结点。

默认规则：

- 已确认模块默认冻结；
- 只有上游变更才解冻；
- 不能静默覆盖；
- 如果只打回一个模块，默认最小解冻，只解冻该模块及其直接依赖。

如果用户在蓝图确认后大改：

- 新壳；
- 集数；
- 改写力度；
- 核心关系；
- 核心设定；

则默认回上游重开，并整份蓝图重新确认。

---

## 13. 失败类型

| 失败类型 | 含义 | 处理入口 |
|---|---|---|
| 洗飞 | 源本让人追下去的有效性丢了 | 回到源本有效性分析 |
| 改名复刻 | 源本事件链和强桥段整串保留，只换身份和名词 | 回到洗稿边界和蓝图层 |
| 短剧降级 | 事件变新了，但场面变弱、变冷、变文件对白 | 回到蓝图或场景施工层 |
| 正文长剧化 | 剧情逻辑对，但台词多、人物冷、站着聊天 | 回到场景施工层或正文层 |
| 状态漂移 | 分批写作后，关系、误会、伏笔、人物状态接不上 | 回到连续性层 |

这些失败不能互相抵消。

---

## 14. reviewer 定位

reviewer 默认每次首批都跑，但只兜底，不负责生产短剧感。

reviewer 只查：

- 洗飞；
- 复刻；
- 提前消费；
- 真相矛盾；
- 明显强度漏损；
- 状态漂移。

reviewer 不负责：

- 把短剧感做出来；
- 逐句重写；
- 对用户拍板。

如果用户反馈和 reviewer 冲突：

- 只要不碰红线或自相矛盾，用户目标优先；
- 主控负责归因和决定是否采纳 reviewer。

## 15. 准入规则与例外模式

### 15.1 正式写稿链准入

正式写稿链默认只接受完整文本源本。

摘要、片段、口述、零散材料：

- 默认只能进诊断 / 蓝图模式；
- 不能直接进正式正文链。

### 15.2 坏源本

如果主控判断源本本身质量一般，或不适合直接洗：

- 默认转诊断 / 蓝图模式；
- 不直接承诺正式正文质量。

坏源本判定口径采用：

- 红旗清单 + 主控综合判断

红旗示例：

- 主追问弱；
- 强节点稀；
- 重复水戏多；
- 后半塌；
- 拉扯弱；
- 外部反馈差。

### 15.3 高风险改编例外模式

如果用户坚持超出洗稿器边界、明显更像原创改编的方向：

- 允许继续；
- 但主控必须明确告诉用户：
  - 这不是标准洗稿链；
  - 质量保障下降；
  - 不计入默认 workflow 的正向验证证据。

### 15.4 用户覆盖系统建议

如果主控明确判断用户坚持的改法会伤短剧感或洗飞源本：

- 允许继续；
- 但必须显式写出会损失什么；
- 记录为“用户覆盖系统建议”的例外。

### 15.5 无源稿脚本医生模式

如果用户只给现成新稿，不给源本，却要求“改得更短剧”：

- 不进洗稿链；
- 只当脚本医生模式处理；
- 不计入默认 workflow 的正向验证证据。

## 16. 关键质量门

这不是新增 reviewer，而是每个关键工位自己的内置责任。

关键层：

- S4 创作蓝图包
- S5 场景施工稿
- S6 正文
- S8 连续性 / 信息边界

默认规则：

- 每个关键工位先自检自己过没过线；
- 主控只做 gate，检查是否真的过线、证据够不够、是否带病下传；
- 过线证据直接写在产物头部，不只写在 `run log`。

如果某层没过线，不允许继续往下。

## 17. 成功标准

第一阶段成功，不等于已经能自动产出完整爆款。

第一阶段成功是：

1. 用户需求被确认。
2. 源本有效性被对齐。
3. 洗稿方案摘要被确认。
4. 创作蓝图包被用户确认。
5. writer 不再拿任务补戏，而是拿场景施工稿写正文。
6. 高压节点由动作、空间、身体、物件、当众后果承载，而不是文件、会议、冷对白。
7. 每集有明确追看理由和结尾追问。
8. 正文经过台词压缩和视听 polish。
9. 状态追踪能支撑关系、误会、证据、伏笔连续。
10. 主创链先把明显失败处理掉，reviewer 没指出致命阻断。
11. 剧本正文满足 page-to-screen 合同。
12. 用户能判断：继续、调整后继续，或停止。

## 18. workflow 升默认门槛

新的候选 workflow 不能只跑一个样本就升默认。

默认门槛：

- 先跑飞书三星样本首批 `1-10` 集；
- 再跑一个明显不同题材、不同气质的样本首批 `1-10` 集；
- 第二个样本由主控按规则选；
- 两个样本都过，才允许升默认 workflow。

“过”的双门槛：

1. 整体显著强于 `V22 / V24 / V25 / V32`
2. 追看强度至少保住 `V5`

这里的“保住 V5”不是逐集对齐，也不是照抄 V5，而是看首批整体体验信号是否成立：

- 开场能抓人；
- 压力持续升级；
- 至少一次阶段释放或足够强悬置；
- 结尾让人想继续看；
- 高压节点主要靠可见承载；
- 人物拉扯 / 对立压力源不是平的。

这个体验门由主控结合样本对照来判，不由 reviewer 单独拍板。

no-ship gate 规则：

- 主控执行；
- 只保留 4-6 条一票否决项；
- 来源只取历史复发根因 + 首批整体体验门；
- 不从零碎反馈里长；
- 一次性判定，不过就退回链路层，不在同一版里无限 review。
