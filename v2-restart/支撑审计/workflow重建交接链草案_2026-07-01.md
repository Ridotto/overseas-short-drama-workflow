# workflow 重建交接链草案

> 日期：2026-07-01  
> 当前 stage：交接链设计  
> 上游：`全版本能力回收总账_2026-07-01.md`、`能力归层与交接矩阵_2026-07-01.md`  
> 用途：为下一版正式候选 workflow spec 提供链路草案。  
> 注意：这不是正式 workflow spec，不能直接用来声称“按 workflow 跑完”。

---

## 0. 这份草案解决什么

之前失败的共同点不是“缺文档”，而是：

```text
分析、短剧手艺、外部项目、writer、reviewer 都存在，
但中间产物没有稳定把能力交给下一层。
```

新版 workflow 应该先解决交接，而不是继续增加规则。

---

## 1. 产品运行形态

产品第一阶段仍然是：

```text
主控 agent + 主创链 + skill 工位 + 干净 reviewer
```

不是网页 App，不是工程实现，不是每一步固定一个长期 agent。

### 1.1 用户视角

用户只和主控 agent 对话：

- 上传源本；
- 说明新壳、集数、语言、市场、改写力度；
- 给导演反馈或修改要求；
- 收到最终剧本和必要说明。

### 1.2 内部视角

主控负责：

- 判断当前 stage；
- 锁定目标和边界；
- 选择 workflow；
- 建 run log；
- 调用主创链；
- 调用干净 reviewer；
- 归因返修层；
- 交付。

主创链负责：

```text
S1 源本拆文
-> S2 洗稿边界
-> S3 新本故事架构
-> S4 场面承载原料
-> S5 writer 施工稿
-> S6 正文
-> S7 polish
-> S8 continuity
-> S9 作者自检
```

同一个主创 agent 可以连续执行这些工位，但每一步必须显式切换 skill / 工位口径。

reviewer 负责：

- 干净隔离；
- 第一轮不读作者自检；
- 查洗飞、复刻、提前消费、真相矛盾、明显强度漏损；
- 给回滚层和 suggestion；
- 不生产短剧感。

## 2. 主交接链

| Step | 工位 | 输入 | 输出 | 进入下一步前必须成立 |
|---|---|---|---|---|
| 0 | 主控接需求 | 用户需求、源本入口 | `最小需求确认` | 源本、新壳、市场、语言、集数、力度、禁区、正文用途已确认或有默认说明 |
| 1 | S1 源本拆文 | `最小需求确认 + 源本` | `源本有效性包` | 不是剧情摘要；能说明为什么能追、能付费 |
| 2 | S2 洗稿边界 | `源本有效性包` | `洗稿边界包` | 明确强继承体验、可等效迁移、自由变量、必须删除和 forbidden shape |
| 3 | S3 新本故事架构 | `洗稿边界包` | `新本故事开发包` | 有故事核、真相账本、错信、阶段骨架、分集功能、角色行动方式；没有具体高压外形 |
| 4 | S4 场面承载原料 | `新本故事开发包` | `场面承载原料包` | 只给原料，不排完整戏；有压力、被迫行动、可见后果、DNC |
| 5 | S5 writer 施工稿 | `场面承载原料包` | `单集短剧施工稿` | 像戏的粗稿，不像任务单；短剧手艺在写稿当下生效 |
| 6 | S6 screenplay 正文 | `单集短剧施工稿` | `screenplay 正文初稿` | 可看可听，page-to-screen，没有内部标签和说明书感 |
| 7 | S7 polish | `screenplay 正文初稿` | `台词视听 polish 稿` | 台词有功能、角色声线清楚、动作/声音/身体/空间承载情绪 |
| 8 | S8 continuity | `台词视听 polish 稿` | `story memory checkpoint` | 真相、错信、DNC、伏笔、状态不冲突 |
| 9 | S9 作者自检 | `polish 稿 + checkpoint` | `作者自检与返修路由` | 硬风险必须打回对应层，不能交给 reviewer 兜底 |
| 10 | S10 干净 reviewer | 需求、边界摘要、最终稿、必要源本片段 | `reviewer report` | 独立审；只给问题、严重度、回滚层、suggestion |
| 11 | S11 主控拍板 | 最终稿、自检、reviewer、run log、版本对比 | `主控版本对比与拍板` | 不把 run log 当质量证明；做全维度横向对比 |

## 3. 每个工位的核心动作

### S0 需求确认

必须确认：

- 源本；
- 新壳；
- 目标市场；
- 输出语言 / 审稿语言；
- 集数是否改变；
- 改写力度；
- 用户禁区；
- 正文用途：人审稿、AI 视频生成、字幕版、双语版等。

S0 不写剧情。

### S1 源本拆文

必须拆：

- 核心期待；
- 留存骨架；
- 钩子 / 卡点；
- 付费节点；
- 爽点和压抑释放；
- 情绪 / 信息 / 情节密度；
- 人物关系和拉扯；
- 信息差 / 真相揭露；
- 反派压力；
- DNC；
- 水分 / 烂点；
- 源本高辨识外形。

S1 不设计新壳。

### S2 洗稿边界

必须输出：

- 强继承体验；
- 等效迁移体验；
- 自由改写变量；
- 必须删除 / 修复；
- forbidden shape；
- 新壳不能承载的源本外形。

S2 不写新剧情。

### S3 新本故事架构

必须输出：

- 新本故事核；
- 题材 / 世界观边界；
- 真相账本；
- 角色错信；
- 主追问和阶段追问；
- 阶段骨架；
- 分集功能；
- 人物行动方式；
- 反派压力与反扑燃料；
- DNC 排程。

S3 不写高压场完整外形。

### S4 场面承载原料

必须输出：

- 这场承载源本哪个体验功能；
- 新壳里靠什么权力、关系、空间、物件或行动承载；
- 谁压谁；
- 谁旁观；
- 谁被迫行动；
- 压力如何形成；
- 释放后造成什么可见后果；
- 禁止外形；
- DNC；
- 可选动作方向。

S4 不排完整戏。

### S5 writer 施工稿

必须把 S4 原料排成单集戏：

- 本集观众正在等什么；
- 开场第一个可见动作/声音/冲突；
- 中段如何升压；
- 爽点前如何压低；
- 谁行动，谁被迫反应，谁失去什么；
- 关系/权力/信息如何变化；
- 结尾停在哪个未完成动作/声音/画面；
- 本集不能提前消费什么；
- 禁止复用什么源本外形；
- 本集短剧手艺怎么用。

S5 不是检查表，不是审计报告。

### S6 screenplay 正文

必须写成：

- 剧本页上的内容基本就是最终成片能看见、听见的东西；
- 人物通过动作、身体、空间、声音、物件和当众后果表达情绪；
- 台词短、准、有功能；
- 每集结尾卡在答案前一拍。

S6 不回头读 S1-S4。

### S7 台词 / 视听 polish

只修表达：

- 删水词；
- 删解释腔；
- 强化角色声线；
- 把可视听表达的内容从对白改成动作、声音、身体、空间；
- 确保台词承担攻击、防守、逼迫、拒绝、试探、威胁、揭露、反击或选择。

S7 不重设主结构。

### S8 连续性 / 信息边界

必须查：

- 真相账本；
- 角色错信；
- 已播 / 未播；
- DNC；
- 伏笔；
- 关系 / 权力状态；
- 同一信息是否重复当主刀。

S8 不评价审美。

### S9 作者自检

必须判断：

- 是否洗飞；
- 是否复刻；
- 是否短剧降级；
- 是否正文长剧化；
- 是否状态漂移；
- 是否提前消费；
- 失败应回哪层。

S9 的输出本质上是 `rewrite report / 返修诊断`：

- 先判失败层；
- 再说明为什么必须返修；
- 再写返修时必须保留的源本体验、新壳设定和短剧手艺；
- 最后写禁止破坏的内容。

S9 不能把硬风险写成“看主控偏好”，也不能直接给正文打逐句补丁。

### S10 干净 reviewer

必须保持降权：

- 不负责把剧写好；
- 不逐句代写；
- 不读作者自检做第一轮；
- 不无限挑刺；
- issue 必须带回滚层和 suggestion。

### S11 主控拍板

必须做：

- 主创自检 vs reviewer 对照；
- 与 V5 / V22 / V24 / V25 / 上一版对比；
- 判断是否显著提升；
- 判断是否可交付；
- 判断用户 / 导演反馈回哪层。

## 4. 外部项目接入方式

0xsline / short-drama 不照搬，只接三件事：

1. 分阶段加载参考；
2. 每个命令 / skill 有清楚输入输出；
3. 短剧手艺在写作时生效，而不是只在 reviewer 里出现。

| 外部能力 | 接入位置 | 变成什么生产动作 |
|---|---|---|
| `genre-guide.md` | S0/S3/S7 | 题材承诺、市场语感、角色说话方式 |
| `opening-rules.md` | S5/S6 | 开场第一个可见动作/声音/冲突 |
| `rhythm-curve.md` | S3/S5/S8 | 阶段压放、单集节奏、重复风险 |
| `hook-design.md` | S1/S3/S5/S10 | Hook Type、未回答问题、结尾卡住点 |
| `paywall-design.md` | S1/S3/S5/S11 | 付费追问、兑现义务、不可提前消费 |
| `satisfaction-matrix.md` | S1/S4/S5/S10 | 爽点类型、压低动作、释放动作 |
| `villain-design.md` | S1/S3/S5/S8 | 反派资源、误判、反扑燃料 |
| `compliance-checklist.md` | S0/S10/S11 | 红线辅助，不做创作发动机 |
| `rewrite report` / 返修诊断 | S9/S11 | 失败归层、保留项、禁破坏项和回滚建议 |

## 5. 回滚规则

| 失败 | 回滚层 |
|---|---|
| 目标市场 / 语言 / 集数 / 用途不清 | S0 |
| 源本追看功能没拆出来 | S1 |
| 防同构压过防洗飞 | S2 |
| 主追问、真相账本、错信、阶段骨架不清 | S3 |
| 新壳承载弱、只有证据 / 文件 / 系统 / 会议 | S4 |
| 施工稿像任务单，没有行动链和压放 | S5 |
| 正文像说明书、站桩聊天、内部标签、不可视听 | S6 |
| 台词水、同声、解释腔 | S7 |
| 真相矛盾、DNC 失效、重复信息主刀、状态漂移 | S8 |
| 作者把硬风险写成偏好 | S9 |
| reviewer 越权生产短剧感或无限挑刺 | S10 |
| 只看 run log / 只看 reviewer / 只和烂版比 | S11 |

## 6. 新版 workflow spec 的写作要求

下一步若写正式候选 workflow spec，必须：

1. 以这份交接链为骨架；
2. 合并 `全版本能力回收总账` 和 `能力归层与交接矩阵`；
3. 不复活旧设计提纲规则池；
4. 不把 0xsline 原创流程照搬；
5. 不让 writer 读厚材料；
6. 不让 reviewer 变质量发动机；
7. 明确每层输入、输出、禁区、回滚；
8. 跑样本时用 run log 记录执行，但用剧本质量和横向对比判断结果。

---

## 7. 验证计划

正式候选 workflow spec 写完后，再跑同一飞书三星样本首批 1-10 集。

对比：

- V5：短剧感和场面发动机正向证据；
- V22：较稳但偏平；
- V24：完整链路但 writer 渲染感明显；
- V25：skill 工位链方向有效但未达标；
- 新版样本。

验收看：

1. 是否没有洗飞源本；
2. 是否没有改名复刻；
3. 是否高压节点像戏，不像报告；
4. 是否保住 V5 的场面发动机，但不复刻 V5；
5. 是否比 V25 更少旧外形回流；
6. 是否钩子、节奏、爽点、人物拉扯、反派压力、DNC、视听表达都进入正文；
7. 是否 reviewer 降权；
8. 是否主控能正确判断回滚层；
9. 是否最终产物显著强于 V22/V24/V25，并至少不低于 V5 的追看强度。
