# 海外短剧洗稿器 · 机制重构根因与改法

> 日期：2026-06-30  
> 状态：机制层 checkpoint，不是执行模板  
> 目的：把 V1-V13 暴露的问题从样本症状里抽出来，避免继续在正文层补丁式修稿。

---

## 1. 最底层目标

本项目不是为了把某一个测试剧本修到最好。

本项目要做的是：

**把一个已经有效或值得参考的短剧源本，洗成新壳下仍然好看、能追、能付费的新剧本。**

所以 workflow 的核心不是“证明每个源本功能都迁移了”，而是：

```text
读懂源本为什么好看
-> 抓住源本让用户继续付费的留存锚点
-> 找到新壳下能承载同一体验的人物冲突
-> 让新剧本从人物欲望、阻碍、误判、选择、代价里长出来
-> 再独立检查有没有洗飞、同构、降级
```

这里的“留存锚点”包括：

```text
核心期待链；
钩子和卡点；
情绪密度和情绪峰谷；
爽点压抑-释放；
关系拉扯；
信息差和揭露节奏；
付费前后的追问惯性；
全剧/阶段发展节奏。
```

v4 不是从零原创写作法。  
v4 的“故事状态”和“压力链”只解决一件事：**把源本有效性换到新壳里重新长出来。**

---

## 2. 已确认症状

V3 到 V13 的测试暴露了几类重复症状：

1. 高压场容易滑成文件、录音、屏幕、key、授权、董事会、系统确认；
2. 人物虽然有动作，但动作像“完成流程”，不是“互相逼迫”；
3. 台词短了，但人物仍然冷，像成年人谈判，不像短剧强冲突；
4. 视听语言补上后，常变成贴血、手抖、跪下，而不是戏本身变强；
5. 返修越来越像加料，不像重写，产物越来越像流程说明；
6. manifest / pass / reviewer pass 容易被误解成“剧本质量已过”；
7. 分析层读到了情绪、节奏、钩子、拉扯，但正文层没有稳定承接。

这些不是彼此独立的问题。

---

## 3. 根因

根因不是“缺视听语言”，也不是“缺情绪描写”，更不是“缺原创能力”。

更准确是：

**源本里活的强戏和留存锚点，在中间交接时被压成了功能、结果、证据和卡点；writer 拿到的是任务单，不是源本留存锚点、故事当前状态和压力链。**

于是 writer 会自然选择最安全、最容易过检的写法：

```text
要揭露真相 -> 放文件 / 录音 / 屏幕
要权力变化 -> 放授权 / key / 董事会 / 系统
要情绪外化 -> 补血 / 手抖 / 跪下
要证明没洗飞 -> 回指源本功能编号
```

这些写法能通过“有没有动作、有没有信息、有没有权力变化”的检查，但不一定像短剧强戏。

---

## 4. 层级错位

当前问题主要出在四个交接层。

### 4.1 分析层到创作层

分析层能读出源本的钩子、爽点、节奏、情绪、拉扯。

但下传给 writer 时，常变成：

```text
本集要揭露 X
本集要打脸 Y
本集要让 A 追悔
本集要卡在 Z
```

这会诱导结果式写作。

正确下传应该是：

```text
这个故事现在活到哪儿了？
谁想要什么？
谁挡住谁？
谁误判谁？
压力如何逼到不得不选择？
这个选择让谁立刻失去什么？
观众此刻等的是什么？
```

### 4.2 Story Memory 位置

当前 workflow 里已经有 Story Memory / 状态追踪。

但它偏向写完后记录：

```text
关系状态
信息差状态
伏笔 / 证据状态
角色状态
```

这只能防穿帮，不能稳定提升写作。

Story Memory 必须拆成两种用途：

```text
写前：Story Operating State，告诉 writer 故事当前怎么活着；
写后：Story Memory Checkpoint，保存本批写完后的状态变化。
```

### 4.3 Writer Brief

旧 brief 的问题不是字段少，而是容易结果导向。

它会告诉 writer：

```text
最大刺激点是什么；
结尾停在哪；
不能提前消费什么。
```

这些都需要，但还不够。

高压场真正需要的是压力链：

```text
起压 -> 阻挡 -> 加码 -> 误判/反击 -> 被迫选择 -> 可见代价 -> 卡点
```

没有这条链，writer 会把“爆点”安排出来，而不是让爆点从人物冲突里炸出来。

### 4.4 Reviewer Note 到返修

V13 证明 v3 草案方向有正向效果。

但 V14 局部修稿又暴露出另一个断点：

reviewer 说“这个揭露太方便 / 这个文件太纸面 / 这个场面太弱”之后，如果没有返修 brief，writer 会直接换道具、换信息源、补动作。

这仍然是补丁。

返修前必须先把 reviewer note 改写成新的压力链，而不是直接改正文。

## 5. 现成项目里能直接用什么

这些不是从零发明，现成资产里已经有可用部件。

| 来源 | 可用部分 | 应该怎么用 |
|---|---|---|
| `oh-story` | 情节节点、角色状态、伏笔、上下文追踪 | 用来做 Source Bible 和 Story Operating State |
| `short-drama` | 钩子、爽点、节奏、分集功能 | 用来判断留存和情绪体验，不搬固定比例 |
| `how-to-make-script` | scene_draft、screenplay_draft、rewrite_report、dialogue_polish、quality gate | 用产物合同和失败层诊断，不把标签写进正文 |
| `workflow_spec_v2` | 动作链、压力推进、人物被迫行动链 | 保留思想，避免变成大表 |
| `workflow_spec_v3` | 分析/创作/审计/治理拆层，高压节点粗验，薄 writer brief | 作为 v4 基础 |

重要的是接法：

**这些资产不能再被接成证明链，要接成创作前的故事状态和压力链。**

## 6. 底层改法

### 6.0 先锁源本留存锚点

写新本之前必须先压出一份薄的源本留存锚点。

它不是源本剧情摘要，也不是完整 source ref。

它只回答：

```text
源本靠什么让用户一直想看下去？
哪些钩子 / 卡点 / 追问必须等效迁移？
哪些情绪密度和节奏惯性不能丢？
哪些爽点必须先压后爆？
哪些关系拉扯和信息差是付费动力？
```

后面的 Story Operating State 和压力链都必须服务这些锚点。

如果压力链写得很顺，但丢了源本的追问、卡点、情绪密度或爽点节奏，那不是成功改写，而是写成了另一个原创故事。

### 6.1 不再把“有无”当核心

不要只问：

```text
有没有钩子？
有没有动作？
有没有视听语言？
有没有情绪？
有没有权力变化？
```

这些容易变成补丁。

要先问：

```text
这一场戏是被谁的欲望和谁的阻碍逼出来的？
爆点是不是人物不得不做的选择造成的？
如果拿掉文件、屏幕、系统，这场戏还剩下人和人的冲突吗？
这场结束，谁立刻失去东西，谁获得主动权？
```

### 6.2 Writer 写正文前必须先写压力链

高压场不能直接进正文。

先写：

```text
谁先施压：
对方怎么挡 / 逃 / 骗：
压力怎么加码：
谁被迫做不可逆选择：
爆点出现后谁立刻失去什么：
卡点停在哪个未完成动作 / 选择 / 后果前：
```

正文只服务这条链。

### 6.3 Story Memory 前移

每批写作前，writer 应该拿到薄的当前状态：

```text
主要人物现在想要什么：
主要人物现在怕什么：
关系现在卡在哪里：
谁掌握主动权：
观众正在等什么：
本批必须推进什么：
本批不能提前消费什么：
```

这不是大表，是写戏前的坐标。

### 6.4 返修先改压力链，再改正文

如果 reviewer 指出局部问题，不能直接正文补丁。

必须先写：

```text
原场问题属于哪层：
原场为什么是结果式/证明式：
新的压力链是什么：
需要删掉什么：
哪些旧道具/证据只能保留为辅助：
```

然后才改正文。

---

## 7. 不该做什么

不要继续：

- 为每个导演反馈加一条规则；
- 把视听语言、情绪、拉扯感、台词都做成单独大表；
- 把 Story Memory 做成 20 列状态表；
- 让 writer 看完整 source ref、manifest、quality gate、版本对比；
- 用 manifest pass 证明剧本质量；
- 把返修当成在旧稿上加血、加动作、加台词。

---

## 8. 下一步应该怎么做

下一步不是继续修 V14，也不是直接跑完整样本。

先把 workflow 草案改成：

```text
Source Bible
-> Adaptation Boundary
-> Story Operating State
-> High-pressure Scene Viability
-> Episode Plan
-> Pressure-chain Writer Brief
-> Screenplay Draft
-> Rewrite Report
-> Pressure-chain Patch Brief
-> Dialogue Polish
-> Reviewer
-> Story Memory Checkpoint
```

然后再拿同一源本做小验证：

```text
同一个高压节点
-> 旧 brief 写法
-> 新 pressure-chain brief 写法
-> 只比较这一个节点是否更像短剧强戏
```

小验证通过后，再跑 1-10 集。
