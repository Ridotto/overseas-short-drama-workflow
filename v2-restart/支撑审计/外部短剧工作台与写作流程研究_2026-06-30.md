# 外部短剧工作台与写作流程研究

日期：2026-06-30  
用途：给 Cloud / Claude 做独立 review 的外部资料汇总  
项目上下文：本项目是“出海短剧洗稿器”，不是从零原创短剧生成器，也不是视频生产工具。外部资料只能作为“维度候选”和“流程参照”，不能压过源本有效性。

---

## 0. 先说结论

这轮外部资料里，真正有价值的不是某一套完整方案，而是一批可以拆下来用的“短剧写作/分析零件”。

最值得吸收的核心零件：

1. `源本分集功能表`：不要只分析剧情，要分析每集功能。
2. `Hook / Friction / Spike / Button`：单集短剧结构可以拆成开头钩、正面摩擦、最大刺激点、结尾按钮。
3. `三大密度`：情绪密度、信息密度、情节密度，可作为分析/写作/自检总分层。
4. `power shift`：每场戏结束后，权力关系和局势有没有变化。
5. `visual executability`：情绪、压迫、爽点是否能被观众直接看见，而不是只靠文件/对白/解释。
6. `source_ref / provenance`：新本关键场面要能回指源本有效功能，防止洗飞。
7. `台词功能标签`：攻击、防守、羞辱、逼迫、揭露、反击、埋伏笔；低信息闲聊要删。
8. `acting direction`：高压节点的人物情绪必须外化到表情、肢体、空间、身体反应、当众后果。

最大风险：

- 不能把外部资料里的固定秒数、固定集数、固定付费比例拿来当硬规则。
- 不能把“视频生产工作台”的可生产性，误当成“剧本好看”的标准。
- 不能把通用短剧方法论变成新的规则怪谈。
- 对我们来说，标准永远是：源本为什么有效，新本有没有把这种有效性迁移过去，同时避免同构到像只改名字。

## 1. 研究目标与边界

### 1.1 这次要解决的问题

用户的核心担心不是“PRD有没有更多规则”，而是：

- 我们到底知不知道短剧应该长什么样？
- 钩子、卡点、dramatic、视听语言、短剧感，具体怎么落到写作里？
- 现在项目里写作环节是否太薄，导致结构对了但正文不像短剧？
- 分析、写作、自检是不是应该环环相扣，而不是各看各的？

所以本轮外部研究的目标是：

1. 看成熟短剧/竖屏剧工作台怎么拆流程。
2. 看短剧写作工具和教程反复关注哪些维度。
3. 找出可以补进我们项目“源本分析 / 变量迁移 / 新本写作 / 自检 reviewer”的零件。
4. 同时防止被外部项目带偏，因为它们大多不是洗稿工具。

### 1.2 不做什么

本轮不做工程实现判断，不讨论网页 App、数据库、API、部署。

本轮也不直接改 PRD。本文是外部研究材料和候选维度池，供 Cloud 独立 review。

## 3. 逐来源可借内容

## 3.1 0xsline/short-drama

### 它的完整流程

该项目是一个短剧创作 skill。主流程是：

`/start -> /plan -> /characters -> /outline -> /episode -> /review -> /compliance -> /export`

对应产物：

- `creative-plan.md`
- `characters.md`
- `episode-directory.md`
- `episodes/ep001.md...`
- `compliance-report.md`
- export 完整剧本

### 我们可以直接借的部分

#### A. 分集目录作为必经中间件

它不会从创意直接写正文，而是先做全剧方案、人物、分集目录，再写单集。

对我们项目的改造：

`源本分集功能表 -> 变量迁移表 -> 新本分集功能表 -> 正文`

这里的“分集功能表”不是摘要，而是回答：

- 这一集观众在等什么？
- 这一集制造了什么情绪？
- 这一集有没有新信息 / 新局势 / 新关系变化？
- 这一集结尾钩子是什么？
- 这集在全剧里承担什么功能？

#### B. 钩子类型表

它把钩子拆成五类：

| 钩子类型 | 机制 | 对我们怎么用 |
|---|---|---|
| 悬念钩 | 接近答案，在揭晓前切断 | 标源本每集结尾观众想知道什么 |
| 反转钩 | 最后一秒颠覆预期 | 标哪些反转要迁移，哪些只是壳 |
| 情绪钩 | 情绪推到最高点切断 | 标男女主拉扯/崩溃/反击前一拍 |
| 信息钩 | 关键信息只给一半 | 标文件、录音、监控、身份信息如何承载 |
| 危机钩 | 突发重大威胁，主角来不及反应 | 标车祸、暴露、闯入、倒计时等高冲击节点 |

这个表可以直接进入我们的源本分析表和 reviewer 表。

#### C. 单集微结构

它把单集拆成：

- 开头：承接上集悬念 / 抛出本集冲突。
- 中段：冲突展开、压力升级、新信息介入。
- 结尾：阶段兑现或极致悬念。

对我们的用法：

新本每集不能只是“发生几件事”，而要能填出：

| 单集环节 | 要回答的问题 |
|---|---|
| 开头 | 这一集开头凭什么不被划走？ |
| 中段 | 冲突是否升级，不是平行重复？ |
| 最大刺激点 | 观众最爽/最气/最心疼的是哪一下？ |
| 结尾 | 观众为什么必须看下一集？ |

### 不能照搬的部分

它有很多固定比例，例如关键集占比、付费集比例、集数规划。这些不能变成我们的硬规则。我们只能借结构和维度。

## 3.2 ReelForge-YAML

### 它的完整流程

这是“网文章节 -> 竖屏短剧 YAML/JSON”的工具。它强调：

- 每集第一镜必须是 `opening_hook`。
- 每集最后一镜必须是 `cliffhanger`。
- 心理描写必须转成可见动作、表情、站位、道具、光影、声音。
- 低冲突段落要把手机、屏幕、账单、录音、监控、文件、针管、印章等前置成可拍镜头。
- 每个 episode / shot 保留 `source_ref / source_map`。

### 我们可以直接借的部分

#### A. 质量评估维度

它的 evaluator 有几项非常适合我们：

| ReelForge 维度 | 我们项目里的改造 |
|---|---|
| hook_score | 开头有没有短剧冲击力 |
| cliffhanger_score | 卡点是不是卡在答案前一拍 |
| power_shift_score | 场面结束后谁压过谁，局势有没有变 |
| visual_executability | 情绪/冲突是否能被画面直接承载 |
| continuity_score | 角色、空间、动作、道具是否连续 |
| provenance_score | 新本关键点是否能追溯源本有效功能 |
| dialogue_language | 台词是否像短剧，而不是解释性长对白 |

其中最值得加入的是 `power_shift`。

因为导演反馈里反复说“站桩聊天”“像两个冷静成年人对话”，本质就是：这场戏没有权力变化，没有压迫/反击/失控/后果，只有低信息对白。

#### B. visual_executability

它把“抽象情绪”判为坏 case。

对我们来说，可以转成自检口径：

- 如果高压节点只写“她崩溃了”“他黑化了”“气氛凝重”，不合格。
- 必须写成能看见的东西：手抖、下跪、摔门、撕文件、围观者沉默、血迹、屏幕弹窗、监控画面、酒杯碎裂、被推出门外。

#### C. source_ref

这个对洗稿很重要。

我们的新本不应该复制源本文字和道具，但关键场面的功能必须能回指源本：

- 源本第2集车祸：功能可能是强视觉危机 + 情感冲击 + 男主无情/误会升级。
- 新本不能只变成档案室钥匙交还，因为那只迁移了“冷漠功能”，没迁移“冲击强度”。

所以新本关键场面表里应有一列：

`源本对应功能 / 源本强度 / 新本等效场面 / 是否丢失冲击`

### 不能照搬的部分

它是小说转视频 YAML，偏分镜和视频生产。它要求每集固定 shot 数，这不适合我们当剧本文档标准。

## 3.3 Toonflow-app

### 它的完整流程

Toonflow 里最有价值的链路是：

`项目参数 -> 故事骨架 -> 改编策略 -> 单集剧本 -> 监督审核 -> 分镜/资产/视频`

它不是洗稿器，但“骨架/策略/正文/监督”的分层很成熟。

### 我们可以借的部分

#### A. 故事骨架不是剧情摘要，而是“卖点结构”

Toonflow 的故事骨架会先定：

- 故事核
- 隐线 / 主角弧
- 大三角核心角色
- 三幕结构
- 分集策略
- 全局删减决策表
- 付费卡点
- 反转登记表

我们不能照搬原创剧做法，但可以做洗稿版：

| Toonflow 骨架项 | 我们洗稿版 |
|---|---|
| 故事核 | 源本核心期待：观众到底追什么 |
| 隐线 | 主角/关系情绪变化轨迹 |
| 大三角 | 源本核心压迫/误会/对抗结构 |
| 三幕结构 | 源本阶段任务与节奏惯性 |
| 分集策略 | 源本分集功能和留存机制 |
| 删减决策 | 哪些是水分，哪些是有效机制 |
| 反转登记 | 哪些反转必须功能迁移 |

#### B. 三大密度

Toonflow 反复强调：

- 情绪密度
- 信息密度
- 情节密度

这三个是非常好的总分层，能防止我们的分析表变散。

对我们来说：

| 密度 | 分析源本时看什么 | 写新本时看什么 | 自检时看什么 |
|---|---|---|---|
| 情绪密度 | 每集最强情绪是什么，如何制造 | 新本有没有等效情绪压力/爽点 | 是否冷、平、像聊天 |
| 信息密度 | 每集给了什么新信息/信息差 | 新本有没有新信息推进 | 是否空转、注水 |
| 情节密度 | 局势/关系/权力是否变 | 新本每场是否有不可逆变化 | 是否重复同一矛盾 |

这可以替代一堆散规则。

#### C. “改编策略”这个中间层

Toonflow 在骨架和正文之间有 adaptation strategy。这个对我们非常有启发。

我们现在的问题之一是：分析做了，变量表也有，但写作时可能直接跳正文。  
中间应该有一个轻量的“本次洗稿策略”，不是规则表，而是本单的写作决策。

洗稿版可以是：

1. 本次改写力度：中度 / 深度。
2. 绝对保留：源本哪些体验功能不能丢。
3. 必须替换：哪些身份、职业、道具、场景、台词形式不能同构。
4. 场面等效策略：源本强视觉节点在新壳里怎么等效。
5. 台词策略：哪些场面要少说多做。
6. 风险点：哪些地方最容易只改名。

这个不是新主流程，可以放在“写作前工作稿”里。

#### D. 监督层

Toonflow 有一个监督层 agent，不做修改，只出问题清单。这个适合我们 reviewer。

可借原则：

- Reviewer 不改稿，只判问题。
- 严重问题要指向具体位置。
- 不把轻微问题堆满。
- 先判能不能用，再判美不美。

### 不能照搬的部分

Toonflow 有很多硬比例，比如付费点按总集数百分比、前10集投流素材等。那些是原创/改编生产流程里的经验，不适合作为我们洗稿器硬标准。

## 3.4 huobao-drama

### 它的关键价值

huobao-drama 把“剧本改写”和“分镜拆解”分得很清楚。

剧本改写阶段：

- 保留核心情节。
- 增强画面感。
- 用对白推动情节，减少旁白。
- 每场戏适合短视频节奏。
- 不写景别、角度、运镜，这些属于分镜拆解。

分镜拆解阶段：

- 每个镜头聚焦单一动作。
- 补景别、角度、运镜、视频提示词、音效、时长。

### 对我们当前争议的帮助

导演说 v3 “不够视听化”，但这不等于剧本正文要写满“特写、仰拍、推镜”。

正确边界应该是：

- 剧本正文要写清楚可拍动作、表情、空间压迫、道具、身体反应、当众后果。
- 不必在剧本阶段大量写技术镜头语言。
- 真到视频/分镜阶段，再拆景别和运镜。

所以 PRD 里的“短剧场面等效”应理解为：

高压节点不能只靠文件、会议、冷对白推进；必须有观众能直接看见的动作、反应、后果。

不是：

每场都写镜头术语。

## 3.5 AgentCine

### 它的有价值文件

重点读到：

- `agent_storyboard_plan.zh.txt`
- `agent_storyboard_detail.zh.txt`
- `agent_acting_direction.zh.txt`
- `screenplay_conversion.zh.txt`
- `agent_storyboard_insert.zh.txt`

### 可借内容

#### A. source_text 保留

AgentCine 处理分镜时要求保留 `source_text`。  
我们洗稿不保留原文文字，但要保留“源本功能引用”。

建议新本关键场景草案加：

- 源本节点
- 源本功能
- 新本承载方式
- 同构风险

#### B. 表演指导

它的 acting direction 很适合解释“人物情绪外化”。

可借成我们的正文要求：

| 抽象写法 | 短剧写法 |
|---|---|
| 她很害怕 | 她后退一步，手指抓住桌沿，指节发白 |
| 他很愤怒 | 他一脚踹开椅子，声音压低，眼神死盯对方 |
| 她崩溃了 | 她跪在地上，抓住对方裤脚，哭到说不完整话 |
| 气氛紧张 | 所有人都停住，酒杯悬在半空，没人敢出声 |

#### C. 空间关系

AgentCine 很强调：

- 谁面对谁。
- 谁挡在谁前面。
- 谁被逼到哪里。
- 谁在围观。

这对短剧很重要，因为短剧的压迫常常不是“说出来”的，而是空间上被逼、被围、被当众审判。

对我们自检：

高压场面如果没有空间关系，容易变站桩聊天。

## 3.6 Jellyfish

### 它的完整流程

Jellyfish 是一个 AI 短剧生产工作台，核心链路是：

`剧本输入 -> 剧本理解/分镜拆解 -> 角色/场景/道具/服装抽取 -> 候选确认 -> shot ready -> 视频生成工作区`

### 可借内容

它提醒我们：

剧本正文最后要能被拆成：

- 场景
- 角色
- 道具
- 服装
- 对白
- 分镜
- 生成任务

对我们不是要求现在做生产链，而是自检时要问：

这段剧本如果给导演/分镜师，能不能拆？

如果一段全是：

- “她想起过去的痛苦”
- “他内心复杂”
- “两人气氛微妙”

那它就不可生产，也不像短剧。

### 不能照搬

Jellyfish 强在生产，不强在故事判断。  
它能告诉我们“可不可拆”，不能告诉我们“爆不爆”。

## 3.7 story-shot-agent / PenShot

### 可借内容

PenShot 的核心是：

`script_parser -> shot_segmenter -> video_splitter -> prompt_converter -> quality_auditor -> continuity_guardian`

对我们来说，最有用的是两个点：

#### A. continuity guardian

短剧节奏快，但不能乱跳。  
新本不能为了爽点硬切，角色状态、空间、道具、情绪要连续。

这能补 reviewer：

- 上一场人物刚崩溃，下一场是否无理由冷静？
- 证据/道具是否突然出现？
- 角色位置和行动是否接得上？
- 情绪升级是否有承接？

#### B. shot segmentation 思维

如果一场戏拆不出动作段，就说明它可能是“解释型剧情”。

例如：

- 不合格：男主看完文件，知道真相。
- 更短剧：男主点开监控，屏幕里女主被推倒、流产报警声响起，男主手里的文件被攥皱。

## 4. 跨来源共同结论

### 4.1 短剧不是电影/电视剧缩短版

多个来源都在说同一个意思：

短剧不是把长剧压短，而是以极短单位持续制造：

- 期待
- 情绪
- 信息差
- 关系变化
- 权力变化
- 结尾追问

所以“像电视剧”的常见症状是：

- 多轮低信息对白。
- 人物冷静谈判。
- 关键情绪靠解释，不靠动作。
- 场面没有权力变化。
- 结尾把话说完，没有卡在答案前。
- 开头先铺背景，而不是直接把冲突甩出来。

### 4.2 单集不是一段剧情，而是一个留存闭环

综合外部资料，一个单集至少要回答：

| 维度 | 问题 |
|---|---|
| 承接 | 上集问题怎么回应 |
| 本集期待 | 这集观众想看什么 |
| 正面摩擦 | 谁和谁发生直接冲突 |
| 压力升级 | 事情怎么变糟或变强 |
| 最大刺激 | 本集最强画面/情绪是哪一下 |
| 阶段兑现 | 观众有没有吃到一点东西 |
| 新问题 | 为什么还要看下一集 |

这比单纯“一集有几个钩子”更健康，不容易变规则怪谈。

### 4.3 高压节点必须有“场面落点”

场面落点不是镜头术语，而是：

- 动作
- 空间
- 身体反应
- 道具
- 当众后果
- 围观反应
- 证据可见化

例如：

| 弱写法 | 强写法 |
|---|---|
| 男主发现女主流产真相 | 男主看到监控里女主被推倒、警报声响、血迹、病床文件和录音同时出现 |
| 女二害怕 | 女二跪着抱住男主腿，被一脚踹开，围观者后退 |
| 男主无情 | 男主当众收走女主唯一权限/钥匙/身份牌，保安立刻拦她 |
| 女主霸气回归 | 女主不说话，墨镜未摘，全场停住，男二/男主先看见她 |

### 4.4 台词必须功能化

外部资料和导演反馈完全一致：

短剧台词不是自然聊天。每句台词最好承担至少一个功能：

- 攻击
- 防守
- 羞辱
- 逼迫
- 揭露
- 反击
- 埋伏笔
- 触发动作
- 改变关系

如果一段对白没有新信息、没有情绪变化、没有权力变化，就算写得自然，也是在拖。

### 4.5 “可生产”不是“好看”，但能发现一类坏剧本

Jellyfish、AgentCine、PenShot、小镜故事板这类工具不能告诉我们什么是爆款，但能告诉我们：

如果一段剧本拆不成镜头、动作、人物反应、道具和结果，它大概率不适合短剧。

所以“可视化/可拍性”应该是底线，不是最高标准。

## 6. 对当前项目 PRD / 工作流的影响

### 6.1 当前主流程不用大改

现有 PRD 的大方向是对的：

`源本分析 -> 变量迁移 -> 新本规划 -> 写首批 -> 自检/reviewer -> 反馈改稿`

外部资料不支持“再加很多大环节”。它支持的是把现有环节做实。

### 6.2 需要补强的不是“规则”，而是中间产物

最应该补的是三个中间产物：

1. `源本短剧语法标注表`
2. `关键场面等效表`
3. `新本分集功能表`

这三个能解决：

- 分析和写作脱节。
- 自检只能泛泛说好坏。
- 新本功能迁移了，但短剧味没迁移。
- 改得像只换名字。

### 6.3 写作方式应是批次/阶段，不是全剧一次性正文

外部项目普遍不会让模型一次吞完整长剧本再直接吐完整剧本。

更合理的工作方式：

1. 全剧/首批源本标注。
2. 首批新本分集功能表。
3. 关键场面草案。
4. 写 1-10 集正文。
5. 作者自检。
6. 干净 reviewer 审。
7. 根据反馈改首批。
8. 再进入后续批次。

这也能缓解上下文污染和上下文爆炸。

## 7. 需要 Cloud 重点 review 的问题

请 Cloud 不要泛泛评价“资料是否丰富”，重点判断以下问题：

1. 这些外部维度里，哪些确实应该进入我们项目？
2. 哪些会把项目带偏成原创短剧生成器，而不是洗稿器？
3. `源本短剧语法标注表` 是否足够覆盖“短剧味”？
4. `Hook / Friction / Spike / Button` 是否适合我们做单集分析，还是会过度格式化？
5. `三大密度` 是否适合作为总分层，能否替代零散规则？
6. `power shift` 是否应该成为自检必查项？
7. `visual executability` 应该放在正文写作要求里，还是只放 reviewer？
8. `source_ref / provenance` 在洗稿场景下应该怎么表述，才能防洗飞但不鼓励照抄？
9. 当前 PRD 是否需要改，还是先用这些模板跑一次三星样本验证？
10. 是否有任何外部资料明显不适合我们，应该明确剔除？

## 8. 我当前的产品判断

我的判断是：

1. 不要新写一份“通用短剧写作基准”当最高标准。
2. 应该先做“源本短剧语法标注表”，让短剧感从已验证源本里长出来。
3. 外部资料只作为标注维度候选，不作为最终标准。
4. 当前最小有效动作是：拿三星源本 1-10 集标注一遍，再拿 v3 新稿对照。
5. 如果对照能解释导演反馈，例如第二集冲击力降级、第七集台词过多、第八集动作外化不足，就说明这套标注表有效。
6. 如果解释不了，再回头补维度，不要先在 PRD 里堆规则。

一句话：

外部资料证明，我们缺的不是“短剧规则大全”，而是把源本的短剧有效性拆出来、迁移过去、再用同一把尺审新本的中间工作台。

## 10. 风险清单

### 10.1 外部资料风险

1. 低星 GitHub 仓库可能只是 prompt 壳。
2. 商业工作台官网多是营销语言，不能当方法论证据。
3. 国内短剧方法论可能偏中文市场，出海短剧需二次判断。
4. 数字比例、固定集数、付费点位置很容易变规则怪谈。
5. 视频工作台强调可生产，但“能生成视频”不等于“剧本好看”。

### 10.2 我们项目内部风险

1. 把“场面等效”理解成每场都要炸。
2. 把“钩子”理解成疯狂堆悬念。
3. 把“视听化”理解成堆镜头术语。
4. 把“源本有效性迁移”理解成照搬卡点位置/原事件。
5. 把“去同构”理解成改得越大越好，导致源本有效机制丢失。
6. 把 reviewer 变成打分机器，而不是定位问题。

## 11. 给 Cloud 的推荐审查口径

请 Cloud 按这个顺序审：

1. 先判断本文是否准确服务“洗稿器”，有没有被原创短剧/视频生产工具带偏。
2. 再判断候选维度是否覆盖用户担心的“短剧味”：钩子、卡点、dramatic、视听语言、情绪密度、台词功能。
3. 再判断是否有规则怪谈风险：有没有把候选维度变成硬规则。
4. 最后判断下一步是否应该先跑样本标注，而不是继续改 PRD。

我倾向的下一步：

`先标三星源本 1-10 集 -> 对照 v3 新稿 -> 解释导演反馈 -> 再决定是否更新 PRD`

而不是现在继续扩文档。

## 12. 如何反补到源本分析和新本自检

可以反补，而且应该反补。  
但反补方式不是“新增一堆规则”，而是把同一组维度放到两个位置：源本分析先用它看清“源本为什么有效”，新本自检再用它检查“新本有没有把有效性迁移过去”。

换句话说：

`源本分析` 和 `新本自检` 应该用同一把尺。

### 12.1 反补到源本分析

源本分析不能只写：

> 第1集发生了什么，第2集发生了什么。

而要写：

> 这集为什么让人想看，靠什么情绪、信息、场面、卡点留住观众。

建议把源本分析表升级成三层。

#### 第一层：分集功能分析

这是每集都要看的基础层。

| 字段 | 要回答的问题 | 目的 |
|---|---|---|
| 本集功能 | 这一集是立期待、升压、反击、揭露、误会、关系推进，还是卡点？ | 防止只做剧情摘要 |
| Hook | 开头凭什么抓人？ | 看源本开头机制 |
| Friction | 谁和谁发生正面摩擦？ | 看冲突是否外显 |
| Spike | 本集最大刺激点是哪一下？ | 找真正的高能点 |
| Button | 结尾卡在哪个问题上？ | 看留存机制 |
| 主情绪 | 羞辱、心疼、爽、焦虑、甜、怒、期待？ | 看情绪密度 |
| 新信息 | 本集给了什么新信息？ | 看信息密度 |
| 权力变化 | 场面结束谁压过谁？局势怎么变？ | 看情节密度 |

这个表的核心作用：把源本从“剧情流水账”拆成“体验功能表”。

#### 第二层：关键场面分析

不是每场都重分析，只分析高压节点：

- 开场强钩子
- 羞辱/压迫场面
- 反击/打脸场面
- 关系强拉扯场面
- 信息揭露场面
- 每集结尾卡点
- 付费/阶段性强节点

建议字段：

| 字段 | 要回答的问题 |
|---|---|
| 场面功能 | 这场戏是压迫、揭露、反击、误会、崩溃，还是卡点？ |
| 冲击来源 | 它为什么有冲击？车祸、当众羞辱、身体伤害、身份暴露、证据出现、关系撕裂？ |
| 视听承载 | 观众能直接看到什么？脸、手、血、屏幕、监控、围观者、空间压迫？ |
| 身体反应 | 角色有没有可见反应？跪、抖、后退、抓住、摔倒、沉默、失控？ |
| 当众后果 | 这场戏有没有公开羞辱、身份变化、权力变化、关系撕裂？ |
| 台词功能 | 关键台词是攻击、防守、逼迫、揭露、反击，还是废话？ |
| 卡点位置 | 停在答案前、动作前、反应前，还是把话说完了？ |

这能直接解决导演反馈里的问题：  
为什么“车祸”强，而“安全档案室”弱？因为前者有突发危机、身体伤害、视觉冲击、情绪震荡；后者更多是信息/权限动作，冲击来源不等效。

#### 第三层：源本有效机制提取

每集最后要提炼一句：

> 这一集必须迁移的不是事件，而是哪个用户体验？

例如：

| 源本事件 | 不该机械照搬的部分 | 必须迁移的体验 |
|---|---|---|
| 车祸 | 车祸这个具体事件 | 突发危机 + 男主无情 + 女主受伤/被抛弃的情感冲击 |
| 女主回归 | 同一句台词/同一场地 | 霸气回归 + 众人反应 + 下一集打脸期待 |
| 男主看录音 | 录音这个唯一证据形式 | 真相可视化 + 男主情绪被击穿 |

这一步很重要。它能防止模型把“源本分析”误解成“照搬源本事件”。

### 12.2 反补到新本自检

新本自检应该镜像源本分析。

源本分析问：

> 源本为什么有效？

新本自检问：

> 新本有没有把这个有效性迁移过去？如果没有，是丢了情绪、丢了信息、丢了场面，还是丢了卡点？

建议自检分两层：硬底线 + 质量返修。

#### 硬底线：有没有洗飞

| 检查项 | 不合格信号 |
|---|---|
| 核心期待迁移 | 源本观众追的是身份/复仇/误会/关系，新本写散了 |
| 分集功能迁移 | 源本这集是强危机，新本变成平淡过渡 |
| 情绪曲线迁移 | 源本在升压，新本提前释放或情绪变平 |
| 关键卡点迁移 | 源本卡在答案前，新本把答案说完 |
| 高压场面等效 | 源本是强视觉/情感冲击，新本变成文件、会议、冷对白 |
| 权力变化迁移 | 源本场面有胜负变化，新本只是站桩聊天 |

如果这些不合格，不是“小修文笔”，而是洗飞。

#### 质量返修：新本自身好不好看

| 检查项 | 返修信号 |
|---|---|
| 情绪密度 | 角色太冷静，冲突不够 dramatic |
| 信息密度 | 台词很多但新信息少 |
| 情节密度 | 场面结束局势没变 |
| 台词功能 | 多轮低信息对白、像电视剧聊天 |
| 视听承载 | 关键节点靠文件/对白解释，缺动作、反应、后果 |
| 人物外化 | 人物没有身体反应、走位、blocking |
| 同构度 | 事件、场次、台词功能太像源本，只是改名换背景 |
| 连续性 | 情绪、动机、道具、证据出现不顺 |

### 12.3 源本分析和新本自检的对应关系

最理想的状态是：每个源本关键节点都有一条对应的新本自检问题。

| 源本分析字段 | 新本自检字段 |
|---|---|
| 源本 Hook 是什么 | 新本开头是否有等效 Hook |
| 源本 Spike 是什么 | 新本最大刺激点是否等效 |
| 源本 Button 卡在哪 | 新本是否卡在同等追问上 |
| 源本主情绪 | 新本主情绪是否同强度迁移 |
| 源本新信息 | 新本是否有等效信息推进 |
| 源本权力变化 | 新本是否有同等局势变化 |
| 源本视听承载 | 新本是否有可见动作/反应/后果 |
| 源本台词功能 | 新本台词是否承担同等功能 |

这样自检就不会变成抽象评价“好不好看”，而是具体检查“哪种有效性丢了”。

### 12.4 不要让反补变成注意力涣散

维度很多，但执行时不应该让模型一次平铺 40 个问题。

推荐分层：

#### 必填主表：每集只填 8 个字段

1. 本集功能
2. Hook
3. Friction
4. Spike
5. Button
6. 主情绪
7. 新信息
8. 权力变化

#### 重点节点表：只分析高压场面

只在开场、强冲突、强反击、强揭露、结尾卡点处填：

- 场面功能
- 冲击来源
- 视听承载
- 身体反应
- 当众后果
- 同构风险

#### 自检表：只看失败信号

自检时不需要长篇夸优点，重点抓：

- 是否洗飞
- 是否降级
- 是否同构
- 是否站桩
- 是否低信息对白
- 是否卡点说完

### 12.5 反补后的最小工作流

不需要改成复杂新流程。最小版本可以是：

1. 源本 1-10 集分集功能标注。
2. 源本关键场面等效标注。
3. 新本 1-10 集分集功能规划。
4. 写正文。
5. 作者自检：逐项对照源本有效机制是否迁移。
6. 干净 reviewer：不看作者自检，独立检查是否洗飞/降级/同构。

这不是新增规则系统，而是把分析、写作、自检接到同一条线上。

## 13. 给 Cloud 的追加问题

请 Cloud 额外判断：

1. 上面这套反补是否会让源本分析更有效，还是会过重？
2. `Hook / Friction / Spike / Button` 是否适合做每集必填字段？
3. `权力变化 power shift` 是否应该成为每集必填？
4. `重点节点表` 是否能覆盖导演反馈里的“视听化/短剧感不足”问题？
5. 新本自检是否应该严格镜像源本分析，还是需要另设独立维度？
6. 这些字段是否会诱导模型机械套模板？如果会，哪些字段要降级成 reviewer 提醒？
