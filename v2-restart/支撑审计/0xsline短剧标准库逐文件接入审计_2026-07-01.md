# 0xsline 短剧标准库逐文件接入审计

> 日期：2026-07-01  
> 状态：审计文档，不是新 workflow，不直接改 PRD / workflow。  
> 范围：`参考资产/1-short-drama_短剧标准库/` 下全部文件。  
> 目的：逐项确认 0xsline/short-drama 的能力，我们是否已经用到，是否只是借了名词，是否应该接入为 skill / agent 交接产物。

---

## 0. 总判断

我们没有完全白造轮子，但确实没有充分继承 0xsline 的完整短剧生产链。

当前项目已经吸收了不少短剧概念：

- 钩子 / 卡点 / End Button
- 爽点 / 压抑释放 / Max Spike
- 节奏 / Pressure Progression
- 台词短、准、有功能
- page-to-screen / 可拍正文
- 独立 reviewer

但主要问题是：

```text
我们借了 0xsline 的零件，
没有继承它的“短剧原生工作台链路”。
```

0xsline 的强处不是某条规则，而是：

```text
/start -> /plan -> /characters -> /outline -> /episode -> /review -> /compliance -> /export
```

它把题材、受众、角色、节奏、爽点、钩子、正文、审稿做成连续产物链。  
我们当前链路更强在“源本迁移 / 防洗飞 / 防同构”，但短剧创作底盘被拆散了。

所以推荐关系是：

```text
0xsline = 短剧创作底盘
我们自己的能力 = 源本有效性迁移 + 防同构 + 新壳等效承载
```

不是照搬 0xsline，也不是继续自己重造一套短剧底盘。

## 2. 入口与流程文件

### 2.1 `README.md`

**0xsline 原生能力**

它把短剧创作定义成一条完整工作台：

```text
/start
/plan
/characters
/outline
/episode
/review
/compliance
/export
```

还定义了工作目录：

```text
.drama-state.json
creative-plan.md
characters.md
episode-directory.md
episodes/
reviews/
compliance-report.md
export/
```

这不是散文档，而是非常明确的 skill 产物链。

**我们当前是否接入**

部分接入。

我们有对应物：

| 0xsline | 我们当前近似物 |
|---|---|
| `/start` | 最小需求确认 / 二次需求确认 |
| `/plan` | Creative Development Pack / Stage Skeleton |
| `/characters` | Character Operating State / Relationship State |
| `/outline` | Episode Pursuit Map |
| `/episode` | Screenplay Draft |
| `/review` | 作者自检 / 独立 reviewer |
| `/compliance` | 当前只有零散边界，未成正式主链 |
| `/export` | 交付判断 / 版本归档 |

但问题是：我们是自己拆出来的，不是沿用 0xsline 的稳定产物链。  
所以很多东西在概念上对齐了，交接上没对齐。

**缺口**

- 缺少一个类似 `.drama-state.json` 的轻量状态索引，用来让主控知道当前已经有什么产物、写到哪、哪些已 pass。
- `creative-plan / characters / episode-directory` 这三个 0xsline 的基础产物，在我们这里被拆进多个文档，writer 取用时容易丢层次。
- `/review` 五维体系没有作为稳定审稿镜片接入，只散落在 quality gate 里。

**建议处理**

改造后采用。

不要照搬它的命令名，但可以采用它的产物链思想：

```text
run_state
source_bible
adaptation_boundary
creative_plan
character_relationship_bible
episode_pursuit_directory
scene_packets
screenplay
review_report
delivery_package
```

### 2.2 `SKILL.md`

**0xsline 原生能力**

这是实际执行说明。它明确每个命令加载哪些 reference：

- `/start` 读 `genre-guide.md`
- `/plan` 读 `opening-rules.md`、`paywall-design.md`、`rhythm-curve.md`、`satisfaction-matrix.md`
- `/characters` 读 `villain-design.md`
- `/outline` 读 `paywall-design.md`、`rhythm-curve.md`
- `/episode` 读 `opening-rules.md`、`rhythm-curve.md`、`satisfaction-matrix.md`、`hook-design.md`
- `/review` 五维评分
- `/compliance` 读 `compliance-checklist.md`

**我们当前是否接入**

弱接入。

`workflow_spec_v2.md` 明确写过引用 `short-drama /start`、`/plan`、`episode-directory`、`/episode`、`hook-design`、`rhythm-curve` 等。  
`workflow_spec_v11` 保留了其中一部分字段。

但目前接入方式像“借参考维度”，不像“按 skill 阶段调用 reference”。

**缺口**

- 我们没有明确写：哪个 agent / skill 在哪个阶段读取哪份 reference。
- 我们没有明确写：该 reference 产出的字段如何进入下一环节。
- `writer brief` 被压薄后，很多 reference 的能力没有以“当前集可写信息”的形式传入正文。

**建议处理**

直接采用它的“按阶段加载 reference”思想。  
不要把 8 份 reference 全塞给 writer。应该让对应 skill 在对应阶段读取，输出薄产物。

## 3. Reference 文件逐项审计

### 3.1 `genre-guide.md`

**0xsline 原生能力**

提供：

- 13 种题材
- 每类题材的核心受众、典型爽点、常见设定、关键要素
- 题材叠加规则
- 出海题材映射
- 欧美爆款元素
- 欧美文化元素速查
- 中式叙事陷阱
- 英文台词写作要点

**我们当前是否接入**

部分接入。

我们在需求确认、Creative Development Pack 里有：

- 新壳
- 目标市场
- 题材类型
- 风格元素
- 世界观
- 输出语言

但题材语法没有稳定支配后续每集写法。  
例如“豪门商战 / Billionaire Romance / Revenge”应该影响场景、台词、反派压力、爽点类型，但在 V22 writer brief 里更多变成背景设定。

**缺口**

- 题材承诺没有成为 writer 的显式输入。
- 受众口味没有成为 reviewer 维度。
- 出海文化陷阱没有稳定进入边界包。

**处理建议**

改造后采用。

接入方式：

| 环节 | 接入字段 |
|---|---|
| 需求确认 | 主副题材、目标受众、目标市场、调性 |
| Creative Plan | Genre Promise / Audience Promise |
| Writer Brief | 本集题材爽点一句话，不放完整题材表 |
| Reviewer | 是否兑现题材承诺，是否仍是中式逻辑 |

不能硬规则化：

- S/A/B 爆款元素不能直接变成必须使用。
- 题材模板不能压过源本有效性。

### 3.2 `opening-rules.md`

**0xsline 原生能力**

提供：

- 前 5 秒 / 前 30 秒抓人原则
- 6 种开场模板
- 开场质量自检
- 不同题材推荐开场

**我们当前是否接入**

部分接入。

我们有 `Opening Hook`，也一直强调前三集和首集抓人。  
但当前常见问题是：开场字段存在，正文不一定真的写成“第一眼能抓住”的场面。

**缺口**

- 首集和每集开场没有稳定由 `opening-rules` 这类规则生成可演场面。
- reviewer 能指出开头弱，但不一定能回滚到“开场场面设计”。

**处理建议**

改造后采用。

接入方式：

| 环节 | 接入字段 |
|---|---|
| Episode Pursuit Directory | Opening Hook |
| Scene Packet | 首场冲突 / 第一眼画面 / 观众疑问 |
| Writer Brief | 本集开场必须从什么动作或冲突开始 |
| Reviewer | 是否慢铺、旁白、信息过载、无主角处境 |

不能硬规则化：

- “第 1 秒 / 第 3 秒 / 第 5 秒”只能做提醒，不能机械卡秒。
- 6 种模板只能做候选，不可套模板。

### 3.3 `rhythm-curve.md`

**0xsline 原生能力**

提供：

- 单集微结构：前 30 秒钩子、中段冲突升级、最后 30 秒爽点/钩子
- 全剧四阶段节奏波形
- 加速/减速工具
- 不同题材节奏差异
- 节奏自检

**我们当前是否接入**

部分接入。

我们有：

- Stage Skeleton
- Pressure Progression
- Episode Pursuit Map
- Do Not Consume
- End Button

但没有形成一个“全剧节奏仪表盘”。  
V22 能保住顺序，但短剧感弱，说明节奏不是完全没接，而是没有稳定指导“哪里该压、哪里该放、哪里该停”。

**缺口**

- 阶段骨架更偏剧情阶段，不够像情绪/爽点/付费波形。
- 每集 pressure progression 有字段，但 writer brief 容易压成“起手、升级、反转”三句。
- reviewer 不应只看有没有变化，还要看节奏是否像短剧。

**处理建议**

改造后采用。

接入方式：

| 环节 | 接入字段 |
|---|---|
| Source Bible | 源本节奏惯性 / 压放曲线 |
| Stage Skeleton | 阶段追问、阶段兑现、阶段留白 |
| Episode Pursuit Directory | 本集起压 / 升压 / 爆点 / 收住 |
| Story Memory | 已兑现爽点、未兑现追问、节奏风险 |
| Reviewer | 是否空转、拖沓、一直炸、提前消费 |

不能硬规则化：

- 15% / 30% / 35% / 20% 这种比例不适合洗稿硬套。
- 总集数不同、源本节奏不同，必须以源本有效性为准。

### 3.4 `hook-design.md`

**0xsline 原生能力**

提供：

- 五种钩子：悬念钩、反转钩、情绪钩、信息钩、危机钩
- 分阶段钩子配置
- 不同题材钩子侧重
- 钩子叠加规则
- 钩子与付费卡点关系
- 钩子质量自检

**我们当前是否接入**

部分接入。

我们有：

- 钩子与卡点表
- End Button
- Last Frame / Sound / Action
- Do Not Consume

但 hook 类型和 hook 质量没有稳定进入 writer。  
很多时候只剩“最后卡什么”，没有写清“观众正在等哪个答案，停在答案前哪一拍”。

**缺口**

- 钩子类型没有稳定作为分析和 writer 交接字段。
- 结尾按钮容易变成最后一帧画面，而不是付费追问。
- 下集承接关系不够强时，reviewer 容易只记风险。

**处理建议**

直接采用其维度，改造后进入我们链路。

接入方式：

| 环节 | 接入字段 |
|---|---|
| Source Bible | 源本每个强卡点的钩子类型 |
| Episode Pursuit Directory | 本集 End Button + Hook Type |
| Writer Brief | 最后一拍的画面 / 动作 / 声音 + 未回答问题 |
| Reviewer | 是否卡在答案前，是否虚假悬念，是否下集承接 |

不能硬规则化：

- 每集都同等强钩子会导致硬堆。
- 钩子类型表是语言，不是模板。

### 3.5 `paywall-design.md`

**0xsline 原生能力**

提供：

- 付费卡点位置规划
- 悬念未解原则
- 情绪峰值原则
- 沉没成本原则
- 五种经典卡点套路
- 卡点禁忌
- 免费集与付费集节奏配合

**我们当前是否接入**

弱到部分接入。

我们有“能付费”“留存节点”“End Button”“Do Not Consume”，但没有稳定用 paywall 视角去看：

```text
观众为什么愿意为下一集付费？
付费后下一集必须兑现什么？
这个悬念是不是只卡了一次？
```

**缺口**

- 付费卡点没有成为阶段骨架和 Episode Map 的强字段。
- Reviewer 不应只看“有钩子”，还应看“钩子是不是付费级追问”。
- 付费后兑现旧悬念 + 生成新悬念的机制不稳定。

**处理建议**

改造后采用。

接入方式：

| 环节 | 接入字段 |
|---|---|
| Source Bible | 源本付费/留存卡点功能 |
| Stage Skeleton | 关键付费窗口 / 不可提前消费 |
| Episode Pursuit Directory | 本集未解问题 / 下集必须兑现 |
| Reviewer | 假悬念、卡点说完、同一悬念重复卡 |

不能硬规则化：

- 第 8-12 集、10-15% 付费比例不能照搬。
- 我们是洗稿，源本付费惯性优先于固定比例。

### 3.6 `satisfaction-matrix.md`

**0xsline 原生能力**

提供：

- 爽点本质：压抑 -> 释放
- 五大爽点类型：身份碾压、打脸复仇、逆袭翻盘、情感爆发、悬念揭秘
- 爽点分布策略
- 爽点类型搭配
- 不同题材爽点侧重
- 爽点质量自检

**我们当前是否接入**

部分接入。

我们有“爽点 / 压抑释放”和 `Max Spike`。  
但 V22 暴露的问题是：`Max Spike` 有时只是“最大信息点”或“最大机制结果”，不一定是“压抑后的爽点释放”。

**缺口**

- 爽点类型没有稳定继承到每集。
- 压抑和释放周期容易被压成“本集最大刺激点”。
- “同类型爽点重复”和“爽点强度递进”没有稳定进入 reviewer。

**处理建议**

改造后采用。

接入方式：

| 环节 | 接入字段 |
|---|---|
| Source Bible | 源本爽点类型 / 压抑来源 / 释放方式 |
| Creative Plan | 新壳核心爽感 / 辅助爽感 |
| Stage Skeleton | 阶段爽点递进 |
| Episode Pursuit Directory | 本集 Max Spike 属于哪类爽点 |
| Scene Packet | 爽点前如何压低，释放动作是什么 |
| Reviewer | 无铺垫爽、爽点被推翻、同类重复、强度不递进 |

不能硬规则化：

- 不能让所有戏都写成“羞辱 -> 亮身份 -> 全场跪”。
- 爽点类型是诊断语言，不是固定套路。

### 3.7 `villain-design.md`

**0xsline 原生能力**

提供：

- 四层反派体系
- 反派层级关系
- 反派塑造技巧
- 反派台词风格
- 不同题材反派配置
- 反派设计自检

**我们当前是否接入**

弱接入。

我们有“主要反派为什么敢继续作死”，也有角色状态和关系状态。  
但反派压力没有成为持续发动机。  
很多时候反派只是提供证据链、误会、陷害，而不是持续制造追看压力。

**缺口**

- 反派层级不一定要完整四层，但“阶段压力升级”需要有。
- 反派为什么有能力、为什么不直接失败、为什么继续作死，需要进入 Creative Plan 和 Episode Map。
- 反派每阶段赢/输什么，需要进入 Story Memory。

**处理建议**

改造后采用。

接入方式：

| 环节 | 接入字段 |
|---|---|
| Source Bible | 源本反派压力如何制造追看 |
| Creative Plan | 反派动机 / 能力来源 / 节制理由 |
| Stage Skeleton | 每阶段反派压力升级 |
| Episode Pursuit Directory | 本集谁压谁，反派赢/输什么 |
| Story Memory | 反派资源、败局、伏笔状态 |
| Reviewer | 反派为恶而恶、太弱、靠主角降智、无伏笔 |

不能硬规则化：

- 不要强塞四层反派。
- 不要为了反派可恨而堆合规高风险桥段。

### 3.8 `compliance-checklist.md`

**0xsline 原生能力**

提供：

- 红线清单
- 高风险灰区
- 短剧常见踩坑
- 正向价值观
- 平台审核重点
- 出海合规补充
- P0-P3 优先级

**我们当前是否接入**

弱接入。

PRD 里有“烂点/必须删除”和一些风险判断，但没有独立合规 skill。  
当前项目是 AI 出海短剧，合规不该压过创作，但也不能完全靠最后人工补。

**缺口**

- 目标市场合规边界没有在需求确认阶段稳定确认。
- 高风险内容没有独立报告。
- Reviewer 里合规和剧作质量容易混在一起。

**处理建议**

改造后采用，但放在护栏层，不放进创作发动机。

接入方式：

| 环节 | 接入字段 |
|---|---|
| 需求确认 | 目标市场 / 平台尺度 / 用户禁区 |
| Adaptation Boundary | 源本高风险外形 / 必删烂点 |
| Writer Brief | 当前集必要尺度边界 |
| Reviewer / Compliance Agent | P0 删除、P1 必修、P2 建议 |
| Delivery | 合规风险登记 |

不能硬规则化：

- 不要把“正向价值观”硬塞成宣传片。
- 出海合规要按目标市场重判。

## 4. 和我们当前项目的逐项结论

### 4.1 已经保留得比较好的

| 能力 | 我们现状 | 判断 |
|---|---|---|
| 洗稿产品锚点 | PRD_v4 清楚 | 保留 |
| 源本有效性迁移 | Source Bible / 高价值节点 | 保留 |
| 防同构 | Boundary / Forbidden Shape | 保留 |
| End Button / Do Not Consume | Episode Map 有 | 保留，但要加强承接 |
| page-to-screen | Screenplay Contract 有 | 保留，但执行要靠上游场面包 |
| 干净 reviewer | PRD / workflow 有 | 保留 |

### 4.2 已经有，但没接稳

| 能力 | 当前问题 | 建议 |
|---|---|---|
| 题材/受众/调性 | 进了开发包，但不持续支配每集 | 加 `Genre Promise` 到每集/每阶段 |
| 节奏曲线 | 有 Stage Skeleton，但不是短剧节奏仪表盘 | 加阶段压放/留白/兑现 |
| 爽点矩阵 | 有 Max Spike，但不一定是爽点释放 | 加爽点类型、压抑来源、释放动作 |
| 反派压力 | 有角色状态，反派升级弱 | 加反派阶段压力 |
| 付费卡点 | 有 End Button，但付费动机弱 | 加未解问题和下集兑现 |
| Review 五维 | 散在 quality gate | 改成 reviewer lens，不做总分 |

### 4.3 我们现在重复造轮子的地方

我们自己造了：

- Creative Development Pack
- Stage Skeleton
- Episode Pursuit Map
- Scene Option
- Writer Brief
- Reviewer 三档

这些并不全错。  
错的是：很多功能本来在 0xsline 里已经有原生位置，我们重新命名后没有完整继承它们的上下游关系。

例如：

| 0xsline 原生链 | 我们重造后问题 |
|---|---|
| `/plan` 里同时有三幕、节奏、付费、爽点 | 我们拆成多个包，但 writer 不一定拿到统一创作方向 |
| `/characters` 里有人物、关系、反派、弧线 | 我们有角色状态，但反派压力/关系拉扯传递弱 |
| `/outline` 里每集有钩子、标记、节奏段落 | 我们有 Episode Map，但付费/爽点/题材口味弱 |
| `/episode` 直接按短剧格式写场次 | 我们改成 page-to-screen 对，但容易把短剧感放到上游口号 |
| `/review` 五维审查 | 我们 reviewer 更复杂，但短剧原生五维反而不稳定 |

## 5. 推荐的 agent / skill 链路

不建议每一步固定一个长期 agent。  
建议固定 skill 和中间产物，agent 可以按需临时开。

推荐形态：

```text
主控 Orchestrator
  ↓
需求确认 skill
  输出 Run Contract / User Need
  ↓
源本分析 skill
  输出 Source Bible
  ↓
洗稿边界 skill
  输出 Adaptation Boundary
  ↓
短剧创作方案 skill
  借 0xsline /plan
  输出 Creative Plan
  ↓
人物关系与反派压力 skill
  借 0xsline /characters + villain-design
  输出 Character & Relationship Bible
  ↓
分集追剧目录 skill
  借 0xsline /outline + rhythm/hook/paywall
  输出 Episode Pursuit Directory
  ↓
场面/beat 开发 skill
  结合我们自己的洗稿迁移和 V5 scene packet 经验
  输出 Scene Packets / Episode Writing Cards
  ↓
正文写作 skill
  借 0xsline /episode + 我们 page-to-screen
  输出 Screenplay Draft
  ↓
作者自检 skill
  输出 Author Ready Check
  ↓
独立 reviewer skill
  借 0xsline /review 五维 + 我们三红线
  输出 Layered Review Report
  ↓
合规 / 出海适配 skill
  借 compliance-checklist，按需跑
  输出 Compliance Report
  ↓
主控交付判断
```

关键不是 agent 数量，而是每一步的输入输出合同。

## 6. 接入优先级

### P0：必须先接

1. `README/SKILL` 的完整产物链思想  
   目的：避免我们继续散装造轮子。

2. `genre-guide` 的题材/受众/出海映射  
   目的：让短剧味随题材变化，不再用一套“失去/可见后果”打所有剧。

3. `rhythm-curve` 的单集微结构和阶段节奏  
   目的：解决稳定但平、能看但不够追。

4. `satisfaction-matrix` 的压抑-释放  
   目的：修正 `Max Spike` 只剩信息点的问题。

5. `hook-design / paywall-design` 的 End Button / 未解问题 / 下集兑现  
   目的：让卡点变成追付费，而不是最后一帧。

### P1：第二批接

1. `villain-design`  
   目的：补反派压力和关系拉扯，不让剧情只靠证据链推进。

2. `opening-rules`  
   目的：首集和每集开场更稳。

3. `/review` 五维体系  
   目的：给 reviewer 一个短剧原生镜片，但不能做总分机器。

### P2：护栏接入

1. `compliance-checklist`  
   目的：出海/平台/红线护栏。

2. `/export`  
   目的：交付格式和归档，不影响当前核心创作质量。

## 7. 不要做的事

不要把 0xsline 整套照搬，因为它是从零原创短剧，不是洗稿。

不要把固定比例变成硬规则：

- 关键集 25-35%
- 付费集 10-15%
- 前 10 集几个付费卡点
- 四阶段固定百分比

不要把 reviewer 变成 50 分评分器。  
我们需要分层审稿，不需要总分幻觉。

不要让 writer 直接吃完整 reference。  
writer 只吃本集需要的薄卡片。

不要为了“短剧感”硬堆钩子、爽点、反派、失去、跪求。  
短剧感要服务源本有效性和题材承诺。

---

## 8. 最终判断

这不是小补丁。

也不是推翻式大重构。

这是一次中等重构 / 底盘校准：

```text
保留我们的洗稿主线；
正式把 0xsline 的短剧创作底盘接进来；
把散落在分析、开发包、writer brief、reviewer 里的短剧能力重新接成产物链。
```

最重要的改法不是“加字段”，而是：

```text
每个 skill 只负责一层；
每层输出一个可交接产物；
下一层按产物继续写，不重新猜上游意图；
reviewer 能打回具体产物层，而不是无限要求正文更好。
```
