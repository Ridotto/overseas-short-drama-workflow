# 0xsline 底盘接入架构审计

> 日期：2026-07-01  
> 状态：只读审计结论，不是新 workflow，不改 PRD / workflow / 样本产物。  
> 审计目标：判断当前项目与 0xsline/short-drama 标准库之间，哪些能力必须保留、哪些应该合并、哪些不能照搬，以及下一版产品形态应该如何避免继续丢能力。

---

## 0. 结论先行

这不是小补丁，也不是推倒重来。

更准确的判断是：

```text
中等重构 / 短剧底盘校准。

保留我们的洗稿主线；
保留 v11 已验证有效的分层；
正式接入 0xsline 的短剧生产底盘；
把散在 Source Bible、Creative Pack、Episode Map、Writer Brief、Reviewer 里的短剧能力接成连续产物链。
```

不能继续只加零散规则。根因不是“缺一条视听语言规则”或“缺一条台词规则”，而是多条能力链在下游被压成单条任务链。

证据：

- PRD v4 的产品目标是“把一个已被市场验证或值得参考的短剧源本，洗成一个新壳下仍然好看、能追、能付费的新剧本”，且交付的是剧本文字，不是分析报告：`PRD_v4.md:31-43`。
- PRD v4 明确失败类型不能互相抵消：洗飞、改名复刻、短剧降级、正文长剧化、状态漂移分别有不同回滚入口：`PRD_v4.md:116-130`。
- 原项目能力对账表已经指出，当前问题不是没有能力，而是“能力被保留但没有真的传到正文”“能力放错层”“后续草案把能力压窄”：`原项目能力对账表_2026-07-01.md:10-35`。
- 0xsline 的强处不是单条规则，而是 `/start -> /plan -> /characters -> /outline -> /episode -> /review -> /compliance -> /export` 的连续短剧生产链：`0xsline短剧标准库逐文件接入审计_2026-07-01.md:30-37`。

## 1. 根因判断

### 1.1 当前 v11 已解决了一部分问题

v11 的正确方向是把分析线、创作线、审计线、治理线拆开，规定 writer 只读薄的 Playable Writer Brief：`workflow_spec_v11_写作前故事包候选版.md:99-115`。

V21 证明这个方向有效但不够稳定：

- V21 不是 pass，但不是 block；
- writer brief 已经能交付主链；
- 仍没稳定卡住核心概念、源本动作簇残留、证据链转人物行动链这三类问题。

证据：`PRD_v21_v11workflow_vs_v3_v5_v20_整体对比.md:18-49`。

V22 证明 v11 小修有效：

- workflow execution complete through reviewer；
- creative outcome 是 `pass-pending-lock`；
- source carry-through 是 `pass with registered risks`；
- delivery outcome 仍不是 direct pass。

证据：`PRD_v22_latest_v11_交付判断.md:3-57`。

### 1.2 当前 v11 仍缺“短剧底盘”

V22 的 Source Bible、Creative Pack、Episode Map、Scene Options 都比 V21 稳，但它仍暴露一个结构问题：

```text
writer brief 薄了，这是对的；
但短剧底盘里的 Hook Type、付费级追问、爽点类型、阶段压放、反派压力没有稳定进入 writer brief 和 reviewer lens。
```

证据链：

- v11 Writer Brief 明确只能保留可写信息，不能塞完整 Source Bible、manifest、reviewer、历史失败清单：`workflow_spec_v11_写作前故事包候选版.md:484-579`。
- V22 Writer Brief 确实做到了薄，逐集只有“观众等什么 / 兑现什么 / 最后卡什么 / 高压场面 / 谁压谁 / 谁失去什么 / 起手动作 / 信息边界 / Last Frame”：`PRD_v22_latest_v11_Playable_Writer_Brief.md:19-167`。
- 但 0xsline 的钩子体系要求钩子是“必须知道答案的问题，然后不给答案”，且需要 Hook Type 与质量自检：`references/hook-design.md:1-15`、`references/hook-design.md:294-310`。
- 0xsline 的付费卡点要求情绪峰值、悬念未解、付费后兑现旧悬念并制造新悬念：`references/paywall-design.md:1-11`、`references/paywall-design.md:182-202`。
- 0xsline 的爽点矩阵要求压抑到释放、爽点类型搭配和强度递进：`references/satisfaction-matrix.md:1-18`、`references/satisfaction-matrix.md:273-319`。
- 0xsline 的节奏曲线要求单集前 30 秒、中段递进、最后 30 秒爽点/钩子，并每 5 集检查情绪曲线和未完成感：`references/rhythm-curve.md:1-56`、`references/rhythm-curve.md:244-263`。
- 0xsline 的反派设计要求反派可恨、可信、递进，避免反派太弱或为恶而恶：`references/villain-design.md:1-15`、`references/villain-design.md:303-319`。

所以：v11 的“写作前故事包”方向对，但还没完整吸收 0xsline 的短剧创作底盘。

## 2. 最终产品形态判断

当前产品形态不该变成网页 App，也不该变成一个巨型 agent。

建议形态：

```text
主控 Orchestrator
  -> 需求确认 / Run Contract
  -> 源本分析 / Source Bible
  -> 洗稿边界 / Adaptation Boundary
  -> 源本迁移版 Creative Plan
  -> Character & Adversary Pressure Bible
  -> Episode Pursuit Directory
  -> Scene Packet / Episode Writing Card
  -> Thin Playable Writer Brief
  -> Screenplay Draft
  -> Author Ready Check
  -> Clean Layered Reviewer
  -> Delivery Judgment
```

关键是固定 skill 和产物合同，不固定长期 agent 实例。

证据：

- PRD v4 第一阶段就是 skill / workflow 原型，先稳定“源本分析 -> 新本写作 -> 自检/reviewer”的创作链路：`PRD_v4.md:72-86`。
- v11 总流程已经有 Source Bible、Boundary、Creative Pack、Stage Skeleton、Episode Map、Scene Option、Writer Brief、Screenplay、Reviewer、Delivery：`workflow_spec_v11_写作前故事包候选版.md:72-93`。
- 0xsline 原生链路明确有 creative-plan、characters、episode-directory、episodes、review、compliance、export 等连续产物：`SKILL.md:5-21`、`README.md:321-340`。

## 5. V3 / V5 / V21 / V22 证据判断

### 5.1 V3 / V5 不能只抽成标签

V3 / V5 合并方案已经指出，V3 / V5 至少有十条链路要同时保住：留存/钩子、情绪/爽点、主线期待、人物关系、权力变化、同构扣除、场面动作、正文表达、连续性、审稿/版本对比：`V3_V5完整对读与合并方案_2026-07-01.md:33-46`。

V5 的价值不是“更刺激”，而是 Scene packet 给到了 writer 能写正文的材料：动作链、权力变化、证据触发动作、信息边界、卡点：`V3_V5完整对读与合并方案_2026-07-01.md:132-198`。

不能照搬 V5 的最终正文格式，因为它带内部标签，读起来不像干净剧本：`V3_V5完整对读与合并方案_2026-07-01.md:226-236`。

### 5.2 V21 证明 v11 分层方向成立，但短剧记忆点不稳

V21 的判断：

- workflow complete；
- creative revise；
- source carry-through pass with risks；
- delivery not pass。

证据：`PRD_v21_v11workflow_vs_v3_v5_v20_整体对比.md:18-49`。

V21 主要硬伤：

- 核心骗局定义混乱，不是台词问题，而是 Creative Pack / Writer Brief 之前未锁定：`PRD_v21_v11workflow_vs_v3_v5_v20_整体对比.md:118-135`。
- Ep1 源本动作簇保留太密：`PRD_v21_v11workflow_vs_v3_v5_v20_整体对比.md:136-153`。
- V21 比 V5 干净，但短剧场面记忆点不如 V5：`PRD_v21_v11workflow_vs_v3_v5_v20_整体对比.md:169-192`。

### 5.3 V22 证明小修有效，但仍不能 full pass

V22 的有效点：

- Source Bible 抓住源本体验功能、高价值节点、信息差和源本冲突；
- Boundary 修复假孕/有孩子冲突，锁定双胞胎；
- Creative Pack 主追问成立；
- Scene Options 有餐桌名牌、服务路火场、授权拍卖、酒店徽章、信托门、露台安全线；
- Writer Brief 没塞 reviewer / manifest / 历史失败噪音；
- screenplay 可看可拍。

证据：`PRD_v22_latest_v11_reviewer_round2_源本审计.md:23-36`。

V22 仍不该 full pass：

- 7-10 集功能排序与源本高度一致；
- Ep7 Adrian 对抗强度偏弱；
- 后续不能让 Blackwood / Serena / Julian 全部失去反扑能力。

证据：`PRD_v22_latest_v11_交付判断.md:19-29`、`PRD_v22_latest_v11_交付判断.md:41-55`。

## 6. 架构合并方案

### 6.1 应合并的重复功能

| 现有功能 | 0xsline 对应能力 | 合并后产物 |
|---|---|---|
| Creative Development Pack | `/plan` 的故事骨架、节奏、付费、爽点 | Source-Migrated Creative Plan |
| Character Operating State / Relationship State | `/characters` + `villain-design` | Character & Adversary Pressure Bible |
| Stage Skeleton | `rhythm-curve` 全剧节奏波形 | Stage Rhythm Skeleton |
| Episode Pursuit Map | `/outline` + hook/paywall/rhythm | Episode Pursuit Directory |
| HighPressure Scene Options | V5 Scene Packet + 0xsline `/episode` 前置结构 | Scene Packet / Episode Writing Card |
| Playable Writer Brief | 0xsline `/episode` 必要输入 + 我们的薄输入原则 | Thin Writer Brief |
| Layered Reviewer | 0xsline `/review` 五维 + PRD 三红线 | Layered Review Report |

### 6.2 不应合并的内容

以下能力相似但不能合并，否则会丢产品目的：

| 不合并项 | 原因 |
|---|---|
| Source Bible 与 0xsline `/plan` | Source Bible 是洗稿源本有效性，不是原创故事策划 |
| Adaptation Boundary 与 compliance | Boundary 是防洗飞/防复刻，compliance 是平台/尺度护栏 |
| Scene Option 与 reviewer | Scene Option 是创作判断，reviewer 是验收判断 |
| Dialogue polish 与 Screenplay rewrite | Dialogue 只能修表达，不能修坏场面发动机 |
| Manifest 与质量门 | manifest 只证明步骤和证据，不证明剧本质量 |

1. 保留 v11 全链路和 manifest 协议。
2. 在 Creative Pack 引入 Genre Promise / Audience Promise，但只写会影响场面、台词、尺度的部分。
3. 在 Character Operating State 后新增或合并 Adversary Pressure：反派资源、阶段压力、反派赢/输、下一阶段反扑。
4. 把 Stage Skeleton 改成 Stage Rhythm Skeleton：追问、兑现、留白、压放、关键付费窗口。
5. 把 Episode Pursuit Map 升级为 Episode Pursuit Directory：加 Hook Type、Payoff Question、Paywall Function、Satisfaction Type、Next-Episode Payoff。
6. 把 Scene Options 输出转成 Scene Packet / Episode Writing Card：可见后果、爽点前压低、释放动作、信息边界、卡点最后一拍。
7. Writer Brief 继续薄，但必须保留本集短剧发动机：题材爽点一句话、Hook Type、未回答问题、爽点类型、反派压力、最后一拍。

不做：

## 8. 本轮审计的 root cause hypothesis

```text
Root cause hypothesis:
当前架构失败不是单项规则缺失，而是“源本迁移能力”和“短剧创作底盘”没有合成一条稳定产物链。

v11 修好了 writer brief 被审计噪音污染的问题；
但它把 writer brief 压薄后，没有把 0xsline 的题材、节奏、爽点、钩子、付费、反派压力这些短剧发动机用薄字段传进去。

结果就是：
上游有能力；
正文能变干净；
但短剧追看、付费、反派压力、爽点递进仍然靠主创临场发挥，不够稳定。
```

验证状态：

- V21 支持该假设：分层更好，但核心真相、动作簇和证据链仍出问题。
- V22 支持该假设：核心真相修好了，正文可拍了，但 7-10 集爆点密度、强机制同构和后续反扑仍需登记风险。
- 0xsline 支持该假设：它的价值在连续短剧生产链，不在单条规则。

## 11. 结论

下一步不应该直接改 PRD，也不应该直接重跑样本。

```text

```
