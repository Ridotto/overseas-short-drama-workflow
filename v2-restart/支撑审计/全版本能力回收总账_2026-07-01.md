# 全版本能力回收总账

> 日期：2026-07-01  
> 当前 stage：全版本能力回收  
> 用途：在写新版 workflow spec 之前，把 V1-V15、PRD、设计提纲、外部项目和样本验证里真正有用的能力收回。  
> 注意：这不是新 workflow，不是 writer 输入，也不是规则库。

---

## 0. 总判断

不能再用任何单一版本当母版。

过去的问题不是“某一版全错”，而是：

```text
每一版都抓到过一块真实问题；
每一版又都丢掉过另一块重要能力。
```

所以现在要做的不是选 V5、V11、V14 或 V15，而是把所有被样本或审计证明有价值的能力按层回收。

本项目的最高目标仍然是：

```text
把已被市场验证或值得参考的短剧源本，
洗成新壳下仍然好看、能追、能付费的新剧本。
```

任何能力如果不能服务这个目标，不能进入主链。

## 1. 版本能力回收

| 来源 | 可以回收什么 | 不能继承什么 | 证据 / 理由 |
|---|---|---|---|
| `PRD_v4.md` | 产品锚点、失败类型、主控/主创/reviewer 分工、用户输入、首批交付、自检回滚 | 不把 PRD 写成执行大表 | 当前主产品定义仍清楚：洗稿，不是原创、评分器、换名器 |
| `设计提纲_v8.md` | 7 个迁移动作、五层分析、变量迁移策略、力度按变量控制、短剧标准库降级为参考 | 不复活成规则池，不让变量表机械化 | 它保住了“源本有效性迁移”的产品定义，比后续很多局部修法更完整 |
| `workflow_spec_v1.md` | 需求确认、源本有效性摘要、事件载体拆解、变量迁移、新事件载体、阶段骨架、分集功能、关键场面、状态追踪、写作包、返修 | 过早把许多东西做成表，容易把 writer 拉进证明题 | 是完整链路早期形态，能力覆盖广，但执行口径还不够干净 |
| `workflow_spec_v2.md` | 当前最重要的成熟链路：Source Bible、Boundary、beat sheet、Episode map、Scene packet、Screenplay、Rewrite report、Dialogue polish、Quality gate、Story memory、Reviewer、Version comparison | 不能把运行记录 / quality gate 变质量证明；不能让 scene packet 变审计包 | 跑出 V5；V5 证明“场景目的 / 动作链 / 权力变化 / 可见后果 / DNC”能显著改善产物 |
| `workflow_spec_v3.md` | 分析线 / 创作线 / 审计线 / 治理线；减少 writer 被审计材料污染；clean reviewer 分两轮 | 不把分析 agent 和 writer 拆成互不相干的人 | V13 有正向信号：同构下降、机制感减弱、短剧感提升 |
| `workflow_spec_v4.md` | Source Retention Anchor、Story Operating State、Pressure-chain Writer Brief、Pressure-chain Patch Brief | 不能把压力链当万能模板；不能漂成原创生成器 | 解决“writer 拿到结果任务单，不拿故事状态和人物压力链”的问题 |
| `workflow_spec_v5.md` | Source Strong-Scene Read、Source Drama Engine Cards、Scene Option Pitch、Scene Choice Gate、Writer Self-Trim、Version Preference Review | 不能把“高压场面优选”放大全局；不能只修场面强度 | V5 能 pass，核心价值是场面发动机；但它不是最终目标，更不能替代源本/节奏/关系/连续性 |
| `workflow_spec_v6.md` | 能力回收、修连接不堆规则、writer-facing brief 不吃审计底稿、Scene Sequence、Beat survival check | 不能只做单集级简报，不能让 beat survival check 变证明题 | 它准确指出“分析有，writer 没吃到；能力放错层；局部修法挤掉全局能力” |
| `workflow_spec_v7.md` | 创作优先、流程减负、剧本文字必须像剧本、运行记录轻量化 | 不能过度减负到丢掉骨架、钩子、爽点、连续性 | 提醒我们“流程完整不等于剧本好”，但单独用会太薄 |
| `workflow_spec_v8.md` | 改写力度、关键场面迁移卡、主控版本对比 | 不能只恢复关键场面迁移卡；V18 已证明不够 | V18 失败证明：迁移卡壳恢复了，但完整创作链没恢复 |
| `workflow_spec_v9.md` | V3/V5 合并意识、多链路并行、版本对比 | 不能粗暴抽象成“V3 保方向 / V5 保强度” | 它指出 V3/V5 都有多个能力，但抽象仍偏粗 |
| `workflow_spec_v10.md` | V3/V5 更完整回收、Source/Boundary/Beat/Scene/Screenplay/Memory/Reviewer 全链合并 | 不能让运行记录变成质量中心 | 是一次更全的回收尝试，但仍需要按能力归层 |
| `workflow_spec_v11.md` | Source Bible、Creative Development Pack、Stage Skeleton、Episode Pursuit Map、HighPressure Scene Options、Playable Writer Brief、核心真相自洽 | 不能让 writer brief 过薄后丢短剧发动机；不能把风险全留给 reviewer | V22 稳但偏平，说明分层方向对，但强度生产不足 |
| `workflow_spec_v12.md` | 0xsline 底盘接入、Genre/Audience Promise、Character & Opposition、Stage Rhythm、Episode Pursuit Directory、Scene Carrier Pack | 不能照搬原创短剧工作台；不能用固定比例压过源本 | 把短剧知识接入主链，但仍有“字段在、动作弱”的风险 |
| `workflow_spec_v13.md` | writer 层重接：先写单集短剧施工稿，再写正文；强度轴在 writer 写作时使用 | 不能变成只测一集；不能让施工稿成为新规格 | EP8 验证说明 writer 渲染模式是关键风险之一 |
| `workflow_spec_v14.md` | 主控 + 主创链 + skill 工位、0xsline 分阶段加载、S5 writer 施工稿、reviewer 降权 | 不能认为工位存在就等于生效；不能让每个 skill 都变长期 agent | V25 比 V24 更像短剧，但仍 revise，说明方向有效但交接不干净 |
| `workflow_spec_v15.md` | 工位合同、唯一交接链、run log 不等于质量、S3/S4/S5/S8/S9 硬合同 | 不能直接正式化；尚未通过完整样本验证 | 当前最新候选，它把问题压到“每层产什么、禁什么、下游只吃什么” |

## 2. 样本证据回收

| 样本 / 版本 | 证明了什么 | 不能误读成什么 |
|---|---|---|
| V3 | 主线可读、新壳能站住、源本追看功能没有明显洗飞 | 不是最终写作质量合格样本 |
| V5 | `workflow_spec_v2` 的场面发动机有效，能把证据链改成动作链和权力变化 | 不是最终目标，不是所有剧都要复刻 V5 外形 |
| V13 | 分析/创作/审计/治理拆层能降低证明题和同构感 | 不是正式 workflow pass |
| V16 | v6 的连接链能跑通，但仍 delivery revise | 不证明 v6 可正式化 |
| V18 | 只恢复关键场面迁移卡不够 | 不说明迁移卡没用，而是它不能单独当主轴 |
| V22 | 分层、真相、边界更稳，但整体偏平 | 不证明 workflow 已能产出强短剧 |
| V24 | 外部底盘和完整链路更多，但 writer 仍偏渲染 | 不证明“文档越完整越好” |
| V25 | skill 工位链和 writer 施工稿有效，短剧感强于 V24 | 不达送审标准；S3/S4/S5/S8/S9 仍有断口 |
| V26 run log | v15 run log 能记录交接，但不是质量证明 | 不能因为 run log 完成就说 workflow 通过 |

## 3. 必须保留的核心能力

这些能力不能被任何新版 workflow 剪掉。若必须降级，必须写清楚代价。

| 能力 | 归属判断 | 丢掉后的失败 |
|---|---|---|
| 源本有效性迁移 | 项目根能力，S1/S2/S3/S5/S11 都要保护 | 洗飞，变原创或空壳 |
| 防改名复刻 | S2/S3/S4/S10/S11 | 只换名字和背景 |
| 新壳等效承载 | S2/S3/S4/S5 | 新事件不自然，或变弱 |
| 留存 / 钩子 / 卡点 | S1/S3/S5/S10/S11 | 观众不想点下一集 |
| 付费 / 关键留存节点 | S1/S3/S5/S11 | 能看但不一定愿意付费 |
| 爽点 / 压抑释放 | S1/S4/S5/S7/S10 | 只有剧情推进，没有爽感 |
| 情绪密度 / 信息密度 / 情节密度 | S1/S3/S5/S8/S10 | 空转、注水、冷、散 |
| 主线期待 / 信息差 / 真相揭露 | S1/S3/S8/S11 | 观众不知道追什么，或真相提前吃完 |
| 人物关系 / 拉扯 / 角色状态 | S1/S3/S5/S8 | 人物冷、关系平、火葬场没痛感 |
| 反派压力 / 反扑燃料 | S1/S3/S5/S8 | 主角一路碾压，剧情变平 |
| Do Not Consume | S1/S3/S4/S5/S8 | 后续爆点被提前消费 |
| Beat / Episode / Stage 骨架 | S3/S5/S11 | 只剩单场强，不成剧 |
| Scene packet / 场面发动机 | S4/S5/S6 | 高压节点退回文件、会议、证据播报 |
| Page-to-screen / 视听表达 | S6/S7/S10 | 剧本像说明书，不像 AI 短剧脚本 |
| 台词功能 / 角色声线 | S6/S7/S10 | 多轮水词、角色同声、电视剧聊天 |
| Story memory / continuity | S8/S11 | 分批写作状态漂移 |
| Clean reviewer | S10/S11 | 作者自说自话，或 reviewer 污染 writer |
| 主控版本对比与反馈归层 | S11 | 每条反馈变新规则，或只修单本 |
| 运行记录 | 主控治理层 | 需要记录执行，但不能当质量证明 |

## 4. 外部项目原名对齐

这张表只解决“叫法对齐”，避免后续 agent 因为外部项目使用英文名，就误判本项目没接入对应能力。

| 外部原名 / 旧产物名 | 本项目归属名 | 归属层 | 使用方式 |
|---|---|---|---|
| `Genre Promise` | 题材承诺 / 题材语感 | S0/S3/S7 | 确认新壳的题材承诺、世界观边界和角色说话方式 |
| `Audience Promise` | 受众承诺 / 目标市场语感 | S0/S3/S7 | 确认目标观众期待，不让英文对白等于出海语感 |
| `Opening` / `opening-rules` | 短剧开场法 | S5/S6 | 写作时决定第一眼看到或听到什么冲突，不做空泛“开场要强” |
| `Paywall` / `paywall-design` | 付费追问 / 关键留存节点 | S1/S3/S5/S11 | 迁移源本付费前后的追问和兑现义务，不照搬固定集数比例 |
| `Satisfaction` / `satisfaction-matrix` | 爽点 / 压抑释放 | S1/S4/S5/S10 | 让爽点落到压低、释放和可见后果，不做爽点类型填空 |
| `Villain` / `Adversary` | 反派压力 / 反扑燃料 | S1/S3/S5/S8 | 保留反派继续作死、误判和反扑的资源，不只贴坏人标签 |
| `Rewrite report` | 返修诊断 / 返修路由 | S9/S11 | 先判失败层，再给返修建议；不能变成正文补丁清单 |
| `Dialogue polish` | 台词 / 视听 polish | S7 | 删水词、保声线、把能演的内容从解释改成动作和声音 |
| `Quality gate` | 质量闸门 / reviewer lens | S9/S10/S11 | 只兜底和归层，不生产短剧感，不用分数替代判断 |

## 5. 不能再拿的东西

| 内容 | 不拿原因 |
|---|---|
| 0xsline 固定比例 / 固定付费集数 | 会压过源本节奏，变机械模板 |
| 0xsline 从零原创假设 | 我们是洗稿，不是原创生成器 |
| 0xsline 50 分总分 review | 会制造分数幻觉，且 reviewer 重新变质量发动机 |
| V5 正文内部标签 / 格式残留 | V5 的链路有价值，正文格式不能照搬 |
| V8 只靠关键场面迁移卡 | V18 已证明不够 |
| 旧设计提纲大规则池 | 只能做参考，不进入执行大表 |
| “短剧感”硬堆跪求、失去、反派、枪、血 | 会从机制修复滑回单本补丁 |
| 把 reviewer 作为短剧感生产者 | reviewer 应兜底和归层，不负责把剧写好 |

## 6. 本总账对下一步的约束

下一步写新版 workflow spec 时，必须遵守：

1. 不以任何单一版本为准；
2. 不把 V5 当目标，只回收它的场面发动机；
3. 不把 v15 直接正式化，只吸收它的工位合同思路；
4. 不让短剧手艺挤掉洗稿主线；
5. 不让运行记录、reviewer、自检进入 writer 输入；
6. 每个能力都必须有归属层、交接物、正文落点和失败回滚层；
7. 样本验证必须看产物质量，不看流程是否跑完。
