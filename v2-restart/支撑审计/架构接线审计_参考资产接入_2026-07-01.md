# 架构接线审计：参考资产如何接入当前洗稿器

> 日期：2026-07-01  
> 状态：审计文档，不是 PRD / workflow 正式改版。  
> 范围：当前 PRD / workflow / V3-V5-V20-V22 样本证据 + 参考资产 0xsline、oh-story、how-to-make-script、Dramatron、shortdrama-pipeline。  
> 原则：不凭感觉删东西；没有证据不做产品取舍；所有判断回到“海外短剧洗稿器”这个产品目的。

---

## 1. 产品锚点

本项目不是从零原创短剧工具，也不是评分器。

产品目标仍然是：

```text
把一个已被市场验证或值得参考的短剧源本，
洗成新壳下仍然好看、能追、能付费的新剧本。
```

因此，任何能力只要直接服务以下目标，就不能直接丢：

- 源本为什么让观众继续看；
- 源本为什么让观众愿意付费；
- 新壳是否能承载同等体验；
- 新本是否不像改名复刻；
- 新本自身是否成立；
- 正文是否像可拍的出海短剧剧本；
- 后续批次是否能接住人物、关系、伏笔和追问。

这次审计的核心问题不是“还缺哪条规则”，而是：

```text
现有项目里很多能力都出现过，
但没有稳定接成一条会生效的产物链。
```

---

## 2. 总结论

### 2.1 不是推倒重来

当前 v11 / V22 方向有价值，不能推翻。

已成立的部分：

- Source Bible 能读出源本有效性；
- Adaptation Boundary 能防洗飞和复刻；
- Creative Development Pack 解决了“新本先成为一个故事”的问题；
- Stage Skeleton / Episode Pursuit Map / HighPressure Scene Options 让写作前材料更可写；
- Playable Writer Brief 比厚审计包更适合正文；
- Screenplay 已经比旧版更像剧本；
- Reviewer 两轮读取边界是对的；
- Manifest-first 是必要治理，不是质量证明。

### 2.2 也不是小补丁

这不是加一条“视听语言规则”或“钩子规则”能解决。

V3 / V5 的正向证据、V20-V22 的失败/改善证据、0xsline 的完整短剧链都指向同一件事：

```text
短剧原生能力没有稳定接成 writer 和 reviewer 都使用的同一组合同。
```

所以这次应该叫：

```text
中等重构 / 底盘校准
```

不是大重构，不改产品目标；也不是小补丁，不再逐症状加规则。

---

## 3. 已确认根因

### 根因一句话

当前链路里，“洗稿迁移层”变强了，但“短剧创作底盘”没有稳定接上。

更具体说：

```text
钩子、付费、爽点、情绪密度、视听语言、台词、反派压力、关系拉扯、信息边界，
都在文档或样本里出现过，
但它们经常只是散点字段、检查项或口号，
没有全部进入同一套：
源本分析 -> 创作开发包 -> 分集追剧提纲 -> writer brief -> 正文 -> reviewer 回查
```

### 证据

V5 表现较好，不是因为它少规则，而是因为它把信息变成了现场动作和后果：

```text
Scene purpose + action chain + power shift + relationship shift + do not consume
```

V20 / V21 / V22 证明：

- v11 分层有效；
- writer brief 变薄是对的；
- 但短剧发动机仍然容易在压缩时丢失；
- V22 虽然 complete through reviewer，但仍是 `pass-pending-lock`，不是 full pass；
- 风险集中在 7-10 集爆点密度、源本功能排序偏贴、后续追看资产不足、部分对抗弱。

0xsline 证明：

它有价值的不是某几个规则，而是一条短剧生产链：

```text
/start -> /plan -> /characters -> /outline -> /episode -> /review -> /compliance -> /export
```

它的 reference 也是按阶段加载，不是全部塞给 writer。

## 4. 参考资产逐项取舍

### 4.1 0xsline / short-drama

定位：

```text
短剧创作底盘
```

能拿：

- 阶段产物链；
- 按阶段加载 reference；
- 题材 / 受众 / 出海映射；
- 开场抓人；
- 节奏曲线；
- 钩子类型；
- 付费卡点；
- 爽点矩阵；
- 反派压力；
- 合规风险分级。

不能照搬：

- 从零原创流程；
- 50-100 集默认原创规模；
- 15 / 30 / 35 / 20 固定节奏比例；
- 10-15% 付费卡点比例；
- 每集硬塞强钩子；
- 四层反派硬塞；
- 50 分评分器；
- 国内合规正能量口径；
- 把所有 reference 直接塞给 writer。

接入方式：

```text
0xsline = 短剧底盘
我们自己的 workflow = 洗稿迁移层
```

它不能替代 Source Bible / Adaptation Boundary；它应该补 Stage Skeleton、Episode Map、Scene Options、Writer Brief、Reviewer 的短剧原生能力。

### 4.2 oh-story

定位：

```text
连续性、状态追踪、去 AI 味、文件驱动交接
```

能拿：

- 分析产物作为 source of truth；
- 角色状态、时间线、伏笔、上下文追踪；
- 读旧文后续写的状态恢复；
- 去 AI 味 / 表达漂移检查；
- “先有大纲和状态，再写正文”的 guard。

不能照搬：

- 网文长篇日更结构；
- 扫榜选材和平台榜单逻辑；
- 长篇章节规模；
- 用网文节奏替代短剧节奏。

接入方式：

用于 Story Memory / Continuity，不用于决定短剧爽点标准。

### 4.3 how-to-make-script

定位：

```text
输出契约、表达 lens、视觉语言、multi-agent 交接
```

能拿：

- screenplay / scene / dialogue / rewrite_report / quality_gate 的输出契约思想；
- 对话不能是裸对话，必须有动作 / 表演 / 前置行为支撑；
- 视觉语言必须服务当前任务，不做全量词典；
- 质检 lens 要按任务选择，不做一张万能表；
- multi-agent 交接要有边界和产物合同。

不能照搬：

- 泛影视 / 广告 / 品牌片的一整套输出类型；
- 带内部标签的最终剧本正文；
- 过度机器检查导致正文像程序。

接入方式：

用于正文合同、表达层、reviewer lens、agent handoff。

### 4.4 shortdrama-pipeline

定位：

```text
生产状态、人工审核门、warning-only 质检
```

能拿：

- 剧本确认后再进入下游；
- 人工审核门；
- issue 必须带 suggestion；
- warning 不等于阻断；
- 状态流和 latest / runs 分离。

不能照搬：

- 视频生成 pipeline；
- shot JSON 作为当前剧本文字主流程；
- 工程 CLI / API 状态机。

接入方式：

用于治理层和交付状态，不用于替代创作链。

### 4.5 Dramatron

定位：

```text
分层生成和选择性上下文启发
```

能拿：

- 从 logline / characters / scenes / dialogue 分层展开；
- 上层骨架控制下层输出；
- 选择性上下文，而不是一次性塞满。

不能照搬：

- 从零生成假设；
- 戏剧 / 电影式自由创作；
- 纯 top-down 不回写。

接入方式：

只作为分层思想补充，不作为短剧标准库。

### 4.6 商业平台产物

定位：

```text
最终产物形态观察
```

能看：

- 它最后会给用户哪些材料；
- 它的正文、人物、梗概、大纲是否方便人读；
- 它是否说明“正文前必须有能写的故事材料”。

不能做：

- 不照抄字段；
- 不倒推黑箱中间链路；
- 不因为平台有某栏目，就强塞进我们的 workflow。

## 5. 不能丢的能力清单

以下不是新规则，是本项目为了达成产品目标必须保留的能力。

| 能力 | 为什么不能丢 | 应该接到哪里 |
|---|---|---|
| 源本有效性 | 洗稿的底子，丢了就变原创 | Source Bible / Boundary / Reviewer |
| 钩子 / End Button | 决定下一集追看 | Source Bible / Episode Map / Writer Brief / Reviewer |
| 付费卡点 | 决定“愿不愿意付费继续看” | Stage Skeleton / Episode Map / Story Memory |
| 爽点 / 压抑释放 | 决定短剧满足感 | Source Bible / Stage / Scene Options |
| 情绪密度 / 节奏压放 | 决定是否平、拖、一直炸 | Stage Skeleton / Episode Map / Reviewer |
| 人物关系 / 拉扯 | 女频、男频、亲情、复仇都需要持续张力 | Creative Pack / Story Memory |
| 反派压力 / 对抗发动机 | 没有阻力就没有持续追看 | Creative Pack / Stage / Episode Map |
| 信息差 / 真相路径 | 防提前消费，防追问断 | Source Bible / Stage / Episode Map / Story Memory |
| 权力变化 / 打脸 | 短剧高压节点需要可见变化 | Scene Options / Screenplay |
| 视听语言 / page-to-screen | 最终要变 AI 短剧画面 | Scene Options / Screenplay Contract |
| 台词功能 / 角色声线 | 防电视剧聊天和 AI 味 | Screenplay / Dialogue Polish |
| 状态追踪 / 伏笔 | 防后续批次乱写 | Story Memory |
| Reviewer 分层 | 防无限挑刺，也防放过硬伤 | Reviewer |
| Manifest / 版本对比 | 证明执行，不证明质量 | Governance |

结论：

```text
没有证据支持删除上面任何一项。
```

## 6. 应该合并的同类能力

不能把每项能力都变成一张表，否则又会变规则怪谈。

应该合并成 5 个可交接合同。

### 6.1 Pursuit Button Contract

合并：

- Hook；
- End Button；
- Paywall；
- Do Not Consume；
- Last Frame / Sound / Action；
- Next-Episode Payoff。

它回答：

```text
观众这一集到底在等什么？
结尾停在答案前哪一拍？
下一集必须兑现什么？
哪些信息 / 关系 / 情绪不能提前消费？
这个悬念是不是重复卡过？
```

它不是让每集硬塞强钩子，而是防止“最后一帧有画面，但没有真正追问”。

### 6.2 Pressure-Release Contract

合并：

- Satisfaction；
- Max Spike；
- 情绪密度；
- Pressure Progression；
- 压抑释放；
- 爽点类型。

它回答：

```text
这一集压低什么？
观众憋的是什么气？
用什么行动释放？
释放后谁失去权力、脸面、位置、关系或安全感？
释放后有没有留下下一轮压力？
```

它不是打脸公式，不要求每场都炸。

### 6.3 Scene Carrier Contract

合并：

- 高压场面候选；
- 视听语言；
- page-to-screen；
- AI 可拍；
- 证据触发行动；
- 可见后果。

它回答：

```text
这场戏靠什么人、动作、空间、物件、声音、身体反应成立？
如果删掉屏幕、文件、报告，这场高压戏还剩什么？
证据出现后，现场谁必须行动、失去或选择？
最后观众记住的是哪个可见后果？
```

它不是禁用文件 / 屏幕 / 证据。证据可以存在，但不能替代现场戏。

### 6.4 Relationship & Opposition Engine

合并：

- Character Operating State；
- Relationship / Power State；
- 反派压力；
- 关系拉扯；
- 权力变化；
- 角色动机连续性。

它回答：

```text
谁想要什么？
谁误判什么？
谁怕失去什么？
谁压谁、谁退谁、谁靠近谁？
反派为什么还能继续作死？
下一阶段谁有资源反扑？
```

它不是人物百科，而是场面发动机。

### 6.5 Layered Review Lens

合并：

- 0xsline 五维短剧 lens；
- 我们的三红线；
- hard fail / risk / optional；
- warning-only 思想；
- reviewer 两轮读取边界。

它回答：

```text
这个问题伤不伤产品红线？
是致命阻断、应改风险，还是可选优化？
失败应该回到 Source、Boundary、Creative Pack、Stage、Episode、Scene、Brief、Screenplay、Dialogue、Continuity 哪一层？
```

它不能变成总分机器，也不能变成无限挑刺入口。

## 7. 当前链路哪些要保留、重接、降级

### 7.1 保留

- PRD_v4 的产品定义；
- v11 的分层方向；
- manifest-first；
- Source Bible；
- Adaptation Boundary；
- Creative Development Pack；
- Stage Skeleton；
- Episode Pursuit Map；
- HighPressure Scene Options；
- Playable Writer Brief；
- Screenplay / page-to-screen；
- Dialogue polish；
- Story memory；
- 干净 reviewer；
- 主控版本对比。

### 7.2 重接

| 断点 | 当前问题 | 重接方式 |
|---|---|---|
| Source Bible -> Creative Pack | 源本有效性读出来后，短剧类型/爽点/卡点未必稳定变成新本发动机 | Creative Pack 接 `Genre Promise`、`Core Satisfaction`、`Opposition Engine` |
| Stage Skeleton -> Episode Map | 阶段骨架有追问/兑现，但不够像短剧节奏波形 | 接 `Pursuit Button` 和 `Pressure-Release` |
| Episode Map -> Writer Brief | 当前 brief 有等什么/兑现什么/卡什么，但付费兑现、爽点类型、反派压力容易被压没 | brief 保持薄，但补短剧发动机薄字段 |
| Scene Options -> Screenplay | 证据可能仍滑成说明或机制 | 用 `Scene Carrier Contract` 强制证据触发现场行动 |
| Reviewer -> 下一批 | reviewer 风险如果不回 Story Memory / Episode Map，会变成一次性评论 | reviewer 风险必须回写为下一批追问、禁提前消费、反扑资产 |

### 7.3 降级或删除

| 内容 | 处理 | 理由 |
|---|---|---|
| 固定比例公式 | 降级为参考 | 洗稿以源本有效性为准 |
| 50 分评分器 | 删除 / 不采用 | 总分会制造虚假客观性 |
| 四层反派硬塞 | 降级 | 需要反派压力，不需要四层模板 |
| writer 读完整 reference | 禁止 | 上下文污染，writer 会写成规则执行 |
| V5 正文内部标签 | 删除 | 最终剧本不能像说明书 |
| 平台字段照抄 | 禁止 | 黑箱倒推会偏离产品 |
| 旧大变量矩阵 | 不恢复 | 容易机械套模板 |
| 只补关键场面卡 | 不采用为完整解 | V18 已证明不够 |

## 8. 建议的新主链形态

这不是最终 workflow，只是审计后建议的产品链路。

```text
0. Run Contract / 需求确认
   - 新壳、市场、语言、集数、改写力度、用户禁区

1. Source Bible / 源本有效性包
   - 源本为什么能追、能付费
   - 钩子、爽点、卡点、关系、信息差、权力变化、表达手法

2. Adaptation Boundary / 洗稿边界
   - 必须保留的体验
   - 必须避开的源本外形
   - 可以自由改的变量
   - 必须删除/修复的烂点

3. Short Drama Development Pack / 新本短剧开发包
   - 不是百科
   - 只锁题材承诺、受众口味、核心爽感、核心真相、关系/反派发动机

4. Stage Rhythm & Pursuit Plan / 阶段节奏与追问计划
   - 阶段压放、兑现、留白、付费/追问窗口、反扑资产

5. Episode Pursuit Directory / 分集追剧目录
   - 每集追问、压力释放、End Button、Do Not Consume、Next-Episode Payoff

6. Scene Carrier Pack / 高压场面包
   - 高压节点 2-3 个候选
   - 选最能承载源本体验且最有短剧现场性的方案

7. Playable Writer Brief / 可写剧本简报
   - 薄
   - 只给当前批次/当前集能写戏的信息
   - 不给 manifest、reviewer、历史争论、完整 reference

8. Screenplay Draft / 正文
   - page-to-screen
   - 台词英文默认，但需求确认必须确认
   - 动作/场景描述按需求中文、英文或双语

9. Author Ready Check / 作者自检
   - 先判失败层，不直接润色

10. Clean Reviewer / 独立审稿
   - 第一轮只读正文和最小需求
   - 第二轮再读源本和中间产物
   - 三档：致命阻断 / 应改风险 / 可选优化

11. Story Memory / 状态追踪
   - 角色状态、关系距离、权力位置、信息差、已兑现、未兑现、伏笔、禁提前消费

12. Delivery / 主控交付判断
   - workflow execution
   - creative outcome
   - source carry-through
   - delivery outcome
```

## 9. Agent / skill 形态判断

建议固定的是 skill 和产物合同，不是固定长期 agent。

原因：

- 长期 agent 容易上下文变脏；
- 每轮 reviewer 应保持干净；
- 主创需要继承源本分析判断，不能被彻底切成互不理解的两个人；
- 但每一层可以由对应 skill 支撑，主控编排。

推荐形态：

```text
主控 Orchestrator
  -> 需求确认 skill
  -> 源本分析 skill
  -> 洗稿边界 skill
  -> 短剧开发包 skill
  -> 阶段/分集追剧 skill
  -> 场面载体 skill
  -> 正文写作 skill
  -> 作者自检 skill
  -> 干净 reviewer skill / 临时 agent
  -> continuity/story memory skill
```

主控负责方向和拍板。  
skill 负责产物合同。  
agent 可以临时调度，不必固定工位。

## 10. 不让重构丢东西的检查法

每次改 PRD / workflow 前，必须过这张表。

| 问题 | 如果答案是否定，不能改 |
|---|---|
| 这个改动是否仍然服务“洗稿”，不是从零原创？ | 否则偏离产品 |
| 源本有效性是否还能一路传到正文？ | 否则洗飞 |
| 钩子 / 付费 / 爽点 / 节奏 / 拉扯 / 反派 / 信息边界是否有归属层？ | 否则丢短剧感 |
| writer 是否只收到可写信息，而不是审计噪音？ | 否则程序化 |
| reviewer 是否能回查同一组合同，而不是另起一套标准？ | 否则无限挑刺 |
| 风险项是否和致命阻断分开？ | 否则永不 pass |
| 固定比例 / 评分 / 模板是否只是参考？ | 否则规则怪谈 |
| 新增内容是否合并进现有合同，而不是新开一张表？ | 否则膨胀 |

## 11. 下一步建议

不要直接把这份审计全文塞进 PRD。

建议顺序：

1. 先让 Cloud 或另一干净 reviewer 审这份审计，重点看有没有漏接、误接、过度工程化。
2. 通过后，再小范围改 `workflow_spec_v11`，不是改 PRD 主定义。
3. 改动只做三件事：

## 13. 最终口径

当前项目不是缺“更多规则”，而是缺“短剧底盘和洗稿迁移层的稳定接线”。

正确改法不是：

```text
视听语言不够 -> 加视听规则
台词不够 -> 加台词规则
钩子不够 -> 加钩子规则
```

正确改法是：

```text
把 0xsline 的短剧原生生产链，
oh-story 的状态追踪，
how-to-make-script 的输出契约，
shortdrama-pipeline 的审核门，
接到我们自己的洗稿迁移层上。
```

同时守住：

```text
源本有效性迁移是核心；
短剧感是底盘；
writer brief 不能吃噪音；
reviewer 不能无限挑刺；
最终交付必须是好看的、能追的、能付费的出海短剧剧本。
```
