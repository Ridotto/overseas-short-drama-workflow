# 原项目能力对账表

> 日期：2026-07-01
> 状态：审计表，不是新 workflow。
> 上游：`原项目能力全集盘点_2026-07-01.md`
> 目的：检查原项目已有能力在当前正式链路里是“已保留、部分保留、放错层、只写了但没传下去、还是被压窄/丢失”。

---

## 0. 当前判断

当前不是“原项目没有这些东西”。

当前更像是三种问题混在一起：

1. **能力被保留了，但执行时没有真的传到正文。**
   例如：源本拆文里有情绪、钩子、爽点、拉扯，但 writer 最后拿到的是结果任务单或证明链。

2. **能力放错层了。**
   例如：视听承载、台词功能、拉扯感如果只在 reviewer 里出现，就会变成写完后返修，不会变成写作发动机。

3. **后续草案把能力压窄了。**
   例如：v5 把“高压节点场面优选”放大成主轴，就挤掉了留存、爽点、节奏、人物关系、变量迁移、连续性等原本已经有的东西。

所以后续不该问：

```text
还要加什么规则？
```

而应该问：

```text
这些能力有没有一路进到 writer 输入、正文、reviewer 判错里？
```

## 2. 现在最容易被误判的地方

### 2.1 “文档里有”不等于“workflow 生效”

例如：

```text
workflow_spec_v2.md 已经写了 page-to-screen、动作行、身体状态、台词功能。
```

但如果 writer 的输入仍然是：

```text
本集必须揭露 X，必须证明 Y，必须回指源本 Z
```

那正文还是会写成证明题。

结论：

```text
下一轮检查不看文档有没有写，而看中间产物是否把它传到 writer。
```

### 2.2 “高压场面优选”不能替代“源本有效性迁移”

高压场面优选只能解决：

```text
这一场怎么更有戏。
```

它解决不了：

```text
这 10 集观众到底追什么；
爽点周期有没有被打乱；
男女主关系拉扯有没有推进；
卡点有没有保留气口；
信息差有没有接住；
角色状态有没有连续。
```

所以 v5 只能局部保留。

### 2.3 “防同构”不能压过“防洗飞”

如果为了不像源本，把车祸改成档案室，把人戏改成系统，把关系冲突改成文件审批，这不是创新，是短剧降级。

正确顺序应该是：

```text
先保住源本有效体验；
再避开源本高辨识外形；
再找新壳里同等强度的承载方式。
```

### 2.4 “写作前状态”比“写完后追踪”更关键

Story memory 写完后保存当然需要。

但真正影响正文的是写之前 writer 是否知道：

```text
谁现在误会谁；
谁在忍；
谁已经快崩；
谁握有主动权；
观众正等哪一下；
这一集不能提前说破什么。
```

这也是 v4 草案里 `Story Operating State` 有价值的地方。

## 3. 下一步重构时的保留策略

### 必须保留为主链路

- 源本拆文包
- 洗稿边界包
- Beat sheet / outline
- Episode function map
- Scene packet
- Screenplay draft
- Rewrite report
- Dialogue polish
- Quality gate
- Story memory checkpoint
- Independent reviewer
- Version comparison

这些不应该被 v5 或任何新草案替换掉。

### 必须补强连接

- 源本拆文包 -> Episode function map
- Episode function map -> Scene packet
- Scene packet -> Screenplay draft
- Source Bible / Story state -> Writer brief
- Author gate -> Reviewer -> 主控回滚点

补强连接不是加规则，而是防止分析停在上游。

### 可以从 v4 草案吸收

- `Source Retention Anchor`
- `Story Operating State`
- `Pressure-chain Writer Brief`
- `Pressure-chain Patch Brief`

但吸收方式不能让项目变成原创剧本生成器。

它们只能回答：

```text
源本好看的东西现在在故事里如何继续发动。
```

### 可以从 v5 草案吸收

- `Scene Option Pitch`
- `Scene Choice Gate`
- `Version Preference Review`

但只放在：

```text
高压节点 / 场面迁移 / 版本优选
```

不能升成全局主轴。

## 4. 当前最短人话结论

现在不是“再补一条规则”的阶段。

也不是“把 v5 继续修好”的阶段。

当前最该做的是：

```text
拿这张对账表，检查正式 workflow 每一步有没有把原项目已有能力传下去。
```

如果没传下去，修连接。

如果放错层，调位置。

如果被 v5 这类局部草案挤掉，恢复。

如果只是某个样本不好，不要把样本补丁写回主机制。
