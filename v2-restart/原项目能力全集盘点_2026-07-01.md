# 原项目能力全集盘点

> 日期：2026-07-01
> 状态：能力总账 / 防漂移盘点，不是新 workflow。
> 目的：把原项目已经沉淀过的分析、写作、自检、追踪、外部资产能力全部拉回一张总图，防止后续因为某个局部失败点，把项目重新缩成另一个窄产品。

---

## 0. 先说结论

我之前把 `workflow_spec_v5_创作优选草案.md` 放大成主轴，这是错的。

v5 只抓住了一个局部问题：

```text
高压节点要做场面候选和优选，不能让新本降级成弱场面。
```

但原项目远不止这个。

原项目至少已经包含这些能力：

```text
需求确认
源本拆文
源本有效性骨架
留存 / 钩子 / 卡点
爽点 / 压抑释放
节奏 / 情绪密度 / 信息密度 / 情节密度
主线期待 / 信息差 / 真相揭露
人物关系 / 拉扯 / 反派压力 / 角色状态
变量迁移 / 同构扣除 / 新壳等效承载
阶段骨架 / 分集功能 / 高压场面
scene draft / screenplay draft / dialogue polish
视听承载 / 可生产性 / page-to-screen
story memory / continuity
author gate / clean reviewer / rewrite report
版本对比 / 样本验证 / 用户导演反馈回流
```

所以当前正确动作不是继续修 v5，也不是再发明一套主流程。

正确动作是：

```text
先确认这些能力在当前正式链路里有没有保留；
如果没保留，判断是丢了、放错层、还是只写了但没传到下游。
```

---

## 1. 产品锚点不能变

当前产品仍以 `PRD_v4.md` 为准：

**把一个已被市场验证或值得参考的短剧源本，洗成一个新壳下仍然好看、能追、能付费的新剧本。**

这句话里有三个硬点：

| 硬点 | 含义 | 不能滑向 |
|---|---|---|
| 已验证源本 | 标准首先来自源本为什么有效 | 从零原创工具 |
| 洗成新壳 | 不是改名复刻，也不是照搬桥段 | 查词替换 / 换名稿 |
| 仍然好看能追能付费 | 不只证明流程执行，还要产出短剧强文本 | manifest 完成 / 字段完整 / 证明题 |

因此所有能力都必须服务一件事：

```text
源本让用户继续看和付费的有效性，是否被迁移到新壳里。
```

## 2. 原项目能力总表

### 2.1 需求确认能力

| 能力 | 原有来源 | 应该解决什么 |
|---|---|---|
| 最小需求确认 | `workflow_spec_v2.md` Step 0；`short-drama /start`；`how-to-make-script routing` | 不一开始问爆用户，但先拿到能读本子的最低信息 |
| 二次需求确认 | `workflow_spec_v2.md` Step 3 | 读完源本后，再确认真正影响写法的取舍 |
| 改写力度按变量控制 | `设计提纲_v8.md` 第五部分；`PRD_v4.md` 需求输入 | 不把“中度/深度”当固定模式，也不默认深度重构 |
| 输出语言 / 审稿语言 / 正文用途 | `PRD_v4.md`、`workflow_spec_v2.md` | 出海短剧默认英文对白，但仍需确认是否中文、英文或双语，以及是否给 AI 视频链路使用 |

这部分现在没有根本问题，但必须注意：

```text
需求确认不是表单化用户，而是让 agent 判断后给用户拍板。
```

### 2.2 源本拆文与有效性分析能力

| 能力 | 原有来源 | 应该保留的东西 |
|---|---|---|
| 一句话故事核 / 核心期待 | `设计提纲_v8.md`、`workflow_spec_v2.md` | 观众到底追什么，不让新本写散 |
| 五层分析 | `设计提纲_v8.md` | 留存骨架、情绪爽点、主线推进、人物压力、干净度 |
| 源本拆文包 | `workflow_spec_v2.md` Step 1-2；`oh-story` 拆文报告 / 情节节点 / 写作手法 | 把源本拆成下游可用节点，不是剧情摘要 |
| 高价值节点表 | `workflow_spec_v2.md`；`oh-story/demo_拆文样本/情节节点.md` | 节点类型、情绪强度、涉及人物、写作手法、观众体验功能、关系/权力变化、信息差 |
| 源本水分 / 烂点 / 禁止复用外形 | `设计提纲_v8.md`；`workflow_spec_v2.md` | 区分有效机制、外壳、水分、烂点，防止继承坏东西 |

这部分不是可有可无。

如果这里变薄，下游就只会拿到：

```text
某集要揭露什么 / 某节点要完成什么剧情功能
```

而不是：

```text
这一集为什么让人心痒、痛、气、爽、期待下一集。
```

### 2.3 留存 / 钩子 / 卡点能力

| 能力 | 原有来源 | 应该进入哪里 |
|---|---|---|
| Hook 类型 | `short-drama/references/hook-design.md` | 源本拆文、Episode function map、reviewer |
| End Button / 卡点 | `workflow_spec_v2.md`；外部研究 `Hook / Friction / Spike / Button` | 分集功能、集尾设计、版本对比 |
| 付费 / 关键留存节点 | `设计提纲_v8.md`；`short-drama/paywall-design.md` | 阶段骨架、节奏重排，不当硬比例 |
| 卡在答案前一拍 | 导演反馈；`hook-design.md` | 正文结尾画面/声音/动作，不把答案说完 |

这里要防两个错：

1. 不要变成“一集必须几个钩子”的机械规则。
2. 也不能因为怕规则怪谈，把源本的留存惯性丢掉。

正确口径：

```text
强继承的是追看功能和卡点气口；
具体位置、形式、道具、台词按新壳和改写力度等效迁移。
```

### 2.4 爽点 / 压抑释放能力

| 能力 | 原有来源 | 应该看什么 |
|---|---|---|
| 爽点矩阵 | `short-drama/references/satisfaction-matrix.md` | 身份碾压、打脸复仇、逆袭翻盘、情感爆发、悬念揭秘等 |
| 压抑 -> 释放 | `设计提纲_v8.md`；导演 note | 爽之前有没有先压低、羞辱、误判、看衰、危险 |
| 爽点兑现周期 | `设计提纲_v8.md` | 不能拖到泄气，也不能太早掏空 |
| 爽点形式等效 | `设计提纲_v8.md` 变量表 | 原形式可换，但观众吃到的释放要在 |

v5 最大遗漏之一，就是容易只看“场面有没有强”，但没有把爽点的蓄力、释放、兑现周期放回全局。

### 2.5 节奏 / 密度能力

| 能力 | 原有来源 | 用途 |
|---|---|---|
| 情绪密度 | Toonflow 研究记录；`设计提纲_v8.md` 情绪峰谷 | 看情绪是否平、冷、无爆点 |
| 信息密度 | `设计提纲_v8.md` 主线推进；外部研究 | 看每集/每场有没有新信息，而不是对白空转 |
| 情节密度 | 外部研究；ReelForge `power_shift` | 看局势、权力、关系有没有变化 |
| 单集微结构 | `short-drama/rhythm-curve.md`；外部研究 | 开头钩、中段升级、最大刺激、结尾钩 |
| 节奏惯性 / 阶段比例 | `设计提纲_v8.md` 变量表 | 集数不变时可贴近；集数变化时按阶段重排 |

这部分必须保留在分析、写作和自检三处。

但它不应该变成数值打分。

正确用法：

```text
分析源本：这集靠什么密度留住人。
写新本：这一集有没有等效推进和刺激。
自检：有没有空转、注水、冷场、提前释放。
```

### 2.6 主线期待 / 信息差 / 真相揭露能力

| 能力 | 原有来源 | 用途 |
|---|---|---|
| 主线期待链 | `设计提纲_v8.md` | 观众一直在等什么，防止新本散掉 |
| 信息差状态 | `workflow_spec_v2.md` 源本拆文包；story memory | 观众、主角、反派分别知道什么 |
| 真相揭露路径 | `workflow_spec_v2.md` | 什么时候给答案，什么时候升级问题 |
| Do Not Consume | `workflow_spec_v2.md` Episode map / Scene packet | 防止把下一集爽点提前说完 |

这部分如果丢了，新本会出现两种坏形态：

```text
表面很热闹，但观众不知道在追什么；
或者每集都解释很多，但没有新的追问。
```

### 2.7 人物关系 / 拉扯 / 反派压力能力

| 能力 | 原有来源 | 应该保留什么 |
|---|---|---|
| 人物关系功能 | `设计提纲_v8.md` 变量表 | 压迫者、裁决者、见证者、保护者、背叛者等功能不能无故消失 |
| 拉扯 / 误会 / 靠近退开 | 导演 note；`workflow_spec_v2.md` 通过标准；`workflow_spec_v4` Story Operating State | 关系不是静态设定，要在场面中进退、误判、试探、伤害、保护 |
| 反派压力升级 | `设计提纲_v8.md`；`short-drama/villain-design.md` | 反派身份可换，压力阶梯要在 |
| 角色状态 | `oh-story` 追踪文件；`story_memory_checkpoint` | 人物上一场的情绪、身体、主动权不能断 |
| 主角驱动力 | `设计提纲_v8.md` 人物压力层 | 人设必须驱动行动，不是贴标签 |

这部分不能被“场面优选”吞掉。

例如一个场面动作很强，但男女主关系没有推进、误会没有升级、拉扯没有变化，仍然可能不是好短剧。

### 2.8 变量迁移 / 同构扣除能力

| 能力 | 原有来源 | 用途 |
|---|---|---|
| 四类迁移策略 | `设计提纲_v8.md` | 强继承、等效迁移、自由改写、必须删除 |
| 洗稿边界包 | `workflow_spec_v2.md` Step 4 | 必须保留体验、必须换源本外形、自由变量、删除烂点 |
| 同构红线 | `PRD_v4.md` 失败类型；`workflow_spec_v2.md` | 防止源本事件链和强桥段整串保留 |
| 内部源本回指 | `workflow_spec_v2.md` | 新本关键场面能回到源本功能，但正文不复制源本外形 |

这部分是项目自有能力，外部项目不能直接替代。

但它必须薄。

它不能变成：

```text
全剧万能变量矩阵
```

它应该只回答：

```text
保什么体验，避什么外形，哪些可自由改，哪些必须删。
```

### 2.9 新壳承载 / 场面迁移能力

| 能力 | 原有来源 | 正确位置 |
|---|---|---|
| 新壳承载判断 | `设计提纲_v8.md` 动作4；`PRD_v4.md` | 用户新壳能不能承载源本有效性 |
| 新壳场面迁移卡 | `workflow_spec_v2.md` Step 6 | 高风险节点从源本功能变成新场面动作链 |
| Power shift / visual executability | ReelForge 研究记录；`外部短剧工作台...` | 判断场面结束有没有权力变化、能不能看见 |
| High-pressure Scene Viability | `workflow_spec_v4_机制重构草案.md` | 只可作为机制草案参考，不可直接替代正式 workflow |
| v5 场面优选 | `workflow_spec_v5_创作优选草案.md` | 只能作为此层局部模块：多个候选场面选哪个更强 |

这里是 v5 有价值的地方，也是 v5 过窄的地方。

v5 可以补：

```text
高压节点不要只有一个安全方案，要做候选和优选。
```

但 v5 不能替代：

```text
源本拆文、留存骨架、钩子卡点、爽点周期、人物关系、节奏密度、变量迁移、连续性。
```

### 2.10 骨架 / 分集功能 / 首批闭环能力

| 能力 | 原有来源 | 作用 |
|---|---|---|
| Beat sheet / outline | `workflow_spec_v2.md`；`how-to-make-script`；Dramatron 分层思路 | 先搭骨架，再进单集和场面 |

| 首批留存小闭环 | `设计提纲_v8.md`；`PRD_v4.md` | 首批不是固定集数，而是覆盖第一个可验证闭环 |
| 分批推进 | `PRD_v4.md` | 首批通过后再续写，避免整本方向错了再返工 |

Episode function map 不是剧情摘要，必须至少回答：

```text
Opening Hook
Core Friction
Max Spike
Change
End Button
Do Not Consume
Pressure Progression
Carry-through From Source
```

### 2.11 正文写作能力

| 能力 | 原有来源 | 应该保证什么 |
|---|---|---|
| Scene packet | `workflow_spec_v2.md`；`scene_card / scene_draft` | writer 能只看它写本集，不是信息清单 |
| Screenplay draft | `workflow_spec_v2.md`；`output-format-contracts.md` | 剧本页上写的东西，基本等于 AI 成片里能看见听见的东西 |
| Page-to-screen | `workflow_spec_v2.md` Step 9；how-to-make-script | 不写作者解释、功能说明、心理判断替代画面 |

| AI 出海短剧边界 | `PRD_v4.md`；`workflow_spec_v2.md` | 使用 Hollywood screenplay 基本容器，但服务 AI 生成链路，不是 Final Draft 排版比赛 |

这里的关键不是“格式漂亮”，而是：

```text
剧本写出来的东西能直接变成画面、声音、动作、人物状态。
```

### 2.12 台词与表达能力

| 能力 | 原有来源 | 用途 |
|---|---|---|
| Dialogue polish | `workflow_spec_v2.md`；how-to-make-script | 不只是口语化，而是逐句看功能、潜台词、角色声音、节奏 |
| Expression lens | `expression-lens-triggers.md` | 查解释腔、情绪标签句、完整句病、角色同声 |

如果场面发动机坏了，只润台词不会救。

### 2.13 视听承载 / 可生产性能力

| 能力 | 原有来源 | 正确边界 |
|---|---|---|
| 视听承载 | `设计提纲_v8.md` 自检；`workflow_spec_v2.md` 正文合同；导演反馈 | 关键情绪、大事件、人物转变必须能被看见或听见 |
| 不写抽象概念 | `设计提纲_v8.md` 镜头感说明 | 不能只写“黑化、崩溃、惨烈战争、关系破裂” |
| 可生产拆解 | Jellyfish / shortdrama-pipeline / story-shot-agent | 能拆成场景、角色、道具、动作、声音，不等于做分镜 |
| 不堆镜头术语 | huobao-drama 研究记录 | 剧本阶段写动作、表情、空间、后果；分镜阶段再写景别运镜 |

这不是一个孤立规则。

它应该在三个地方同时存在：

```text
源本分析：源本强场面靠什么可见东西成立。
写作：新本用什么可见动作/声音/身体/空间承载。
自检：如果只剩文件、对白、解释、屏幕播报，是否短剧降级。
```

### 2.14 连续性 / Story memory 能力

| 能力 | 原有来源 | 用途 |
|---|---|---|
| 关系状态 | `oh-story` 追踪/关系.md；`workflow_spec_v2.md` Story memory | 男女主、主反、配角功能状态不能跳 |
| 信息差状态 | `workflow_spec_v2.md`；`oh-story` 上下文 | 谁知道什么，谁误会什么，观众知道什么 |
| 伏笔 / 证据状态 | `oh-story` 伏笔.md；story memory checkpoint | 防止证据突然出现、伏笔忘记 |
| 时间线 / 道具 / 空间连续性 | `oh-story` 时间线；story-shot-agent continuity guardian | 防止为了爽点硬跳 |

这部分不是后勤文档。

短剧节奏越快，越需要它防止：

```text
上一场崩溃，下一场无理由冷静；
上一集卡点，下一集不回应；
证据突然出现；
关系状态无故复位。
```

### 2.15 自检 / Reviewer / 返修能力

| 能力 | 原有来源 | 用途 |
|---|---|---|
| Rewrite report | `workflow_spec_v2.md`；how-to-make-script | 先判断坏在哪层，不直接润色 |
| Quality gate report | `workflow_spec_v2.md`；adaptive quality checking | 先检查产物合同，再查洗稿迁移 |
| 作者自检 | `workflow_spec_v2.md` Step 12 | 作者不能用剧情逻辑对来放行剧本 |
| 干净 reviewer | `PRD_v4.md`；`workflow_spec_v2.md` | 第一轮只读正文，第二轮再查源本迁移 |
| 版本对比 | `workflow_spec_v2.md` Step 15 | 永远和用户指定基线、上一版比，不只看本版自嗨 |
| 反馈回流 | `PRD_v4.md`；样本验证 | 导演/用户反馈要定位失败层，不逐条堆规则 |

这里要特别区分：

```text
execution pass != creative pass != source audit pass != delivery pass
```

manifest 只能证明流程执行，不证明剧本好。

### 2.16 样本验证能力

样本验证不是为了把某一版剧本修到能用，而是为了证明 workflow 哪里有效、哪里失效。

已经沉淀过的样本教训包括：

| 样本 / 阶段 | 主要教训 |
|---|---|
| V3 导演反馈 | 不是只改名；还暴露台词多、视听弱、情绪冷、卡点说完、关系拉扯不足 |
| V5 / V7 | scene/action 链比纯证据表有效，但仍可能术语重、机制感重 |
| V8-V12 | 只加不减会让文本越来越像程序；机制词、key、授权、屏幕、档案会吞掉人戏 |
| V13 / V14 | 分析/创作/审计/治理拆层有正向信号，但仍会被结果任务单污染 |
| V15 | manifest complete 和 reviewer pass 不等于最终剧本稳定合格；仍需判断是否程序化、是否保留广义短剧有效性 |

样本教训不能变成某一集的硬规则。

它应该回到机制层：

```text
哪个能力没传下去？
哪个中间产物诱导了错误？
```

## 4. 当前 v5 草案错在哪里

v5 的有效部分：

```text
高压节点需要候选场面、优选判断和版本偏好，不要只有一个安全方案。
```

v5 的错误部分：

| 错误 | 具体表现 |
---|---|
| 把局部模块放大成主 workflow | 用“场面优选”覆盖了整条洗稿链 |
| 丢掉源本广义有效性 | 留存、钩子、卡点、爽点、节奏、关系、信息差只剩背景 |
| 太像原创场面生成 | 容易从“源本为什么让人追”漂到“新场面哪个更好” |
| 没有承接完整变量体系 | 强继承 / 等效迁移 / 自由改写 / 必须删除被压窄 |
| 没有覆盖全局追踪 | Story memory、角色状态、伏笔、时间线弱化 |
| 容易继续单点修复 | 只修“高压场好不好”，不查前后链条是否丢信息 |

因此 v5 只能降级为：

```text
新壳场面迁移 / 高压节点优选 的局部参考。
```

它不能替代：

```text
PRD_v4.md
workflow_spec_v2.md
设计提纲_v8.md
workflow_spec_v4_机制重构草案.md
```

## 5. 当前要查的不是“还缺哪条规则”，而是三类断点

### 5.1 能力是否被保留

例如：

```text
源本分析里有钩子、爽点、拉扯、节奏；
但当前 workflow 是否还把它们作为正式输入？
```

如果只在参考文档里有，而执行链路不消费，就等于没有。

### 5.2 能力是否放错层

例如：

```text
视听承载如果只放 reviewer，自然会变成写完后补丁；
它必须在源本分析、scene packet、screenplay draft 都有位置。
```

同理：

```text
拉扯感如果只在自检里说“加强拉扯”，就没用；
它必须先进入关系状态、场面压力、人物选择和卡点设计。
```

### 5.3 能力是否传到下游

这是目前最危险的断点：

```text
分析层读懂了；
写作包没带下去；
正文自然写不出来；
reviewer 再怎么骂也只能返修局部。
```

判断标准不是“文档有没有写”，而是：

```text
分析项 -> episode map -> scene packet / writer brief -> screenplay -> reviewer
```

能不能一路追到。

## 6. 下一步建议

不要继续修 V15，也不要继续补 v5。

下一步应该做一张对账表：

```text
原项目能力项
当前正式 workflow 是否保留
保留在哪一步
是否传到 writer
是否传到 reviewer
当前样本暴露的失败是否与它有关
结论：保留 / 恢复 / 降级 / 删除 / 只作参考
```

对账对象至少包括：

- `PRD_v4.md`
- `workflow_spec_v2.md`
- `设计提纲_v8.md`
- `workflow_spec_v4_机制重构草案.md`
- `外部短剧工作台与写作流程研究_2026-06-30.md`
- `PRD_v4_workflow_环节来源审计_2026-06-30.md`
- `参考资产/`
- `样本验证/飞书三星样本/` 的 V3-V15 对比和 reviewer 结论

只有对账完成后，才能判断：

```text
是小修 workflow_spec_v2；
还是把 v4 草案某些机制并回正式链路；
还是需要重写一个正式 workflow_spec_v3。
```

现在直接改 workflow，风险很高，因为很容易再次把一个局部修法放大成全局方案。
