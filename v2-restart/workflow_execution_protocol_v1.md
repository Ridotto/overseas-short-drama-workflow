# 海外短剧洗稿器 · workflow 运行记录协议 v1

> 作用：防止 agent 口头说“跑了流程”，但不把每一层实际产物和交接关系留下来。  
> 注意：这是运行记录，不是质量证明，不是审计表，不是 writer 输入。

---

## 1. 核心原则

任何 agent 只要声称“跑 workflow / 重跑 / 按流程产出”，必须留下本轮 `run_log`。

`run_log` 只回答三件事：

1. 本轮到底用了哪个 workflow；
2. 每个工位实际产出了什么；
3. 产物交给了下一层什么。

`run_log` 不能回答：

- 剧本好不好；
- 是否可以送审；
- 最终创作结论；
- writer 写得是否有短剧感；
- reviewer 是否同意。

这些必须由最终剧本和横向对比判断，不能由运行记录替代。

---

## 2. 禁止把运行记录变成证明系统

运行记录里禁止使用以下写法：

- 流程证明术语；
- “本步骤通过”；
- “本步骤质量合格”；
- “短剧感已经解决”；
- “源本有效性迁移成功”；
- “complete 所以可交付”。

允许使用的状态只有：

| 状态 | 含义 |
|---|---|
| `done` | 这一层产物已经生成 |
| `needs_rework` | 这一层发现必须回滚的问题 |
| `blocked` | 缺输入或冲突，不能继续 |
| `skipped` | 本轮明确不需要，必须写原因 |

`done` 只表示“这个工位有产物”，不表示“产物质量好”。

---

## 3. 文件位置和命名

建议命名：

```text
样本验证/<样本名>/<版本目录>/run_log_<版本号>.md
```

运行记录应放在本轮产物同目录。

它不要求必须是第一个正式产物，但必须在主控最终判断前补齐。

如果本轮没有运行记录，只能说：

```text
本轮有产物，但无法确认每层交接是否按 workflow 执行。
```

不能说：

```text
已完整按 workflow 跑完。
```

## 5. writer 输入隔离

`run_log` 永远不能给 writer 当输入。

writer 不读取：

- `run_log`；
- reviewer 报告；
- 作者自检；
- 旧版横向对比；
- 临时诊断；
- 流程状态；
- 流程证明语言。

writer 输入分两层，不能混在一起：

- P5 场景施工稿的项目事实输入只读取 `P4 创作蓝图包`。P4 必须已经把当前批的分集承载、追看债务、角色状态线、DNC 和禁用外形消化进去。
- P5 仍可加载 workflow 指定的方法资产，但方法资产只提供写法协议，不能提供新的剧情事实、角色关系、洗稿边界或付费排程。
- P6 正文初稿的项目事实输入只读取 `P5 场景施工稿`。P6 不能再读取当前故事状态、洗稿边界、外部手艺库或 P4 原料。
- P7 正文修订稿只读取 `P6 首批正文初稿`，只修表达，不重设结构。

如果 P5、P6 或 P7 需要回头读更早项目材料，说明上一层交接产物不完整，不能假装继续写好。

---

## 6. 主控最终判断

主控最终不能只看 `run_log`。

主控必须看：

1. 最终剧本正文；
2. writer 施工稿是否真的驱动正文；
3. 源本有效性是否活到新本；
4. 0xsline / short-drama 手艺是否变成写作动作；
5. reviewer 是否越权；
6. 与 V5 / V22 / V24 / V25 等实际产物的横向对比。

`run_log` 只负责告诉主控：

```text
这一轮每个工位到底产出了什么，交接有没有脏。
```

它不能替主控判断：

```text
剧本是否好看、能追、能付费。
```
