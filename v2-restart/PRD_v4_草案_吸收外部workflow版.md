# 海外短剧洗稿器 · PRD v4 草案

> 日期：2026-06-30
> 状态：给 Cloud / Claude review 的草案，不直接替换 `PRD_v3.md`
> 基础版本：`PRD_v3.md`
> 本版核心变化：直接吸收外部成熟项目中可用的 workflow 工序，补齐“源本分析 -> 新本正文”之间的断层

---

## 0. 这版到底改什么

v3 的大方向是对的：产品仍然是“把一个已验证短剧源本，洗成新壳下仍然好看、能追、能付费的新剧本”。

v4 草案不改产品定义，不把产品变成评分器，不做网页 App，不做视频生产。

本版只解决一个问题：

**分析源本之后，不能直接靠模型写正文。中间必须有足够具体的改编工序，把源本有效性转成新本事件、新本场面和单集写作包。**

这次直接吸收外部项目里能用的 7 个工序：

| 直接吸收的工序 | 来源 | 用来解决什么 |
|---|---|---|
| 源本事件载体拆解表 | Toonflow 的事件提取 + oh-story 的情节节点拆解 | 不只知道“源本有效”，还知道它靠什么事件、画面、证据、动作成立 |
| 新事件载体迁移表 | Toonflow 的 adaptation strategy | 防止源本事件链整串换名复刻 |
| 正文前策略审核 | Toonflow 的 supervision | 写正文前先拦住“改名复刻”和“场面降级” |
| 单集写作包 | Toonflow 逐集输入 + 0xsline 分集目录 | 避免写作时上下文过大，被源本事件链拖着跑 |
| 正文后台词压缩 pass | how-to-make-script 的 dialogue polish / rewrite doctor 思路 | 解决台词水、解释腔、电视剧聊天 |
| 短剧场面 badcase 检查 | ReelForge 的 power shift / visual executability / badcase repair | 抓站桩、文件替代场面、没有权力变化、卡点说完 |
| 状态追踪 | oh-story 的关系/时间线/伏笔/角色状态追踪 | 防止人物关系、误会、信息差、伏笔在分批写作中漂移 |

注意：这些不是新增规则大全，而是新增或强化“中间产物”和“通过关口”。

---

## 1. 产品一句话

把一个已被市场验证过的短剧源本，洗成一个新壳下仍然好看、能追、能付费的新剧本。

产品交付的是剧本文字，不是分析报告。

核心判断不变：

**新本没有洗飞源本有效性，同时不是源本改名复刻，并且作为一个新短剧自己站得住。**

v4 增加一句执行口径：

**源本有效性必须通过新事件载体和新短剧场面承载出来；不能只迁移剧情功能，也不能只换名词。**

---

## 2. 产品不是

- 不是评分器。
- 不是从零原创工具。
- 不是短剧标准库套模板工具。
- 不是查词替换式换皮工具。
- 不是把导演反馈拆成一百条规则的工具。
- 不是视频、分镜、投流片段或成片工具。
- 不是一键生成完整长剧的黑箱工具。

---

## 3. 核心失败类型

v4 把失败类型拆清楚，避免混在一起。

| 失败类型 | 含义 | 例子 | 处理 |
|---|---|---|---|
| 洗飞 | 源本让人追下去的有效性丢了 | 源本核心期待、卡点、情绪走势、关系功能被改没 | 回到源本有效性分析 |
| 改名复刻 | 源本事件链和强桥段整串保留，只换身份和名词 | 黑手党/医院/亲子鉴定换成董事会/档案/信托文件，但结构一眼同源 | 回到新事件载体迁移表 |
| 短剧降级 | 事件变新了，但场面变弱、变冷、变文件对白 | 车祸冲击换成档案室交钥匙；情绪爆发换成会议宣读 | 回到关键场面草案 |
| 正文长剧化 | 剧情逻辑对，但台词多、人物冷、站着聊天 | 第 7 集多轮低信息对白 | 回到正文后短剧化返修 |
| 状态漂移 | 分批写作后，关系、误会、伏笔、人物状态接不上 | 第 8 集角色反应不符合前面情绪债 | 回到状态追踪 |

---

## 4. 第一阶段总 workflow

第一阶段仍然先跑 skill / workflow 原型，不先做网页 App。

v4 的完整内部流程：

```text
源本 + 用户需求
  -> 1. 需求对齐
  -> 2. 源本有效性摘要
  -> 3. 源本事件载体拆解表
  -> 4. 变量迁移策略
  -> 5. 新事件载体迁移表
  -> 6. 正文前策略审核
  -> 7. 洗稿方案给用户确认
  -> 8. 新本阶段骨架
  -> 9. 分集功能表
  -> 10. 关键场面草案
  -> 11. 状态追踪初始化
  -> 12. 单集写作包
  -> 13. 首批正文初稿
  -> 14. 正文后短剧化返修
  -> 15. 作者自检
  -> 16. 独立 reviewer 分层审稿
  -> 17. 主控汇总交付
  -> 18. 用户 / 导演反馈回流
```

相比 v3，新增或强化的是第 3、5、6、11、12、14、16。

## 5. 详细步骤

### 1. 需求对齐

目标：确认这次到底怎么洗，不让模型默认深度重构，也不让它默认改名复刻。

必须问清：

| 问题 | 作用 |
|---|---|
| 新壳是什么 | 决定身份、职业、场景、证据形态怎么重建 |
| 集数是否变化 | 决定留存节点是否重排 |
| 改写力度 | 决定哪些贴近源本、哪些必须大换 |
| 目标市场 | 决定常识、本地化、敏感设定 |
| 用户特别想保留什么 | 防止误删用户在意的爽点 |
| 用户特别不想要什么 | 防止继承雷点 |

输出：`需求确认摘要`

通过标准：

- 用户能确认方向；
- 主控能判断这次更接近贴近改写、中度改写，还是深度重构；
- 不把“改写力度”当成单一档位，而是后续按变量分别判断。

---

### 2. 源本有效性摘要

目标：先回答源本为什么让观众继续看。

仍按五层看：

| 层 | 看什么 |
|---|---|
| 留存骨架 | 开头、每集钩子、关键卡点、付费或留存节点 |
| 情绪爽点 | 压抑、释放、爽点兑现、高潮递进 |
| 主线推进 | 核心期待、信息差、秘密揭露、每集/每场新进展 |
| 人物压力 | 主角驱动力、关系功能、反派层级、压迫升级 |
| 干净度 | 水分、烂点、AI 味、非视听语言、不可拍描述 |

输出：`源本有效性摘要`

注意：这一步不是写剧情摘要，也不是写长篇报告。它只保留后续改写必须用的判断。

---

### 3. 源本事件载体拆解表

来源：Toonflow 的事件提取 + oh-story 的情节节点拆解。

目标：把“源本为什么有效”拆成可迁移、可禁止复刻、可改写的节点。

适用范围：

- 前 1-3 集；
- 每集结尾钩子；
- 付费点或强留存点；
- 大误会、大羞辱、大危机、大反转、大打脸；
- 身份揭露、火葬场、关系质变；
- 所有同构风险高的桥段。

不要求全剧每场都填，避免流程拖死。

字段：

| 字段 | 要回答什么 |
|---|---|
| 源本节点编号 | 第几集 / 哪一段 / 大致位置 |
| 源本事件载体 | 具体发生了什么事件，不是功能词 |
| 源本场面承载 | 观众直接看到了什么动作、画面、身体反应、空间压力、道具或证据 |
| 观众体验功能 | 让观众爽、急、痛、气、期待、误会、想追哪一种 |
| 信息差状态 | 谁知道、谁不知道、观众知道多少 |
| 关系 / 权力变化 | 这一段结束后谁压过谁，谁靠近谁，谁误会谁，谁失去主动权 |
| 情绪走势 | 从哪里压低，到哪里爆发，或在哪里悬住 |
| 结尾钩子 / 卡点 | 如果是集尾，停在什么问题或动作上 |
| 源本高辨识外形 | 哪些事件、道具、证据、场景、台词太像源本，后续不能复用 |

示例判断：

```text
源本：第二集开头车祸
源本事件载体：女主流产/危机与男主冷酷选择绑定在同一高压事件里
源本场面承载：车祸、血、身体伤害、救不救的即时选择
观众体验功能：危机、心疼、男主误判/无情、追悔火葬场伏笔
不能复用外形：车祸、医院抢救、同样的流产证据链如果新壳承载不了要重建
```

通过标准：

- 不能只写“危机”“反转”“爽点”这种抽象词；
- 必须写清源本靠什么具体事件和场面成立；
- 必须标出不能照搬的外形。

---

### 4. 变量迁移策略

目标：决定每类内容如何处理。

四类策略不变：

| 策略 | 含义 |
|---|---|
| 强继承 | 追看功能、情绪走势、关键期待不能无故丢 |
| 等效迁移 | 原形式不适合新壳，但功能必须保留 |
| 自由改写 | 外壳、身份、职业、地点、对白、桥段包装可换 |
| 必须删除或修复 | 水分、烂点、假动机、虚假悬念、不适配设定 |

v4 的补充：

强继承不是继承事件外形，而是继承观众体验和结构功能。

---

### 5. 新事件载体迁移表

来源：Toonflow 的 adaptation strategy。

目标：把“不能照搬源本外形”真正落到新事件上，防止新本只是换名。

这是 v4 最关键的新中间产物。

字段：

| 字段 | 要回答什么 |
|---|---|
| 对应源本节点 | 回指源本事件载体拆解表 |
| 必须保留的体验 | 观众必须感受到什么 |
| 源本不能复用的外形 | 哪些事件、道具、证据、场景、台词不能再用 |
| 新事件载体 | 新壳下具体发生什么事 |
| 新压力来源 | 谁在压迫，压力从哪里来 |
| 新证据形态 | 用什么承载信息：人证、物证、视频、直播、账目、监控、身体痕迹等 |
| 新行动链 | 人物不是说了什么，而是做了什么 |
| 同构风险 | 高 / 中 / 低，为什么 |
| 场面强度风险 | 新事件会不会变成文件、会议、冷对白 |
| 结论 | 可进入骨架 / 需重建 / 放弃该方案 |

示例：

```text
源本节点：车祸流产危机
必须保留体验：女主身体受伤 + 男主冷酷误判 + 观众心疼/愤怒 + 后续追悔伏笔
不能复用：车祸、医院抢救、同样的二选一事故
新事件载体：必须找一个豪门商战壳里同样即时、可见、有身体代价的危机场面
风险：如果只改成“安全档案室交钥匙”，就是短剧场面降级
结论：不能直接进入正文，需重建事件载体
```

通过标准：

- 如果只是把源本名词替换成新壳名词，不通过；
- 如果新事件只能靠文件和对白解释，不通过；
- 如果新事件保不住源本体验，也不通过；
- 必须能说明“为什么这不是改名复刻”。

---

### 6. 正文前策略审核

来源：Toonflow 的 supervision。

目标：正文写出来之前，先拦掉方向性错误。

审核对象：

- `源本事件载体拆解表`
- `新事件载体迁移表`
- `变量迁移策略`

审核问题：

| 问题 | 判定 |
|---|---|
| 源本有效体验是否明确 | 不明确则回到源本分析 |
| 源本高辨识外形是否扣掉 | 没扣掉则回到事件载体拆解 |
| 新事件载体是否只是换名 | 是则打回 |
| 新事件是否能承载同等短剧冲击 | 不能则打回 |
| 新壳是否自洽 | 不自洽则重建 |
| 是否为了去同构把源本钩子/情绪/关系功能洗没 | 是则打回 |

执行方式：

- 普通节点由主控审核；
- 高风险节点可单开临时 reviewer；
- 审核不写正文，只判断能不能进入写作。

输出：`正文前策略审核意见`

结论：

| 结论 | 含义 |
|---|---|
| 通过 | 可以进入洗稿方案和骨架 |
| 局部返修 | 少数节点需要重建 |
| 阻断 | 当前改写方案不能进入正文 |

---

### 7. 洗稿方案给用户确认

目标：让用户确认方向，而不是让用户看内部大表。

用户可见内容：

- 新壳怎么落；
- 哪些体验强继承；
- 哪些源本外形不会照搬；
- 哪些强节点会用新事件承载；
- 首批覆盖哪个留存闭环；
- 是否存在需要用户拍板的取舍。

注意：

用户不需要看到所有内部字段。用户要看得懂“这版为什么不是换名复刻，也不会洗飞”。

---

### 8. 新本阶段骨架

来源：0xsline 的 plan / outline 思路 + Toonflow 的 story skeleton。

目标：先确定全剧大方向和阶段比例。

要回答：

- 前段怎么抓人；
- 第一个留存小闭环在哪里；
- 第一次大爽点、反转或关系质变在哪里；
- 中段靠什么持续拉扯；
- 后段怎么揭真相、兑现、收尾；
- 如果集数改变，哪些节点按阶段比例重排；
- 哪些源本有效节点必须留在相似阶段，哪些可以重排。

输出：`新本阶段骨架`

注意：

阶段骨架不是固定三幕模板。它服从源本和用户需求。

---

### 9. 分集功能表

来源：0xsline 的 episode-directory + Toonflow 的逐集决策。

目标：保证每集都有功能，不空转，不让模型一口气写飞。

每集只写薄信息：

| 字段 | 要回答什么 |
|---|---|
| 集数 | 第几集 |
| 源本功能引用 | 对应源本哪个追看功能，不是照搬事件 |
| 本集新事件 | 新壳下这一集发生的核心事件 |
| 本集变化 | 关系、局势、信息、情绪至少哪一项发生变化 |
| 最大刺激点 | 这一集最强的一下是什么 |
| 结尾钩子 | 观众为什么必须看下一集 |
| 禁止复用 | 本集不能照搬的源本外形 |

通过标准：

- 不能只写剧情梗概；
- 每集必须有变化；
- 每集结尾必须有追看理由；
- 不要求每集都爆炸，但不能空转。

---

### 10. 关键场面草案

来源：Toonflow 改编策略 + ReelForge visual executability / power shift。

目标：高压节点写正文前，先确认这一场如何成为短剧场面。

适用范围：

- 前 1-3 集；
- 首批每集结尾；
- 强危机、强羞辱、强误会、强打脸；
- 身份揭露；
- 大爽点兑现；
- 关系质变；
- 同构风险高的桥段。

字段：

| 字段 | 要回答什么 |
|---|---|
| 场面目的 | 这一场要制造什么体验 |
| 新事件载体 | 这场具体发生什么 |
| 观众直接看到什么 | 不靠解释，画面上有什么 |
| 人物动作 | 主角、反派、关键配角分别做什么 |
| 身体反应 | 哭、跪、摔、退、僵住、颤抖、失控、被拖走等，按剧情需要使用 |
| 空间压力 | 当众、封闭空间、门口、台上台下、众人围观、直播、监控等 |
| 证据/道具 | 证据怎么被看见，不只是被读出来 |
| 权力变化 | 这场结束谁压过谁 |
| 关系变化 | 谁靠近、谁退开、谁误会更深、谁第一次动摇 |
| 台词原则 | 哪些必须说，哪些必须用动作替代 |
| 卡点 | 停在哪个动作、信息、选择或后果前 |

通过标准：

- 高压节点不能只有文件、会议、冷对白；
- 如果把台词静音，观众仍应大致看懂这场发生了什么情绪/权力变化；
- 不是每场都夸张，但高压节点不能端着。

---

### 11. 状态追踪初始化

来源：oh-story 的关系状态、时间线、伏笔、角色状态。

目标：分批写作时防止人物、误会、信息差、伏笔漂移。

建立四个轻量追踪表：

| 追踪表 | 记录什么 |
|---|---|
| 关系状态 | 主角之间、主角与反派之间的信任/误会/依赖/敌意变化 |
| 信息差状态 | 观众知道什么、主角知道什么、反派知道什么 |
| 伏笔与证据 | 哪些证据已出现、谁看见了、还没兑现什么 |
| 角色状态 | 身体伤害、情绪债、身份暴露程度、当前目标 |

这些不是用户可见交付物，主要给主创和 reviewer 用。

通过标准：

- 首批写作前至少初始化主角关系、核心误会、核心秘密、关键证据；
- 每批写完更新一次；
- reviewer 可以用它判断人物反应是否漂移。

---

### 12. 单集写作包

来源：Toonflow 逐集输入 + 0xsline episode-directory。

目标：主创写某一集时，不直接吞全量源本和全量分析，避免被源本事件链拖走。

每集写作前生成一个小包：

| 字段 | 内容 |
|---|---|
| 上一集结束状态 | 上一集卡在哪里，人物/信息/关系状态是什么 |
| 本集功能 | 本集继承哪个源本追看功能 |
| 本集新事件载体 | 新壳下具体发生什么 |
| 必须保留体验 | 爽、痛、急、误会、反转、拉扯等 |
| 必须发生的变化 | 关系/局势/信息/情绪哪一项变化 |
| 禁止复用外形 | 不准照搬的源本桥段、道具、证据、台词 |
| 关键场面落点 | 本集高压场面怎么拍出来 |
| 台词提醒 | 哪些信息不要靠对白解释 |
| 结尾钩子 | 卡在哪个动作/信息/选择前 |

通过标准：

- 主创能只看这个包写当前集；
- 不需要反复翻大段源本；
- 写出来后能回查是否执行了本集功能。

---

### 13. 首批正文初稿

目标：写出可读剧本文字。

写作原则：

- 主创 agent 必须同时负责分析和写作，不能把源本理解交给另一个 agent 后自己只按标签写。
- 正文必须执行单集写作包。
- 高压节点必须执行关键场面草案。
- 普通过渡集可以克制，但必须快、清楚、少废话。
- 文件、会议、录音、报告、对白可以使用，但只能辅助场面，不能替代关键情绪和冲击。

输出：`首批正文初稿`

---

### 14. 正文后短剧化返修

来源：how-to-make-script dialogue polish / rewrite doctor + ReelForge badcase repair。

目标：正文初稿写完后，不直接进入作者自检，先做一次短剧化返修。

这一步不是重新设计全剧，只改正文层坏味道。

#### 14.1 台词压缩

处理：

- 删重复信息；
- 删寒暄；
- 删解释腔；
- 删“我现在很痛苦/我黑化了/我很愤怒”这种抽象直说；
- 把能用动作、物件、画面、身体反应表达的台词改掉。

判断：

```text
这句台词有没有功能？
它是在逼迫、攻击、揭露、反击、选择、拒绝、威胁、试探，还是只是聊天？
连续两三轮对白有没有新信息、新压力或新关系变化？
```

#### 14.2 场面强化

处理：

- 高压节点补动作、身体反应、空间压迫、当众后果；
- 文件/证据尽量转成可见证据，不只让角色读；
- 让权力变化被看见；
- 让关系拉扯被动作化。

例子：

```text
弱：男主看完文件，沉默。
强：男主听见录音，同时监控画面里出现女主倒地的瞬间；他手指停在暂停键上，会议室里没人敢出声。
```

#### 14.3 卡点修正

处理：

- 不把结果说完；
- 卡在选择落下前、身份揭开前、惩罚执行前、直播按钮按下前、证据播放前；
- 如果一集已经把期待兑现完，补新一轮追问。

#### 14.4 重复功能场删除

处理：

- 如果一场戏只是在重复上一场的信息或情绪，删掉或合并；
- 能用一句话 cover 的场，不单开一场；
- 例如导演指出的 2-1，如果男主一句“把档案室钥匙交还”能完成冷酷功能，就不需要拖一场。

输出：`首批正文修订稿`

---

### 15. 作者自检

目标：主创先对自己的稿子负责。

自检不打分，只回答：

| 问题 | 如果失败 |
|---|---|
| 是否洗飞源本有效性 | 回到源本有效性摘要 |
| 是否改名复刻 | 回到新事件载体迁移表 |
| 是否短剧场面降级 | 回到关键场面草案 |
| 是否台词水、解释腔、电视剧聊天 | 回到正文后短剧化返修 |
| 是否状态漂移 | 回到状态追踪 |
| 是否每集有变化和钩子 | 回到分集功能表 |

作者自检必须说明：

- 自己最担心的问题是什么；
- 哪些地方可能需要 reviewer 重点看；
- 哪些导演反馈风险已经处理过。

---

### 16. 独立 reviewer 分层审稿

来源：how-to-make-script 的 quality gate / rewrite report 思路。

Reviewer 仍然是单开干净临时 agent，不做长期固定工位。

Reviewer 第一轮不要读作者自检。

输入：

- 用户需求；
- 源本有效性摘要；
- 源本事件载体拆解表；
- 新事件载体迁移表；
- 正文前策略审核意见；
- 新本阶段骨架；
- 分集功能表；
- 关键场面草案；
- 状态追踪；
- 首批正文修订稿；
- PRD 验收标准。

Reviewer 输出：

| 结论 | 含义 |
|---|---|
| pass | 可给用户 / 导演确认 |
| revise | 方向对，但有必须修改项 |
| block | 不能继续写后续，必须回到前面某一步 |

Reviewer 必须分层判错：

| 失败层 | 说明 |
|---|---|
| 源本分析失败 | 没读出源本为什么有效 |
| 事件载体迁移失败 | 新本仍是源本事件换名，或新事件承载不了 |
| 场面承载失败 | 高压节点变文件、会议、冷对白、站桩 |
| 分集功能失败 | 单集空转，没有关系/局势/信息/情绪变化 |
| 正文对白失败 | 台词水、解释腔、长剧感 |
| 卡点失败 | 集尾说完，没停在下一集欲望点 |
| 状态追踪失败 | 人物、误会、证据、伏笔前后不接 |

Reviewer 不能只说“加强视听语言”。必须指出具体失败层和返修入口。

---

### 17. 主控汇总交付

主控对比：

- 作者自检；
- 独立 reviewer 意见；
- 必要时的导演反馈。

主控决定：

- 直接交付；
- 局部返修；
- 回到某个上游工序；
- 停止继续写后续。

用户看到：

1. 首批新本；
2. 简短改动说明；
3. 作者自检摘要；
4. reviewer 结论；
5. 需要用户判断的问题。

---

### 18. 用户 / 导演反馈回流

目标：反馈不是无限测试，而是回流到具体工序。

反馈分类：

| 反馈类型 | 回流位置 |
|---|---|
| 只是改名 | 新事件载体迁移表 |
| 视听冲击弱 | 关键场面草案 |
| 台词太多 | 正文后短剧化返修 |
| 拉扯不够 | 关键场面草案 + 关系状态追踪 |
| 节奏拖 | 分集功能表 + 重复功能场删除 |
| 卡点不准 | 分集功能表 + 卡点修正 |
| 人物反应怪 | 状态追踪 + 角色动机 |

反馈不能直接变成新规则。必须先定位失败层。

## 7. 通过标准

第一阶段成功，不等于自动产出爆款。

第一阶段成功是：

1. 用户需求被确认。
2. 源本有效性被读出来。
3. 源本事件载体被拆清楚。
4. 高辨识源本外形被标出。
5. 新事件载体能说明为什么不是改名复刻。
6. 高压节点有短剧场面承载，而不是文件/会议/冷对白。
7. 每集有明确功能和结尾追看理由。
8. 正文经过台词压缩和短剧化返修。
9. 状态追踪能支撑人物关系、误会、证据、伏笔连续。
10. 独立 reviewer 没指出硬失败。
11. 用户 / 导演能判断继续、调整后继续，或停止。

---

## 8. 硬失败

| 硬失败 | 处理 |
|---|---|
| 源本有效性没迁过去 | 回到源本有效性摘要 |
| 源本事件外形整串复用 | 回到源本事件载体拆解表和新事件载体迁移表 |
| 新事件只是名词替换 | 回到新事件载体迁移表 |
| 新事件变弱、变文件、变会议、变冷对白 | 回到关键场面草案 |
| 高压节点只有台词，没有动作/身体/空间/当众后果 | 回到关键场面草案 |
| 台词水、解释腔、电视剧聊天 | 回到正文后短剧化返修 |
| 单集没有关系/局势/信息/情绪变化 | 回到分集功能表 |
| 卡点把结果说完 | 回到分集功能表和卡点修正 |
| 关系、误会、伏笔、证据前后漂移 | 回到状态追踪 |
| 为刺激硬堆钩子爽点 | 回到关键场面草案和分集功能表 |

---

## 9. 风险与边界

| 风险 | 表现 | 应对 |
|---|---|---|
| 规则怪谈 | 每集都填巨大表 | 只对关键节点做事件载体和关键场面草案 |
| 工作稿过厚 | 中间产物拖死流程 | 分层：阶段薄、分集薄、关键场面重点写 |
| 默认深度化 | 为了避同构，把源本好看的节奏和关系洗掉 | 事件载体可换，体验功能不能随便丢 |
| 改名复刻 | 源本事件链整串带入 | 新事件载体迁移表 + 正文前策略审核 |
| 短剧降级 | 新事件变冷、变会议、变文件 | 关键场面等效 + badcase 检查 |
| 写作和分析断裂 | 分析给标签，写作写飞 | 主创同时负责分析和写作，写作使用单集写作包 |
| Reviewer 只看文本好不好 | 没发现失败发生在哪层 | Reviewer 必须分层判错 |
| 过早视频化 | 把分镜/镜头术语塞进 PRD | 剧本阶段只要求可拍场面，不写镜头工程 |

| v3 | v4 草案 |
|---|---|
| 源本节点拆解稿 | 强化为“源本事件载体拆解表” |

| 独立 reviewer | 改成分层审稿 |

## 11. 当前样本问题如何被 v4 处理

### 第 2 集：车祸 -> 档案室降级

v4 会在两个地方拦住：

- `新事件载体迁移表`：判断档案室是否能承载车祸级别的身体危机和情绪冲击。
- `关键场面草案`：如果只能靠交钥匙、文件和对白推进，判定短剧场面降级。

### 第 7 集：台词多、像电视剧聊天

v4 会在正文后短剧化返修处理：

- 删重复对白；
- 标每句台词功能；
- 把解释性信息转成动作、物件、表情、空间压迫；
- 如果关系没有变化，回到关键场面草案。

### 第 8 集：不够 dramatic、站桩

v4 会在关键场面草案处理：

- 明确人物身体反应；
- 明确空间压迫和当众后果；
- 明确权力变化；
- 明确谁跪、谁退、谁失控、谁被踢出局，按剧情需要选择，不机械撒狗血。

### 第 3 集：女主回归不该说太多

v4 会在卡点修正处理：

- 如果视觉姿态和众人反应已经能形成期待，就少说；
- 集尾卡在“她出现，但还没兑现反击”的瞬间。

### 第 10 集：导演觉得好

v4 会把它作为正例：

- 压力到位；
- 拉扯升级；
- 动作卡点明确；
- 可以卡在直播按钮按下前。

## 12. 最终判断

v4 草案不是把 PRD 写厚，而是把成熟外部 workflow 里真正能用的工序拿来补上。

本版最重要的变化不是新增概念，而是：

```text
源本分析不能直接进入正文。
必须先经过事件载体重建、正文前策略审核、单集写作包。
正文写完后还必须做短剧化返修，再进入作者自检和独立 reviewer。
```

这条链路如果成立，能同时处理三个当前最严重的问题：

1. 不是改名复刻；
2. 不因为去同构而洗弱；
3. 不写成电视剧式冷对白。
