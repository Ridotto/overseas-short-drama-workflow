# workflow_spec_v15 工位合同候选版

> 日期：2026-07-01  
> 状态：候选版，未正式化。  
> 上游：`PRD_v4.md`、`workflow_spec_v2.md`、`skill_chain_spec_v1.md`、`层级归位表_2026-07-01.md`、`原项目能力对账表_2026-07-01.md`、V5/V25 样本证据。  
> 目的：不再只写“每层应该做什么”，而是写清“每层只能产什么、不能产什么、下游只能吃什么、失败回哪层”。

---

## 0. 本版为什么存在

v14/V25 显示：

- skill 工位链方向有效；
- writer 施工稿有效；
- 干净 reviewer 基本归位；
- 但产物仍未达送审标准。

V25 的失败不是 reviewer 没有努力，也不是短剧知识没写进文档，而是：

```text
S3 / S4 / S5 / S8 / S9 的交接物还不够干净。
```

具体表现：

- S3/S4 仍可能把源本高辨识外形塞进新本骨架和场面原料；
- S4 仍可能替 writer 排完整高压戏；
- S5 仍可能吃到半成品场面、证明语言和旧外形；
- S8 记录信息重复消费但没有真正拦住；
- S9 发现硬风险却容易降级成“看主控偏好”。

所以 v15 不是大改 PRD，也不是继续修单本剧本。

v15 只改一件事：

```text
把工位链变成工位合同。
```

---

## 1. 产品锚点

产品仍然是：

```text
把已被市场验证或值得参考的短剧源本，
洗成新壳下仍然好看、能追、能付费的新剧本。
```

本版不能漂成：

- 从零原创；
- 0xsline 原创工作台复刻；
- 评分器；
- reviewer 驱动返修机；
- 为单个样本打补丁。

## 1.1 本版不是只管短剧感

v15 的重点是让短剧手艺真正进入 writer 生产动作，但它不能把项目收窄成“只追求短剧感”。

本项目的主目标仍然是洗稿。也就是说，短剧手艺必须服务于源本有效性迁移，而不是替代它。

每次执行都必须同时保住：

- 源本有效性：源本为什么好看、为什么能追、为什么能付费；
- 洗稿边界：哪些体验继承，哪些外形必须替换；
- 新壳承载：新事件必须从新壳自然长出；
- 留存 / 钩子 / 卡点：原本吊住用户的功能不能丢；
- 爽点 / 压抑释放：不能只剩剧情推进；
- 情绪密度 / 信息密度 / 情节密度：不能空转，也不能解释化；
- 主线期待 / 信息差 / 真相揭露：观众追问和兑现路径必须稳；
- 人物关系 / 拉扯 / 角色状态：不能所有角色只负责交代信息；
- 权力变化 / 打脸 / 反派压力：高压节点必须改变局势；
- Do Not Consume：后续火葬场、反转、关系击穿不能提前消费；
- 台词功能 / 视听表达 / page-to-screen：最终剧本必须可看可听；
- 连续性 / 伏笔 / story memory：长链路不能状态漂移；
- 海外市场语感和合规禁区；
- 用户 / 导演反馈归层回流。

如果某个候选 workflow 只让“短剧手艺”更强，但让以上任一能力断掉，不能通过。

## 2. 执行链路

```text
0. 主控建 run state / run log
1. S0 需求确认
2. S1 源本拆文
3. S2 洗稿边界 / 变量迁移
4. S3 新本故事架构
5. S4 场面承载原料
6. S5 writer 单集施工稿 / 写作卡
7. S6 screenplay 正文写作
8. S7 台词 / 视听 polish
9. S8 连续性 / 真相 / 信息边界
10. S9 作者自检与返修路由
11. S10 干净 reviewer
12. S11 主控版本对比与拍板
```

## 2.1 唯一交接链

本 workflow 的交接原则很简单：

```text
A 环节产出 X；
B 环节只吃 X；
B 环节产出 Y；
C 环节只吃 Y。
```

每一层可以把上游必要信息带进自己的产物，但下游不回头翻旧包。  
如果下游必须回头读更早的材料，说明上一层产物没做好，应该回上一层重做，而不是让下游加读材料。

| 环节 | 吃什么 | 产什么 | 下一层只吃什么 |
|---|---|---|---|
| S0 需求确认 | 用户需求、源本入口 | `最小需求确认` | S1 只吃 `最小需求确认 + 源本` |
| S1 源本拆文 | `最小需求确认 + 源本` | `源本拆文包` | S2 只吃 `源本拆文包` |
| S2 洗稿边界 | `源本拆文包` | `洗稿边界包` | S3 只吃 `洗稿边界包` |
| S3 新本故事架构 | `洗稿边界包` | `新本故事开发包` | S4 只吃 `新本故事开发包` |
| S4 场面承载原料 | `新本故事开发包` | `场面承载原料包` | S5 只吃 `场面承载原料包` |
| S5 writer 施工稿 | `场面承载原料包` | `单集短剧施工稿` | S6 只吃 `单集短剧施工稿` |
| S6 screenplay 正文 | `单集短剧施工稿` | `screenplay 正文初稿` | S7 只吃 `screenplay 正文初稿` |
| S7 台词/视听 polish | `screenplay 正文初稿` | `台词视听 polish 稿` | S8 只吃 `台词视听 polish 稿` |
| S8 连续性/信息边界 | `台词视听 polish 稿` | `story memory checkpoint` | S9 只吃 `polish 稿 + checkpoint` |
| S9 作者自检 | `polish 稿 + checkpoint` | `作者自检与返修路由` | S10/S11 只吃自检结论和最终稿 |
| S10 干净 reviewer | 用户需求、最终稿、必要边界摘要 | `reviewer report` | S11 只吃 `reviewer report` |
| S11 主控拍板 | 最终稿、自检、reviewer、run log、版本对比 | `主控版本对比与拍板` | 用户只看交付结论和必要说明 |

注意：

- “只吃 X”不是说 X 只能很薄，而是 X 必须把下一层需要的信息消化好；
- S5 的 `单集短剧施工稿` 必须内嵌本集必要故事状态、禁用外形、DNC、输出语言和短剧手艺使用；
- S6 正文 writer 不再打开 S1-S4；
- run log、reviewer、自检、旧版对比永远不能进入 S5/S6。

本版继续保留 v14 的产品形态：

- 用户只和主控 agent 对话；
- 主创链可由同一个 agent 连续执行，但每步必须显式切换工位；
- reviewer 干净隔离；
- 主控负责反馈归层和最终拍板。

## 3. 本版新增的硬合同

### C1. run log 不等于创作合格

run log 只记录每个工位是否有产物、交接给下游什么、是否混入不该混入的材料。

run log 不能写成创作质量证明。

每份 run log 顶部必须写：

```text
run log 已补齐，不等于剧本可交付。
工位 done 只代表该工位有产物，不代表剧本可交付。
```

writer 产物中禁止出现：

- 本工位 done；
- 本工位加载了哪些文档；
- 本轮流程状态判断；
- reviewer / run log / 流程证明术语。

这些只出现在 run log 或主控文件里，且不能进入 writer 输入。

### C2. S3 不得锁死具体高压外形

S3 可以锁：

- 真相账本；
- 错信；
- 主追问；
- 阶段追问/兑现/留白；
- 分集功能；
- 角色行动方式；
- 关系/权力/信息状态。

S3 禁止锁：

- 枪抵眉心；
- 车祸弃救；
- 厨房签离婚；
- 吞纸；
- 与源本高辨识外形同构的整串动作；
- 任何已经是正文级高压场面的具体桥段。

如果 S3 写出具体高压外形，不能交给 S4，必须重写为：

```text
体验功能 + 新壳承载方向 + 禁止源本外形。
```

### C3. S4 只给原料，不排戏

S4 必须产出：

- 源本体验功能；
- 新壳承载事实；
- 谁压谁；
- 谁旁观；
- 谁被迫行动；
- 压力如何形成；
- 释放后必须造成什么可见后果；
- 禁止复用的源本外形；
- 不可提前消费信息；
- 可选新壳动作方向。

S4 禁止产出：

- 完整节拍；
- 正文段落；
- 台词；
- 成片级最后一帧；
- 已经替 writer 排好的高压戏。

如果 S4 写成“完整场面方案”，S5 会退化成抄写，必须回 S4 重做原料层。

### C4. S5 只吃 S4 原料包，S6 只吃 S5 施工稿

S5 输入只能是：

- S4 场面承载原料包。

S4 必须已经把以下内容消化进原料包：

- 当前故事状态；
- 本集故事功能；
- 洗稿边界；
- forbidden shape；
- 本集 Do Not Consume；
- 本阶段短剧手艺需求；
- 新壳场面原料。

S5 不读：

- reviewer report；
- 作者自检；
- run log；
- 历史版本对比；
- 厚源本分析全文；
- S1/S2/S3 原文；
- 外部标准库全文；
- 临时诊断；
- 流程状态。

S5 必须产出本集写作卡，而不是任务单。

本集写作卡必须回答：

1. 本集观众正在等什么；
2. 开场第一个可见动作/声音/冲突是什么；
3. 中段如何升压；
4. 爽点前如何压低；
5. 谁做动作，谁被迫反应，谁失去什么；
6. 关系/权力/信息如何变化；
7. 结尾停在哪个未完成动作、声音或画面；
8. 哪些信息不能提前消费；
9. 禁止复用的源本外形；
10. 本集短剧手艺如何使用。

S6 正文 writer 只能读取 S5 写作卡 / 单集施工稿。  
如果 S6 还需要回头读 S1-S4，说明 S5 交接失败，不能继续写正文。

### C5. 外部短剧能力必须变成生产动作

0xsline / short-drama 不能只是“参考过”。

每份参考的接入方式如下：

| 参考能力 | 接入层 | 必须变成什么 |
|---|---|---|
| `opening-rules.md` | S5 | 开场第一个可见动作/声音/冲突 |
| `rhythm-curve.md` | S3/S5/S8 | 本集开场-中段-结尾压放；首批节奏重复风险 |
| `hook-design.md` | S5 | 未回答问题、卡住动作、下一集兑现义务 |
| `satisfaction-matrix.md` | S4/S5 | 压低动作、释放动作、可见后果 |
| `paywall-design.md` | S3/S5 | 付费追问和兑现义务，不照搬固定集数比例 |
| `villain-design.md` | S3/S5 | 反派资源、误判、反扑燃料 |
| `genre-guide.md` | S0/S3/S7 | 题材、市场、语感、角色声线 |
| `compliance-checklist.md` | S10/S11 | 红线辅助，不做创作发动机 |

如果这些能力只出现在“已加载参考”或 reviewer 里，不算生效。

### C6. S8 记录重复信息就必须判断是否回滚

S8 不评价短剧感。

但 S8 必须判断：

- 同一信息是否被重复当作主刀；
- 终局需要的新刀口是否被提前说完；
- Do Not Consume 是否失效；
- 角色错信是否前后冲突；
- 核心真相是否自相矛盾。

如果 S8 记录问题但仍放行，必须说明：

```text
为什么它不是信息边界问题。
```

否则默认回 S3/S8。

### C7. S9 不能把硬风险交给 reviewer 兜底

作者自检发现以下问题时，不能写成“看主控偏好”：

- 源本高辨识外形回流；
- 新壳载体降级；
- 施工稿强度没有进入正文；
- 信息提前消费；
- 核心真相混乱。

必须直接路由：

| 问题 | 回滚层 |
|---|---|
| 源本体验丢失 | S1/S2 |
| 高辨识外形回流 | S2/S3/S4 |
| 场面承载不住 | S4 |
| 施工稿像任务单 | S5 |
| 正文像说明书 | S6/S7 |
| 真相/信息乱 | S3/S8 |
| 台词水 | S7 |

## 4. 每层产物合同

| Step | 产物 | 必须有 | 禁止有 |
|---|---|---|---|
| S0 | `最小需求确认.md` | 源本、新壳、市场、语言、集数、力度、禁区 | 创作方案 |
| S1 | `源本拆文包.md` | 追问、钩子、爽点、压放、关系/权力变化、DNC、源本高辨识外形 | 新壳设计 |
| S2 | `洗稿边界包.md` | 强继承体验、等效迁移、自由改写、必须删除、forbidden shape | 具体新剧情 |
| S3 | `新本故事开发包.md` | 真相账本、错信、追问兑现、阶段骨架、分集功能、角色行动方式 | 具体高压外形 |
| S4 | `场面承载原料包.md` | 压力来源、被迫行动、可见后果、候选新壳动作、禁止外形 | 完整节拍、台词、正文 |
| S5 | `writer单集施工稿.md` | 本集写作卡、一拍一拍行动链、压放、结尾卡点、DNC | 流程证明/run log/reviewer |
| S6 | `screenplay正文初稿.md` | 可看可听、动作/空间/身体/声音、台词有功能、卡点 | 内部标签、说明书、站桩聊天 |
| S7 | `台词视听polish稿.md` | 删水词、保声线、强动作和声音 | 重设主线、替 S4/S5 写戏 |
| S8 | `story_memory_checkpoint.md` | 真相、错信、已播/未播、伏笔、重复信息风险 | 审美评价 |
| S9 | `作者自检与返修路由.md` | 失败层级、返修范围、保留体验、禁止外形 | 把硬风险写成偏好 |
| S10 | `reviewer_report.md` | 致命阻断/应改风险/可选优化、回滚层、suggestion | 逐句代写、生产短剧感 |
| S11 | `主控版本对比与拍板.md` | 是否交付、为何、和 V5/上一版对比、返修层 | 把运行记录完成当质量通过 |

## 5. 关键能力保活检查

每次跑完 S5 后，主控必须抽查：

| 源头能力 | S5 中必须出现的形态 |
|---|---|
| 源本核心期待 | 本集观众正在等什么 |
| 钩子/卡点 | 未回答问题 + 卡住的最后动作/声音/画面 |
| 爽点/压抑释放 | 爽点前如何压低，释放落在哪个动作 |
| 情绪密度 | 本集情绪如何升温，不靠空喊 |
| 信息密度 | 哪个信息推进局势，哪个信息不能说 |
| 人物关系/拉扯 | 谁靠近，谁后退，谁误判，谁怕失去 |
| 权力变化/打脸 | 谁当众失去位置、脸面、资源或主动权 |
| 角色状态 | 当前谁忍、谁崩、谁伪装、谁握主动 |
| 新壳承载 | 该动作是否从新壳自然长出 |
| 短剧手艺 | 开场、节奏、爽点、钩子、台词、视听是否进入写作动作 |

如果 S1/S2/S3 有，S5 没有，不是 writer 自由发挥，而是交接失败。

## 6. 和 v14 的差异

v14 已经推进了：

- 主控 + 主创链 + reviewer 的产品形态；
- skill 工位概念；
- writer 施工稿；
- 分阶段加载 0xsline；
- reviewer 降权。

v15 继续保留这些。

v15 只补 v14 未解决的断口：

1. S3/S4 不准把具体旧外形塞给 writer；
2. S4 不准替 writer 排完整戏；
3. S5 输入必须生产化，不能混入流程证明；
4. 0xsline 能力必须变成写作动作；
5. S8 不能记录信息重复却软放行；
6. S9 不能把硬风险交给 reviewer 兜底；
7. run log 语义固定为执行记录，不是质量证明。

## 7. 验证方式

如果继续验证，使用同一飞书三星样本首批 1-10 集。

对比对象：

- V5：短剧感和场面发动机正向证据；
- V22：较稳但偏平；
- V24：完整链路但 writer 渲染感明显；
- V25：v14 工位链样本，方向有效但未达送审标准；
- 新版 v15 样本。

验收不看“流程是否跑完”，而看：

1. 是否没有洗飞源本追看功能；
2. 是否没有改名复刻；
3. 是否比 V25 更少旧外形回流；
4. 是否比 V25 更稳定地产生动作后果链；
5. 是否至少保住 V5 的场面发动机优势；
6. 0xsline 的开场/节奏/钩子/爽点是否真的进入 S5/S6；
7. S8 是否能拦住重复信息消费；
8. S9 是否能在主创侧打回硬风险；
9. reviewer 是否仍然降权；
10. run log 是否没有被误读成质量通过；
11. 是否保住主线期待、人物关系、拉扯、反派压力、信息差、情绪密度和 story memory；
12. 是否没有为了短剧感牺牲新壳自然性、源本有效性或海外市场语感。

## 8. 当前状态

本版只是候选 workflow。

它不证明问题已经解决。

它只把目前确认的根因转成可执行合同：

```text
工位清楚；
交接物干净；
writer 只吃生产卡；
外部短剧能力变成写作动作；
检查层只回滚，不生产短剧感。
```

正式化前必须跑样本。
