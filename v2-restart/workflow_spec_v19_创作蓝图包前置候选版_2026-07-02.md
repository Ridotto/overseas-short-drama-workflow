## 0. 文档定位

这份文档只定义执行合同，不重复解释产品是什么。

它回答：

- 每层只负责什么；
- 每层只能吃什么；
- 每层必须产出什么；
- 哪些词和写法禁止出现；
- 哪些情况硬阻断；
- 谁负责判定过线；
- 过线证据写在哪里；
- workflow 何时退役、何时能升默认。

它不回答：

- 网页 App 长什么样；
- 分镜怎么做；
- AI 视频链路怎么接；
- 每个提示词具体怎么写。

---

## 1. 总目标

把一个已被市场验证或值得参考的短剧源本，洗成一个新壳下仍然好看、能追、能付费的新剧本。

总通过轴只有一个：

```text
观众看完当前集 / 当前批次后，是否更想继续看，是否愿意为下一段答案付费。
```

视听、钩子、爽点、拉扯、反派 / 对立压力、DNC、台词和连续性都不是独立 checklist。  
它们都必须服务这个追看 / 付费效果。

---

## 2. 四种模式

### 2.1 诊断 / 蓝图模式

只跑到 `创作蓝图包`，不进正文。

适用：

- 用户只想判断源本值不值得洗；
- 用户只想先确认方向、角色、粗纲和当前批戏稿；
- 输入不是完整文本源本，只是摘要、片段、零散材料；
- 系统和用户对“源本为什么好看”尚未对齐。

### 2.2 标准首批模式

默认模式。

跑完整链路到首批 `1-10` 集正文。

### 2.3 长篇多批次模式

全剧粗、当前批细。

规则：

- S3 负责全剧骨架和批次地图；
- S4 只给当前批蓝图；
- S5 只给当前批场景施工稿；
- S6 只写当前批正文；
- 每一批开始前，重新确认当前批细层蓝图；
- 每一批正文后，跑轻版 reviewer。

### 2.4 局部改稿 / 定点返修模式

只接受已有项目上下文。

适用：

- 用户只想修某几集；
- 只想改某个卡点；
- 只想重写某段场面；
- 已有蓝图和正文基础，不需要重跑整条正式链。

这条模式的结果不算默认 workflow 的正向验证证据。

## 3. 准入与例外

### 3.1 正式写稿链准入

正式写稿链默认只接受一个完整文本的主源本。

摘要 / 片段 / 口述 / 零散材料：

- 默认只能进诊断 / 蓝图模式；
- 不能直接进正式正文链。

### 3.2 坏源本

如果主控判断源本质量一般，或不适合直接洗：

- 默认转诊断 / 蓝图模式；
- 不直接承诺正文质量。

坏源本判定采用：

- 红旗清单 + 主控综合判断

红旗示例：

- 主追问弱；
- 强节点稀；
- 重复水戏多；
- 后半塌；
- 拉扯弱；
- 外部反馈差。

### 3.3 高风险改编例外模式

如果用户坚持的方向已经明显超出洗稿器边界，更像原创改编：

- 可以继续；
- 但主控必须明确写出：
  - 这不是标准洗稿链；
  - 质量保障下降；
  - 不计入默认 workflow 的正向验证证据。

### 3.4 用户覆盖系统建议

如果主控明确判断某改法会伤短剧感或洗飞源本：

- 允许继续；
- 但必须显式写出会损失什么；
- 记录为“用户覆盖系统建议”的例外。

### 3.5 无源稿脚本医生模式

如果用户只给现成新稿，不给源本，却要求“改得更短剧”：

- 不进洗稿链；
- 只当脚本医生模式处理；
- 不计入默认 workflow 的正向验证证据。

## 4. 用户可见主链

```text
需求确认
-> 源本有效性摘要 + 洗稿方案摘要（一次前置确认）
-> 创作蓝图包（用户确认）
-> 场景施工稿
-> 正文
-> reviewer 轻版硬伤检查
-> 完整版交付
-> 用户反馈回流
```

用户只和主控 agent 对话。

默认不展示给用户：

- 故事控制包；
- 场景施工稿；
- 完整 reviewer 报告；
- 后台 run log。

如果用户主动要求看 reviewer 结论，主控只提供精简摘要。

## 5. 固定交付物

### 用户可见

- `需求确认摘要`
- `源本有效性摘要`
- `洗稿方案摘要`
- `创作蓝图包`
- `首批正文修订稿`

### 内部关键产物

- `故事控制包`
- `场景施工稿`
- `台词视听 polish 稿`
- `story memory checkpoint`
- `run log`

## 6. 总工位链

```text
S0 需求确认
-> S1 源本拆文
-> S2 洗稿边界
-> S3 故事控制包
-> S0b 二次需求确认
-> S4 创作蓝图包
-> 用户确认蓝图
-> S5 场景施工稿
-> S6 screenplay 正文
-> S7 台词 / 视听 polish
-> S8 continuity / 信息边界
-> S9 作者自检
-> S10 干净 reviewer
-> S11 主控拍板
```

唯一交接原则：

```text
A 层产出 X；
B 层只吃 X；
B 层产出 Y；
C 层只吃 Y。
```

下游不能回头翻上游厚材料。  
如果下游必须回头翻，说明上一层产物不合格，应回上一层重做。

---

## 7. 关键层硬门槛

关键层只拦四层：

- `S4 创作蓝图包`
- `S5 场景施工稿`
- `S6 screenplay 正文`
- `S8 continuity / 信息边界`

门槛责任：

- 每个关键工位先自检自己过没过线；
- 主控只做 gate，检查是否真的过线、证据够不够、有没有带病往下传。

不新增 checker，不靠 reviewer 兜底。

过线证据必须写在产物头部，不只写在 `run log` 里。

## 8. 工位合同

### S0 需求确认

**只负责：** 拿到可开工边界。

**输入：**

- 用户需求；
- 源本入口；
- 用户已知的新壳、集数、语言、市场、改写力度和禁区。

**必须输出：** `需求确认摘要`

必须回答：

- 源本是什么；
- 新壳是什么；
- 目标市场是什么；
- 输出语言 / 审稿语言是什么；
- 集数是否改变；
- 改写力度按哪些变量控制；
- 用户禁区是什么；
- 正文用途是什么；
- 是否有额外用户材料。

**禁止：**

- 分析源本；
- 设计剧情；
- 默认深度重构；
- 不确认输出需求就往下走。

**硬阻断：**

- 目标市场、语言、集数、用途不清。

### S1 源本拆文

**只负责：** 拆源本为什么能追、能付费。

**输入：**

- `需求确认摘要`
- 完整主源本文本

**必须输出：** `源本有效性摘要`

必须拆出：

- 核心期待；
- 留存骨架；
- 钩子 / 卡点；
- 付费节点；
- 爽点和压抑释放；
- 情绪密度 / 信息密度 / 情节密度；
- 人物关系和拉扯；
- 权力变化；
- 角色状态；
- 反派 / 对立压力源；
- DNC；
- callback / payoff 债务；
- 源本高辨识外形；
- 源本自相矛盾或含混点。

**禁止：**

- 设计新壳；
- 提前改剧情；
- 只写剧情流水账。

**硬阻断：**

- 用户不同意系统对“源本为什么好看”的理解。

### S2 洗稿边界

**只负责：** 区分什么必须继承，什么必须替换。

**输入：**

- `需求确认摘要`
- `源本有效性摘要`
- 用户额外材料（如有）

**必须输出：** `洗稿方案摘要`

必须写清：

- 强继承体验；
- 等效迁移外形；
- 自由改写变量；
- 必须删除的烂点；
- forbidden shape；
- 新壳可承载范围；
- 例外风险（如用户坚持高同构桥段）。

**禁止：**

- 把防同构压过防洗飞；
- 为了不像源本，把人戏改成文件、审批、屏幕；
- 写新剧情。

**硬阻断：**

- 用户与系统对保什么、换什么没有对齐。

### S3 故事控制包

**只负责：** 生成内部控制层。

**输入：**

- `源本有效性摘要`
- `洗稿方案摘要`
- 用户需求

**必须输出：** `故事控制包`

必须包含：

- 真相账本；
- 角色错信；
- 源本强节点重排表；
- 全剧阶段骨架；
- 批次地图；
- 每批追看债务；
- 反派 / 对立压力源；
- DNC；
- callback / payoff 债务和兑现项；
- 新壳承载硬边界。

强节点映射规则：

- 保阶段顺序；
- 保追看债务和兑现关系；
- 允许并拆；
- 不允许自由重排。

扩写规则：

- 新增节点只能从源本已有压力 / 债务 / 关系 / 对立压力源里长出来；
- 不允许凭空另起一套爽点系统。

**禁止输出：**

- 具体高压桥段外形；
- 台词；
- 场景动作串；
- 风格卖点话术。

**硬阻断：**

- 没有 `源本强节点 -> 批次落点 -> 追看债务`；
- 没有对立压力源；
- DNC 不成清单。

### S0b 二次需求确认

**只负责：** 在进蓝图前再次对齐方向。

**输入：**

- `故事控制包`

**必须输出：** `二次需求确认结论`

必须确认：

- 源本最需要保住的追看 / 付费效果；
- 改写力度和变量边界；
- 新本故事核、题材、世界观和目标市场语感；
- 全剧阶段骨架和首批 / 当前批次范围；
- 明确不能提前消费的信息；
- 明确不能复用的源本高辨识外形；
- 输出语言和正文用途。

**硬阻断：**

- 用户没确认，不能进入 S4；
- 用户在这里大改方向，回 S0/S2/S3，不从 S4 开始硬拧。

### S4 创作蓝图包

**只负责：** 给用户确认、给 writer 作为主上游。

**输入：**

- `故事控制包`
- `二次需求确认结论`

**必须输出：** `创作蓝图包`

必须分两块：

#### 全剧粗层

- 一句话故事核
- 核心角色设定
- 全剧梗概
- 阶段粗纲
- 主追问

#### 当前批细层

- 当前批每集戏稿
- 每集追看债务
- 关键戏发动方式
- 角色本批状态线
- 关系变化摘要
- DNC 摘要
- 禁用外形摘要

每集戏稿最低要求：

- 开场先发生什么可见动作；
- 谁压谁；
- 压力怎么升级；
- 这集最大的释放点是什么；
- 最后一拍停在哪；
- 下一集欠什么答案。

**禁止退化成：**

- 本集功能；
- hook/spike/button 标签；
- 观看冲动总线；
- 爽感类型；
- 强戏候选空间。

形态上限：

- 不得塞入故事控制包原文；
- 不得塞入 reviewer 语言；
- 不得塞入完整场景施工稿；
- 不得塞入 run log / 旧版对比 / 厚分析术语。

**硬阻断：**

- 用户没确认蓝图；
- 当前批任何一集只有功能没有戏；
- 蓝图包厚到用户读不懂。

### S5 场景施工稿

**只负责：** 把蓝图变成可演场景稿。

**输入：**

- 只读 `创作蓝图包`
- 只能读当前批，不得回头翻 `S1-S3`

**必须输出：** `场景施工稿`

输出形态：轻标签 + 戏稿

每个场景组至少包含：

- `目标`
- `动作链`
- `被迫反应`
- `关系/权力变化`
- `最后一拍`
- `DNC`
- `禁用外形`

高压集：

- 必须细到场景组

过渡集：

- 可以按连续戏块写；
- 但必须推进关系 / 权力 / 信息 / 情绪 / 对立压力源中的至少一项；
- 并留下下一步追问；
- 不能只写“本集承上启下”。

必须加载并执行：

- `opening-rules.md` -> 本集前 5 秒用什么动作/画面抓人
- `rhythm-curve.md` -> 开场 / 中段升压 / 释放或卡断分别是哪一下
- `hook-design.md` -> 最后停在什么未完成动作 / 答案前一拍
- `satisfaction-matrix.md` -> 爽点前怎么压，释放时谁失去什么
- `villain-design.md` -> 本批对立压力源靠什么资源 / 误判继续施压
- `genre-guide.md` -> 仅在确认语感 / 输出语言时使用

**硬阻断：**

- 报告 / 屏幕 / 会议 / 系统播报承担高压戏本体；
- `本集要揭露/要打脸/要卡点` 这类任务单语言；
- `情绪更强一点/短剧感更足` 这类空话；
- 需要回头读厚材料。

### S6 screenplay 正文

**只负责：** 把施工稿写成成片对应的正文。

**输入：**

- 只读 `场景施工稿`

**必须输出：** `首批正文初稿`

格式合同写死：

- 英文 screenplay
- 场景头
- 可见动作
- 人物对白
- `VO/OS`
- 不写心理解释
- 不写内部标签

默认不用明显镜头术语，不写成 shot list。

英文优先级：

- 先保体验，再调英文；
- 目标市场语感优先于直译原表达，但不能洗飞源本体验功能。

**硬阻断：**

- 抽象情绪；
- 报告播报；
- 站桩聊天；
- 多轮低信息对白；
- 正文像说明书；
- 回头读厚材料。

### S7 台词 / 视听 polish

**只负责：** 修表达，不补结构。

**输入：**

- 只读 `首批正文初稿`

**必须输出：** `首批正文修订稿`

只允许：

- 删水词；
- 强化角色声线；
- 把情绪往动作 / 身体 / 空间 / 声音上压。

台词声线默认来自：

- 角色卡
- 本批状态线

不单独拉台词包。

**禁止：**

- 改主结构；
- 重设高压场；
- 替 writer 补关键戏。

如果发现问题不是表达，而是场面太弱，必须打回 `S5`。

### S8 continuity / 信息边界

**只负责：** 查事实、状态、DNC、伏笔和批次交接。

**输入：**

- `故事控制包`
- `首批正文修订稿`

**必须输出：** `story memory checkpoint`

必须检查：

- 真相是否矛盾；
- 角色错信是否一致；
- 哪些信息已经播出；
- 哪些信息还不能播出；
- 角色关系 / 权力状态是否连续；
- 伏笔是否被提前消费；
- callback / payoff 是否被兑现或忘记；
- 下一批追问是否还活着；
- 下一批对立压力源是否还在。

**硬阻断：**

- DNC 失效；
- 真相矛盾；
- 重复信息主刀；
- 角色状态漂移；
- callback 忘记兑现；
- 下一批无追问、无对立压力源或状态断裂。

### S9 作者自检

**只负责：** 先在主创侧发现明显失败，并判断回哪层修。

**输入：**

- S1-S8 全部产物
- 当前正文

**必须输出：** `作者自检与返修路由`

必须判断：

- 是否洗飞；
- 是否改名复刻；
- 是否提前消费；
- 是否核心真相矛盾；
- 施工稿强度是否活到正文；
- 高压节点是否滑成报告 / 会议 / 屏幕 / 冷对白；
- 台词是否水；
- 下一集追问是否还在。

**禁止：**

- 把所有问题都交给 reviewer；
- 为了过检查把正文写得更程序化。

### S10 干净 reviewer

**只负责：** 兜底和分层。

**输入：**

第一轮只读：

- 用户需求
- 最终正文
- 必要边界信息

第二轮才补读：

- 必要源本片段
- S3-S8 关键产物

仍不读：

- 作者自检
- run log
- 主控自评
- 旧版横向对比
- writer 解释稿

**必须输出：** `reviewer 摘要`

只查：

- 洗飞
- 复刻
- 提前消费
- 真相矛盾
- 明显强度漏损
- 状态漂移

每条 issue 必须写：

- 问题在哪
- 为什么影响产品目标
- 应该回哪一层修
- 一个方向性 suggestion

**禁止：**

- 逐句挑刺；
- 把“还能更好”判成必须返修；
- 替 writer 写新场；
- 生产短剧感。

### S11 主控拍板

**只负责：** 对比、归因、反馈回流、交付判断。

**输入：**

- run log
- 本轮全部产物
- 指定对比对象
- reviewer 摘要
- 作者自检

**必须判断：**

- 每层有没有产物；
- 关键层有没有过硬门；
- 新稿是否不低于 V3 的清楚度；
- 是否至少保住 V5 的追看强度；
- 是否显著强于 V22 / V24 / V25 / V32；
- reviewer 是否越权；
- 是否可以交用户；
- 如果不能，回哪一层修。

**禁止：**

- 只看最终剧本；
- 把运行记录完成当质量通过；
- 把样本补丁写成永久规则；
- 让 reviewer 变成质量发动机。

## 9. 过线证据写法

每个关键产物头部都必须有一段简短 gate 结论：

- 当前层通过了什么；
- 因为什么通过；
- 如果没通过，卡在哪一条硬门槛。

只记在 `run log` 里，不算过线证据。

---

## 10. 升默认门槛

workflow 不能只跑一个样本就升默认。

必须：

1. 跑飞书三星样本首批 `1-10`
2. 再跑一个不同题材、不同气质样本首批 `1-10`
3. 第二样本由主控按规则选

只有两样本都过，才允许升默认 workflow。

双门槛：

1. 整体显著强于 `V22 / V24 / V25 / V32`
2. 追看强度至少保住 `V5`

这里的“保住 V5”不是逐集对齐，也不是照抄 V5，而是看首批整体体验信号是否成立：

- 开场能抓人；
- 压力持续升级；
- 至少一次阶段释放或足够强悬置；
- 结尾让人想继续看；
- 高压节点主要靠可见承载；
- 人物拉扯 / 对立压力源不是平的。

这个体验门由主控结合样本对照来判，不由 reviewer 单独拍板。

no-ship gate 规则：

- 主控执行；
- 只保留 4-6 条一票否决项；
- 来源只取历史复发根因 + 首批整体体验门；
- 不从零碎反馈里长；
- 一次性判定，不过就退回链路层，不在同一版里无限 review。
