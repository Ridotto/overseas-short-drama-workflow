# PRD V21 v11 workflow vs V3 / V5 / V20 整体对比

对象：

- V3：`PRD_v3_豪门商战版_首批1-10集.md`
- V5：`PRD_v5_豪门商战版_首批1-10集.md`
- V20：`PRD_v20_v10workflow_豪门商战版_首批1-10集.md`
- V21：`PRD_v21_v11workflow_豪门商战版_首批1-10集.md`

说明：

- V21 主创运行时未读取 V3 / V5 / V20，保持干净运行。
- V21 reviewer 第一轮只读最小需求和正文，第二轮再读源本与本轮中间产物。
- 本文由主控在 reviewer 完成后读取旧版用于整体对比。

---

## 总判断

V21 不是 pass。

V21 的当前状态应该写成：

```text
workflow execution: complete through reviewer
creative outcome: revise
source carry-through: pass with revision risks
delivery outcome: not pass
```

但 V21 也不是 block。它证明 v11 的新增分层有实际价值：

- 源本主体验链没有洗飞；
- 新壳不只是改名，债权、港口、冷链、信托、CEO seal 等已经进入故事；
- Stage Skeleton / Episode Map / Scene Options 在 reviewer round2 里没有被判失败；
- 正文比 V20 更少程序感，不再主要靠认证台和屏幕自动判死；
- Ep8/Ep9 的信息边界比 V20 稳，Ep9 仍保留双胞胎旧物火葬场。

V21 没过的原因不是“流程完全无效”，而是：

```text
新本开发包和 writer brief 已经能把主链交给 writer，
但还不能稳定卡住三类关键问题：
1. 核心概念定义含混；
2. 源本表面动作簇残留；
3. 证据 / 商战动作因果不够可拍、可感、可信。
```

换句话说，v11 解决了“writer 没拿到可写故事包”的一部分问题，但还没有解决“故事包里哪些东西必须定清楚、哪些源本外形必须拆散、哪些证据链必须变成人物行动链”的稳定性问题。

## V21 相比 V20：确实有进步

### 1. 7-8 集从“系统认证台”往“人物崩塌”挪了一步

V20 的 7-8 集核心是：

```text
继承墙认证 -> 屏幕 NOT VERIFIED -> 样本盒结果 -> Chloe 反咬
```

这导致人物像在等系统判决。

V21 改成：

```text
信托命名礼 -> 护士 / 维修工 / Owen 出场 -> Leo 本能扑向亲生父亲 -> Serena 当众失控 -> 反咬 Ethan
```

这比 V20 好。它至少把一部分爆点从屏幕结果搬回了人身上。

但它没有完全过关，因为证人连续出现太顺，护士证词也不够硬，Leo 的反应容易显得被 Ava 当工具用。

### 2. Ep8/Ep9 信息边界比 V20 好

V20 的 Ep8 直接让 Chloe 说出：

```text
你听见她求救、你切电、你选择我、她流产有你签名。
```

这几乎把 Ep9 火葬场提前说完。

V21 的 Ep8 只把 Serena 反咬和 Ethan 的共同罪责打出来，Ep9 仍保留：

- `TWINS. SIX WEEKS.`
- 婴儿袜；
- Ava 写给自己的卡片；
- Ethan 雨中下跪。

所以 V21 的 8/9 分层比 V20 稳。

### 3. 正文更像剧本，不像流程记录

V21 没有把 `Purpose / Action / Performance / Source Ref` 这些内部字段写进正文。动作行和英文对白也比早期版本更干净。

这个说明 v11 的 `writer brief` 分层有用。

## V21 相比 V20：仍然没解决的硬伤

### 1. 核心骗局定义混乱

这是 V21 最大的新问题。

上游写了“假孕”，正文又写 Leo 五岁、Owen 是亲生父亲。这样会让观众和 reviewer 都卡住：

```text
Serena 到底是没怀孕？
还是怀了别人孩子？
还是伪造孩子属于 Ethan？
```

这不是台词问题，是新本开发包里的核心骗局没有定死。

正确层级应该在 Creative Development Pack / Writer Brief 之前解决。否则正文越写越乱。

### 2. Ep1 源本动作簇保留太密

V21 Ep1 仍然有：

```text
private kitchen
加密笔记本
pregnancy scan
怀孕女配进家宴
divorce papers
假摔
耳光
签字离开
女配打电话害女主
```

这和源本第一集动作簇太接近。它不是全稿改名复刻，但第一集会让人产生“这不还是那场戏吗”的感觉。

### 3. Ep5 有源本惩罚残影

V21 没有直接踩碎手指，但用了：

```text
Serena 的手掌按在碎玻璃上；
Ava 的高跟鞋停在她指尖前。
```

这仍然贴着源本“高跟鞋踩手指”的外形。

这说明 Adaptation Boundary 写了“禁止复用外形”，但没有稳定拆掉“动作残影”。

## V21 相比 V5：更干净，但不如 V5 有短剧记忆点

V5 的强项不是“流程更先进”，而是它抓住了短剧场面里的可见后果：

- 胸针被摘；
- 黑卡被剪 / 冻结；
- 继承墙封条；
- 物证室银铃和周年卡；
- 第 8 集只给非亲生和切电命令，第 9 集才给旧物。

V21 比 V5 更干净，也更符合 page-to-screen 的正文格式，但场面记忆点不如 V5 硬。

V21 的弱点是有些场面虽然有动作，但动作不够成为“记得住的惩罚 / 失去 / 反扑”：

- Ep4-Ep5 的港口、冷链、董事倒戈还不够可见；
- Ep7 的护士和维修工更像证据口播；
- Ep10 的 Skybridge 边缘有效，但 `One life for two` 太接近源本原句。

结论：

```text
V21 的 workflow 分层优于 V5；
V21 的正文短剧场面记忆点不如 V5。
```

所以不能简单说“新版一定比旧版好”。V11 要继续吸收 V5 的场面后果能力。

---

## V21 相比 V3：更会写剧本，但概念一致性不如 V3 稳

V3 的问题很明显：

- 第二集安全档案层不如源本车祸有冲击；
- 7-8 集偏股东大会 / 材料公开；
- 整体更像“稳态商战剧本”。

但 V3 有一个优点：它的概念比较直，读者不容易问“这个骗局到底是什么”。

V21 比 V3 更像 screenplay，也更有可拍动作：

- Ep2 火门比 V3 档案层强；
- Ep9 旧物和雨中下跪有画面；
- Ep10 高空边缘卡点有冲击。

但 V21 的 Serena 核心骗局混乱，会直接伤害连续性。这个问题一旦出现，比单场弱更危险，因为它会污染 7-10 集整条链。

## 对 v11 workflow 的判断

### 成立的部分

v11 的方向是成立的。证据是：

1. Reviewer 没有判 Stage Skeleton / Episode Map / Scene Option 失败；
2. V21 没有回到 V20 那种明显认证台程序感；
3. writer brief 能产出干净 screenplay；
4. 源本主体验链没有洗飞；
5. 信息边界比 V20 稳。

### 不成立或不够稳的部分

v11 还没有把以下三件事稳定接好：

1. **核心概念锁定**
   - 比如假孕 / 假继承人 / 假父权到底是哪一个。
   - 这是新本故事发动机，不是后期 continuity 小修。

2. **源本动作簇拆散**
   - 不是只列 Forbidden Shape，而是要检查一集里是否连续继承了太多源本表面动作。
   - Ep1 现在就是动作簇太密。

3. **证据链变成人物行动链**
   - 证据不能只是人上来说、屏幕上显示。
   - 它必须造成某人当场失去、反扑、选择、崩溃或暴露。
   - V21 比 V20 好，但 Ep7 仍然证据因果太顺。

## 下一步建议

不建议直接改 V21 正文。

应该先把这次 reviewer 的结论回填到 v11 的机制层，但只动小范围，不继续堆大表。

建议只补三处：

1. 在 Creative Development Pack 加一条“核心骗局 / 核心真相模型”
   - 先定清楚反派到底骗了什么、真相是什么、每一层什么时候爆。
   - 未定清不能进入 Stage Skeleton。

2. 在 Adaptation Boundary 或 Scene Option 加一条“动作簇密度检查”
   - 单集不能连续继承源本多个高辨识动作。
   - 不是禁止强继承体验，而是避免外形连环复刻。

3. 在 Scene Option / Writer Brief 加强“证据必须触发人物行动”
   - 证据出现后必须接人物选择、权力剥夺、关系断裂或身体反应；
   - 如果只是证人说完、众人震惊，不算强场面。

这三处都是机制修正，不是为了 V21 这个本子打补丁。

如果修完再跑，V21 的目标不是追求“完美剧本”，而是验证：

```text
新 workflow 能不能同时保住：
源本有效性、新壳成立、短剧场面、信息边界、正文可拍。
```

目前 V21 只证明了前两项和一部分第三项，还没证明稳定可交付。
