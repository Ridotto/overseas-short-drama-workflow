## 0. 文档定位

这份文档是从 `PRD_v4_产品契约_spec_v2.md` 重新编译出来的候选 workflow。

它不是：

- PRD；
- 工程 spec；
- 多 agent 产品架构；
- reviewer 规则集；
- 打回机制；
- v19 的补丁版。

它只做一件事：

```text
把 PRD_v4 的产品契约编成可执行的创作流程，
同时避免再次滑成字段碎片、gate 表、reviewer 兜底和 S 编号规则怪谈。
```

v20 可以继承 v19 已经验证过的能力：

- `故事控制包 -> 创作蓝图包 -> 场景施工稿 -> 正文` 这条中间承接层；
- 外部短剧写法协议进入实际写作动作；
- writer 输入隔离；
- run log 只记录执行，不证明质量；
- clean reviewer 只做硬伤兜底。

v20 必须修掉 v19 的变形：

- 用户可见主链不能再写入 `场景施工稿 / reviewer`；
- `创作蓝图包` 不能只是内部字段集合；
- 内部产物不能抢用户主确认文档的位置；
- 质量责任不能写成末端 gate 或无限打回；
- 外部项目不能作为固定比例模板覆盖源本和用户需求。

## 1. Source of Truth

### 1.1 当前上游顺序

本 workflow 的优先级固定为：

1. 用户确认的需求、收费边界、输出语言、集数、改写力度和禁区；
2. `PRD_v4.md`；
3. `PRD_v4_产品契约_spec_v2.md`；
4. `项目基础说明.md` 的地基原则；
5. 源本本身；
6. v19 已验证的有效能力；
7. 外部项目和商业项目参照。

外部项目、商业项目和历史样本只能校准写法与产物形态，不能反过来覆盖用户、PRD、源本和洗稿边界。

### 1.2 当前非上游

以下材料不得直接决定 v20：

- 旧线程聊天记忆；
- `PRD_v4_执行系统_spec_v1.md`；
- v8-v18 旧 workflow；
- v19 的 S 编号叙述；
- reviewer 全报告；
- run log；
- 历史样本文本本身；
- 外部项目的固定比例和完整交接链。

它们只能作为反例、证据或参考。

## 2. 产品目标

产品必须稳定完成：

```text
把一个已被市场验证或值得参考的短剧源本，
洗成一个新壳下仍然好看、能追、能付费的新剧本。
```

最终交付物是剧本文字，不是分析报告。

“好看、能追、能付费”必须落实到产物链中：

- 源本有效性被拆出来；
- 新壳能承载源本有效机制；
- 用户在正文前能确认一份完整商业项目形态的创作蓝图包；
- 场景施工稿能自然扩成正文，不需要 writer 回翻厚材料补戏；
- 正文有可见行动、冲突、爽点兑现和下一步追问；
- 连续性状态能支撑长篇多批次继续写；
- reviewer 不负责生产短剧感。

## 3. 四种运行模式

### 3.1 诊断 / 蓝图模式

只跑到 `创作蓝图包`，不进入正文。

适用：

- 用户只想判断源本值不值得洗；
- 用户只想先看方向、角色、粗纲、当前批集纲；
- 用户只给摘要、片段或零散材料；
- 系统和用户还没对齐源本为什么好看。

产物上限：

- 需求确认摘要；
- 源本有效性摘要；
- 洗稿方案摘要；
- 创作蓝图包。

### 3.2 标准首批模式

默认模式。

产物目标：

```text
跑完整链路到当前批正文，默认首批 1-10 集。
```

验证重点：

- 用户需求被确认；
- 源本有效性没有洗飞；
- 新壳自洽；
- 创作蓝图包可确认；
- 场景施工稿能支撑正文；
- 正文能追、能爽、有继续看的冲动；
- 首批整体体验至少保住既有强样本的关键体验信号。

### 3.3 长篇多批次模式

适用：

- 用户要 100 分钟、100 集或多批次交付；
- 不能一次写完整本；
- 每一批都需要重新确认当前批细层蓝图。

固定原则：

```text
全剧粗，当前批细，批次滚动推进。
```

每批都必须继承上一批连续性状态，不得重置人物关系、误会、伏笔、承诺、证据、DNC 和下一债务。

### 3.4 局部改稿 / 定点返修模式

只接受已有项目上下文。

适用：

- 用户只想修某几集；
- 用户只想改某个卡点；
- 用户只想重写某段场面；
- 已有蓝图和正文基础，不需要重跑整条正式链。

这条模式不计入默认 workflow 正向验证。

---

## 4. 用户可见主链

用户只和主控 agent 对话。

默认用户可见主链固定为：

```text
需求确认
-> 源本有效性摘要 + 洗稿方案摘要
-> 创作蓝图包
-> 首批正文修订稿
-> 完整版交付 / 用户反馈回流
```

用户默认看不到：

- 故事控制包；
- 场景施工稿；
- 完整 reviewer 报告；
- 作者自检；
- run log；
- proof-flow；
- gate 表；
- 候选演法表；
- 多 agent 分工记录。

如果用户主动要求看 reviewer 结论，主控只给精简摘要。

### 4.1 用户确认节点

用户必须确认：

1. 需求方向；
2. 源本有效性摘要和洗稿方案摘要；
3. 创作蓝图包；
4. 首批正文结果；
5. 是否继续、调整后继续，或停止。

用户不确认：

- 内部工位细节；
- agent 分工；
- reviewer 全量意见；
- run log；
- 质量检查表。

## 5. 总产物链

v20 用产物链表达流程，不把 S 编号当产品事实。

```text
P0 需求确认摘要
-> P1 源本有效性摘要
-> P2 洗稿方案摘要
-> P3 故事控制包
-> P4 创作蓝图包
-> P5 场景施工稿
-> P6 首批正文初稿
-> P7 首批正文修订稿
-> P8 连续性状态
-> P9 作者自检摘要
-> P10 clean reviewer 轻版硬伤摘要
-> P11 主控交付判断
```

其中：

- P0、P1、P2、P4、P7 是默认用户可见产物；
- P3、P5、P8、P9、P10、P11 是内部生产或决策产物；
- P6 是写作过程产物，默认不直接交付用户；
- run log 是运行记录，不属于创作产物。

### 5.1 自然承接原则

每个产物都必须自然给下游提供继续生产所需的材料。

下游不能靠临场补锅替上游完成创作责任。

```text
上一层没有自然做出来，下游就不要假装能继续写好。
```

但这不是无限打回机制。v20 的处理方式是：

- 缺必要输入：停在当前层，向主控说明缺什么；
- 用户方向变化：回到对应用户确认节点；
- 创作材料不足：重做当前产物或上一个产物，不进入后续写作；
- 正文质量不达标：记录根因层，下一轮只修最小必要链路。

## 6. P0 需求确认摘要

### 6.1 作用

确认用户到底要洗什么、怎么洗、写给谁、写成什么形态。

### 6.2 必须问清

| 输入项 | 必须得到什么 |
|---|---|
| 源本 | 完整文本源本，或明确降级为诊断 / 蓝图模式 |
| 新壳方向 | 题材、身份、世界观、核心场景或关系迁移方向 |
| 改写力度 | 贴近、等效迁移、深度重构、局部保留、删除修复 |
| 集数和批次 | 全剧目标、当前批范围、是否改变原集数 |
| 目标市场 | 地域、文化常识、敏感边界 |
| 输出语言 | 用户可见材料语言、正文语言 |
| 正文用途 | 用户审稿、导演审稿或进入 AI 视频生成 |
| 用户保留项 | 主观偏好和必须保留体验 |
| 用户禁区 | 雷点、不要继承的源本烂点、不可碰内容 |
| 收费边界 | 用户已确认的付费点、免费区间或发行方式 |

收费边界必须问，但系统不得默认具体哪一集付费。`8-12` 只能作为经验校验，不是默认排程。

### 6.3 输出形态

`需求确认摘要` 必须简洁可拍板：

- 本轮模式；
- 源本状态；
- 新壳方向；
- 改写力度；
- 当前批范围；
- 输出语言；
- 用户保留项；
- 用户禁区；
- 已知收费边界；
- 缺失信息和风险。

## 7. P1 源本有效性摘要

### 7.1 作用

不是复述剧情，而是说明源本为什么值得洗、为什么能追、为什么能付费。

### 7.2 必须拆出

- 主追问；
- 观众沉没成本如何建立；
- 源本的钩子、压放、爽点、付费冲动机制；
- 源本强节点和兑现节点；
- 哪些未解问题最容易逼用户继续看；
- 付费前后分别兑现什么、再欠什么；
- 反派或对立压力如何推动主角行动；
- 不能洗丢的体验；
- 不能继承的烂点；
- 高辨识外形和同构风险。

### 7.3 外部协议接入

P1 可以使用：

- `hook-design.md` 拆源本钩子；
- `satisfaction-matrix.md` 拆爽点类型和压放；
- `paywall-design.md` 拆追付费机制；
- `villain-design.md` 拆对立压力源。

这些协议只帮助拆源本，不替用户排付费集数。

### 7.4 输出形态

`源本有效性摘要` 必须让用户看出系统已经读懂源本，而不是只会讲短剧概念。

---

## 8. P2 洗稿方案摘要

### 8.1 作用

把源本有效机制迁移到新壳，并说明保什么、换什么、风险什么。

### 8.2 必须说明

- 新壳方向；
- 源本变量如何贴近保留；
- 源本变量如何等效迁移；
- 源本变量如何自由改写；
- 源本变量如何删除或修复；
- 新壳如何承载源本有效机制；
- 如何避免改名复刻；
- 如何避免洗飞；
- 用户偏好和禁区如何进入改编；
- 收费边界和源本原节奏如何优先于外部方法库。

### 8.3 输出形态

`洗稿方案摘要` 是用户前置确认文档。

它必须像改编策略，不像灵感清单。

---

## 9. P3 故事控制包

### 9.1 作用

故事控制包只对内。

它负责把 P1 和 P2 压成新剧可持续运转的内部事实系统。

它不得直接给 writer 当正文输入，也不得默认给用户展示。

### 9.2 必须包含

- 真相账本；
- 角色错信；
- 源本强节点重排；
- 全剧阶段骨架；
- 当前批批次地图；
- 当前批 `episode pursuit directory`；
- 每批追看债务；
- 对立压力源；
- callback / payoff 债务和兑现项；
- DNC；
- 新壳承载硬边界；
- proof-flow 风险节点；
- 高价值节点最低强度地板。

### 9.3 `episode pursuit directory`

采用 `0xsline /outline` 的有用部分：

- 当前批每集一行；
- 每集一句核心冲突 / 爽点 / 欠债；
- 当前批强弱变化；
- 当前批强窗口候选；
- 当前批节奏波形。

不得照搬：

- 固定 🔥 / 💰 比例；
- 前 10 集必须几个卡点；
- 具体付费集数默认排程；
- 外部原创工作台的整条交接链。

### 9.4 高价值节点最低强度地板

凡是源本强节点、付费窗口候选、强揭露或强爽点，都必须回答：

- 观众最想看什么不可逆变化；
- 这一节点压抑了什么；
- 这一节点释放什么；
- 哪些低强度替代必须排除；
- 最低强度必须落在哪个可见后果上。

可见后果包括：

- 人物被迫行动；
- 关系反转；
- 权力变化；
- 体面损失；
- 身体 / 空间后果；
- 当众位置变化；
- 对立压力源升级。

证据、文件、屏幕、系统、检测、证人只能是触发器，不能当高压戏本体。

## 10. P4 创作蓝图包

### 10.1 定位

创作蓝图包是正文前的正式用户确认文档，同时是场景施工稿的主上游。

它不是字段表，也不是内部控制包。

### 10.2 固定形态

```text
创作蓝图包

一、故事梗概
二、人物设定
三、全剧粗纲
四、当前批集纲
五、当前批状态与禁区说明
```

### 10.3 故事梗概

必须让用户看懂：

- 新剧讲什么；
- 新壳是什么；
- 主角为什么必须行动；
- 对立压力来自哪里；
- 源本有效内核如何迁移；
- 本剧为什么有追问；
- 本剧为什么不是源本改名复刻。

不得只是设定说明或源本剧情压缩。

### 10.4 人物设定

必须包含：

- 核心角色；
- 角色欲望；
- 角色压力；
- 角色秘密或错信；
- 关键关系；
- 关系变化潜力；
- 与源本角色功能的迁移关系；
- 新壳下的人物合理性。

人物设定必须服务行动和冲突，不能只有身份标签。

### 10.5 全剧粗纲

必须回答：

- 全剧大阶段如何推进；
- 每阶段核心压力是什么；
- 每阶段主追问如何变化；
- 关键爽点和释放大致落在哪里；
- 源本强节点如何被重排或迁移；
- 长篇多批次时，当前批处在全剧什么位置。

全剧粗纲必须比单集集纲更宏观、更完整，不能薄到只剩阶段标签。

### 10.6 当前批集纲

当前批集纲必须是“已经像戏”的分集内容。

每集至少自然包含：

- 本集发生什么具体人事；
- 谁逼谁行动；
- 本集如何承接上一集；
- 本集兑现什么；
- 本集留下什么追问；
- 本集爽点或压力在哪里；
- 本集结尾为什么让人继续看；
- 本集如何服务整批关系和状态变化。

当前批集纲可以吸收内部四问：

- 为什么继续追 / 继续付；
- 先兑现什么；
- 再欠什么；
- 最强承载方式是什么。

但它不能只展示四问答案，必须写成用户能读的分集集纲。

### 10.7 高刺激集处理

高刺激集包括：

- 当前批开头集；
- 用户确认的付费窗口前后；
- 源本强节点迁移集；
- 关系或权力发生不可逆变化的集；
- 当前批结尾集。

高刺激集必须在集纲里看见：

- 安全弱方案为什么不够；
- 新壳自然方案为什么成立；
- 高冲动方案为什么更值得选；
- 最终选哪个承载方向；
- 证据 / 文件 / 屏幕如果存在，只是触发器，不是戏本体；
- 这一集先兑什么，下一集先接什么。

这些内容必须融进集纲，不要另开成内部候选表给用户看。

### 10.8 当前批状态与禁区说明

必须记录：

- 当前批开始时人物状态；
- 当前批结束时人物状态；
- 关系变化；
- 误会变化；
- 真相 / 证据 / 伏笔状态；
- 已经兑现的债；
- 仍然欠着的债；
- DNC；
- 禁用外形；
- 用户特别要求保留或避免的内容。

## 11. P5 场景施工稿

### 11.1 作用

场景施工稿是正文前最后一层，只服务当前批 / 当前集 / 当前场景组。

它必须让 writer 不回头翻厚材料也能写正文。

P5 只读 P4。

### 11.2 必须采用的写法协议

P5 必须把 `0xsline /episode` 和 `scene-writing` 接成实际施工稿。

必须加载并执行：

- `opening-rules.md`；
- `rhythm-curve.md`；
- `hook-design.md`；
- `satisfaction-matrix.md`；
- `villain-design.md`；
- `scene-writing/SKILL.md`；
- `wp-scene-writing.md`；
- `rb-scene-draft.md`。

`paywall-design.md` 不在 P5 临场补逻辑。付费机制必须已经被 P1-P4 吸收，P5 只负责场面化。外部黄金卡点区间不能当默认排程。

### 11.3 每集施工稿必须包含

- opener；
- scene chain；
- conflict escalation；
- pressure / release；
- visible consequence；
- end hook；
- next debt；
- 本集兑现义务；
- 下一集承接义务；
- 不能写什么。

### 11.4 每场必须包含

- scene function；
- conflict；
- change point；
- playable action；
- dialogue carrier；
- scene end state。

每场至少改变以下一项：

- 信息；
- 关系；
- 权力；
- 行动条件。

场景前后局面不变，说明它不是场景。

### 11.5 proof-flow 硬边界

以下只能做触发器，不能做高压戏本体：

- 报告；
- 屏幕；
- 会议；
- 系统；
- 文件；
- 检测结果；
- 证人供述。

决定性 change point 必须是：

- 某个人被迫做动作；
- 某个人当场失去关系；
- 某个人当场失去权力；
- 某个人当场丢体面；
- 某个人身体或空间位置发生危险变化。

如果删掉证据展示后场景就没有戏，它不是 scene-writing，是 proof-flow。

### 11.6 高刺激集施工要求

高刺激集必须显式写出：

- 开场第一拍；
- 中段升压；
- 压低动作；
- 释放动作；
- 结尾 hook；
- 本集兑现；
- 下集承接。

P5 不得自己发明新的商业刺激逻辑。如果 P4 给出的集纲仍是冷方案、制度方案、证据方案，P5 必须停线并说明 P4 缺哪种场面承载。

## 12. P6 首批正文初稿

### 12.1 输入隔离

P6 只读 P5。

不得读取：

- P1；
- P2；
- P3；
- P4；
- run log；

- 临时诊断。

### 12.2 正文合同

正文必须是可读、可拍、可继续反馈的短剧文本。

必须满足：

- page-to-screen；
- 人物行动推动；
- 对白不裸飘；
- 信息揭示服务冲突；
- 高压节点可见；
- 爽点有兑现；
- 结尾有追问；
- 状态连续；
- 不复刻源本；
- 不洗飞源本。

正文不是施工稿扩字，也不是对白版剧情提纲。

## 16. P10 clean reviewer 轻版硬伤摘要

P10 由干净上下文 reviewer 执行。

- 逐句重写；
- 对用户拍板；
- 替上游补设计；
- 成为新增生产工位。

reviewer 输出只给主控。用户主动要求时，主控只给精简摘要。

## 18. run log 规则

任何 agent 声称“跑 workflow / 重跑 / 按流程产出”，必须遵守：

```text
v2-restart/workflow_execution_protocol_v1.md
```

run log 只记录：

- 本轮使用哪个 workflow；
- 每个产物实际在哪；
- 产物交给下游什么；
- 是否混入不该给下游的材料；
- 每层状态是 done / needs_rework / blocked / skipped。

run log 不能证明：

- 剧本质量；
- 是否可以送审；
- 短剧感是否达标；
- 源本有效性是否迁移成功。

## 19. 准入与例外

### 19.1 正式写稿链准入

正式写稿链默认只接受完整文本源本。

摘要、片段、口述、零散材料默认只能进入诊断 / 蓝图模式。

### 19.2 坏源本

坏源本红旗包括：

- 主追问弱；
- 强节点稀；
- 重复水戏多；
- 后半塌；
- 拉扯弱；
- 外部反馈差。

坏源本默认转诊断 / 蓝图模式，不承诺正式正文质量。

### 19.3 高风险改编例外

用户坚持超出洗稿边界、明显更像原创改编的方向时，可以继续，但必须说明：

- 这不是标准洗稿链；
- 质量保障下降；
- 不计入默认 workflow 正向验证。

### 19.4 无源稿脚本医生模式

用户只给现成新稿、不提供源本，却要求“改得更短剧”，不进入洗稿链。
