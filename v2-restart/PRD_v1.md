# 海外短剧洗稿器 · PRD v1

> 日期：2026-06-29
> 状态：第一版 PRD 草案
> 上游文档：`项目基础说明.md`、`设计提纲_v8.md`

---

## 1. 产品定位

### 一句话

把一个已被市场验证过的短剧源本，洗成一个新壳下仍然好看、能追、能付费的新剧本。

### 产品不是

- 不是评分器。
- 不是分析报告工具。
- 不是从零原创工具。
- 不是一键成片或视频工具。
- 不是查词替换式换皮工具。
- 不是短剧标准库套模板工具。

### 核心判断

产品最终交付的是剧本文字，不是方法论。

分析、变量判断、自检都只是为了服务一件事：**新本既没有洗飞源本的有效性，又能作为一个新剧本自己站住。**

---

## 2. 目标用户与场景

### 目标用户

写短剧本的人：用户本人或团队里的编剧。

暂不服务导演、监制、甲方、投放团队、视频制作团队。

### 典型场景

用户手上有一部短剧源本，认为它值得参考或已被市场验证。用户想把它改成另一个题材、身份、背景、世界观或集数版本，但不想从零原创，也不想让大模型把源本好看的东西洗丢。

例子：

- 把狼人短剧洗成都市霸总。
- 把真假千金复仇洗成商战家族斗争。
- 保留追看节奏和情绪结构，只换人物身份和背景。
- 原剧 40 集，想改成 60 集或 80 集。

---

## 3. 第一阶段范围

### 第一阶段要做

第一阶段只跑通“源本分析 → 洗稿方案 → 首批新本 → 自检 → 用户确认”的闭环。

默认不一次性洗完整本，而是先交付首批。首批通过后，再继续分批写。

### 第一阶段不做

- 不做从零原创。
- 不做完整评分系统。
- 不做多剧本排名。
- 不做投流片段设计。
- 不做视频分镜、拍摄方案、成片生成。
- 不把 short-drama 标准库数字变成硬规则。
- 不把多 agent 设计暴露成产品主线。

---

## 4. 输入

### 必填输入

1. 源剧本：用户提供一部短剧文本。
2. 改写目标：用户用自然语言说明想怎么洗。

### 需求对齐时必须问清

| 问题 | 为什么要问 |
|---|---|
| 想换成什么新壳？ | 决定题材、身份、世界观、场景如何重建 |
| 改写力度多大？ | 决定哪些变量高度贴源本，哪些做等效迁移 |
| 集数是否改变？ | 决定留存节点和节奏落点是否重排 |
| 目标市场是否明确？ | 决定哪些设定需要本地化或替换 |
| 哪些东西绝对不能丢？ | 锁定用户主观偏好 |
| 哪些东西必须删或不喜欢？ | 避免把源本烂点或用户雷点带进新本 |

### 改写力度的理解

改写力度不是固定三档开关，而是影响变量迁移策略。

- 改动很轻、集数不变、用户想贴近源本：可以高度复刻源本节奏和卡点。
- 改动中等或新壳承载方式不同：保留追看功能，形式和位置做等效迁移。
- 改动很大或集数改变：重排节奏，但不能丢源本的核心商业发动机。

---

## 5. 核心流程

### 流程总览

```text
源本 + 用户需求
  ↓
1. 分析源本有效性骨架
  ↓
2. 把源本内容分层
  ↓
3. 形成迁移基准与变量迁移策略
  ↓
4. 判断新壳/新集数能否承载
  ↓
5. 迁移、等效替换或删除
  ↓
6. 生成洗稿方案
  ↓
7. 生成首批新本并自检
  ↓
首批新本 + 改动说明 + 用户确认
```

### 1. 分析源本有效性骨架

要回答：这个源本为什么能让观众继续看、愿意付费？

分析不是为了输出长报告，而是为了找出改写时不能乱动的迁移基准。

### 2. 把源本内容分层

| 类别 | 含义 | 改写态度 |
|---|---|---|
| 有效性骨架 | 让源本能抓人的运行方式 | 默认继承 |
| 外壳 | 题材、身份、场景、世界观、具体桥段包装 | 可改写 |
| 水分 | 注水、拖沓、无效过渡、重复桥段 | 删除或压缩 |
| 烂点 | 假动机、逻辑洞、虚假悬念、不适合目标市场的设定 | 修掉，不继承 |

### 3. 形成变量迁移策略

每个重要变量都要标明迁移策略：

- 强继承
- 条件强继承 / 等效迁移
- 自由改写
- 必须删除或修复

### 4. 判断新壳能否承载

要回答：用户想要的新题材、新身份、新市场、新集数，能不能承载源本有效性？

如果承载不了，不是放弃源本，而是进入等效迁移：换一种新壳里成立的方式，达到同样的观众体验功能。

### 5. 生成洗稿方案

方案必须先给用户确认。方向错了，不进入写作。

### 6. 生成首批新本

首批不固定 3 集、5 集或 10 集。默认覆盖第一个完整留存小闭环。

### 7. 自检与用户确认

自检通过后，交付首批新本和改动说明。用户确认方向后，再继续分批写。

---

## 6. 分析维度

分析按五层看，不输出五份报告。

| 层 | 看什么 | 用来决定什么 |
|---|---|---|
| 留存骨架 | 开头、每集钩子、关键卡点、付费/留存节点 | 源本怎样让观众不划走、不弃剧 |
| 情绪爽点 | 情绪峰谷、压抑释放、爽点兑现、高潮递进 | 源本怎样压住观众情绪，再释放或反转 |
| 主线推进 | 核心期待、信息差、秘密揭露、每集/每场新进展、信息增量、推进效率 | 期待链和信息密度如何迁移 |
| 人物压力 | 主角驱动力、关系功能、反派层级、压迫升级 | 谁推动故事，谁制造压力，谁放大爽点 |
| 干净度 | 水分、烂点、AI 味、非视听语言、不可拍描述、目标市场不适配 | 哪些内容要删除、修复或替换 |

---

## 7. 变量迁移策略

| 变量 | 默认策略 | 说明 |
|---|---|---|
| 留存/追看功能链 | 强继承 | 保留源本让观众继续看的核心方式 |
| 关键卡点位置 / 钩子形式 / 节奏落点 | 条件强继承 / 等效迁移 | 轻度换壳可贴近源本；中重度改写按新壳重排 |
| 付费/关键留存节点 | 强继承 | 集数改变时按阶段比例重排 |
| 情绪峰谷走势 | 强继承 | 继承压、爽、虐、甜、反转的体验走势，不照抄数值 |
| 爽点兑现周期 | 强继承 | 不能无故拖延，也不能太早清空 |
| 主线期待链 | 强继承 | 深度重写时可重构，但必须有新的同等期待链 |
| 主角处境变化节奏 | 强继承 | 可换事件，不应丢阶段功能 |
| 反派压力升级 | 强继承 / 等效迁移 | 身份可换，压力阶梯要在 |
| 信息差结构 | 等效迁移 | 表现形式可换，信息控制不能乱 |
| 钩子类型 | 等效迁移 | 类型服务留存功能，不是硬模板 |
| 爽点呈现形式 | 等效迁移 | 形式可换，压抑-释放必须成立 |
| 关键道具/秘密/证据 | 等效迁移 | 道具可换，证明、揭露、反杀、引爆功能要保 |
| 人物关系功能 | 等效迁移 | 压迫源、保护者、背叛者、见证者等功能不能无故消失 |
| 题材 / 世界观 / 背景 | 自由改写 | 新壳主要动这里 |
| 身份 / 职业 / 组织 / 地点 | 自由改写 | 必须服务新壳自洽 |
| 具体桥段包装 | 自由改写 | 关键情绪和信息功能要对上 |
| 台词和表达 | 自由改写 | 必须像新剧本，不能保留 AI 味、翻译腔、解释腔 |
| 集数与篇幅 | 条件改写 | 重排留存节奏，不是简单加戏删戏 |
| 水分 / 重复 / 无效过渡 | 必须删除 | 不进入迁移基准 |
| 假动机 / 逻辑漏洞 / 虚假悬念 | 必须修复或删除 | 不能因为源本有就继承 |
| 目标市场不适配设定 | 必须替换 | 换成目标市场能理解的等效压力 |

---

## 8. 用户参与点

用户不是来填表的，但必须在关键节点拍板。

| 节点 | 用户要确认什么 | 不确认的风险 |
|---|---|---|
| 需求对齐 | 新壳、目标市场、集数、改写力度、偏好和雷点 | 工具按错误方向动手 |
| 迁移基准确认 | 哪些变量强继承、等效迁移、自由改写、删除 | 源本有效性被洗飞 |
| 洗稿方案确认 | 新本方向、变量处理方式、首批范围 | 写出来后整批返工 |
| 首批交付确认 | 方向是否对、是否继续、哪里调整 | 后续批次沿着错误方向扩大 |
| 续写确认 | 分批继续或回退修正 | 长剧本中后段漂移 |

---

## 9. 首批交付

### 原则

首批要覆盖第一个完整留存小闭环，而不是固定集数。

一个完整留存小闭环至少包括：

1. 开头抓人。
2. 核心压力或期待建立。
3. 至少一次阶段性兑现，或一次足够强的悬置。
4. 新一轮追剧问题成立。

判断标准：观众看完首批，既觉得已经看到一点兑现，又想知道接下来会怎样。只有兑现没有新问题，会清空追剧动力；只有问题没有兑现，会损失信任。

### 首批交付内容

1. 首批新剧本文字。
2. 简短改动说明：新壳怎么落、保留了什么、等效替换了什么、删了什么。
3. 自检结果：有没有洗飞，是否有硬失败或需要用户判断的问题。

---

## 10. 自检与验收

### 自检原则

自检不是评分器，不合成总分。

自检有两层标准：

1. 硬底线：源本有效性是否迁移。洗飞就失败。
2. 质量标准：新本自身是否自洽、好看、不假、不 AI。

两者同时检查，但前者优先级更高。

### 硬失败

出现以下问题，不能交付为通过：

| 硬失败 | 处理 |
|---|---|
| 源本有效性骨架没有迁移 | 回到分析和变量策略 |
| 强继承项被无故改掉 | 回到迁移基准 |
| 新壳承载不住源本功能 | 回到承载判断 |
| 主线期待断掉或空转 | 回到主线推进 |
| 人物动机和行为站不住 | 回到人物与新壳重写 |
| 水分烂点被继承 | 删除或修复 |
| 为刺激硬堆钩子爽点 | 回到剧本重写 |
| 镜头感/可拍性不过关 | 回到场景重写 |

### 重警告

以下问题不一定直接失败，但必须说明并优先修：

| 重警告 | 处理 |
|---|---|
| 情绪波形变平或塌掉 | 调整节奏和情绪递进 |
| 爽点缺少蓄力或兑现 | 回到爽点设计 |
| 伏笔、秘密、信息差断线 | 回到连续性追踪 |
| 角色状态、关系、时间线漂移 | 回到连续性追踪 |
| 台词有 AI 味、解释味、翻译腔 | 回到表达重写 |

### 镜头感/可拍性要求

剧本不能用抽象概念替代场面。关键情绪、大事件、人物转变，都必须能被摄影机拍出来。

不合格写法：

- “角色突然黑化。”
- “发生一场惨绝人寰的战争。”
- “两人关系彻底破裂。”

合格方向：

写出观众能看到的动作、选择、冲突、后果或场面变化。

## 11. 交付物

### 第一阶段用户可见交付物

| 交付物 | 作用 | 备注 |
|---|---|---|
| 需求确认摘要 | 确认用户到底要怎么洗 | 短，不做长问卷报告 |
| 源本有效性摘要 | 告诉用户源本靠什么有效 | 只写会影响改写的部分 |
| 变量迁移策略表 | 说明哪些继承、哪些等效、哪些改写、哪些删除 | 用户确认后再写 |
| 洗稿方案 | 写首批前的方向方案 | 用户确认后进入写作 |
| 首批新本 | 第一批可读剧本文字 | 覆盖首个留存小闭环 |
| 改动说明 + 自检结果 | 说明有没有洗飞、哪里需要用户判断 | 随首批交付 |

### 暂不固定的模板

各交付物字段先保持轻量，等真实样本跑完后再定模板。现在过早固定字段，会把产品重新推回填表和规则堆叠。

---

## 12. 成功标准

第一阶段成功，不等于产品已经能自动产出完整爆款。

第一阶段成功是：

1. 用户给一个源本和改写方向后，产品能完成需求对齐。
2. 产品能读出源本有效性骨架，并给出合理变量迁移策略。
3. 用户能看懂并确认洗稿方案。
4. 产品能交付首批新本。
5. 首批新本没有洗飞源本有效性。
6. 首批新本自身没有明显硬失败。
7. 用户能明确判断：继续、调整后继续，或停止。

---

## 13. 风险

| 风险 | 表现 | 应对 |
|---|---|---|
| 规则堆叠 | 文档越来越像检查清单，模型注意力涣散 | 五层分析，不新增碎规则 |
| 机械套模板 | 标准库数字压过源本 | 标准库只做参考和异常提醒 |
| 洗飞源本 | 壳换了，但追看功能没了 | 变量迁移策略 + 首批自检 |
| 新壳不自洽 | 人物动机、关系、场景在新背景下不成立 | 承载判断前置 |
| 硬堆钩子爽点 | 一分钟塞多个刺激点，剧本发疯 | 自检硬失败 |
| 源本烂点被继承 | 把旧本水分、假动机、虚假悬念带进新本 | 水分/烂点单独分层 |
| 表达不可拍 | 用抽象概念代替场面 | 镜头感/可拍性自检 |
| 用户方向没对齐 | 工具写了一批用户不想要的方向 | 需求确认 + 方案确认 |

---

## 14. 待决策点

这些不阻塞 PRD v1，但会影响下一轮细化。

1. 需求对齐问题是否要做成固定问卷，还是保持自然语言访谈。
2. 首批默认长度是否需要给出推荐范围，还是继续只按留存小闭环判断。
3. 变量迁移策略表是否需要用户逐项确认，还是只确认工具标出的关键变量。
4. 自检结果是否只给硬失败和重警告，还是也给轻度建议。
5. 真实样本验证先选强本换壳，还是选中等本修正后换壳。

---

## 15. PRD v1 后的下一步

1. 用一个真实源本跑第一阶段小样。
2. 根据小样补齐需求对齐问题清单。
3. 根据小样收敛交付物模板。
4. 再决定是否进入 skill 或具体产品形态设计。
