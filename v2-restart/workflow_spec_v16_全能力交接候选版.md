## 0. 执行需求卡

### 0.1 用户可见目标

把一个已被市场验证或值得参考的短剧源本，洗成一个新壳下仍然好看、能追、能付费的新剧本。

交付物是剧本文字，不是分析报告。

### 0.2 本 spec 解决什么

v16 只解决一件事：

```text
把 PRD、原项目能力、外部短剧手艺和历史样本里已确认有用的能力，
接成一条可执行、可交接、可回滚的写稿链。
```

它不是新 PRD，不是资料池，不是短剧规则大全。

### 0.3 明确不做什么

- 不从零原创。
- 不照搬 0xsline / short-drama 的原创工作台流程。
- 不把 V5 当目标样本复刻。
- 不把短剧感理解成钩子堆砌或强度硬加。
- 不继续修 V25 正文。
- 不让 reviewer 生产短剧感。
- 不让 run log 证明剧本质量。
- 不创建真实 `.codex/skills/SKILL.md`。
- 不进入网页 App 或工程实现。

### 0.4 通过标准

v16 只有跑样本后才能正式化。静态通过不等于剧本通过。

总通过轴只有一个：

```text
观众看完当前集 / 当前批次后，是否更想继续看，是否愿意为下一段答案付费。
```

视听语言、短剧感、钩子、爽点、拉扯、反派压力、DNC、台词和连续性都不是孤立 checklist。它们都必须服务这个追看 / 付费效果。  
如果某项能力“有了”，但没有让观众更想追，仍然视为没有真正生效。

追看 / 付费效果不是钩子堆砌，而是主追问统住，压放、兑现和追问节奏成立。

样本通过必须同时满足：

- 没洗飞源本有效性；
- 不是改名复刻；
- 新壳下自己站得住；
- 观众有继续追的理由；
- 有付费前后的追问和兑现义务；
- 高压节点像戏，不像报告、会议、文件播报；
- 台词短、准、有功能；
- 动作、身体、空间、声音、物件进入正文；
- 钩子、爽点、拉扯、反派压力、DNC、连续性没有被短剧感修复挤掉。

## 1. 产品锚点与非目标

### 1.1 产品锚点

本项目是洗稿器，不是原创短剧生成器。

简写口径：洗稿，不是原创。

核心判断：

```text
新本没有洗飞源本有效性；
新本不是源本改名复刻；
新本作为一个新短剧自己站得住。
```

v16 的关键口径：

```text
源本有效性必须通过新事件载体和新短剧场面承载出来。
不能只迁移剧情功能，也不能只换名词。
```

### 1.2 三条红线

| 红线 | 含义 | 失败回滚 |
|---|---|---|
| 洗飞 | 源本让人追下去、愿意付费的有效性丢了 | S1/S2/S3 |
| 改名复刻 | 源本事件链和强桥段整串保留，只换身份、名词、地点 | S2/S3/S4 |
| 新本不站住 | 新壳不自然、人物动机弱、连续性乱、剧本不像短剧 | 对应失败层 |

### 1.3 非目标

- 不是评分器：不使用 50 分制或总分幻觉。
- 不是模板器：不把开场、爽点、付费点固定成硬比例。
- 不是强度堆叠器：不要求每场都爆、每句都狠、每个节点都反转。
- 不是 reviewer 驱动返修机：reviewer 只兜底和归层。
- 不是视频分镜工具：正文必须可看可听，但不写镜头工程。
- 不是用户要理解的多 agent 展示产品：用户只和主控 agent 对话。

## 2. 总执行链

v16 固定 S0-S11：

```text
S0 需求确认
-> S1 源本拆文
-> S2 洗稿边界
-> S3 新本故事架构
-> S0b 洗稿方案二次确认
-> S4 场面承载原料
-> S5 writer 批次 / 单集 / 场景组施工稿
-> S6 screenplay 正文
-> S7 台词 / 视听 polish
-> S8 continuity / 信息边界
-> S9 作者自检与返修诊断
-> S10 干净 reviewer
-> S11 主控拍板
```

唯一交接原则：

```text
A 环节产出 X；
B 环节只吃 X；
B 环节产出 Y；
C 环节只吃 Y。
```

下游不能回头翻上游厚材料。  
如果下游必须回头翻，说明上一层产物不合格，应回上一层重做。

长篇 / 长批次执行原则：

```text
S3 控制全剧、阶段、批次和分集功能；
S4 给当前批次或当前高压节点准备可写戏原料；
S5 只排当前批次 / 当前集 / 当前场景组的施工稿；
S6 只写当前施工稿对应的正文。
```

如果用户要 100 分钟、100 集或多批次交付，不能让 S5 一次性写完整本施工稿。  
必须先由 S3 产出全剧阶段骨架和批次地图，再按批次进入 S4-S8，并在每批结束后更新 story memory 和下一批 DNC。

## 3. 全链工位合同

### S0 需求确认

**只负责：** 拿到可开工边界。

**输入：**

- 用户需求；
- 源本入口；
- 用户已知的新壳、集数、语言、市场、改写力度和禁区。

**必须输出：** `最小需求确认`

必须回答：

- 源本是什么；
- 新壳是什么；
- 目标市场是什么；
- 输出语言 / 审稿语言是什么；
- 集数是否改变；
- 改写力度按哪些变量控制；
- 用户禁区是什么；
- 正文用途是什么：用户审稿、导演审稿、AI 视频生成、字幕或双语。

**禁止：**

- 分析源本；
- 设计剧情；
- 默认深度重构；
- 默认英文或中文，不确认输出需求。

**下游只能读：**

- S1 读取 `最小需求确认 + 源本`；
- S2 可读取需求边界。

**失败回滚：**

- 目标市场、语言、集数、用途不清，回 S0。

**S0b 洗稿方案二次确认：**

S0b 不单独编号为新工位，但必须发生在 S3 之后、S4 之前。

S0b 必须把以下内容给用户 / 主控确认：

- 源本最需要保住的追看 / 付费效果；
- 改写力度和变量边界；
- 新本故事核、题材、世界观和目标市场语感；
- 全剧阶段骨架和首批 / 当前批次范围；
- 明确不能提前消费的信息；
- 明确不能复用的源本高辨识外形；
- 输出语言和正文用途。

如果 S0b 没有确认，不能进入 S4。  
如果用户在 S0b 改方向，回 S0/S2/S3，不从 S4 开始硬拧。

---

### S1 源本拆文

**只负责：** 拆源本为什么能追、能付费。

**输入：**

- `最小需求确认`；
- 源本。

**必须输出：** `源本有效性包 / Source Bible`

必须拆出：

- 核心期待；
- 留存骨架；
- 钩子 / 卡点；
- 付费节点；
- 爽点和压抑释放；
- 源本强弱节奏：哪些地方该压、哪些地方该放、哪些地方需要呼吸；
- 情绪密度 / 信息密度 / 情节密度；
- 人物关系和拉扯；
- 信息差 / 真相揭露；
- 反派压力和反扑燃料；
- Do Not Consume；
- 水分 / 烂点；
- 源本高辨识外形。

**禁止：**

- 设计新壳；
- 写新剧情；
- 把剧情摘要当源本有效性；
- 把源本强桥段直接当可继承内容。

**下游只能读：**

- S2 读取源本体验功能、高价值节点、高辨识外形和 DNC。

**失败回滚：**

- 如果后续发现“观众追什么、为什么付费”没拆出来，回 S1。

---

### S2 洗稿边界

**只负责：** 区分必须继承的体验和必须替换的外形。

**输入：**

- `源本有效性包`。

**必须输出：** `洗稿边界包 / Boundary`

必须包含：

- 强继承体验；
- 等效迁移体验；
- 自由改写变量；
- 必须删除 / 修复；
- forbidden shape；
- 新壳不能承载的源本外形；
- 改写力度的变量说明。

**禁止：**

- 写具体新剧情；
- 为了防同构洗飞源本；
- 把“不能复刻外形”误读成“不能用短剧套路”。

**下游只能读：**

- S3 读取继承项、自由项、禁用外形和改写边界。

**失败回滚：**

- 源本强桥段整串回流，回 S2/S3/S4；
- 防同构压过防洗飞，回 S2。

---

### S3 新本故事架构

**只负责：** 搭新壳故事、真相、阶段和分集骨架。

**输入：**

- `洗稿边界包`。

**必须输出：** `新本故事开发包`

必须包含：

- 新本故事核；
- 题材 / 世界观边界；
- 目标市场语感；
- 真相账本；
- 角色错信；
- 主追问和阶段追问；
- 全剧阶段骨架 / 粗纲；
- 批次地图：每批覆盖哪些集 / 分钟、承担什么追看功能、在哪个问题上停；
- 强弱分配：每个阶段 / 批次的波峰、波谷、兑现点和追问点；
- 分集功能 / 单集提纲；
- 人物行动方式；
- 反派压力与反扑燃料；
- DNC 排程；
- 付费追问和兑现义务。

**禁止：**

- 写具体高压场完整外形；
- 锁死源本外形；
- 写正文级动作串；
- 把题材设定写成百科；
- 把短剧固定比例压过源本节奏。
- 把每集都设计成高压爆点，导致主线散、兑现不足或观众疲劳。

**下游只能读：**

- S0b 读取用户需要确认的故事方向和批次范围；
- S4 读取当前批次 / 当前集需要的故事状态、分集功能、角色行动方式、反派压力、DNC 和新壳边界。

**失败回滚：**

- 主追问、真相账本、错信、阶段骨架、批次地图不清，回 S3；
- 人物动机与行为矛盾，回 S3/S5；
- 反派没有继续作死的资源或误判，回 S3/S5。

---

### S4 场面承载原料

**只负责：** 把当前批次 / 当前集 / 当前高压节点的故事功能变成可写戏的原料。

**输入：**

- `新本故事开发包`。

**必须输出：** `场面承载原料包`

必须包含：

- 当前批次或当前写作单元；
- 当前批次承接上一批什么状态，结束后必须留下什么追看问题；
- 这场承载源本哪个体验功能；
- 新壳里靠什么权力、关系、空间、物件或行动承载；
- 谁压谁；
- 谁旁观；
- 谁被迫行动；
- 压力如何形成；
- 释放后造成什么可见后果；
- 候选新壳动作方向；
- 禁止复用的源本外形；
- 本写作单元 DNC。

**禁止：**

- 排完整戏；
- 写完整节拍；
- 写台词；
- 写成片最后一帧；
- 用文件、屏幕、会议、系统替代人和人的现场冲突。

**下游只能读：**

- S5 只读 `场面承载原料包`。

**失败回滚：**

- 新壳承载弱、只有证据 / 文件 / 系统 / 会议，回 S4；
- S4 写成半成品正文，回 S4 重做原料。

---

### S5 writer 批次 / 单集 / 场景组施工稿

**只负责：** 把 S4 原料排成当前批次、当前集或当前场景组的短剧戏。

**输入：**

- `场面承载原料包`。

S5 不直接读取：

- 源本拆文包；
- 洗稿边界包；
- 新本故事开发包；
- story memory 全文；
- 外部标准库全文；
- reviewer report；
- 作者自检；
- run log；
- 旧版对比；
- 临时诊断。

**必须输出：** `writer 施工稿 / 写作卡`

写作单元选择：

- 首批 1-10 集验证：通常按集写施工稿，高压节点可细到场景组；
- 100 分钟 / 多批次：按批次进入 S4-S8，每批内部再按集或场景组拆；
- 低压过渡：可以用场景组施工稿，不强迫每场都写成高压戏；
- 绝对波峰、付费点、身份揭露、火葬场、重大反转：必须细到具体场景组。

必须回答：

1. 当前写作单元观众正在等什么；
2. 开场第一个可见动作 / 声音 / 冲突是什么；
3. 中段如何升压；
4. 爽点前如何压低；
5. 谁做动作，谁被迫反应，谁失去什么；
6. 关系 / 权力 / 信息如何变化；
7. 结尾停在哪个未完成动作、声音或画面，下一集 / 下一段必须兑现什么；
8. 哪些信息不能提前消费；
9. 禁止复用的源本外形；
10. 当前写作单元短剧手艺如何使用。

**短剧手艺必须在写稿当下使用：**

| 外部能力 | 在 S5 里的生产动作 |
|---|---|
| `opening-rules` | 写出开场第一拍，不写“开场要强” |
| `rhythm-curve` | 写出当前写作单元开场、中段、末尾压放 |
| `hook-design` | 写出未回答问题、卡住动作、下一集兑现义务 |
| `satisfaction-matrix` | 写出压低动作、释放动作、可见后果 |
| `paywall-design` | 写出付费追问和兑现义务，不照搬固定比例 |
| `villain-design` | 写出反派资源、误判和下一步反扑 |
| `genre-guide` | 写出题材语感和角色说话方式 |

**禁止：**

- 写流程证明；
- 复读上游字段；
- 写成检查表；
- 写完整正文；
- 照抄源本高辨识外形；
- 让 S6 正文 writer 自己回头悟材料。

**下游只能读：**

- S6 只读 `writer 施工稿 / 写作卡`。

**失败回滚：**

- 施工稿像任务单，没有行动链和压放，回 S5；
- S1/S2/S3 有的核心能力没进入 S5，回对应层或 S5；
- S6 还需要回头读 S1-S4，说明 S5 失败，回 S5。

---

### S6 screenplay 正文

**只负责：** 把施工稿写成可看可听的剧本正文。

**硬线：writer 只读 S5 施工稿。**

**输入：**

- `writer 施工稿 / 写作卡`。

**必须输出：** `screenplay 正文初稿`

必须做到：

- 剧本页上的内容基本就是最终 AI 成片能看见、听见的东西；
- 人物通过动作、身体、空间、声音、物件和当众后果表达情绪；
- 台词短、准、有功能；
- 每集 / 每个交付段落结尾卡在答案前一拍；
- 高压节点像戏，不像报告、会议或证据播报。

**禁止：**

- 内部标签；
- 规则名；
- 工位名；
- 作者解释；
- 抽象心理替代画面；
- 站桩聊天；
- 多轮低信息对白；
- 回头翻 S1-S4。

**下游只能读：**

- S7 只读 `screenplay 正文初稿`。

**失败回滚：**

- 正文像说明书、报告、站桩聊天、不可视听，回 S6/S7；
- 如果问题来自施工稿无戏，回 S5。

---

### S7 台词 / 视听 polish

**只负责：** 修表达，不重设主结构。

**输入：**

- `screenplay 正文初稿`。

**必须输出：** `台词视听 polish 稿`

必须处理：

- 删水词；
- 删解释腔；
- 强化角色声线；
- 把能用动作、声音、身体、空间表达的内容从对白里拿出来；
- 确保台词承担攻击、防守、逼迫、拒绝、试探、威胁、揭露、反击或选择；
- 确保高压场面有可见状态变化。

**禁止：**

- 重设主线；
- 替 S4/S5 重新造场面；
- 为了金句硬塞不合角色的台词；
- 把 polish 写成二次创作大改。

**下游只能读：**

- S8/S9 读取 `台词视听 polish 稿`。

**失败回滚：**

- 台词水、同声、解释腔，回 S7；
- 如果表达问题源于场面无动作，回 S5/S6。

---

### S8 continuity / 信息边界

**只负责：** 查事实、错信、DNC、伏笔、状态。

**输入：**

- `台词视听 polish 稿`。

**必须输出：** `story memory checkpoint`

必须检查：

- 真相账本；
- 角色错信；
- 已播 / 未播；
- DNC；
- 伏笔；
- 关系 / 权力状态；
- 同一信息是否重复当主刀；
- 后续火葬场、反转、关系击穿是否被提前消费。
- 当前批次结束后，下一批开场还能接什么追问、关系状态和反派压力。

**禁止：**

- 审美评价；
- 短剧感打分；
- 逐句改正文。

**下游只能读：**

- S9/S11 读取状态和风险。

**失败回滚：**

- 真相矛盾、DNC 失效、重复信息主刀、状态漂移，回 S3/S8；
- 批次交接后下一批无追问、无反派压力或状态断裂，回 S3/S8；
- 如果 S8 记录问题但放行，必须说明为什么它不是信息边界问题。

---

### S9 作者自检与返修诊断

**只负责：** 主创侧先判明显失败和回滚层。

**输入：**

- `台词视听 polish 稿`；
- `story memory checkpoint`。

**必须输出：** `作者自检与返修诊断`

必须判断：

- 是否洗飞；
- 是否复刻；
- 是否短剧降级；
- 是否正文长剧化；
- 是否状态漂移；
- 是否提前消费；
- 失败应回哪层；
- 返修时必须保留什么；
- 返修时禁止破坏什么。

**禁止：**

- 把硬风险写成“看主控偏好”；
- 把问题丢给 reviewer 兜底；
- 直接给正文逐句补丁；
- 为每个“还能更好”开返修。

**下游只能读：**

- S10/S11 读取结论、最终稿和必要边界。

**失败回滚：**

| 问题 | 回滚层 |
|---|---|
| 源本体验丢失 | S1/S2 |
| 高辨识外形回流 | S2/S3/S4 |
| 场面承载不住 | S4 |
| 施工稿像任务单 | S5 |
| 正文像说明书 | S6/S7 |
| 真相 / 信息乱 | S3/S8 |
| 台词水 | S7 |

### S10 干净 reviewer

**只负责：** 独立兜底，查红线和明显漏损。

**硬线：reviewer 不生产短剧感。**

**输入：**

- 用户需求摘要；
- 洗稿边界摘要；
- 最终稿；
- 必要源本片段。

第一轮 reviewer 不读：

- 作者自检；
- 主创解释；
- 旧版对比；
- run log；
- 临时诊断。

**必须输出：** `reviewer report`

必须包含：

- 致命阻断；
- 应改风险；
- 可选优化；
- 问题所属层级；
- suggestion。

**Reviewer 只查：**

- 洗飞；
- 改名复刻；
- 提前消费；
- 真相矛盾；
- 明显强度漏损；
- 正文长剧化；
- 状态漂移；
- 触碰红线。

**禁止：**

- 生产短剧感；
- 逐句代写；
- 无限挑刺；
- 因“还能更好”打回；
- 把个人口味升成致命阻断。

**下游只能读：**

- S11 读取 issue、严重度、回滚层和 suggestion。

**失败回滚：**

- reviewer 越权生产短剧感或无限挑刺，回 S10/S11 修 reviewer 口径。

---

### S11 主控拍板

**只负责：** 对比、归因、反馈回流、交付判断。

**输入：**

- 最终稿；
- 作者自检；
- reviewer report；
- run log；
- V5 / V22 / V24 / V25 / 上一版对比材料；
- 用户 / 导演反馈。

**必须输出：** `主控版本对比与拍板`

必须判断：

- 是否可交付；
- 为什么；
- 比 V22 / V24 / V25 是否明显提升；
- 是否至少保住 V5 的追看强度；
- 失败应回哪一层；
- 用户 / 导演反馈属于哪一层。

**禁止：**

- 只看 run log；
- 只看 reviewer；
- 只和烂版比；
- 把每条反馈变成新规则；
- 让 reviewer 对用户拍板。

**下游只能读：**

- 用户只看交付结论和必要依据。

**失败回滚：**

- 主控无法归因，回 S11；
- 如果问题属于上游，回对应层，不重开全链，除非触碰产品红线。

## 6. Reviewer 降权协议

Reviewer 是兜底，不是发动机。

### 6.1 Reviewer 可以打回

只有以下问题可以触发 `revise`：

- 伤到三条产品红线；
- 核心真相或主要人物动机自相矛盾；
- 明显洗飞；
- 明显改名复刻；
- 明显短剧降级；
- 明显提前消费；
- 明显状态漂移；
- 高压节点完全没有现场戏。

### 6.2 Reviewer 不能打回

以下只能列为风险或可选优化：

- 某句台词还能更狠；
- 某个桥段可以更刺激；
- 某个套路比较常见但没有复刻具体源本外形；
- 某个节点不如 V5 有标志物；
- 个人觉得还可以更短剧。

### 6.3 Reviewer 输出格式

每个 issue 必须写：

```text
问题：
严重度：致命阻断 / 应改风险 / 可选优化
失败层：
证据：
建议回滚：
不能破坏：
```

没有失败层和建议回滚的 issue，不允许主控直接转给 writer。

## 7. Run log 规则

run log 是执行记录，不是质量证明。

### 7.1 run log 必须记录

- 本轮使用哪个 workflow spec；
- 每层产物路径；
- 每层交给下游什么；
- 是否出现下游回头翻上游；
- writer 是否读了不该读的材料；
- reviewer 是否读了作者自检；
- 哪些步骤只是 execution complete；
- 最终创作判断是什么。

### 7.2 run log 禁止

- 证明剧本质量；
- 给 writer 当输入；
- 混入正文；
- 替代主控横向对比；
- 因为步骤齐全就判 pass。

run log 顶部必须写：

```text
run log 完成不等于剧本可交付。
工位 done 只代表该工位有产物，不代表剧本质量通过。
```

run log 还必须记录本轮输出范围：

- 是首批 1-10 集；
- 还是 100 分钟 / 多批次中的第几批；
- S0b 是否确认过洗稿方案；
- 当前批次承接上一批什么状态，并给下一批留下什么追问。

## 8. 样本验证协议

v16 写完后，必须先跑同一飞书三星样本首批 1-10 集。

首批验证通过只证明首批链路成立。  
如果用户目标是 100 分钟 / 多批次交付，还必须继续做一次批次交接验证：检查 S3 批次地图、S8 checkpoint 和下一批 S4/S5 是否能接住追问、状态、DNC 和反派压力。

### 8.1 对比对象

- V5：场面发动机和追看强度正向证据；
- V22：较稳但偏平；
- V24：链路完整但 writer 渲染感明显；
- V25：skill 工位链方向有效但未达标；
- v16 新样本。

### 8.2 验收问题

必须回答：

1. 是否没有洗飞源本追看功能；
2. 是否没有改名复刻；
3. 是否比 V25 更少旧外形回流；
4. 是否比 V25 更稳定地产生动作后果链；
5. 是否至少保住 V5 的场面发动机优势；
6. 0xsline 的开场、节奏、钩子、爽点是否真的进入 S5/S6；
7. S8 是否拦住重复信息消费；
8. S9 是否在主创侧打回硬风险；
9. reviewer 是否仍然降权；
10. run log 是否没有被误读成质量通过；
11. 主线期待、人物关系、拉扯、反派压力、信息差、情绪密度和 story memory 是否保住；
12. 是否没有为了短剧感牺牲新壳自然性、源本有效性或海外市场语感。

### 8.3 判定

只有三种判定：

| 判定 | 含义 |
|---|---|
| `workflow candidate pass` | 样本产物显著强于 V22/V24/V25，至少保住 V5 追看强度，可考虑升为当前默认 workflow |
| `workflow execution complete / creative revise` | 流程跑完，但产物仍不达标；必须根据失败层返修 workflow |
| `workflow invalid` | 流程本身使 writer 变差、丢能力或复发旧问题，不能继续小修 |

## 9. 静态对齐检查

在跑样本前，主控必须先检查：

- PRD 成功标准是否都能追到 S0-S11；
- 追看 / 付费总效果是否是所有小能力的共同验收轴；
- 每个历史失败类型是否有前置生产动作；
- S0b 二次确认是否存在，且发生在 S3 之后、S4 之前；
- 长篇 / 多批次是否有 S3 全剧阶段骨架、批次地图和 S8 批次 checkpoint；
- writer 是否只读 S5 施工稿；
- S5 是否消化源本有效性、钩子、爽点、拉扯、反派压力和 DNC；
- S6/S7 是否负责 page-to-screen、台词功能和视听表达；
- S8 是否只查连续性和信息边界；
- S9 是否承担返修诊断；
- S10 是否保持降权；
- S11 是否做横向对比和用户反馈归层；
- run log 是否不进入 writer 输入。

如果以上任一项无法追到具体层级，不能跑样本。
