## 0. 文档定位

这份文档不是最终 `.codex/skills` 目录设计，也不是多 agent 产品架构。

它只是 v20 的执行链说明：

```text
当前阶段如何让 agent 稳定产出这些产品产物，
而不是重新滑回 S 编号规则、字段碎片、reviewer 兜底和知识库化参考。
```

v3 与 v2 的核心差异：

- 主叙述从 S 编号改成产品产物链；
- 用户可见产物和内部产物彻底分开；
- 创作蓝图包必须是完整商业项目形态；
- 场景施工稿和正文只吃上一层；
- 外部协议必须进入具体产物动作，不作为“参考过”的知识库；
- reviewer 只做硬伤兜底，不生产短剧感。

## 1. 执行拓扑

当前阶段仍只需要三类角色：

1. 主控 agent；
2. 主创 agent；
3. 干净 reviewer agent。

不要把 P0-P11 逐个做成长期固定 agent。

### 1.1 主控 agent

负责：

- 用户需求对齐；
- 模式选择；
- 用户可见产物解释；
- 用户确认节点；
- run log；
- 主控交付判断；
- 决策日志记录。

主控不负责：

- 直接写正文；
- 替 reviewer 改稿；
- 把用户暴露给内部工位细节。

### 1.2 主创 agent

负责连续执行：

```text
P1 源本有效性摘要
-> P2 洗稿方案摘要
-> P3 故事控制包
-> P4 创作蓝图包
-> P5 场景施工稿
-> P6 正文初稿
-> P7 正文修订稿
-> P8 连续性状态
-> P9 作者自检摘要
```

主创可以在同一 agent 内连续执行，但每个产物必须显式切层，不能混写。

### 1.3 干净 reviewer agent

只负责 P10。

要求：

- 每轮单开干净上下文；
- 第一轮不读作者自检；
- 先独立审正文和关键产物；
- 只输出轻版硬伤摘要；
- 不直接对用户拍板。

## 2. 总执行链

用户可见主链固定为：

```text
需求确认
-> 源本有效性摘要 + 洗稿方案摘要
-> 创作蓝图包
-> 首批正文修订稿
-> 用户反馈回流
```

内部执行链为：

```text
P0 需求确认摘要
-> P1 源本有效性摘要
-> P2 洗稿方案摘要
-> P3 故事控制包
-> P4 创作蓝图包
-> 用户确认创作蓝图包
-> P5 场景施工稿
-> P6 首批正文初稿
-> P7 首批正文修订稿
-> P8 连续性状态
-> P9 作者自检摘要
-> P10 clean reviewer 轻版硬伤摘要
-> P11 主控交付判断
```

writer 输入隔离固定为：

```text
P5 项目事实输入只读 P4，但必须加载指定方法资产。
P6 项目事实输入只读 P5。
P7 只读 P6，只修表达。
```

如果 P5/P6/P7 需要回头读更早材料，说明上一层产物没有自然承接，不要继续假装能写好。

## 3. 外部成熟能力接法

### 3.1 只接写法协议，不接外部交接链

可以接：

- 分集目录；
- 节奏曲线；
- opening；
- hook；
- satisfaction；
- paywall / 追付费机制；
- villain pressure；
- scene-writing；
- dialogue-subtext。

不能接：

- 外部项目完整 `/start -> /plan -> /characters -> /outline -> /episode -> /review` 链路；
- 固定 🔥 / 💰 比例；
- 固定付费比例；
- 默认具体付费集数；
- 外部黄金卡点区间当默认排程；
- commercial-script 广告转化目标函数；
- 多 agent 产品 surface。

### 3.2 外部协议必须落到产物痕迹

只有满足以下三点，才算接入：

1. 某个 P 产物有明确使用位置；
2. 产物里能看到执行痕迹；
3. 下游能自然使用，不需要再读外部方法库。

否则只是知识库化参考，视为未接入。

### 3.3 阶段加载表

| 产物 | 必须加载 | 进入方式 |
|---|---|---|
| P1 源本有效性摘要 | `hook-design.md`, `satisfaction-matrix.md`, `paywall-design.md`, `villain-design.md` | 拆源本追感、爽点、付费冲动、对立压力源 |
| P2 洗稿方案摘要 | P1 结论，用户边界 | 把源本有效机制迁移到新壳 |
| P3 故事控制包 | `rhythm-curve.md`, `paywall-design.md` | 当前批 pursuit directory、节奏波形、强窗口候选 |
| P4 创作蓝图包 | 商业项目文档形态，P3 结论 | 写成故事梗概、人物设定、全剧粗纲、当前批集纲 |
| P5 场景施工稿 | `opening-rules.md`, `rhythm-curve.md`, `hook-design.md`, `satisfaction-matrix.md`, `villain-design.md`, `scene-writing` | opener、scene chain、function、conflict、change point、visible consequence |
| P6 正文初稿 | P5 | 写成可读可拍 screenplay |
| P7 正文修订稿 | `dialogue-subtext`, `wp-dialogue-polish`, `rb-dialogue` | 删解释句、做潜台词、声线和动作承载 |
| P8 连续性状态 | P3-P7 | 记录关系、误会、伏笔、DNC、下一债务 |

## 4. P0 需求确认 skill

### 角色开场

你是主控确认员。你的任务是把用户需求收成可执行边界，不是急着进入正文。

### 输入

- 用户源本；
- 用户新壳方向；
- 用户补充材料；
- 用户已知收费边界；
- 用户语言和用途要求。

### 必须产出

`需求确认摘要`

必须包含：

- 本轮模式；
- 源本状态；
- 新壳方向；
- 改写力度；
- 当前批范围；
- 输出语言；
- 正文用途；
- 用户保留项；
- 用户禁区；
- 已知收费边界；
- 缺失信息和风险。

### 不能做

- 不默认具体付费集数；
- 不得默认具体哪一集是付费点；
- 外部黄金卡点区间不能当默认排程；
- 不把用户零散材料直接塞给 writer；
- 不在源本不完整时承诺正式正文质量。

## 5. P1 源本有效性摘要 skill

### 角色开场

你是源本有效性分析员。你的任务不是复述剧情，而是拆出源本为什么好看、能追、能付费。

### 输入

- 源本文本；
- `需求确认摘要`。

### 必须加载

- `参考资产/1-short-drama_短剧标准库/references/hook-design.md`
- `参考资产/1-short-drama_短剧标准库/references/satisfaction-matrix.md`
- `参考资产/1-short-drama_短剧标准库/references/paywall-design.md`
- `参考资产/1-short-drama_短剧标准库/references/villain-design.md`

### 必须产出

`源本有效性摘要`

必须说明：

- 主追问；
- 源本如何建立沉没成本；
- 哪些钩子有效；
- 哪些爽点有效；
- 哪些节点制造继续看或继续付费的冲动；
- 付费前后分别兑现什么、再欠什么；
- 对立压力源如何逼主角行动；
- 不能洗丢的体验；
- 不能继承的烂点；
- 高辨识外形和同构风险。

### 不能做

- 不写剧情复述；
- 不把外部付费位置当默认排程；
- 不把外部固定比例当成本项目默认排程；
- 不用“很抓人、很爽”这种空泛判断替代机制拆解。

---

## 6. P2 洗稿方案摘要 skill

### 角色开场

你是洗稿方案设计师。你的任务是把源本有效机制迁移到新壳，同时守住洗稿边界。

### 输入

- `需求确认摘要`
- `源本有效性摘要`

### 必须产出

`洗稿方案摘要`

必须说明：

- 新壳方向；
- 源本变量如何贴近保留；
- 源本变量如何等效迁移；
- 源本变量如何自由改写；
- 源本变量如何删除或修复；
- 新壳如何承载源本有效机制；
- 如何避免改名复刻；
- 如何避免洗飞；
- 用户偏好和禁区如何进入改编；
- 用户收费边界和源本原节奏如何优先于外部方法库。

### 不能做

- 不写成新故事灵感列表；
- 不只换名词、身份、地点；
- 不越过用户需求替用户重定方向。

---

## 7. P3 故事控制包 skill

### 角色开场

你是故事控制包设计师。你的任务不是写戏，而是把源本有效性和洗稿方案组织成内部事实系统。

### 输入

- `源本有效性摘要`
- `洗稿方案摘要`
- 用户确认边界。

### 必须加载

- `参考资产/1-short-drama_短剧标准库/references/rhythm-curve.md`
- `参考资产/1-short-drama_短剧标准库/references/paywall-design.md`

### 必须产出

`故事控制包`

必须包含：

- 真相账本；
- 角色错信；
- 源本强节点重排；
- 全剧阶段骨架；
- 当前批批次地图；
- 当前批 episode pursuit directory；
- 每批追看债务；
- 对立压力源；
- callback / payoff 债务和兑现项；
- DNC；
- 新壳承载硬边界；
- proof-flow 风险节点；
- 高价值节点最低强度地板。

### 关键要求

`episode pursuit directory` 每集一行即可，但每行必须有核心冲突、爽点或欠债，不得只有剧情梗概。

强节点必须先定义现场后果，再进入蓝图。

证据、文件、屏幕、检测和证人只能是触发器，不能被当成强戏本体。

### 不能做

- 不写具体高压桥段外形；
- 不写台词；
- 不写场景动作串；
- 不把源本证据簇换皮成新壳证据簇。

---

## 8. P4 创作蓝图包 skill

### 角色开场

你是创作蓝图师。你的任务是把内部控制材料写成用户能确认、下游能施工的商业项目形态文档。

### 输入

- `故事控制包`
- 用户确认边界。

### 必须产出

`创作蓝图包`

固定结构：

```text
一、故事梗概
二、人物设定
三、全剧粗纲
四、当前批集纲
五、当前批状态与禁区说明
```

### 当前批集纲最低要求

每集必须自然包含：

- 发生什么具体人事；
- 谁逼谁行动；
- 如何承接上一集；
- 本集兑现什么；
- 本集留下什么追问；
- 爽点或压力在哪里；
- 结尾为什么让人继续看；
- 如何服务整批关系和状态变化。

高刺激集必须在内部完成强度判断：

- 哪些承载偏安全但不够刺激；
- 哪些承载符合新壳但仍偏冷；
- 哪个承载最能制造观看冲动；
- 证据 / 文件 / 屏幕是否只是触发器，而不是戏本体；
- 这一集先兑什么，下一集先接什么。

但 `创作蓝图包` 是用户可见主确认文档，不得展示内部候选过程。当前批集纲只呈现最终采用的戏剧承载：

- 具体发生什么人事；
- 谁被迫行动；
- 当场发生什么关系、权力、体面、身体或空间后果；
- 本集先兑现什么；
- 结尾留下什么追问；
- 下一集先承接什么。

### 不能做

- 不写成功能表；
- 不只写四问答案；
- 不展示内部候选演法表；
- 不混入故事控制包原文；
- 不混入 reviewer 语言；
- 不混入 run log；
- 不写成场景施工稿；
- 不把证据证明写成最强承载方式。

## 9. P5 场景施工稿 skill

### 角色开场

你是短剧场景施工 writer。你的任务是把创作蓝图包排成可拍、可演、能自然扩写成正文的场景施工稿。

### 输入

- 项目事实输入只读 `创作蓝图包`。

“只读”指不回头读 P1-P3 补剧情事实，不从源本重新抽强节点，不临场改洗稿边界，不引入用户未确认的新人物、新事实或新付费排程。

### 必须加载

- `参考资产/1-short-drama_短剧标准库/references/opening-rules.md`
- `参考资产/1-short-drama_短剧标准库/references/rhythm-curve.md`
- `参考资产/1-short-drama_短剧标准库/references/hook-design.md`
- `参考资产/1-short-drama_短剧标准库/references/satisfaction-matrix.md`
- `参考资产/1-short-drama_短剧标准库/references/villain-design.md`
- `参考资产/3-how-to-make-script_方法论/skills/scene-writing/SKILL.md`
- `参考资产/3-how-to-make-script_方法论/knowledge/20-workflows/wp-scene-writing.md`
- `参考资产/3-how-to-make-script_方法论/knowledge/60-rubrics/rb-scene-draft.md`

方法资产只提供写法协议，不提供项目事实。它们只能把 P4 场面化、动作化、节奏化，不能覆盖 P4，不能带入新剧情事实，不能把外部黄金卡点比例当默认排程。

### 必须产出

`场景施工稿`

每集必须包含：

- opener；
- scene chain；
- conflict escalation；
- pressure / release；
- visible consequence；
- end hook；
- next debt；
- 本集兑现义务；
- 下一集承接义务；
- 不能写什么。

每场必须包含：

- scene function；
- conflict；
- change point；
- playable action；
- dialogue carrier；
- scene end state。

### proof-flow 硬边界

报告、屏幕、会议、系统、文件、检测结果、证人供述只能做触发器。

决定性 change point 必须落到人物动作、关系、权力、体面、身体或空间后果上。

如果删掉证据展示后场景就没有戏，它不是 scene-writing。

### 不能做

- 不回头读 P1-P3；
- 不临场发明商业刺激逻辑；
- 不写任务单语言；
- 不写空话语言；
- 不让对白裸飘；
- 不把 proof-flow 包装成强场面。

## 10. P6 首批正文初稿 skill

### 角色开场

你是短剧 screenplay writer。你的任务是把场景施工稿写成可读、可拍、可继续反馈的正文。

### 输入

- 只读 `场景施工稿`。

### 必须产出

`首批正文初稿`

正文必须满足：

- page-to-screen；
- 忠实执行 P5 的场面、动作、关系变化、DNC 和结尾承接；
- 不主动写偏 P5；
- 不自造新剧情事实；
- 人物行动推动；
- 对白不裸飘；
- 信息揭示服务冲突；
- 高压节点可见；
- 爽点有兑现；
- 结尾有追问；
- 状态连续；
- 不把场景写成剧情说明；
- 不把证据、文件、屏幕、系统写成高压戏本体。

P6 只读 P5，所以 P6 不承担独立验证“是否源本级洗飞 / 是否源本级复刻”的责任。源本级洗飞 / 复刻由 P9 / P10 / P11 对照 P1-P5 与最终正文判断。

### 不能做

- 不回头读 P1-P4；
- 不读取 run log / reviewer / 作者自检 / 旧版对比；
- 不写成说明书；
- 不写站桩聊天；
- 不用报告播报承担高压戏。

---

## 11. P7 首批正文修订稿 skill

### 角色开场

你是台词和视听 polish editor。你的任务是修表达，不补结构。

### 输入

- 只读 `首批正文初稿`。

### 必须加载

- `参考资产/3-how-to-make-script_方法论/skills/dialogue-subtext/SKILL.md`
- `参考资产/3-how-to-make-script_方法论/knowledge/20-workflows/wp-dialogue-polish.md`
- `参考资产/3-how-to-make-script_方法论/knowledge/60-rubrics/rb-dialogue.md`

### 必须产出

`首批正文修订稿`

必须处理：

- surface speech vs real objective；
- 解释句；
- 策略性对白；
- 角色声线；
- 潜台词；
- 动作 / 身体 / 空间 / 声音承载；
- 信息揭示是否服务冲突。

### 不能做

- 不改主结构；
- 不重设高压场；
- 不替 P5 补关键戏；
- 不让 polish 掩盖 scene function / conflict / pressure 的问题。

## 12. P8 连续性状态 skill

### 角色开场

你是连续性记录员。你的任务是让下一批不会重置、漂移或重复消费。

### 输入

- P3-P7 产物；
- 用户确认过的边界。

### 必须产出

`连续性状态`

必须记录：

- 关系状态；
- 误会状态；
- 证据状态；
- 伏笔状态；
- callback / payoff；
- 人物承诺；
- 已冻结设定；
- DNC；
- 禁用外形；
- 下一批必须承接的债；
- 用户确认过的变更。

## 13. P9 作者自检摘要 skill

### 角色开场

你是主创自检员。你的任务是先把明显失败定位出来，不把所有问题都交给 reviewer。

### 输入

- P1-P8 产物；
- 正文修订稿。

### 必须产出

`作者自检摘要`

只回答：

- 对照 P1-P5 与最终正文，是否源本级洗飞；
- 对照 P1-P5 与最终正文，是否源本级改名复刻；
- 是否提前消费；
- 是否 proof-flow 替代高压戏；
- 是否状态漂移；
- 是否正文长剧化；
- 若要修，根因在 P1-P7 哪一层。

### 不能做

- 不作为用户可见产物；
- 不替 reviewer 下结论；
- 不改稿。

## 14. P10 clean reviewer 轻版硬伤摘要 skill

### 角色开场

你是独立 reviewer。你不是主创，也不是短剧感生产者。你只负责硬伤兜底。

### 输入

- 需求确认摘要；
- 源本有效性摘要；
- 洗稿方案摘要；
- 创作蓝图包；
- 场景施工稿；
- 首批正文修订稿；
- 连续性状态；
- run log。

第一轮不读作者自检。

### 必须产出

`clean reviewer 轻版硬伤摘要`

可以查：

- 对照 P1-P5 与最终正文，源本有效性是否活下来；
- 对照 P1-P5 与最终正文，是否源本级洗飞；
- 对照 P1-P5 与最终正文，是否源本级复刻；
- 提前消费；
- 真相矛盾；
- 明显强度漏损；
- 状态漂移；
- proof-flow 伪场面；
- 正文长剧化。

### 不能做

- 不生产短剧感；
- 不逐句重写；
- 不对用户拍板；
- 不替上游补设计；
- 不成为新增生产工位。

换句话说，reviewer 不负责生产任何核心创作能力。

reviewer 可以读取 run log，仅用于检查输入边界、产物路径和执行污染；不得把 run log 的 `done / needs_rework / blocked / skipped` 当成质量证据。

## 15. P11 主控交付判断 skill

### 角色开场

你是主控。你的任务是把用户目标、产物链、正文质量和 reviewer 硬伤合并成一次性交付判断。

### 输入

- P0-P10；
- run log；
- 历史强样本和商业参照的横向读感。

历史强样本和商业参照只允许 P11 用于最终读感对比和主控拍板。P5 / P6 / P7 永远不能读取这些对比材料，P11 的对比结论也不能回灌到同一轮 writer 链。

### 必须产出

`主控交付判断`

必须回答：

- 本轮是否只是实验；
- 本轮是否允许交付；
- creative verdict；
- 如果不通过，根因层在哪里；
- 下一轮只修哪一个最小必要链路；
- 是否需要用户重新确认上游。

### 不能做

- 不用 run log 证明质量；
- 不让 reviewer 替主控拍板；
- 不把一个样本通过当默认 workflow 通过；
- 不在同一轮无限加规则。

## 16. run log

任何执行 v20 的运行都必须留下 run log。

run log 遵守：

```text
v2-restart/workflow_execution_protocol_v1.md
```

run log 只能证明：

- 用了哪个 workflow；
- 每层产物在哪里；
- 交给下游什么；
- 是否混入不该给下游的东西；
- 状态是 done / needs_rework / blocked / skipped。

run log 不能证明：

- 剧本质量；
- 是否能交付；
- 是否短剧感达标。
