## 0. v7 先解决什么

v7 不重写产品定义。

产品仍然是：

```text
把一个已被市场验证或值得参考的短剧源本，
洗成一个新壳下仍然好看、能追、能付费的新剧本。
```

v7 要解决的是过去几轮暴露出的流程问题：

```text
流程越来越像证明题；
中间产物越来越多；
agent 花大量注意力证明“我做了”，
但最终正文不稳定、不像剧本、创作感弱。
```

所以 v7 的核心调整不是加规则，而是减法：

```text
保留洗稿必须判断；
压缩强制产物；
把厚分析变成薄写作输入；
让 workflow 服务写作，而不是服务自证。
```

## 2. v7 的判断原则

### 2.1 洗稿优先，不漂成原创

每次写作都必须先回答：

```text
源本为什么让人追；
源本哪些钩子、卡点、爽点、情绪走势、关系拉扯、信息差、权力变化不能丢；
这些东西在新壳里用什么方式重新成立。
```

新本可以改事件、身份、职业、场景、时代、世界观，但不能把源本让人继续看和继续付费的发动机洗没。

### 2.2 创作优先，不做程序正义

manifest 只证明流程有没有跑，不证明剧本好。

最终判断看正文：

```text
是不是一个能看的短剧本；
是不是不像改名复刻；
是不是保住源本追看冲动；
是不是新壳自己站得住；
是不是能被 AI 出海短剧画面承载。
```

### 2.3 分析可以厚，交给 writer 必须薄

主创可以完整读源本，也可以做厚分析。

但进入正文写作时，writer-facing 输入必须压成：

```text
观众此刻在等什么；
本集/本场谁压谁；
谁想要什么；
谁误会什么；
谁在忍、谁快崩、谁要选择；
压力怎么升；
钩子停在哪一拍；
哪些源本外形不能照搬。
```

不能把完整审计底稿、source_ref、证明链、禁用表、版本对比全部塞给 writer。

### 2.4 剧本文字必须像剧本

正文默认采用：

```text
中文场景/动作描述 + 英文对白
```

除非需求确认另有指定。

正文不是分析报告，也不是说明书。最终剧本里不要出现：

```text
Purpose / Function / Source Ref / Pressure Chain / Gate / 这一场的作用是
```

剧本页上写出的内容，应基本对应最终 AI 成片里能看见和听见的东西。

## 3. v7 主流程

```text
0. 最小需求卡
1. 主创读本笔记
2. 二次需求确认
3. 洗稿方案
4. 首批写作简报
5. 首批正文
6. 作者问题清单
7. 干净 reviewer
8. 主控判定与下一步
```

这是当前候选主流程。

其他材料都只能作为草稿、辅助或追溯，不默认进入强制交付。

---

## 4. 0. 最小需求卡

目标：先拿到足够开工的信息，不一开始把用户问爆。

必须确认：

```text
源本：
新壳：
目标市场：
集数是否变化：
改写力度 / 变量边界：
本轮输出范围：
输出语言 / 审稿语言：
正文用途：人审稿 / AI 视频生成用剧本 / 中英双语 / 字幕台词版
已知禁区：
```

默认：

```text
台词英文；
动作和场景描述中文；
内部分析中文；
产品目标是 AI 出海短剧剧本文字。
```

---

## 5. 1. 主创读本笔记

目标：不是复述剧情，而是让主创真正知道源本为什么好看。

这一步可以参考 v2 的 Source Bible，但不强制产出大表。

必须回答：

```text
1. 一句话故事核
2. 观众追看的核心期待
3. 首批每集凭什么让人继续看
4. 高价值节点：具体发生什么，观众为什么痛、爽、急、气、期待
5. 钩子与卡点：停在哪一拍，为什么想看下一集
6. 爽点 / 压抑释放：哪里压低，哪里兑现
7. 情绪走势和信息走势
8. 人物关系拉扯：谁靠近、谁退开、谁误判、谁压谁
9. 角色状态：谁在忍、谁快崩、谁掌握主动权
10. 源本水分、烂点、禁止复用外形
```

通过标准：

```text
能说清源本“让人心痒继续看”的东西；
能说清哪些是必须迁移的体验，哪些只是源本外形；
能指导新本写作，而不是只给 reviewer 看。
```

失败信号：

```text
只剩剧情摘要；
只剩节点编号；
只剩“有钩子、有爽点、有反转”这种标签；
说不出源本强度靠什么成立。
```

---

## 6. 2. 二次需求确认

目标：读完源本后，主创先判断新壳能不能承载源本，再让用户确认关键取舍。

主创先给建议，不把问题全甩给用户。

必须确认：

```text
新壳是否能承载源本最强追看机制；
哪些变量贴近源本；
哪些变量等效迁移；
哪些变量自由改；
哪些源本外形必须避开；
哪些源本烂点必须删；
集数变化时，钩子和留存阶段如何重排；
用户特别想保留或特别不想要什么。
```

如果用户需求会明显削弱源本有效性，必须直接提示风险。

---

## 7. 3. 洗稿方案

目标：在写正文前，先把“这次怎么洗”讲清楚。

这是给用户和主控看的，不是给 writer 的厚审计表。

结构：

```text
1. 本次改写一句话方案
2. 保留什么：核心期待、钩子惯性、爽点周期、情绪走势、关系拉扯、信息差、权力变化
3. 换掉什么：身份、职业、场景、世界观、源本高辨识事件外形
4. 删除什么：拖沓、水分、烂点、重复功能角色
5. 首批阶段骨架：每集推进什么，卡在哪里
6. 高压节点处理：哪些节点需要重新选场面，哪些可以直接等效迁移
7. 主要风险：洗飞、改名复刻、短剧降级、长剧化、状态漂移
```

通过标准：

```text
用户能看懂方向；
主创知道怎么写；
不会把新本自由度放大成从零原创；
也不会把防同构理解成把强戏洗淡。
```

---

## 8. 4. 首批写作简报

目标：把分析和方案压成真正能写正文的输入。

这是 writer-facing 的核心材料。

必须薄。

每集只写：

```text
上一集停在哪里：
本集观众在等什么：
本集开头怎么接住：
本集主要冲突：
本集最大刺激点：
本集关系 / 局势 / 信息 / 情绪变化：
本集钩子停在哪里：
本集不能提前说破什么：
本集源本追看功能迁移什么：
本集禁止复用什么源本外形：
```

高压场再补：

```text
谁压谁：
谁被迫行动：
谁失控或做选择：
观众能直接看到什么：
这场结束后有什么不可逆后果：
```

普通场不要厚写成表。

---

## 9. 5. 首批正文

目标：交付真正的剧本文字。

正文格式：

```text
EPISODE N

INT./EXT. 场景 - 时间

动作行：只写观众能看见、听见、感受到的外部行为、空间变化、物件、声音、身体反应。

角色名
English dialogue.
```

写作要求：

```text
1. 每场必须有当前行动，不写“角色很痛苦”这种空描述。
2. 高压场不能靠多人低信息对白拖时间。
3. 台词要有功能：推进、压迫、遮掩、误判、试探、反击、卡点。
4. 能用动作、空间、物件、身体反应、当众后果表达的，不优先用解释性对白。
5. 结尾钩子停在动作、反应、选择或后果前一拍，不把下一集爽点说完。
6. 不在正文里写内部标签或创作说明。
```

这里不是要求每场都爆。

低压过渡可以克制，但不能空转，不能用无效聊天冒充推进。

---

## 10. 6. 作者问题清单

目标：作者先诚实指出自己这版最可能哪里不行。

这一步不是作者自夸 pass，也不是长篇自检报告。

只写：

```text
1. 我最担心洗飞的地方；
2. 我最担心像改名复刻的地方；
3. 我最担心短剧降级或长剧化的地方；
4. 我认为旧版或源本局部更强的地方；
5. 我需要 reviewer 特别看的地方。
```

作者问题清单不能替代 reviewer。

---

## 11. 7. 干净 reviewer

Reviewer 每次单开临时干净上下文。

第一轮只读正文和最小需求卡，不读作者问题清单，不读主创解释。

第一轮判断：

```text
这是不是一个能看的短剧首批；
有没有明显长剧化、程序化、说明书化；
有没有台词水、人物冷、视听弱、卡点说完；
是否不看源本也站得住。
```

第二轮再读：

```text
主创读本笔记；
洗稿方案；
首批写作简报；
源本必要片段或源本节点。
```

第二轮判断：

```text
有没有洗飞源本有效性；
有没有改名复刻；
有没有把源本强戏洗成弱场面；
有没有把钩子、卡点、爽点周期、关系拉扯、节奏惯性洗丢；
失败如果存在，应该回到哪一步。
```

Reviewer 输出只允许：

```text
pass / revise / block
```

如果是 revise，必须说明失败层：

```text
需求没对齐；
源本没读透；
洗稿方案错；
写作简报传递断；
正文执行弱；
局部台词/场面可修；
状态连续性断；
```

---

## 12. 8. 主控判定与下一步

主控不只看 manifest，也不只看 reviewer。

主控必须看：

```text
正文；
作者问题清单；
reviewer 第一轮；
reviewer 第二轮；
和基线版本 / 上一版本的对比。
```

主控只能给四种结论：

```text
可给用户/导演看；
局部 revise；
回上游重做方案；
暂停，不再修这个样本。
```

如果问题只在 1-2 个局部场面，才做局部 revise。

如果问题是整体不像剧本、整体程序化、整体丢源本追看冲动，不允许继续在正文上打补丁，必须回到读本笔记、洗稿方案或写作简报。

## 13. v7 运行记录要求

只要声称“按 v7 完整跑”，仍必须建立 `workflow_run_manifest`。

但 v7 manifest 不再强制列 v2 的 0-15 张表。

v7 manifest 只记录：

```text
任务：
版本：
使用产品定义：
使用 workflow spec：
源本：
新壳：
输出范围：

| Step | 名称 | 状态 | 输出文件 | 是否允许进入下一步 | 备注 |
|---|---|---|---|---|---|
| 0 | 最小需求卡 | pass/fail/skipped |  |  |  |
| 1 | 主创读本笔记 | pass/fail/skipped |  |  |  |
| 2 | 二次需求确认 | pass/fail/skipped |  |  |  |

| 4 | 首批写作简报 | pass/fail/skipped |  |  |  |
| 5 | 首批正文 | pass/fail/skipped |  |  |  |
| 6 | 作者问题清单 | pass/fail/skipped |  |  |  |
```

## 14. 不再做什么

当前不做：

```text
不继续修 V16；
不再把某个样本的导演反馈逐条写成通用规则；
不再把每个旧草案都并行当主流程；
不再让 writer 面对完整审计底稿；
不再把 PRD 写厚；
不再把完整 manifest 当成质量 pass。
```

下一步只有一个：

```text
用 v7 跑一次完整样本验证。
只跑一次。
跑完看产物是否比旧路线更稳定、更像短剧本、更少程序感。
```
